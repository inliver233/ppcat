# 第八轮候选摘录

## 1. 真故障 body builder `0x920d7c`

`test3` 共享 xref 库与本地 `capstone` 一致：

- `0x920d7c` 直接 callers 只有 3 个：
  - `0x920b3c`
  - `0xc4cb30`
  - `0xc4d22c`
- 函数字符串命中：
  - `adNum:`
  - `tips`
  - `无法正常联网，导致广告无法显示！...`
- 函数内部自带原生 `return null` 退路：
  - `0x920e34: mov x0, x22`
  - `0x920e38: mov x15, x29`
  - `0x920e3c: ldp x29, x30, [x15], #0x10`
  - `0x920e40: ret`

关键窗口：

```asm
0x00920d94: 0b0dfb97    bl      #0x7e41c0
0x00920dac: 40042036    tbz     w0, #4, #0x920e34
0x00920dc4: 71c2ff97    bl      #0x911788
0x00920ddc: c0022036    tbz     w0, #4, #0x920e34
0x00920e10: a0012037    tbnz    w0, #4, #0x920e44
0x00920e34: e00316aa    mov     x0, x22
0x00920e38: ef031daa    mov     x15, x29
0x00920e3c: fd79c1a8    ldp     x29, x30, [x15], #0x10
0x00920e40: c0035fd6    ret
```

## 2. `0x920b3c` 中对 `0x920d7c` 的调用

这个 caller 对 `0x920d7c` 的返回值并不直接消费；调用后继续使用自己栈里的对象：

```asm
0x00920d14: 700361f9    ldr     x16, [x27, #0x4200]
0x00920d18: f001bfa9    stp     x16, x0, [x15, #-0x10]!
0x00920d1c: 18000094    bl      #0x920d7c
0x00920d20: ef410091    add     x15, x15, #0x10
0x00920d24: a0835bf8    ldur    x0, [x29, #-0x48]
```

这使得：

- `0x920d1c: 18 00 00 94 -> 1f 20 03 d5`

成为一个可单独尝试的 `NOP call` 候选：

- 调用前 `stp` 已把 shadow stack 补齐
- 调用后原本就有 `add x15, #0x10`
- 返回值未直接使用

## 3. `0xc4d22c` 中对 `0x920d7c` 的调用

同样是“调用后立刻丢弃返回值”的形态：

```asm
0x00c4d288: 400180d2    mov     x0, #0xa
0x00c4d28c: 70534091    add     x16, x27, #0x14, lsl #12
0x00c4d290: 103a46f9    ldr     x16, [x16, #0xc70]
0x00c4d294: f001bfa9    stp     x16, x0, [x15, #-0x10]!
0x00c4d298: b94ef397    bl      #0x920d7c
0x00c4d29c: ef410091    add     x15, x15, #0x10
0x00c4d2a0: e00316aa    mov     x0, x22
```

这使得：

- `0xc4d298: b9 4e f3 97 -> 1f 20 03 d5`

也属于“栈仍平衡、返回值未使用”的 `NOP call` 候选。

## 4. `0xc4cb30` 中对 `0x920d7c` 的调用

这一条显式消费了返回值，把它塞进 `x1` 继续传给弹窗链：

```asm
0x00c4cc78: f001bfa9    stp     x16, x0, [x15, #-0x10]!
0x00c4cc7c: 4050f397    bl      #0x920d7c
0x00c4cc80: ef410091    add     x15, x15, #0x10
0x00c4cc84: e10300aa    mov     x1, x0
0x00c4ccac: 7c9ae097    bl      #0x47369c
```

因此：

- 这里**不适合**像前两个 caller 一样直接把 `BL` 改 `NOP`
- 如果要局部压空 body，最多只建议把：
  - `0xc4cc84: e1 03 00 aa -> e1 03 16 aa`
  - 含义：`mov x1, x0 -> mov x1, x22`

但 `backup/ROUND8_FACTS` 已明确：

- 故障窗根因不在这类 `showDialog BL`
- 所以这一条只算旁路候选，不是主推荐

## 5. “喵喵饿了”常驻块 `0xae30e0`

`test3` xref 库：

- `0xae30e0`
  - `callers = 0`
  - `bool_ret = True`
  - 35 个 callees
  - 唯一高价值中文命中：`喵喵饿了`

本地关键分支：

```asm
0x00ae3128: 1f500071    cmp     w0, #0x14
0x00ae312c: c10d0054    b.ne    #0xae32e4
```

特殊分支内的真实字符串装载点：

```asm
0x00ae36f4: 25f4f397    bl      #0x7e0788
0x00ae36f8: 71b34091    add     x17, x27, #0x2c, lsl #12
0x00ae36fc: 31f246f9    ldr     x17, [x17, #0xde0] ; ref=28525 "喵喵饿了"
0x00ae3700: 11b000b8    stur    w17, [x0, #0xb]
```

这给出一个最直接的隐藏候选：

- `0xae312c: c1 0d 00 54 -> 6e 00 00 14`

语义：

- 原来只有 `w0 == 0x14` 才进入这条“喵喵饿了”专用支路
- 改成无条件跳 `0xae32e4`，强制走通用 fallback 分支

注意：

- 这是当前最像“隐藏常驻块”的单点 patch
- 但 `w0 == 0x14` 与具体 widget 类型的对应关系仍然是静态推断
- 所以它属于“高信号候选”，不是运行时已证实结论
