# 第二十八轮代码恢复报告

> 分支：`test1`
>
> 角色：纯静态分析。本轮目标不是重复 `test3` 已经打穿的首页 overlay/banner，而是接力 `test3` 还没真正解开的两件事：
>
> 1. 阅读页 `喵喵饿了` 常驻块，下一步该追哪条对象池线
> 2. `VIP / reward privilege` 哪个点仍然最适合做“功能完整化”方案

## 1. 本轮结论

### 1.1 `test3` 的首页结论我已独立复核，仍然成立

- `0xbd2e1c` 是首页分发器
- `0xbd3540` 是 `立即获取 + admobBannerFurtureView` 那条首页子树
- `0xbd2e24` 的 `SizedBox.shrink` 返回 patch 仍是当前最强的首页清理点

本轮不重复把这部分当新突破。

### 1.2 `slot 0x59a4 / ref 121017` 必须继续降级，不能再写成“喵喵块容器”

我用 `test3` 的 `pool_deserialized.json + named.json + xref_db.json` 继续复核后，确认：

- `ref 121017` 在 `named.json` 里是 `cid=11`
- 同类 `cid=11` 样本里能直接看到 `hashCode / data / _state / floatingActionButton` 这类**字段/成员**名字
- `ref 121017` 周围是一整段 `cid=11, owner_ref_id=65` 的对象，形态更像一簇 `Field`

并且 `slot 0x59a4 (off 0x2cd20)` 的用户群很杂：

- `0x872208` 机器码验证 / 权限 / “而且广告无法正常显示，此功能”
- `0x8937d0`
- `0xa54178`
- `0xb9fc84`
- `0xbd3540`

这说明它最多只能算“共享状态/字段线索”，**不适合继续当阅读页喵块主线**。

### 1.3 真正更值得追的是 `slot 0x59bd / ref 91705`

`ref 91705` 的结构与 `ref 121017` 明显不同：

- `ref 91705` 在 `named.json` 里是 `cid=7`
- 同类 `cid=7` 样本更像 `Function / Method-like` 对象
- `slot 0x59bd (off 0x2cde8)` 的用户群非常集中，只落在 5 个函数里：

| PC | 所属函数 | 角色 |
|---|---|---|
| `0xa55dac` | `0xa54178` | VIP/喂喵页子树 |
| `0xa570e0` | `0xa570c0` | 含文案“而且广告无法正常显示，此功能” |
| `0xb6837c` | `0xb68308` | 每日喂喵失败链的兄弟 helper |
| `0xba0be0` | `0xb9fc84` | VIP / daily / reward 混合状态机内部 |
| `0xbc60a4` | `0xbc5dc8` | `喵喵饿了` 点击后的 reward prompt 链 |

这个交集比 `0x59a4` 干净得多，已经明显收敛到：

- `喂喵 / reward`
- `每日喂喵失败说明`
- `VIP / privilege`

也就是说，**如果 `test3` 还要继续冲“阅读页喵块真正 mount 点”**，下一条主线不该再盯 `0x59a4/ref121017`，而应转向：

- `0x59bd/ref91705`
- 以及它的三个兄弟函数对象：
  - `ref 91703 / owner 68772`
  - `ref 91704 / owner 68773`
  - `ref 91706 / owner 68775`

## 2. `slot 0x59bd` 的角色已经能比 `test3` 多说一层

### 2.1 `0xbc5dc8` 仍然只是点击链，不是可见性根

`0xbc5dc8` 内部字符串仍然非常明确：

- `是否看推荐信息获取喵粮喂喵？`
- `饿了喵~`
- `获取`
- `AdPgl_Reward|AdMob_Reward|AdNet_Reward|AdKs_Reward`

本轮新补的一层是：

- `0xbc60a4` 不是独立入口，而是 `0xbc5dc8` 内部一个小子段
- 这里正好加载 `slot 0x59bd`
- 然后走 `0x90854c`

所以 `0x59bd` 至少已经能确认与**点击喂喵后的 reward 编排**直接相关。

### 2.2 `0xb68234 / 0xb68308` 也证实 `0x59bd` 在“每日喂喵失败”链里

已知：

- `0xb68234` 装载长正文 `ref 22466`
  - `无法加载数据...每天喂一次喵，喵喵就不来找你了~`
- `0xb68234` 调 `0xa56e00`

本轮新增：

- 它的兄弟函数 `0xb68308` 在 `0xb6837c` 也加载 `slot 0x59bd`
- 然后走 `0x7e5e18`

这说明 `0x59bd` 不只是“VIP 页面里顺手出现一次”，而是已经进入**每日喂喵失败 / reward 提示**这条子链。

### 2.3 `0xa570c0` 是目前最值得 test3 继续盯的小函数

`0xa570c0` 只有一条很干净的业务字符串：

- `而且广告无法正常显示，此功能`

它在 `0xa570e0` 处加载 `slot 0x59bd`，随后直接调用 `0x90854c` 并返回 `null`。

这让 `0xa570c0` 变成了当前一个很窄的信号：

- 它不是首页链
- 也不是大状态机入口
- 它又和 `0x59bd` 同簇

所以若 `test3` 下一步要继续从函数对象反推 mount/上层闭包，`0xa570c0` 比 `0x59a4` 更值得当锚点。

## 3. `VIP` 线的本轮补充

### 3.1 `0xb9fc84` 依旧不是“单点 isVip”

这轮继续压实而不是推翻旧结论：

- `0xb9fcb4 -> bl 0x911788`
- 紧接着看 bit5 / bit4
- 然后才继续往下走

关键片段：

```asm
0x00b9fcb4: b5 c6 f5 97    bl   #0x911788
0x00b9fcc0: 40 00 28 37    tbnz w0,#5,#0xb9fcc8
0x00b9fccc: c0 00 20 37    tbnz w0,#4,#0xb9fce4
```

这仍然只能说明：

- `0xb9fc84` 是共享状态驱动的页面/业务状态机
- 不能把它压缩成“VIP=true/false”的单点函数

### 3.2 `0xba0070 / 0xba007c` 只是局部绑定文案分流

这一点没有变化，但值得继续保留：

- `0xba0070` -> `功能，通过捐赠绑定账号获取`
- `0xba007c` -> `通过捐赠绑定设备获取`

它们是 `0xb9fc84` 内部一个局部文案二选一，不是总闸。

### 3.3 当前最实用的“VIP-like / privilege-like”方案仍然不是硬动 `0xb9fc84`

如果目标是：

- 去广告
- 让 app 更接近“免广告特权已生效”
- 同时不去赌厚状态机入口

我这边仍保持之前排序：

1. **首页 overlay/banner**
   - 用 `test3` 已证实的 `0xbd2e24` `SizedBox.shrink` patch
2. **阅读页喂喵点击退出用户路径**
   - 用 `0xbc5dd0..0xbc5dd8`
   - `原始: ef a1 00 d1 a0 0b 40 f9`
   - `patch: e0 03 16 aa c0 03 5f d6`
3. **实验性 privilege gate**
   - 继续优先看 `0x7e861c`
   - 原始字节：`c0 06 20 36`
   - 指令：`tbz w0,#4,#0x7e86f4`
   - 若要强制走“已生效”路径，可实验性改 `NOP`

但这一条我仍然只写成：

- **实验项**
- **未做运行时确认**
- **比硬改 `0xb9fc84 / 0xa54178` 更窄**

## 4. 对 `test3` 的直接接力建议

### 4.1 对象池主线该怎么改

建议 `test3` 下一步把对象池主线从：

- `slot 0x59a4 / ref 121017`

切到：

- `slot 0x59bd / ref 91705`
- 以及兄弟：
  - `ref 91703 / owner 68772`
  - `ref 91704 / owner 68773`
  - `ref 91706 / owner 68775`

理由：

- `0x59a4` 是字段样共享状态，用户群过杂
- `0x59bd` 是方法样对象，用户群已经收敛到 `VIP + 喂喵 + reward`

### 4.2 真正值得继续扫的函数

如果只选 3 个最值得继续深挖的点，我建议顺序是：

1. `0xa570c0`
   - 字符串最干净：`而且广告无法正常显示，此功能`
   - 同时挂在 `0x59bd` 上
2. `0xbc5dc8`
   - 已知是点击链
   - 内部 `0xbc60a4` 也挂 `0x59bd`
3. `0xb68308`
   - 与 `0xb68234` 的“每日喂喵失败正文”是同区兄弟
   - 同样挂 `0x59bd`

### 4.3 这一轮我不再声称的内容

本轮我明确不再声称：

- `0x59a4/ref121017` 已经是喵喵块容器
- `0xbd2e1c` patch 会顺带让阅读页喵块消失
- `0xb9fc84` 已经能安全给出单点 `isVip` patch

这些都还没有足够静态证据支撑。

## 5. 交叉验证结论

与 `test3` 最新提交 `1e0bd9f` 对账后：

- **一致**
  - `0xbd2e1c -> 0xbd3540` 首页链
  - `0xbc5dc8` 是 `喵喵饿了` 点击后的 reward prompt 主链
  - `0xa54178 / 0xb9fc84` 不应再被写成单点 `isVip`
- **补充**
  - `cid=7 ≈ Function-like`、`cid=11 ≈ Field-like`
  - `0x59a4` 应降级为字段线索
  - `0x59bd` 才是更好的 `喂喵 / reward / VIP` 共用方法簇入口
