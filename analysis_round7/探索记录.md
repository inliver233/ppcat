## 3. 官方 Dart 2.19 对照结论

本轮直接对照了 Dart SDK `2.19.0` tag (`5b42fc09c981093abd0475e3630b9448ba6aa188`) 的 raw 源码：

- `runtime/vm/compiler/assembler/object_pool_builder.h`
- `runtime/vm/app_snapshot.cc`

关键结论：

1. **2.19 官方 `ObjectPool` 仍是标准 old-pool 结构**
   - 先 `WriteUnsigned(length)`
   - 再循环 `length × (Write<uint8_t>(entry_bits) + type-dependent payload)`
2. **合法类型仍只有标准几类**
   - `kTaggedObject`
   - `kImmediate`
   - `kNativeFunction`
   - `kSwitchableCallMissEntryPoint`
   - `kMegamorphicCallEntryPoint`
   - `kImmediate128`
3. **因此 `0x67 / 0x89 / 0x8e` 这类字节不像标准 2.19 的普通 `entry_bits`**

结论不是“unflutter 少了两个 enum 就行”，更像：

- `ObjectPool fill` 入口位置有细微漂移；或者
- pool 开头前面还有一段 `ppcat` 特有的非标准前导结构。

## 4. 为什么不是直接使用官方 2.19 profile

命令：

```bash
cd unflutter
go run ./cmd/unflutter _debug clusters \
  --lib ../lib/arm64-v8a/libapp.so \
  --profile 2.19.0
```

结果：

- cluster 从一开始就大量出现 `CID=42949672xx` 这类明显错位值
- `InstructionsTableLen` 等字段也不合理

说明：

- `ppcat-hybrid` 的大方向明显比官方 `2.19.0` profile 更接近真实格式。
- 问题集中在少数旧格式细节，而不是“版本完全选错”。

## 5. ObjectPool 局部实验结论

### 5.1 已确认 anchor 仍稳定

已知 7 个关键 ref 用：

- `analysis_round6/scan_snapshot_pool_refs.py`
- `unflutter/internal/cluster` 的 inspect tests

都可以稳定命中：

- `26842 @ 0x2eb53c`
- `30947 @ 0x2eb540`
- `29112 @ 0x2ece05`
- `27673 @ 0x2eeeca`
- `30922 @ 0x2f0415`
- `21707 @ 0x2f8c81`
- `22466 @ 0x305c5a`

### 5.2 pool 起始字节不符合标准 length

`ObjectPool` fill 起点 raw：

```text
0x2e0bfb: 89 1b 67 88 84 84 10 c1 c0 03 0a 6a 86 38 d2 16 6c ...
```

而 `alloc` 阶段给出的 `cm.Lengths[0] = 51140`，按当前 VLE 编码应写成：

```text
44 0f 83
```

这组字节并未出现在 pool 起点附近。

### 5.3 有效条目区仍表现为 `0x80/0x81` 主导

对已知 anchor 附近扫描可见大量规律片段：

- `0x2e820c`: `80 5a 51 81 80 63 71 81 ...`
- `0x2e9ad5`: `80 38 63 81 80 22 7c 81 00 81 00 6a 86 ...`
- `0x2ebb9a`: `80 19 58 81 00 81 00 6a 86 ...`
- `0x2ed0e5`: `80 4a 71 81 80 2e 3e 85 ...`
- `0x2f5951`: `80 4b 29 81 ...`
- `0x30292a`: `80 42 2f 81 ...`

说明：

- pool 内部大量“有意义对象引用区”依然与 `0x80 + VLE(ref)`、`0x81 + imm` 规律相容；
- 非标准的是 pool 开头一小段，而不是整个对象池。

## 6. Java / reward 侧新增确认

### 6.1 本地 `jadx` 可独立重跑单类

命令示例：

```bash
TMPDIR=/root/tmp analysis_round6/tools/jadx/bin/jadx \
  -d /root/tmp/jadx_gdt \
  main_dex/classes2.dex \
  --single-class p073.C1083
```

这意味着：

- 后续对 `Pangle/GDT/Kwad` 单类/单路径的复查可以不再完全依赖旧 `jadx` 导出目录。

### 6.2 GDT reward 路径仍支持“动态子 channel”结论

单类重跑后仍可确认：

- `p073.C1083` 构造时执行 `new C0932(C1103.f2851.m3684(), m3249(str))`
- `m3249(str)` 对应的明文前缀已在上一轮解出：
  - `plugins.hetian.me/gdt_plugins/RewardVideo/`

因此：

- 本地 GDT reward 不是固定 `EventChannel`
- 更像“每次请求创建子 `MethodChannel` + 由 listener 回调反推事件”

## 7. 当前最值得继续推进的方向

### 7.1 第一优先：增强 unflutter 的 `ppcat-hybrid` ObjectPool 兼容

原因：

- profile 已命中
- fill 绝大多数阶段已跑通
- 只差 `ObjectPool` 局部

### 7.2 第二优先：在 Java 层继续压缩到具体 smali 修改点

当前结构已经很清楚：

- `Pangle`: `MethodChannel + EventChannel`
- `GDT`: 动态 reward 子 `MethodChannel`

下一步应该从“结构性判断”继续往“哪几个方法短路最稳、哪些字段/事件最少必填”收口。

## 8. 本轮结论

本轮最重要的不是“已完全突破”，而是把现状收口成了两条更清晰的工程路线：

1. **工具链层面**：`jadx/apktool` 已本地拉起，后续 dex / smali 可直接继续深挖。
2. **Dart AOT 层面**：`unflutter ppcat-hybrid` 已证实能正确跑到 `ObjectPool` 前，只在 pool 局部卡住；当前更像“pool 前导结构/入口漂移”问题，而不是整个版本或整条 fill 链错了。
