# 分析报告30

> 分支：`test1`  
> 角色：纯静态分析。  
> 本轮任务是吸收并交叉验证 `github/test3` 最新进展，判断它哪里成立、哪里仍停在假设，同时把它没有继续推进的 `VIP / 特权 / reward` 线收成更窄的可执行候选。

## 1. 对 `test3` 最新内容的裁决

### 1.1 成立的部分

`github/test3` 最新提交 `232a043` 对以下事实的判断，我已用真实 `libapp.so` 独立复核，结论成立：

1. `0xa52bb0` 才是当前阅读页 `喵喵饿了` 真 builder 链的核心。
2. `0xa52c60 / 0xa52c68` 确实分别把两个域写进返回对象。
   - `0xa52c60: 01 b0 00 b8` = `stur w1, [x0, #0xb]`
   - `0xa52c68: 01 f0 00 b8` = `stur w1, [x0, #0xf]`
3. `0xa52c40` 的 `mov x17,#2` 确实是给内部聚合器 `0xa52c90` 提供 `Smi(1)` 的元素计数。
   - `0xa52c40: 51 00 80 d2`
   - `0xa52c44: 11 b0 00 b8`
   - `0xa52c90` 内部确实读 `[#0xb]`，再 `sbfx` 解成计数，`cmp x5,x3 ; b.ge` 走空路径。
4. `slot 0x753` 的正确 pool 偏移确实是 `0x3a98`，不是旧文档里误写的 `0xea8`。
   - `0xf3a204: 60 4f 5d f9  ldr x0,[x27,#0x3a98]`
   - `0xb79c60: 71 4f 5d f9  ldr x17,[x27,#0x3a98]`
5. `obj44988` 不能再被当作“整块直接替代 widget”。
   - 它在多个函数里都只是被塞进复合对象字段。
   - `0xb79c64 stur w17,[x0,#0x33]`
   - `0x4f2424 stur w17,[x0,#0x3b]`
   - `0x53f4f8 stur w17,[x3,#0x3f]`

### 1.2 仍然只是合理假设、不能说“已解”的部分

`test3` 现在最值得保留的 patch 候选有两个，但都还不能说“静态已证实最终效果”：

1. `0xa52c68: 01 f0 00 b8 -> 16 f0 00 b8`
   - 把 `field@0xf` 从内容 child 改成 `null`
2. `0xa52c40: 51 00 80 d2 -> 11 00 80 d2`
   - 把 `0xa52c90` 的元素计数从 `1` 改成 `0`

这两条我都接受为**高价值候选**，但原因不同：

- `field-null` 更像“保留返回对象类型，只挖空内容”
- `count->0` 更像“保留返回对象类型，只让内部 children 为空”

它们都优于 `entry-return-null`，但哪条更稳，仍只能靠主控装机裁决。

### 1.3 我不同意 `test3` 过度表述的地方

我不接受把下面这类话直接写成“已确立”：

1. “返回槽一定是 cid0xaf7 敏感”
2. “obj44988 大概率 cid!=0xaf7”
3. “Patch D 一定比 count->0 更优”

原因很简单：这些都还缺最终的对象类证据或运行时结果。

更稳妥的表述应该是：

- `0xa52c80` 的确唯一分配了 `cid 0xaf7` 这一层对象
- 但 `obj44988` 的精确类目前仍未直接解出
- 所以 `obj44988` 不能当“已知安全替代物”
- `field-null` 与 `count->0` 都是**同类型保守修改**，比“整函数 return null/直接替对象”更值得优先试

## 2. 喵块方向的当前正确排序

结合我前一轮 [分析报告29.md](/root/ppcat/repo/分析报告29.md) 与 `test3` 最新补充，本轮对 `喵喵饿了` 的推荐顺序应改成：

### 2.1 第一优先级：`count -> 0`

- 地址：`0x00a52c40`
- 原字节：`51 00 80 d2`
- 补丁字节：`11 00 80 d2`
- 语义：`mov x17,#2` -> `mov x17,#0`

这条的优点：

1. 不改返回对象分配点
2. 不改返回对象的两个字段写入位置
3. 只让内部聚合器走空路径

### 2.2 第二优先级：`field@0xf -> null`

- 地址：`0x00a52c68`
- 原字节：`01 f0 00 b8`
- 补丁字节：`16 f0 00 b8`
- 语义：`stur w1,[x0,#0xf]` -> `stur w22,[x0,#0xf]`

可选叠加：

- `0x00a52c60: 01 b0 00 b8 -> 16 b0 00 b8`

这条的优点：

1. 也不改返回对象分配点
2. 只把内容域挖空

### 2.3 不再推荐

1. `0xa52bb0 entry-return-null`
2. `0xa52920 entry-return-null`
3. 直接返回 `obj44988`
4. 回到 `0xae30e0 / 0xa54178`

## 3. `test3` 没继续收口的 VIP / 特权线

`test3` 的 `分析报告6_补VIP链路.md` 对结构判断基本是对的，但它停在“链路图”上，没有继续把 patch 优先级收成更窄的层次。按我这轮复核，正确排序是：

### 3.1 第一层：reward 反作弊专属旁路 `0x875efc`

这是 `test3` 最值得继续接力、但它自己没有继续收口完的点。

#### 独立复核

关键窗口如下：

```asm
0x00875ef4: 2e351094  bl      #0xc833ac
0x00875ef8: ef410091  add     x15, x15, #0x10
0x00875efc: 40012037  tbnz    w0, #4, #0x875f24
0x00875f00: ... 
0x00875f18: 18000094  bl      #0x875f78
0x00875f1c: ef810091  add     x15, x15, #0x20
0x00875f20: 0a000014  b       #0x875f48
0x00875f24: ...
0x00875f40: 40003fd6  blr     x2
```

这里 `0x875efc` 是 `onReward Cheat Triggered!` 那个 compare 之后的专属岔路点，这一点我接受 `test3` 与我前面几轮的共同判断。

#### 候选 patch

- 地址：`0x00875efc`
- 原字节：`40 01 20 37`
- 候选补丁：`0a 00 00 14`
- 语义：直接跳到 `0x875f24`，绕过 `0x875f78` 那条 cheat handler

#### 当前判断

- `静态确定`：这是 reward 反作弊链最窄的旁路点之一
- `需主控验证`：绕过后是“正常拿到 reward/特权”，还是只是不走某个日志/错误 handler，需要装机确认

### 3.2 第二层：上层特权消费门 `0x7e861c`

这一层比直接改 `0x8cf36c` 明显更窄，值得继续保留。

#### 独立复核

```asm
0x007e8614: 64 db 55 f9  ldr x4, [x27,#0x2bb0]
0x007e8618: 55 9b 03 94  bl  #0x8cf36c
0x007e861c: c0 06 20 36  tbz w0, #4, #0x7e86f4
```

这说明：

1. `0x7e8534` 是更上层的共享广告/特权 gate
2. `0x7e8618` 消费 `0x8cf36c`
3. `0x7e861c` 是最窄的“消费侧判定门”

#### 候选 patch

- 地址：`0x007e861c`
- 原字节：`c0 06 20 36`
- 候选补丁：`1f 20 03 d5`
- 语义：`tbz w0,#4,...` -> `nop`

#### 当前判断

- `静态确定`：这比直接改 `0x8cf36c` 更窄
- `需主控验证`：是否足以放开所需功能、是否会引入额外副作用

### 3.3 第三层：共享特权核心 `0x8cf36c`

这层不该再被写成“单一 isVip 开关”。

#### 独立复核

`xref_db` 显示它有 16 个 caller，并且字符串里同时带：

- `adNum`
- `lastTime`
- `today`
- `totalNum`
- `totalTime`
- `leaveTime:`
- `已过期！`
- `已生效！\n剩余`

它显然是共享的 reward privilege / no-ad 状态计算，而不是单独 donation VIP。

#### 结论

- 不适合作为当前第一手 patch
- 只适合作为更激进实验层

## 4. 对 `VIP / 特权` 的更务实接力建议

用户怀疑 `test3` 没继续想 `VIP / 特权` 去除方法，这个判断基本成立：它把结构判清了，但没有继续压成“主控先测哪一字节”的序列。

我当前建议主控按这个顺序试：

1. `0x875efc: 40 01 20 37 -> 0a 00 00 14`
   - 目标：保留 reward 路线，绕过 cheat 专属 handler
2. `0x7e861c: c0 06 20 36 -> 1f 20 03 d5`
   - 目标：上层特权消费门更容易走开
3. 只有前两条仍不足时，才考虑更宽的 `0x8cf36c` 共享特权态实验

这比再回头去粗动：

- `0xa54178`
- `0xb9fc84`
- “单一 isVip 总闸”

都更合理。

## 5. 本轮结论

一句话总结：

1. `test3` 最新关于 `0xa52bb0` 的字节事实大体成立，值得学习和吸收。
2. 但它对喵块 safe-hide 的最终 patch 仍停在多个候选并存，不能说“已解”。
3. 它确实没有把 `VIP / 特权` 继续收成最窄 patch 序列；这部分现在最值得接力的是：
   - `0x875efc` reward 反作弊专属旁路
   - `0x7e861c` 上层特权消费门
4. `0x8cf36c` 仍然只是共享特权核心，不该再被包装成“单一 VIP 开关”。
