# 第二十轮代码恢复补充报告

> 分支：`test1`
>
> 角色：纯静态分析。本轮只补强一条前面已经提出、但语义还不够硬的线：`reward 反作弊`。
>
> 目标不是再重复“`0x875efc` 可能有关”，而是把它在 `0x8758bc` 调度链里的**精确位置和意义**压实，并和现有两条主线区分开：
>
> 1. `纯净稳定运行`：故障窗消失、`喵喵饿了` 退出用户路径
> 2. `保留 reward 特权能力`：尝试让点击喵块后真的拿到喵粮 / 触发 reward privilege

## 1. 本轮结论

这轮最重要的新结论只有一个：

**`0x875efc: 40012037 -> 0a000014` 不是模糊候选，而是 `onReward Cheat Triggered!` 专属 handler 的精确绕过点。**

也就是说：

1. `0x8758bc` 是广告类型 dispatcher，这一点 `test2/test3` 已经做对。
2. `0x875ee8` 加载的对象池槽位正是 `ref=11545 "onReward Cheat Triggered!"`。
3. `0xc833ac` 在这段 dispatch 链里表现为“**不相等则 bit4=1**”。
4. 因此 `0x875efc` 当前这条 `tbnz`，在 reward 类型命中 `"onReward Cheat Triggered!"` 时，会落入 **cheat 专属 helper `0x875f78`**。
5. 把 `0x875efc` 改成无条件 `b #0x875f24`，本质上就是**强制走“非 cheat”路径，跳过 cheat handler**。

这让 `0x875efc` 的地位发生了变化：

- 它仍然**不能取代** `0x7e861c`
- 但它已经从“模糊 reward 候选”升级成了“**reward-only 的窄反作弊旁路点**”

## 3. `0xc833ac` 在 dispatcher 里的真实语义

这轮最关键的静态突破，是把 `0xc833ac` 在这段链里的语义定下来了。

### 3.1 为什么能判断它是“!=”风格比较

看 `0x8758bc` 中前面的类型分发：

```asm
0x00875b68: bl   #0xc833ac
0x00875b74: tbnz w0, #4, #0x875b90
0x00875b78: ...  ; call 0x88ed60
```

这里 `0x18ff0` 对应的是一个已知 ad-type 字符串槽位。如果 `tbnz` 的含义是“相等就跳”，那整条分发链就不成立了，因为命中当前 type 时反而不会调用当前 handler。

而整条 dispatcher 是标准的“逐项比较，命中当前项就调当前 handler，不命中才继续往后比”结构。

所以唯一自洽的解释是：

1. `bl #0xc833ac` 的结果是一个布尔对象
2. `bit4=1` 表示“**不相等**”
3. `tbnz w0,#4` 的作用是：**不相等则跳到下一个 case**
4. fall-through 才是“命中当前项，调用当前 handler”

这和 reward 尾端的 `0x875efc` 正好能严丝合缝解释通。

## 4. `0x875efc` 的精确语义

### 4.1 当前结构

reward 调度尾端如下：

```asm
0x00875ee8: ldr   x16, [x27 + 0x19078]   ; ref=11545 "onReward Cheat Triggered!"
0x00875ef4: bl    #0xc833ac
0x00875efc: tbnz  w0, #4, #0x875f24
0x00875f00: ...
0x00875f18: bl    #0x875f78
0x00875f24: ...
0x00875f40: blr   x2
```

结合上节结论：

- `tbnz -> 875f24` = 当前类型 **不等于** `"onReward Cheat Triggered!"`
- fall-through 到 `0x875f00 -> 0x875f18` = 当前类型 **等于** `"onReward Cheat Triggered!"`

也就是说：

**`0x875f78` 只在 cheat 类型命中时被调用。**

### 4.2 为什么 `0x875f78` 可以认定为 cheat 专属 helper

本地与 `test3 xref_db` 交叉确认：

1. `0x875f78` 只有 **1 个 caller**：`0x8758bc`
2. 它正好只从 `0x875efc` 的 fall-through 进入
3. 这个 fall-through 的前置 compare 对象正是 `ref=11545 "onReward Cheat Triggered!"`

因此，即便 `0x875f78` 自身字符串不显眼，它也已经满足结构上的单义性：

- 它不是普通 splash/interstitial/fullscreen/reward handler
- 它就是 **cheat 专属 handler**

### 4.3 patch 之后到底会发生什么

若改：

- `0x875efc: 40012037 -> 0a000014`

那么效果不是“改真假值”这么抽象，而是：

1. 不再看 `c833ac` 的比较结果
2. 无论当前是否命中 `"onReward Cheat Triggered!"`
3. **都直接跳到 `0x875f24`**
4. **永远不进入 `0x875f78` 这条 cheat 专属 helper**

所以这条 patch 的精确表述应改成：

**reward 反作弊 handler 旁路**

而不只是泛泛的“reward 分支候选”。

## 5. 它和 `0x7e861c`、`0xbc5dd0` 的关系

### 5.1 和 `0x7e861c` 不是替代关系

`0x7e861c` 的定位仍然是：

- 上层 privilege gate
- 直接吃 `0x8cf36c`
- 有自带 canonical true/false 尾声

它更像“**特权态消费门**”。

而 `0x875efc` 现在能确认的是：

- reward-only
- anti-cheat-only
- 作用点更贴近“拿奖励之前不要被判作弊”

所以：

- `0x7e861c` = 上层特权门
- `0x875efc` = 下层 reward 反作弊旁路

两者不是替代关系，层级不同。

### 5.2 和 `0xbc5dd0..0xbc5dd8` 是互斥策略

`0xbc5dd0..0xbc5dd8` 的作用是：

- 让 `喵喵饿了` 点击链直接 return null
- 结果是：不再点、不卡、等效退出用户路径

而 `0x875efc` 的意义恰好相反：

- 它只在你**还保留 reward 点击链**时才有意义
- 因为它试图让 reward 不被判作弊

所以这两条线是互斥的：

1. 如果目标是 `纯净稳定运行`
   - 用 `0xbc5dd0..0xbc5dd8`
   - 就不需要再试 `0x875efc`
2. 如果目标是 `保留点击喵块并尝试真拿喵粮/特权`
   - 就不能同时把 `0xbc5dc8` 提前砍掉
   - 此时才值得试 `0x875efc`

## 6. 两套建议方案

### 6.1 方案 A：纯净稳定运行

仍推荐：

1. `0x920d90: 692c0054 -> 29000014`
2. `0xbc5dd0..0xbc5dd8: efa100d1 a00b40f9 027041b8 -> e00316aa fd79c1a8 c0035fd6`
3. `0x7e861c: c0062036 -> 32000014`

这套的目标是：

- 故障窗不弹
- `喵喵饿了` 不再进入 reward / 卡加载
- privilege 路径走更窄上层门

### 6.2 方案 B：尝试保留 reward 特权能力

若目标改成“点喵块还能尝试得喵粮/触发 privilege”，建议测试顺序改成：

1. `0x920d90: 692c0054 -> 29000014`
2. **不要**上 `0xbc5dd0..0xbc5dd8`
3. `0x875efc: 40012037 -> 0a000014`
4. 如仍需 privilege 侧增强，再试 `0x7e861c: c0062036 -> 32000014`

这套的含义是：

1. 先把故障窗正文链堵掉
2. 保留点击链
3. 绕过 cheat 专属 handler
4. 再视情况叠加更上层的特权门

## 7. 一句话结论

这轮之后，`0x875efc` 的定位可以从“reward 候选”正式升级成一句更准确的话：**它是 `onReward Cheat Triggered!` 专属 handler 的旁路点，适合在“保留喵块点击、尝试真拿 reward privilege”这条路线上单独实验；而如果目标只是纯净稳定运行，它仍然排在 `0x920d90 + 0xbc5dd0..0xbc5dd8 + 0x7e861c` 之后。**
