## 一、★ 锁定过程

### 1.1 bool格式(主控破解)
- TRUE = 0xffeb00008071 (null_base+0x30)
- FALSE = 0xffeb00008061 (null_base+0x20)
- 字节序列: 71 80 00 00 eb ff 00 00 (TRUE) / 61 80 00 00 eb ff 00 00 (FALSE)

### 1.2 全单例扫描(Python frida API)
- singleton(thread.field@0x80) 前0x2800字节
- 扫8字节对齐字段: **69个TRUE(排除@0x1028/@0xed0) + 22个FALSE**

### 1.3 ★ 逐个flip验证(自动)
- 基线猫build次数=51
- **第一个候选 sing+0x20: flip TRUE→FALSE, 等3秒, 猫build次数仍=51(Δ=0=停止build)**
- rebuild(切页)后仍=51(不恢复=确认消失)
- **isHungry = singleton.field@0x20 = TRUE → 猫显示; FALSE → 猫消失**

### 1.4 验证确认
- field@0x20 原值=0xffeb00008071(TRUE), 猫饿显示
- flip到0xffeb00008061(FALSE)后, 猫(0xa52920)停止被调用
- 确认: isHungry控制猫widget是否build/显示

## 二、★★ 下一步: 抓guard PC → on-disk patch 根除

### 方法1: 静态反汇编(主控)
在libapp反汇编搜 `ldr xR,[xR,#0x80]` → `ldr xR,[xR,#0x20]`(读thread.field@0x80→field@0x20) 的代码 = guard. 它后面有 `tbnz/tbz/cmp w0, #4` 判断TRUE/FALSE → 条件跳转决定是否构造猫widget.

### 方法2: MemoryAccessMonitor(frida, 但ARM64可能不支持)
监控singleton+0x20地址, 触发rebuild, 抓读取PC.

### 方法3: frida hook读field@0x20的点
全.text搜读[sing,#0x20]的指令(libapp中`ldr xR,[sing,#0x20]`), 找guard函数.

### 主控出patch
guard读field@0x20后: TRUE→构造猫/显示; FALSE→不构造/隐藏.
**on-disk patch: 让guard恒判FALSE(不饿)→永不显猫.**
- 方案A: patch guard的条件跳转(反转/恒跳false分支)
- 方案B: 直接把singleton+0x20在初始化时写FALSE(但这需改初始化代码)
- 方案C: patch读field@0x20的指令, 恒返回FALSE
