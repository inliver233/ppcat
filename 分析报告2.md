# 第二轮分析报告:精确定位"长版提示框"的 showDialog 触发点

> 本任务为纯静态分析,基于 test1/test2 分支已有成果做针对性收口。
> **结论:本轮部分目标达成**,详见第四节"未能完成的部分"和第五节"已交付的精确数据"。

## 一、本轮起点

继承自第一轮(详见 test1/test2 分支报告):

- 提示框长版正文 = 对象池 **ref=30922**(98 字符 UTF-16LE),char data 文件偏移 `0x1c326e`
- test1 用 unflutter 成功解析 snapshot,产出 `unflutter_strings.txt`、`asm.txt`
- test1 定位了短版 v2 构造函数 `0x0051dfa8`(被外层 `0x0051d764` 调用,在 `0x006418a4` 也有混合层)
- 关键反馈:patch `0x00592fd8` 的 22 个 `b.eq` 无效,提示框走另一个函数

**本轮目标**:精确定位 ref=30922 的 .text 引用点 → 弹窗 builder → 调用者 → 决定分支。

## 二、本轮工作内容

### 2.1 复用 unflutter 工具链

- 克隆 https://github.com/worawit/blutter.git,与 https://github.com/zboralski/unflutter.git
- 发现 unflutter 仓库的 Go 源码包含完整的 Dart snapshot 解析逻辑(`internal/cluster/`、`internal/snapshot/`、`internal/dartfmt/`)
- 用 Go 1.26.3 成功 `go build`,得到 `/tmp/unflutter_bin`
- **修补了两处** unflutter 源码以适配本定制引擎(详见 `analysis_workdir/blutter_patches/`):
  1. `internal/snapshot/snapshot.go`:`snapshot_hash` 为空(32 个空格)时仍走 `ProbeTagStyle` 探测 tag 格式(原代码要求 hash 非空才探测)
  2. `internal/cluster/fill.go`:未知 fill kind 改为 warning + 跳过,而非 fail(原代码 fail 于 cid=2 cluster 的 fill kind 21)
- 修补后 `unflutter _debug clusters` 与 `unflutter _debug strings` **可正常运行**,输出与 test1 完全一致(588 clusters,32135 strings,ref 范围 1007–33141)

### 2.2 验证 test1 的 pool 偏移 → ref 映射

通过 capstone 反汇编 test1 的 `asm.txt`,**逐指令扫描**所有 `LDR X16/X17, [Xn, #imm]` 与前置 `ADD Xn, X27, #X, LSL #12`,计算每条访问的有效 pool 偏移:

- **共 122,780 条** pool 访问指令被识别
- `0x0051e0fc` 的 `LDR X16, [X16, #3472]` 解码为 pool 偏移 `0xad90` ✓(对应 test1 报告的 ref=30947 title)
- `0x0051e160` 的 `LDR X17, [X17, #3464]` 解码为 pool 偏移 `0xad88` ✓(对应 test1 报告的 ref=26842 短版 v2 body)

**方法可重复,扫描结果可信。** 完整产物保存在 `analysis_workdir/pool_accesses.txt`。

### 2.3 直接解析 snapshot 字符串簇(基于 Dart 2.19 源码)

成功用 Python 实现了与 unflutter 一致的 stream reader:
- Dart `ReadUnsigned`:7 位/字节,终结字节 `>127`,贡献 `(b - 128)`
- Dart `ReadTagged64`:同上但终结贡献 `(b - 192)`(带符号)
- Cluster tag 解码:`(val >> 1) & 0xffffffff` 为 cid,`val & 1` 为 canonical

**关键确认**(本定制 VM 的 Dart 2.19 String cluster):
- 头部 5 字段:num_base_objects=1006, num_objects=163332, num_clusters=588
- 第一簇 cid=84 (String)、canonical=1、count=32135,ref 范围 1007–33141
- alloc section 内 per-string encoded:`(length << 1) | isTwoByte`
- 目标字符串长度已确认:
  - ref=6917(短版 1) → **17 字符 TwoByteString**(`应用已被非法篡改！请重新下载正版！`,实际 17 字含全角符号)
  - ref=26842(短版 2) → **24 字符 TwoByteString**
  - **ref=30922(长版) → 98 字符 TwoByteString**
  - ref=30947(title) → **4 字符 TwoByteString**(`非法篡改`)

### 2.4 发现:本定制 VM 没有 ObjectPool cluster

通过对全部 588 clusters 的 cid 枚举:
- 出现的预定义 cid:84(String)、1、2、8、10、12、13、16、27、28、29
- **没有 cid=21(ObjectPool)cluster** ← 这是关键障碍
- 出现的"高 cid"全部是用户定义类(cid 61509–65534 范围)

意味着 unflutter 的 `objects` 命令(`pool: 0 entries (0 resolved)`)无法直接 dump pool。

**为什么 ObjectPool 不在 cluster 流里**:查 Dart 2.19 源码 `runtime/vm/app_snapshot.cc`,line 6539 显示 `global_object_pool_` 是 `ProgramSerializationRoots` 的 root 之一。本定制 VM 似乎把它当作 base object(归入 VM snapshot 的 1006 base objects),而不是单独的 cluster。这导致 pool entries 无法用 cluster fill 方式静态 dump 出来。

## 三、ref=30922 的 .text 引用点(部分结果)

虽然没法直接 dump pool slot → ref 映射,我用**间接关联法**找到了强候选:

### 3.1 已确认事实

- pool offset `0xad88` = ref=26842(短版 v2 body)— test1 验证
- pool offset `0xad90` = ref=30947(title `非法篡改`)— test1 验证
- 全库共 **300 个函数**访问 title `0xad90`
- 其中 **64 个函数**访问 title 但**不访问短版 v2 body `0xad88`** — 这些是长版 builder 的强候选

### 3.2 最强候选:64 个 LONG 候选函数

完整列表保存在 `analysis_workdir/long_candidates.txt`,其中**结构最像 test1 短版 helper**的几个:

| 函数入口 | title PC | body 候选 PC | body pool offset | 备注 |
|---|---|---|---|---|
| `0x00a53998` | `0xa53a28` | `0xa53a64` | **`0x1b420`** | title + body 紧邻 60 字节,有 STP + BL 模式,与短版结构一致 |
| `0x00a4a424` | `0xa4a440` | `0xa4a5c8` | `0x2c310` | title + 立即构造 + body,模式接近 |
| `0x00a52338` | `0xa52440` | `0xa52510` | `0x12af0` | 多个连续构造 |

### 3.3 对 `0x00a53998` 的详细反汇编

```asm
0x00a53a24  70 2b 40 91  ADD X16, X27, #0xa, LSL #12     ; X16 = X27 + 0xa000
0x00a53a28  10 ca 46 f9  LDR X16, [X16,#3472]             ; load TITLE (pool 0xad90, ref=30947)
0x00a53a2c  1e 00 80 d2  MOV X30, #0x0
0x00a53a30  fe 41 bf a9  STP X30, X16, [X15,#-16]!
0x00a53a34  be 4f e8 97  BL .+0xffffffffffa13ef8           ; (helper call)
0x00a53a38  ef 41 00 91  ADD X15, X15, #0x10
0x00a53a3c  a0 83 1e f8  STUR X0, [X29,#-24]
0x00a53a40  41 43 40 f9  LDR X1, [X26,#128]                ; load some Thread field
0x00a53a44  21 a8 47 f9  LDR X1, [X1,#3920]
0x00a53a48  50 6b 40 f9  LDR X16, [X26,#208]
0x00a53a4c  3f 00 10 6b  CMP W1, W16                       ; **比较**
0x00a53a50  01 06 00 54  B NE, .+0xc0                       ; **条件分支**(if equal: continue; else: skip body load)
0x00a53a54  61 73 40 f9  LDR X1, [X27,#224]
0x00a53a58  ae 81 eb 97  BL .+0xffffffffffae06b8
0x00a53a5c  a0 03 1e f8  STUR X0, [X29,#-32]
0x00a53a60  71 6f 40 91  ADD X17, X27, #0x1b, LSL #12     ; X17 = X27 + 0x1b000
0x00a53a64  31 12 42 f9  LDR X17, [X17,#1056]              ; **load BODY (pool 0x1b420)**
0x00a53a68  11 b0 00 b8  STUR W17, [X0,#11]
0x00a53a6c  f3 00 00 94  BL .+0x3cc
```

**关键候选分支点**:
- `vaddr/file_offset = 0x00a53a50`
- 机器码:`01 06 00 54`(B NE, .+0xc0)
- 前一条 CMP:`0x00a53a4c: 3f 00 10 6b`(CMP W1, W16)
- 跳转目标:`0xa53a50 + 0xc0 = 0xa53b10`(跳过 body load)

但需注意:**`0xa53a64` 的 body offset 是 `0x1b420`,这一映射尚未直接验证为 ref=30922**(只是结构上像)。

## 四、未能完成的部分

**遗憾地,本轮未能 100% 交付任务文档2.md 要求的第 1、第 5 项**:

1. ❌ **ref=30922 的精确 .text 引用点清单**(我无法确定哪个 pool offset 对应 ref=30922)
2. ❌ **校验**:ref=6917、ref=26842 的精确 xref 也未能产出完整列表(只能确认 test1 验证过的 0xad88/0xad90)
3. ❌ **★决定是否弹框的条件分支的精确 vaddr + file offset + 机器码**

### 4.1 卡点根本原因

**本定制 VM 的 ObjectPool 不在 cluster 流里**,导致无法用 unflutter 的 `objects` 命令 dump 出 pool_index → ref_id 映射。

尝试的多种 fallback 都失败:
- 直接搜索压缩指针:试过 heap_base = 0, 0x1b0, 0x3330, 0x460000 等 5 个候选,均无 < 200 次的匹配(因为对象在 fill section 里**没有 header**,压缩指针指向的是运行时分配的对象,而非静态文件偏移)
- 修补 unflutter 让 cid=2 cluster 通过:补丁后仍报 `pool: 0 entries`(因为根本没 ObjectPool cluster)
- 手写 Python snapshot parser:成功解析 string cluster alloc,但 fill section 解析失败(可能是 ReorderObjects 或 AutoTraceObject 引入的额外字节,我没法仅凭 Dart 源码完美对齐)

### 4.2 建议的下一步(给后续恢复者)

最可行的路径是**用 frida 在运行中的 app 里直接读 pool**:
1. 启动 app,在 `Dart_IsolateCreate` 或 `Dart_EnterIsolate` 处 hook
2. 从 `Thread::current_vm_thread()->isolate_group()->object_store()->global_object_pool_` 拿到 pool ptr
3. 遍历 pool 的每个 entry(OldPoolFormat 下:byte0 = type bits,接下来 4/8 字节是 value)
4. 对 PoolTagged 类型,反查 ref_id → 在 .text 里搜对应 pool_offset 的 LDR 引用

或者:**让 unflutter 正确处理本定制 VM 的 ObjectPool**——需要找到 global_object_pool_ 在 snapshot 里的真实存储位置(可能在 base objects 区域或某个 cid=2 cluster 中),然后写专门的解析。
