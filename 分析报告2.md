# 皮皮喵 Flutter AOT 静态分析报告2

## 1. 本轮范围与方法

- 本轮只收口《任务文档2.md》的单一缺口: 精确定位长版篡改弹窗正文 `ref=30922` 的 `.text` 引用,并给出“决定是否弹框”的最终条件分支。
- 实际使用产物:
  - `unflutter_strings.txt`
  - `unflutter_dump/asm.txt`
  - `unflutter_dump/snapshot.json`
  - `unflutter` 仓内本地 `ppcat-hybrid` profile 与 `internal/cluster/zz_inspect_test.go`
- 关键限制:
  - `unflutter` / `blutter` 都无法完整解析该定制 snapshot 的 `ObjectPool`; `go run ./cmd/unflutter disasm ...` 仍报 `pool 0 entry 1: unknown type 103`
  - 因此本轮的 `ref -> pool slot -> xref` 采用两步法:
    1. 先用 `unflutter_strings.txt` 与 `TestInspectPpcatPrePool` 确认字符串 `ref`
    2. 再对 `unflutter_dump/asm.txt` 做 PP(`X27`) 访问追踪,恢复相邻 pool slot 的 `.text` xref
- 地址约定:
  - 本样本关注区满足 `vaddr == file offset`
  - 下文 `vaddr` 与 `file offset` 记为同值

## 3. `ref=30922` 的 `.text` 引用点

### 3.1 结论

当前能稳定恢复出的“长版正文簇” `.text` 引用点共有 4 处:

| pool slot | `.text` 地址 | 所在函数 | 证据级别 | 说明 |
|---|---:|---:|---|---|
| `pp+0x16c10` | `0x008de460` | `0x008de378` | 高置信相邻簇 | `LDP X5, X30, [X16]` 的第一项 |
| `pp+0x16c18` | `0x008de460` | `0x008de378` | 高置信相邻簇 | 同一条 `LDP` 的第二项 |
| `pp+0x16c20` | `0x008de54c` | `0x008de514` | **最高置信, 直接候选** | 长版正文唯一直接加载点 |
| `pp+0x16c28` | `0x008cdf24` | `0x008cdf00` | 高置信相邻簇 | 相邻 `ref=30925(checkAppMetaData::\n)` |

其中:

- `pp+0x16c20 -> 0x008de54c` 是本轮最重要的直接证据
- `pp+0x16c28 -> 0x008cdf24` 明确落在 `checkAppMetaData::\n` 标签簇
- `0x008de378` / `0x008de514` / `0x008cdf00` 这三个函数在地址上连续,并共同消费 `0x16c10..0x16c28` 这一小段 pool cluster

### 3.2 为什么把 `pp+0x16c20` 认作 `ref=30922`

这是**高置信推断**,不是完整 pool 反序列化后的数学证明。依据如下:

1. `ref=30922` 与 `ref=30925(checkAppMetaData::\n)` 在字符串表中紧邻。
2. `pp+0x16c28` 的唯一 xref 是 `0x008cdf24`,而该函数正是连续装填 `3112/3120/3128...` 一串相邻 slot 的 packer。
3. `pp+0x16c20` 的唯一直接 xref 是 `0x008de54c`,且该值立刻参与一次 helper 调用与 `CMP/B.EQ` 判定,形态上最像“长版正文是否可用,进而决定是否继续弹框”。
4. `0x008de378` 的 `pp+0x16c10/0x16c18` 是同簇相邻项,进一步说明 `0x16c10..0x16c28` 是一段集中使用的 dialog/check 相关对象。

因此本轮把:

- `pp+0x16c20` 视为 `ref=30922` 的唯一直接 `.text` 加载点
- `pp+0x16c28` 视为 `ref=30925` 的直接 `.text` 加载点

## 5. 弹窗 builder / 判定函数 / 最终分支

### 5.1 builder 函数入口

当前最稳妥的说法:

- `0x008de378` = **长版弹窗 builder / tear-off 候选**

理由:

- 只消费长版正文相邻簇 `pp+0x16c10/0x16c18`
- 以 `LDP X5, X30, [X16] ; BLR X30` 形式做间接调用,非常像 closure / tear-off 入口

但要强调:

- 本轮**没有恢复出它的直接 `BL` 调用者**
- 它大概率是通过 pool 装载后的 `BLR` 间接到达

### 5.2 调用者 / 判定函数入口

本轮最强的判定函数是:

- `0x008de514` = **判定函数候选**

它的动作调用点是:

- `0x008de564  BL -> 0x00d99300`

也就是说,在当前证据链下:

- `0x008de514` 负责“是否继续弹框”
- `0x008de564` 是“继续执行动作”的调用点
- `0x008de378` 是同一长版对象簇中的 builder/tear-off 候选

### 5.3 ★决定是否弹框的最终条件分支

这是本轮最重要交付物。

| 项目 | 值 |
|---|---|
| 判定函数入口 | `0x008de514` |
| 长版正文加载 | `0x008de54c` |
| 前一条比较指令地址 | `0x008de554` |
| 前一条比较机器码 | `1f 00 16 6b` |
| 前一条比较反汇编 | `CMP W0, W22` |
| 条件分支地址 | `0x008de558` |
| file offset | `0x008de558` |
| 条件分支机器码 | `40 01 00 54` |
| 条件分支反汇编 | `B EQ, .+0x28` |
| 后续动作调用 | `0x008de564  BL -> 0x00d99300` |

对应片段:

文件: `unflutter_dump/asm.txt`

```asm
0x008de52c  00 30 55 f9  LDR X0, [X0,#10848]
0x008de530  70 17 40 f9  LDR X16, [X27,#40]
0x008de534  1f 00 10 6b  CMP W0, W16
0x008de538  80 00 00 54  B EQ, .+0x10
0x008de53c  70 d3 5a f9  LDR X16, [X27,#13728]
0x008de540  1f 00 10 6b  CMP W0, W16
0x008de544  81 00 00 54  B NE, .+0x10
0x008de548  60 5b 40 91  ADD X0, X27, #0x16, LSL #12
0x008de54c  00 10 46 f9  LDR X0, [X0,#3104]
0x008de550  d1 8f 1a 94  BL .+0x6a3f44
0x008de554  1f 00 16 6b  CMP W0, W22
0x008de558  40 01 00 54  B EQ, .+0x28
0x008de55c  b0 0f 40 f9  LDR X16, [X29,#24]
```

解释:

- `0x008de54c` 装载长版正文所在 slot

- 不等于时才走 `0x008de55c..0x008de564`

从“字节级 patch 让提示框不再弹出”的角度看,这就是当前最可信的最终门控分支。

## 7. 本轮结论

1. `ref=30922` 的唯一直接长版正文加载点是 `0x008de54c`，所在函数入口 `0x008de514`。
2. 当前最可信的“决定是否弹框”条件分支是:
   - `vaddr/file offset = 0x008de558`
   - `hex = 40 01 00 54`
   - 前一条 `cmp` 在 `0x008de554`
   - `cmp hex = 1f 00 16 6b`
3. `0x008de378` 是当前最可信的 builder / tear-off 候选,但仍未恢复出它的直接 `BL` 调用者。
4. `ref=26842` 的校验 xref 与第一轮结果吻合,说明本轮 PP 追踪方法可靠。
5. `ref=6917` 仍未闭合,但不影响本轮最重要的 patch 地址交付。
