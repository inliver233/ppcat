# 第十九轮代码恢复收口报告

> 分支：`test1`
>
> 角色：纯静态分析。本轮不再扩工具，而是直接吸收 `backup / test2 / test3` 最新成果，用本地 `libapp.so` 做字节级复核，收口到用户当前最关心的三件事：
>
> 1. 去广告后的稳定运行
> 2. `喵喵饿了` 的处理
> 3. `VIP / reward 特权` 的最窄可执行方案

## 1. 本轮结论

这轮之后，`test1` 这条线可以明确收敛成下面的排序：

1. **故障窗主线**仍然以 `0x920d90` 为首选 choke point，不回退到旧的 showDialog/NOP 思路。
2. **`喵喵饿了` 主线**仍然以 `0xbc5dd0..0xbc5dd8` 的“不可点击不卡”作为主交付；“完全消失”仍只保留为实验项。
3. **`VIP / 特权` 方向里，`0x7e861c` 仍然是目前最窄、最干净、最适合主控 patch-and-test 的上层 gate。**
4. `0xa54178 / 0xb9fc84` 继续不应被表述成“单一 VIP 总开关”；它们是**消费共享特权态**的大页面/厚状态机。
5. `0x89fe80` 内部两处局部消费点虽然比 `0x8cf374` 窄，但仍不如 `0x7e861c` 干净，不升格为主推荐。

## 2. 交叉验证后的统一认识

### 2.1 与 `backup` 运行时事实一致

`origin/backup:backup/第八轮-去广告去首启弹窗/ROUND8_FACTS.md` 继续决定优先级：

1. 去广告后的根问题是广告 channel 周期调用失败，不是普通 `showDialog`。
2. 故障窗根因是 `MissingPluginException -> 未捕获 -> 全局错误处理`。
3. `V3/V4 stub` 的统一返回值路线已经被运行时证伪，不能再当主线。
4. 因此真正能稳定推进的仍是：
   - 截断故障正文链
   - 让 `喵喵饿了` 退出用户路径
   - 在 reward / 特权层找更窄的上层 gate

### 2.2 与 `test3` 的一致点

`test3` 现在最有价值的不是单一 patch，而是它已经把 `ref -> slot -> .text -> function` 基本打穿。对本轮最关键的交叉结论：

1. `ref=27673` 故障正文只落在 `0x920d7c`。
2. `ref=28525` `喵喵饿了` 落在 `0xa54178 / 0xae30e0`。
3. `ref=11545` `onReward Cheat Triggered!` 落在 `0x875ee8 / 0x89219c / 0xaa62f4 / 0xaacee8`。
4. `0xbc5dc8` 的字符串命中清楚表明它就是 reward prompt / 喂喵点击链。
5. `0x8cf36c` 是共享 `reward privilege` 状态函数，调用者约 16 个，不能再把它说成单一 `isVip`。

### 2.3 与 `test2` 的一致与分歧

一致点：

1. `0xae30e0` 确实是 `喵喵饿了` 常驻块 builder。
2. `0x8758bc` 确实是广告调度分发器。
3. reward / 广告 / 特权不是一条简单 bool 链。

分歧点：

1. `test2` 对 `0x7e85f4` 一类粗门控的早期乐观判断，现在应继续降级。
2. `0x913bf8` 不应再回到高优先级。
3. `V4 stub / qsa` 方向仍然是高复杂度低确定性的备用线，不应压过现有静态 patch 主线。

## 3. `VIP / reward 特权` 的当前正确表述

### 3.1 `0x8cf36c` 是共享特权态，不是单一 VIP 开关

本地与 `test3 xref_db` 交叉复核：

- `0x8cf36c` 为完整 bool-returning 函数
- 字符串包含：
  - `adNum`
  - `lastTime`
  - `today`
  - `totalNum`
  - `totalTime`
  - `leaveTime:`
  - `已生效！\n剩余`
  - `已过期！`
- 调用者约 16 个，包含：
  - `0x7e8054`
  - `0x7e8534`
  - `0x89fe80`
  - `0xa54178`
  - `0xb9fc84` 上游链

这说明它是底层共享“特权是否生效”计算，而不是某个页面专属的 `isVip=true/false`。

### 3.2 `0xa54178` 是 VIP 内容页装配，不是单点总闸

本地复核：

```asm
0x00a541c8: bl  #0x8cf36c
0x00a541cc: stur x0, [x29, #-0x18]
```

并且该结果在页内多次被消费：

- `0xa55684`
- `0xa55870`
- `0xa55a24`

页面字符串也证明它是混合内容装配器，而非单点总闸：

- `通过捐赠绑定设备获取`
- `功能，通过捐赠绑定账号获取`
- `未获得特权将在0点重置浏览次数！`
- `已生效`
- `每日喂喵`
- `喵喵饿了`

所以更准确的表述仍然是：

- `0xa54178` = VIP / 喵块 / 绑定说明 / 若干页面子块的装配函数
- 它消费共享特权态
- 但不适合直接给出“入口 return-null / 单分支强真”的粗 patch

### 3.3 `0xb9fc84` 是更厚的混合状态机

`test3 xref_db` 与本地历史分析一致：

- `0xb9fc84` 调用链更厚
- 字符串同时混有：
  - `未登录`
  - `未获取`
  - `通过捐赠绑定设备获取`
  - `功能，通过捐赠绑定账号获取`
  - `检测更新`
  - `喵喵喵~`

因此它比 `0xa54178` 更不适合当作“单点 VIP 恒真”实验点。

## 4. 为什么 `0x7e861c` 仍然是当前最窄的主推荐

本地反汇编窗口如下：

```asm
0x007e85f8: bl   #0x7e41c0
0x007e8610: tbz  w0, #4, #0x7e86f4
0x007e8618: bl   #0x8cf36c
0x007e861c: tbz  w0, #4, #0x7e86f4
...
0x007e86e4: add  x0, x22, #0x30
0x007e86f4: add  x0, x22, #0x20
```

对应原始字节：

- `0x7e861c: c0 06 20 36`

推荐 patch：

- `0x7e861c: c0062036 -> 32000014`

理由：

1. 它已经在 `0x7e41c0` 的前置检查之后。
2. 它直接消费 `0x8cf36c` 的 bit4 结果。
3. 它有自带的 canonical `true/false` 尾声，语义清晰。
4. 它比 `0x8cf374` 的影响面小。
5. 它比 `0xa54178 / 0xb9fc84 / 0x89fe80` 这些混合 UI/页面状态机更干净。

所以在当前所有候选里，`0x7e861c` 仍然是最适合主控做 `patch-and-test` 的**上层 reward / 特权 gate**。

## 5. 本轮新增但不升格的候选

### 5.1 `0x89fe80` 内部两处局部门

本地反汇编：

```asm
0x008a0448: bl   #0x8cf36c
0x008a044c: tbz  w0, #4, #0x8a0564
...
0x008a054c: bl   #0x47369c
0x008a0554: mov  x0, x22
```

```asm
0x008a05ec: bl   #0x8cf36c
0x008a05f0: tbz  w0, #4, #0x8a07fc
```

原始字节：

- `0x8a044c: c0 08 20 36`
- `0x8a05f0: 60 10 20 36`

它们的问题是：

1. 虽然比 `0x8cf374` 更窄
2. 但仍混在 UI / 登录 / 来源 / 提示链里
3. 上下文明显比 `0x7e861c` 更重

所以它们现在只适合记录为“局部候选”，不适合升格为主推荐。

### 5.2 `0xa54178` 页内局部消费点

本地复核出的三个点：

- `0xa55684: 61002037`
- `0xa55878: 80002037`
- `0xa55a28: 01042036`

这些都证明 `a54178` 在消费特权态，但作用更像：

1. 局部 spacing / 样式选择
2. 某个子块是否显示
3. 某个子块走哪个文案

而不是值得直接提给主控试验的“VIP 页面总开关”。

## 6. 推荐 patch 栈

### 6.1 主推荐

1. `0x920d90: 692c0054 -> 29000014`
2. `0xbc5dd0..0xbc5dd8: efa100d1 a00b40f9 027041b8 -> e00316aa fd79c1a8 c0035fd6`
3. `0x7e861c: c0062036 -> 32000014`

解释：

1. `0x920d90` 负责故障正文链 choke point。
2. `0xbc5dc8` 负责 `喵喵饿了` 退出用户路径，等效消失。
3. `0x7e861c` 负责 reward / 特权方向当前最窄的上层 gate。

### 6.2 次级实验项

1. `0xae3560: 400c0054 -> 1f2003d5`
2. `0x875efc: 40012037 -> 0a000014`
3. `0x8cf374: ef8101d1 -> 2a000014`

解释：

1. `0xae3560` 更像 `喵喵饿了` 显示侧 fallback 门。
2. `0x875efc` 位于 `onReward Cheat Triggered!` 反作弊链，本地与 `test3` 都表明它值得实验，但仍需主控运行时验证收益和副作用。
3. `0x8cf374` 仍然太底层、太共享，只适合作为高风险实验项。

### 6.3 明确撤回 / 不再推荐

1. `0xae312c` 作为主隐藏方案。
2. `0x7e85f4` 作为主总闸。
3. `0xa54178 / 0xb9fc84` 入口 return-null。
4. `0x89fe80` 两处局部门升格为首选。
5. 回到 `V3/V4 stub` 作为当前主路线。

## 7. 一句话结论

这轮之后，`test1` 这条线可以稳定表述为：**故障窗靠 `0x920d90`，`喵喵饿了` 靠 `0xbc5dc8` 退出用户路径，而 `VIP / reward 特权` 方向当前最值得继续推进的不是粗的 `0x8cf374`，也不是厚的 `0xa54178 / 0xb9fc84`，而仍然是更窄、更干净的 `0x7e861c`。**
