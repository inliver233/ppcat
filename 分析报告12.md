# 第十二轮代码恢复补充报告

> 分支：`test1`
>
> 角色：纯静态分析。本轮重点是吸收 `test3` 第八轮下半的新突破，并用 `test1` 本地对原始 `libapp.so` 做二次复核，把 `喵喵饿了` 相关方案重新排序成更偏实用的 patch 清单。

## 1. 本轮结论

### 1.1 新的主结论

上一轮我把 `0xae3560` 作为 `喵喵饿了` 更精细的 fallback 门提出，这条线仍成立，但本轮交叉验证 `test3` 新提交后，我认为：

1. **`0xbc5dc8` 应升级为“喵喵饿了”主推荐 patch 点。**
   - 它不是泛化 UI helper
   - 而是直接命中：
     - `“是否看推荐信息获取喵粮喂喵？”`
     - `“饿了喵~”`
     - `AdPgl_Reward|AdMob_Reward|AdNet_Reward|AdKs_Reward`
   - 也就是“点击喵块 -> 弹 reward 提示 -> 继续奖励链”的直接入口
2. **`0xae3560` 降级为次级实验项。**
   - 它更像“默认喵块 fallback 文案”的精细门
   - 但不如 `0xbc5dc8` 那样直接解决“点了就卡加载”的用户痛点

### 1.2 这一轮后的排序

如果目标是“先让当前 APK 变得更可用”，我现在建议的优先级是：

1. `0x920d90`：残留故障窗 Dart 侧兜底
2. `0xbc5dd0..0xbc5dd8`：喵块点击链直接 return null，不再进入 reward prompt / 卡加载
3. `0xae3560`：只压默认 `"喵喵饿了"` fallback 文案
4. `0x875efc`：reward anti-cheat 实验性旁路
5. `0x8cf374`：reward/no-ad 特权态实验项

## 2. `0xbc5dc8` 的独立复核

### 2.1 xref / 字符串证据

我用 `origin/test3:analysis_workdir/xref_db.json` 对 `0xbc5dc8` 做了独立复核，结果如下：

- `callers = []`
- `bool_ret = False`
- `strings` 命中：
  - `AdPgl_Reward|AdMob_Reward|AdNet_Reward|AdKs_Reward`
  - `获取`
  - `饿了喵~`
  - `是否看推荐信息获取喵粮喂喵？`

这说明它不是外围小函数，而是直接贴着 reward prompt / 喵粮引导链的 async 闭包。

### 2.2 本地反汇编关键窗口

`0xbc5dc8` 开头：

```asm
0x00bc5dc8  fd79bfa9  stp x29, x30, [x15, #-0x10]!
0x00bc5dcc  fd030faa  mov x29, x15
0x00bc5dd0  efa100d1  sub x15, x15, #0x28
0x00bc5dd4  a00b40f9  ldr x0, [x29, #0x10]
0x00bc5dd8  027041b8  ldur w2, [x0, #0x17]
...
0x00bc5dfc  2531f397  bl #0x892290
...
0x00bc5e98  da43fa97  bl #0xa56e00
0x00bc5ea0  e00316aa  mov x0, x22
0x00bc5ea4  ef031daa  mov x15, x29
0x00bc5ea8  fd79c1a8  ldp x29, x30, [x15], #0x10
0x00bc5eac  c0035fd6  ret
```

中段继续构造另一套 UI / reward 对话链，最终第二次自然 `return null`：

```asm
0x00bc60c4  e00316aa  mov x0, x22
0x00bc60c8  ef031daa  mov x15, x29
0x00bc60cc  fd79c1a8  ldp x29, x30, [x15], #0x10
0x00bc60d0  c0035fd6  ret
```

### 2.3 这意味着什么

`0xbc5dc8` 的整体形态已经足够说明：

1. 它本来就是一个“做完 UI / prompt / reward 编排后，返回 null”的 async 闭包
2. 不是那种必须返回复杂对象给调用方继续消费的函数
3. 所以它很适合用“保留已建栈帧，直接回 null”这一类安全 patch 手法

## 3. 主推荐 patch：`0xbc5dd0..0xbc5dd8`

### 3.1 精确 patch

- 地址：`0xbc5dd0`
- 原始字节：
  - `ef a1 00 d1`
  - `a0 0b 40 f9`
  - `02 70 41 b8`
- patch 字节：
  - `e0 03 16 aa`
  - `fd 79 c1 a8`
  - `c0 03 5f d6`

即：

```text
0xbc5dd0: efa100d1 a00b40f9 027041b8
       -> e00316aa fd79c1a8 c0035fd6
```

### 3.2 语义

保留：

- `0xbc5dc8  stp x29,x30,[x15,#-0x10]!`
- `0xbc5dcc  mov x29,x15`

然后直接：

1. `mov x0, x22`
2. `ldp x29, x30, [x15], #0x10`
3. `ret`

也就是：

- 点击喵块
- 进入 reward prompt async 闭包
- 立即正常完成并返回 `null`
- 不再弹 `“是否看推荐信息获取喵粮喂喵？”`
- 不再进入 reward 广告 / “正在加载” / 后续失败链

### 3.3 为什么我现在把它排在 `0xae3560` 前面

因为它更直接满足用户实际诉求：

1. 用户最烦的是“点了喵块就卡住”
2. 这条 patch 直接切断点击链
3. 即使喵块仍可见，至少：
   - 不再弹 reward 提示
   - 不再转进广告链
   - 不再卡“正在加载”

也就是说，它更像一个**功能层止损补丁**，而不只是 UI 层文案 suppress。

## 4. `0xbc5dc8` 的安全性说明

### 4.1 栈平衡

函数前两条已经完成建帧：

```asm
0xbc5dc8  stp x29, x30, [x15, #-0x10]!
0xbc5dcc  mov x29, x15
```

patch 用：

```asm
mov x0, x22
ldp x29, x30, [x15], #0x10
ret
```

能完整回卷这 16 字节的影子栈帧，静态上是平衡的。

### 4.2 返回约定

`0xbc5dc8` 原函数就存在至少两个自然的 `return null` 出口：

- `0xbc5ea0..0xbc5eac`
- `0xbc60c4..0xbc60d0`

这说明“返回 null”不是异常路径，而是函数本来就合法的完成方式。

### 4.3 功能隔离

从字符串命中看，这条函数非常集中在：

- reward prompt
- 喵粮获取
- `饿了喵~`

它不像 `0xb9fc84 / 0xa54178 / 0x7e8534` 那样是混合状态机或全局广告门控，所以入口后直接 return null 的副作用面更窄。

## 5. `0xae3560` 的新定位

### 5.1 继续保留，但降级

上一轮提出的：

- 地址：`0xae3560`
- 原始字节：`40 0c 00 54`
- patch 字节：`1f 20 03 d5`

仍然保留，理由不变：

```asm
0x00ae355c  cmp w0, w22
0x00ae3560  b.eq #0xae36e8
...
0x00ae36fc  ldr x17, [x17, #0xde0] ; ref=28525 "喵喵饿了"
```

这条门确实贴着 `"喵喵饿了"` 的默认 fallback 文案块。

### 5.2 为什么降级

因为和 `0xbc5dc8` 相比：

1. 它解决的是“默认喵块 fallback 是否出现”
2. 不是“点击后卡加载”的直接链
3. 且它所在的 `0xae30e0` 是共享 widget builder，风险边界没有 `0xbc5dc8` 那么窄

所以现在更适合把它定位成：

- **次级实验项**
- 当主控还想继续压“可见喵块”时再试

## 6. 与 `0x920d90 / 0x875efc / 0x8cf374` 的组合

### 6.1 当前最实用组合

我现在建议主控先试：

1. `0x920d90: 69 2c 00 54 -> 29 00 00 14`
2. `0xbc5dd0..0xbc5dd8 -> e00316aa fd79c1a8 c0035fd6`

这两条组合的定位是：

- `0x920d90`：压掉残留故障窗的正文构造链
- `0xbc5dc8`：压掉点击喵块后的 reward prompt / 卡加载

这比：

- 先碰 `0x875efc`
- 或先碰 `0x8cf374`

更适合作为当前 APK 的稳定化方案。

### 6.2 后续实验顺序

若上面两条已经让 APK 基本可用，再继续试：

3. `0xae3560: 400c0054 -> 1f2003d5`
4. `0x875efc: 40012037 -> 0a000014`
5. `0x8cf374: ef8101d1 -> 2a000014`

含义：

- `0xae3560`：压默认喵块 fallback
- `0x875efc`：尝试绕过 reward anti-cheat 分支
- `0x8cf374`：实验性强制特权态 / no-ad

## 7. 对 `test3` 最新结论的交叉验证

### 7.1 一致

我和 `test3` 最新补充报告一致的点：

1. `0xbc5dc8` 确实是 reward prompt / 喂喵点击链的关键 async 闭包
2. `0x920d90` 仍是当前最稳的故障窗兜底
3. `0x8cf374` 仍只能列为实验性特权态方案
4. `V3 / generic stub` 不该再作为主推荐方向

### 7.2 我这里额外保留的点

和 `test3` 相比，我这里继续保留：

- `0xae3560` 作为更细的 UI fallback 门

但它的优先级已经下调，不再压过 `0xbc5dc8`。

## 8. 当前建议的排序

如果只问“下一轮主控最值得先试哪几个点”，我现在的排序改成：

1. `0x920d90 -> 29000014`
2. `0xbc5dd0..0xbc5dd8 -> e00316aa fd79c1a8 c0035fd6`
3. `0xae3560 -> 1f2003d5`
4. `0x875efc -> 0a000014`
5. `0x8cf374 -> 2a000014`

这比我上一轮的排序更贴近用户目标：

- 先不弹故障窗
- 再让喵块“点了别卡”
- 然后才继续追“喵块是否消失 / reward 是否成功 / 特权是否常开”
