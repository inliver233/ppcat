## 一、一句话结论（仲裁后）

1. **0xa52bb0 返回的喵喵块 widget = cid 0xaf7（十进制 2807）实例**（字节铁证：唯一分配 thunk 0xa52c84）。caller 0xa52920 建 14 域内容 child（cid 0xacd，非我初判的 0x7df）后 tail-call 0xa52bb0，返回 `{cid0xaf7: field@0xb=回调/集合, field@0xf=内容child}`——**返回对象恰好只写这 2 个域**（验证者 V3/V1 逐字节确认）。
2. **★ 返回槽 cid 敏感（V5 REFUTED 确立，本轮核心纠偏）**：不是"返回任意合法 Widget 即可"。主控实测 obj91683（builder 内部真实 widget 子件）返回→**阅读页空白**，直接证伪"任意非空 widget 都行"。正确轴是 **cid==0xaf7 vs cid≠0xaf7**。
3. **★ 无 cid-safe 替代对象（E1 确立）**：全 app **没有任何 const cid0xaf7 实例**（cid0xaf7 唯一 thunk 是运行时 0xa52c84，pool 中无对应 const 实例）。obj44988（slot 0x753）虽确定非 String（strings+names 双缺）、是 app 在 0x546760"尺寸==0"分支返回的"渲染空"const Widget（231 加载），但 **cid≠0xaf7**（cid0xaf7 私有、obj44988 通用），按 V5 推断大概率→空白。
4. **→ 唯一 cid-safe 的 safe-hide = 修改 0xa52bb0 输出对象的域（Patch D）**：返回对象仍是合法 cid0xaf7（reader build 不报类型错），但内容 child 被 null 替换 → 块塌缩/不渲染。**这是当前静态能给出的、类型最安全的隐藏方案**，但"块是否真塌缩成零尺寸"仍需主控装机确认（V5 警告：cid 安全≠布局必崩）。
5. **reader build 静态不可定位（V6 + E2 双确认）**：0xa52920/0xa52bb0 外部 BL caller=0（pool 间接，switchable-call t3），目标解析依赖 Code 对象图——blutter cid84 崩溃使其不可解。最佳 reader 区候选 = 0xbf9be8（31112B，reader 模式标志齐全），但喵喵链不可证，仅低置信。

## 二、0xa52bb0 内部结构精解（逐字节，V1/V3 复核）

### 2.1 函数体（224 字节）

```
序言(标准):
  0xa52bb0: stp x29,x30,[x15,#-0x10]!      ; 影子栈压帧
  0xa52bb4: mov x29,x15
  0xa52bb8: sub x15,x15,#0x10              ; [PATCH 落点区]
  0xa52bbc: ldr x16,[x26,#0x38]            ; Thread stack-limit
  0xa52bc0: cmp x15,x16
  0xa52bc4: b.ls 0xa52c78                  ; 栈溢出守卫 → 0xf83fec

  0xa52c0c: bl 0xa53248                             ; ★ CALL_C (5-arg 容器辅助)

  0xa52c54: stur x0,[x29,#-8]                       ; 存 CALL_F 结果
  0xa52c58: bl 0xa52c80                             ; ★ 分配 cid0xaf7 thunk → x0=新喵喵块对象
  0xa52c5c: ldur x1,[x29,#-8]                       ; x1 = CALL_F 结果(回调/集合)

  0xa52c68: stur w1,[x0,#0xf]                       ; ★ 写 field@0xf = 内容 child

```

### 2.2 返回对象数据流（V1 逐字节确认）

- **caller 0xa52920** 在 `0xa52b90: str x1,[x15,#-8]!` 把内容 child x1（cid0xacd，14 域大复合，建于此函数）压影子栈，`0xa52b94: bl 0xa52bb0`。
- **callee 0xa52bb0** 序言使 `x29 = 压栈后值`，caller 的 `[x15,#-8]` = callee 的 `[x29,#0x10]`（帧数学：stp−0x10 + caller−8 = x29+0x10，V1 核对无误）。
- `0xa52c64: ldr x1,[x29,#0x10]` 取回内容 child → `0xa52c68: stur w1,[x0,#0xf]` 写入 field@0xf。**铁证：field@0xf = 内容 child。**
- `0xa52c60: stur w1,[x0,#0xb]` 写入 field@0xb（源 = CALL_F 0xa52c90 结果，回调/集合）。

→ **返回的喵喵块 = {cid0xaf7: field@0xb=回调, field@0xf=内容}**，是 GestureDetector 式结构（onTap+child）。返回对象**仅这 2 个域**（V1 完整性确认）。

### 2.3 各子调用职能（验证者确认 + 簇解析）

| 子调用 | 地址 | 职能 | 证据 |
|---|---|---|---|
| CALL_A | 0xf83044 | slot 分发（slot96447） | 通用 slot-loader stub |
| CALL_B | 0xa532d8 | 分配 cid0xaf3（2803）子件 | movk x2,#0xaf3,lsl16（唯一 thunk 0xa532dc） |
| CALL_C | 0xa53248 | 5-arg 容器辅助（就地写父域@0x13） | 反汇编：ldur w0,[x29,#0x28];stur w0,[x2,#0x1b]…;**返回 null** |
| CALL_F | 0xa52c90 | **循环建集合/children**（x5:0→count） | 反汇编：cmp x5,x3;b.ge;bl 0xf82f38… |
| alloc | 0xa52c80 | **分配 cid0xaf7 主对象** | mov x2,#0x204;movk x2,#0xaf7;ldr x4,[x26,#0x228];br x4 |

→ **"喵喵饿了" label（ref28525）是 const 内嵌**（scan_slot 确认仅 cat 区 0xa54178/0xae30e0 加载，reader/0xa52bb0 区零加载），在内容 child（cid0xacd）的 const widget 树里，不运行时 LDR。

## 三、★★ safe-hide 方案（按置信度，全部标注静态/验证）

### 3.0 核心约束（V5 REFUTED 确立，务必先读）

验证者 V5 驳回了"返回任意合法 Widget 即可避免空白"的乐观。**主控实测 obj91683（builder 内部真实 widget 子件）返回→阅读页空白**，直接证明返回槽对 **cid 敏感**：必须是 cid0xaf7（或 reader build 不报错的等价布局）。因此：

- ❌ **slot0x3975（ref1973"SizedBox.shrink"）= String 陷阱**（前次误判致 SIGSEGV，勿用）。
- ❌ **obj44988（slot0x753）= cid≠0xaf7**，大概率→空白（E1 确认其通用 widget、非喵喵块私有类）。
- ❌ **obj91683/obj51369** 已实测→空白。
- ★ **全 app 无 const cid0xaf7 实例**（E1 全 .text 扫：cid0xaf7 唯一 thunk 0xa52c84 是运行时分配，pool 无 const 对应）。

→ **任何"换一个对象返回"的方案都无 cid-safe 保证**。唯一 cid-safe 的隐藏 = **让 0xa52bb0 返回的对象仍是 cid0xaf7，但把内容域 null 掉**。

### 3.1 ★【主推 A】Patch D：内容 field@0xf null（外科式，cid-safe）

**原理**：返回对象仍是合法 cid0xaf7（reader build 不报类型错），但 field@0xf（内容 child）被 null 替换。若 reader build 按此布局渲染，块塌缩/不渲染内容。

```
地址 0xa52c68 (1 字节改):
  原:  01 f0 00 b8   stur w1, [x0, #0xf]      ; w1 = 内容 child
  改:  16 f0 00 b8   stur w22, [x0, #0xf]     ; w22 = null  (只改寄存器操作数 Rt: 01→16)
```

- **字节编码**：验证者（patch_D）逐位确认。STUR 指令 Rt=bits[4:0]，w22=22=0x16，故低字节 0x01→0x16。XOR(原,改)=0x00000017，全部落在 bits[4:0]，size/opcode/imm9/Rn 不变。**外科级，只动寄存器，不影响控制流/栈/其它域。**
- **为何 cid-safe**：返回对象仍是 0xa52c80 分配的 cid0xaf7，只是 field@0xf=null。
- **崩点预判（需主控验证）**：
  - 若 reader build 对 field@0xf 做 null 检查（如 `.child?.build()`）→ 块塌缩零尺寸 → ✓ 不可见。
  - 若 reader build 直接 `.child.build()` 不检查 → null deref → **可能崩或空白**（与 entry-null 同效）。
- **建议同时 null field@0xb**（回调/集合，见 3.2），降低块残留可见性。

**【需主控 patch-and-test】**：装后观察 (a) 块是否消失 (b) 阅读页内容/翻页/页码正常否 (c) 不崩。

### 3.2 【主推 B】Patch D 全套：field@0xb + field@0xf 都 null

若 3.1 单 null 内容仍有残留，把回调域也 null：

```
地址 0xa52c60 (1 字节改):  原 01 b0 00 b8 → 16 b0 00 b8   (stur w1→w22, field@0xb)
地址 0xa52c68 (1 字节改):  原 01 f0 00 b8 → 16 f0 00 b8   (stur w1→w22, field@0xf)
```

- 返回对象 cid0xaf7 但两域皆 null → 最可能塌缩成空壳。
- 字节同样外科级（验证者确认）。

### 3.3 【实验，需主控验证】Patch obj44988 返回（E1 深挖产物，但 V5 警告）

E1 找到 obj44988（slot0x753）是 app 在 0x546760"尺寸==0"分支返回的"渲染空"const Widget。**理论上是理想隐藏物，但 V5 警告 cid≠0xaf7 可能→空白**。

```
地址 0xa52bb8 (func+8, 16 字节改):
  原:  ef 41 00 d1 | 50 1f 40 f9 | ff 01 10 eb | a9 05 00 54
       (sub x15,#0x10; ldr x16,[x26,#0x38]; cmp x15,x16; b.ls 0xa52c78)
  改:  60 4f 5d f9 | ef 03 1d aa | fd 79 c1 a8 | c0 03 5f d6
       (ldr x0,[x27,#0x3a98]=slot0x753 obj44988; mov x15,x29; ldp; ret)
```

- **slot 算术铁证**：slot 0x753 byteoff = **0x3a98**（0x753×8），**不是主控文档写的 0xea8**（0xea8=slot0x1d5=ref964，错对象）。ldr 直引 `604f5df9`（E1+我独立 capstone 双验证；注意前次报告/任务里写的 `60734091/98454009` 是**误抄的 slot0x3975 配方，错误，勿用**）。
- **栈对称**：func+8 时 x15==x29（序言 stp+mov x29 后未 sub），尾声 mov x15,x29 复位 → ldp 弹出正确帧（V4 框架确认）。
- **崩点预判（V5）**：cid≠0xaf7 大概率→阅读页空白（与 obj91683 同效）。**E1 与主控原"obj44988 渲染不稳"矛盾的解释**：主控文档写的偏移是 0xea8（=ref964，错对象），所以那次测的极可能不是真 obj44988——建议主控用**修正后字节**重测确认。

**【需主控 patch-and-test，且优先级低于 3.1】**：若 3.1 Patch D 仍可见，再用此 obj44988 方案。它是"一击隐藏"的最佳理论候选，但 cid 风险最高。

### 3.4 ❌ 排除：reader build NOP（V6/E2 墙）

reader build 静态不可定位（pool 间接 + blutter cid84 墙）。无法给"NOP 加块进 children"的点。

## 四、reader build 定位（V6 + E2 双确认：不可定位）

- **0xa52920/0xa52bb0 外部 BL caller=0**（pool 间接，switchable-call t3）。V6 纠正：我初判"入口指针零命中=墙"无区分性（所有函数都零命中），墙真正由 **BL 全扫描**确立（0xa52920 外部 caller=0，唯一 BL 是 0xa52bb0 内的自身引用链）。
- **最佳 reader 区候选 = 0xbf9be8**（31112B，0xbf9be8-0xc01020，大帧 0x240，返回单一 widget，reader 模式标志齐全：isCoverPhotoView/isVerticalPhotoView/isChapterDirect/翻页/页数/手势串）。但喵喵链（→0xa52920）经 t3 不可解，体内无 cat 模块 BL、无喵喵簇 slot 0x3678-0x368d → **仅低置信**，不能据此给 NOP 点。
- **根因**：blutter cid84（String 类）崩溃 → Code 对象图不可解 → pool 间接调用目标不可静态解析。与报告 7/8 一致，非本次方法缺陷。

## 八、诚实边界（勿过度承诺）

1. **cid-safe 的隐藏 ≠ 保证不可见**（V5 警告）：Patch D 保证 reader build 不报类型错（返回合法 cid0xaf7），但块是否真塌缩取决于 reader build 对 null 内容域的处理——**静态无法 100% 确定**，需装机。
2. **无 const cid0xaf7 实例**（E1 铁证）：所以无法"返回一个现成的零尺寸喵喵块类实例"。这是本定制 VM 的结构限制。
