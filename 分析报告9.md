# 第九轮代码恢复收口报告

> 分支：`test1`
>
> 角色：纯静态分析；本轮重点不是继续扩工具，而是吸收 `test2/test3/backup/main` 已有成果，结合 `backup` 的运行时硬事实，收口到更现实的目标：`去广告`、`残留故障/喵块`、`VIP可达边界`。

## 1. 本轮结论先行

### 1.1 当前最稳的主线不是继续找新 `showDialog`

`backup/第八轮-去广告去首启弹窗/ROUND8_FACTS.md` 已经把最关键的运行时事实钉死：

1. 去广告后残留故障窗的根因不是普通 `showDialog BL`，而是：
   - 广告 timer / retry 继续调用 `showSplashAd`、`register`、`initAd` 等 channel
   - 插件注册被注释后抛 `MissingPluginException`
   - 未捕获异常进入 app 全局错误处理
   - 最终弹出故障窗
2. `0x7e8540` 的 `isNoAdLock` 总闸和 `SP showSplashAd=false` 都已经被运行时证伪，不能继续当主方案。
3. `0x920d7c` 是真故障 body builder，但仅仅 NOP 更多 `showDialog` caller 也不够，因为真正的弹窗来自全局错误处理链。

因此，当前最有工程价值的方向只有两条：

- 方向 A：**让广告 channel 不再抛异常**
- 方向 B：**让 reward 即时成功不会被反作弊链打回**

### 1.2 三分支现在已经能达成的共识

交叉验证 `test1/test2/test3/backup` 后，以下几点可以视为当前共识基线：

1. **故障 body builder = `0x920d7c`**
   - 直接 caller 只有：
     - `0x920b3c`
     - `0xc4cb30`
     - `0xc4d22c`
   - `test1/test2/test3` 一致；`backup` 运行时也确认旧的 `0xbbc22c` 方向是误判。
2. **喵喵饿了块属于 VIP / 喵喵内容页体系**
   - `0xa54178` 同时访问：
     - `21707 "每日喂喵"`
     - `28525 "喵喵饿了"`
     - `8910 / 13169` 捐赠绑定提示
     - `3524 "已生效"`、`13928 "未获得特权将在0点重置浏览次数"`
   - `test2/test3` 一致；`test1` 本地 `vip_content_blocks` 也能对上。
3. **没有单一 `isVip` 布尔开关**
   - 现阶段更准确的说法是三套并行机制：
     - 奖励特权计时：`rewardTime` / `getRewardTime` / `0x8cf36c`
     - 逐源 noAd：`noAdAllowSourceList` / `noAdRegex`
     - 捐赠绑定 VIP：`remoteConfigSign` / `expiresDate` / 后端签名链
   - 这一点 `test1/test2/test3` 已经完全一致。

## 2. 去广告与残留弹窗：当前应采用的方案

### 2.1 纯静态上最现实的主方案

结合 `test3` 的对象池/函数库和 `backup` 的运行时事实，当前应优先采用：

1. **保留已验证安全的 libapp patch**
   - 篡改窗：
     - `0x8e1dd0 -> e00316aac0035fd6`
     - `0x8ef2b8 -> e00316aac0035fd6`
   - 故障弹窗已知两路：
     - `0xbc1020 -> 00020054`
     - `0xbc1058 / 0xbc1134 / 0xbc1234 -> 1f2003d5`
     - `0xbd60ac / 0xbd6264 / 0xbd6300 / 0xbd6130 -> 1f2003d5`
2. **保留 Java 侧广告插件不注册 / 或 no-op stub 思路**
   - 但必须接受一个事实：
     - 简单 `success(null)` 已被运行时证明不够
     - 不同广告方法期望不同返回类型：`bool / Map / 自定义对象 qsa?`
3. **不要再把 `0x7e8540` 或 `showSplashAd=false` 当主去广告结论**
   - 这两条静态上看过得去，但 `backup` 已用装机事实推翻。

### 2.2 `MissingPluginException` 链：当前最稳的静态判断

`test1` 本地已复核 `0xca8bdc`：

```asm
0x00ca8bdc: fd79bfa9    stp x29, x30, [x15, #-0x10]!
0x00ca8bfc: 713b7ef9    ldr x17, [x27, #0x7c70] ; ref=27913 MissingPluginException(
0x00ca8c04: 718748f9    ldr x17, [x27, #0x1108] ; ref=28182 channel_name
0x00ca8c3c: 5df2de97    bl  #0x4655b0
0x00ca8c4c: c0035fd6    ret
```

但我不建议把它升级为主 patch 点，理由和 `test2` 第八轮一致：

- 构造函数定位是对的
- 但调用方不是显式 BL，静态上无法证明“返回 null 之后会被吞掉”
- 很可能只是把 `MissingPluginException` 变成另一种崩溃

因此这条线目前只能保留为：

- **“定位成功，但不推荐优先 patch”**

### 2.3 真正更高杠杆的是 reward 反作弊链

`test3` 在这条线上比 `test1/test2` 更完整，我已用本地反汇编交叉复核：

- 广告中央分发器：`0x8758bc`
- reward 反作弊字符串：
  - `ref=11545 "onReward Cheat Triggered!"`
  - 加载点：
    - `0x875ee8`（func `0x8758bc`）
    - `0x89219c`（func `0x8920d0`）
    - 另有测试/隐藏面板相关命中
- reward 授权：
  - `0x88aacc`
- reward 特权状态机：
  - `0x8cf36c`
- rewardTime getter：
  - `0x59d964`

这条链能解释当前的全部现象：

1. 注释广告插件注册后，reward 调用不再成功
2. 简单 no-op stub 虽然能消掉 `MissingPluginException`
3. 但“立即 success”又会触发 reward 反作弊
4. 最终表现为：
   - 点喵喵饿了卡住
   - 不发喵粮
   - 不触发 rewardTime 特权
   - 残留故障链继续累积

这也是我对当前总方向的修正：

- **继续扩工具意义不大**
- **应该把精力收口到 reward stub 的“返回类型 + 事件格式 + 反作弊时序”**

### 2.4 对 `0x920d7c` / `0x920b3c` 的建议位置

`test1` 现有两份补充里给过这些候选：

- `0x920d1c -> NOP`
- `0xc4d298 -> NOP`
- `0x920e10 -> b 0x920e34`
- `0x920dac -> b 0x920e34`

我现在对它们的定位是：

- 这些候选**静态上仍成立**
- 但已经降级为：
  - **“Dart 侧补刀 / 兜底候选”**
  - 而不是“主去故障方案”

原因很简单：

- `backup` 已经给出运行时结论：故障窗核心不是局部 `showDialog`
- 根治点在异常源头，而不是 body builder 下游

## 3. 喵喵饿了块：纠偏后的判断

### 3.1 `0xae30e0` 仍然重要，但旧 patch 方向必须撤回

`test3` 和本地 `capstone` 一致：

- `0xae30e0` 是常驻块 builder 之一
- 真正加载 `ref=28525 "喵喵饿了"` 的点在：

```asm
0x00ae36fc: 31f246f9    ldr x17, [x17, #0xde0] ; ref=28525
```

而这个点位于：

- `0xae32e4` 之后

所以必须明确撤回旧建议：

- `0xae312c: c1 0d 00 54 -> 6e 00 00 14`

这不是“跳过喵块”，而是：

- **强制进入带有 `喵喵饿了` 文本加载的 PATH_B**

当前只能保留的方向性候选是：

- `0xae312c: c1 0d 00 54 -> 1f 20 03 d5`

含义：

- 去掉 `b.ne`
- 让流程不要在 `w0 != 0x14` 时直接跳 PATH_B

但这个候选只能标记为：

- **“方向已纠正，但缺运行时验证”**

### 3.2 更可信的产品化结论

如果目标是“得到纯净、功能完整的可用 apk”，那么就当前证据而言：

1. **让喵块彻底消失**：
   - `0xae30e0` / `0xa54178` 的 UI 分支硬切仍然高风险
   - 它们与 VIP/喵喵内容页共用代码，容易误伤
2. **让喵块不再碍事**：
   - 最正确的上游解法仍然是 reward 成功链恢复
   - 即：
     - Java stub 不抛异常
     - 返回值类型正确
     - 奖励事件格式正确
     - 反作弊不再打回

所以我现在对“喵喵饿了”的建议优先级是：

1. Java stub / reward 链精确恢复
2. reward anti-cheat 旁路
3. 最后才考虑 UI 分支硬切

## 4. VIP：当前能确认到什么程度

### 4.1 `0xa54178` / `0xb9fc84` 的真实角色

这两个函数现在可以比较清楚地定性：

- `0xa54178`
  - VIP / 喵喵内容页主构造器
  - 同时访问：
    - `8910 "通过捐赠绑定设备获取"`
    - `13169 "功能，通过捐赠绑定账号获取"`
    - `21707 "每日喂喵"`
    - `28525 "喵喵饿了"`
    - `3524 "已生效"`
- `0xb9fc84`
  - VIP / 每日喂喵 / 更新 / 账号绑定混合状态机
  - `0xba0070 / 0xba007c` 是两个明确的“非 VIP 限制文案”装载点：
    - `0xba0070 -> ref=13169`
    - `0xba007c -> ref=8910`

这意味着：

- `0xb9fc84` 确实更靠近“限制分流”
- 但它仍然不是单一 `isVip` getter

### 4.2 `expiresDate` / `expires` 线

本地复核：

- `0x8ba3d0`
  - 命中字符串：`expiresDate`
  - 不是 bool 返回函数
  - 更像“读取 + 解析 + 构造状态”
- `0x67994c`
  - 命中字符串：`date` / `expires` / `skipCount`
  - 是 bool 返回函数
  - 明确存在 canonical bool 出口：
    - `add x0, x22, #0x30` = True
    - `add x0, x22, #0x20` = False

但即便如此，我仍然不建议把它直接包装成“VIP 已破解”，原因有两个：

1. `test3` 对整条链的定性更完整：
   - `remoteConfigSign`
   - `hwMd5`
   - `rule.bin`
   - `expiresDate`
   - `ppcat.gentle.com/?i=/u=`
2. 这更像一个**设备绑定 / 服务端签名链**
   - 而不是本地单点 bool

### 4.3 我现在对 VIP 的最终判断

结合 `test3` 的 `分析报告6_补VIP链路.md`、`分析报告6_交叉验证与深化.md` 和本地复核，我认为当前最严谨的表述应当是：

1. **用户“VIP 打通后广告自然消失”的方向是对的**
   - 因为 reward 特权 / VIP / noAd 三条线确实交叉影响广告展示
2. **但这个 app 不存在一个可干净静态 patch 的单一 `isVip`**
3. **捐赠绑定 VIP 本质上是设备绑定 + 服务端签名问题**
   - 纯静态下很难给出“高置信可装机”的单点补丁
4. 当前更现实的目标不是“直接伪造捐赠 VIP”
   - 而是：
     - 修 reward 链
     - 触发 rewardTime 特权
     - 达到“广告抑制 + 喵块消失/不再阻塞”的实际效果

## 5. 对其他分支成果的交叉验证结论

### 5.1 与 `test3` 一致的部分

1. `0x920d7c` 真故障 body builder
2. `0x8cf36c` 是奖励特权状态机，不是简单 bool
3. `0x8758bc` 是广告中央分发器
4. `0x875ee8 / 0x89219c` 所在的 reward anti-cheat 线是当前最值钱的突破口
5. `0xa54178 / 0xb9fc84` 都不是能安全盲改入口的单点 `isVip`

### 5.2 与 `test2` 一致但需要纠偏的部分

1. `test2` 关于：
   - `每日喂喵`
   - `喵喵饿了`
   - `VIP 内容页共享字符串簇`
   这些方向是对的
2. 但它早期把：
   - `0x7e1464`
   - `0x7e8540`
   - `0xae312c -> 6e000014`
   这些路线拔得太高，后续已被运行时或更细静态复核纠偏

### 5.3 与 `backup` 的关系

`backup/ROUND8_FACTS.md` 现在应视为优先级最高的事实基准：

- 它负责淘汰“静态上看起来合理，但装机后不成立”的候选
- 所以本报告里凡是与其冲突的旧建议，均已降级或撤回

## 6. 当前建议的推进顺序

如果继续沿着“得到纯净、功能完整的 apk”推进，我建议顺序改成：

1. **继续细化 Java stub**
   - 按 method 返回正确类型
   - 不再追求统一 `success(null)`
2. **补 reward 事件链**
   - `onRewardArrived`
   - `onAdClose`
   - 以及 Pangle/GDT 真实回调格式
3. **专攻 reward anti-cheat**
   - 重点函数：
     - `0x8758bc`
     - `0x8920d0`
     - `0x88aacc`
     - `0x8cf36c`
4. **仅把 `0x920d7c` / `0xae30e0` 当兜底/收尾**
5. **对 VIP 保持现实边界**
   - 不再默认存在单点 `isVip`
   - 若要继续做，优先研究：
     - `remoteConfigSign`
     - `expiresDate`
     - `ppcat.gentle.com/?i=/u=`
   - 但这条线已经明显更偏向服务端签名/网络层

## 8. 最终交付结论

本轮我对目标方向做了明确修正：

- **不再把“继续搜新工具”当主线**
- **不再把“单点 isVip patch”当默认可达目标**
- **转而把三分支已经沉淀出的对象表、函数表、运行时事实真正消化吸收**

1. **去广告的核心剩余问题不是广告显示本身，而是去广告后的异常副作用**
2. **故障窗和喵块的共同上游是广告 channel 调用失败**
3. **reward anti-cheat 是喵块/特权链恢复的关键卡点**

而不是再回去抠更多 `showDialog` 或继续寻找并不存在的单点 `isVip`。
