## 一、test3全部动态成果汇总(22轮)

### 1.1 环境搭建(报告12-13)
- ARM原生redroid 14 arm64-v8a + frida 17.15.1 root
- 突破houdini限制: libapp.so可定位/hook armed/attach不崩
- blutter彻底不可用(任务C关闭)

### 1.2 喵喵块定位(报告13-14)
- cat widget cid = **0x85e** (2142)
- cat constructor = **0xc62438** (alloc cid0x85e)
- cat wrapper = **0xa52920** (经cid虚分派调用, callers=0的原因)
- 构造链 lr = **0xd93204** (func 0xd931c0, blr x30虚分派)

### 1.3 isHungry锁定(报告15-20) ★核心成果
- isHungry = **singleton(thread.field@0x80).field@0x20**
- bool格式: TRUE=**0x8071**, FALSE=**0x8061** (null_base+0x30/+0x20)
- runtime flip sing+0x20→FALSE: **猫停止build, 确认消失(无白屏)** ←铁证

### 1.4 guard定位(报告21-23)
- guard = **0x466e2c** (tbz w0,#4, 在func 0x466dc8内)
- 0x466dc8是**共享函数**(不只猫, 还处理其他overlay)
- guard patch(0x466e2c→b / 0x466e28→add x0,x22,#0x20): **都白屏**
- → guard在共享函数内, **不能直接patch**

### 1.5 排除方案
- obj44988返回(0xa52928 patch): **崩abort**(cid敏感)
- guard patch 0x466e2c/0x466e28: **白屏**(共享函数)
- 反篡改A+B on-disk: **不够**(redroid有第3条路径)
- 反篡改spawn早注入: **不稳**(时序竞争)

## 四、redroid环境问题记录(供未来参考)

### 不稳定原因
- Oracle ARM A1免费实例(4c23G)跑redroid14+frida重负载→adb反复断
- 需反复`docker rm -f` + `docker run`(不能docker restart, binder设备丢)
- adb push/install大文件后adbd经常崩
- 绕过法: docker cp传文件(不走adb), 减少adb操作

### 可用的替代方案
- **LDPlayer**(主控, 反篡改不弹, 稳定, 但houdini限制hook不生效)
- **物理ARM设备**(树莓派5/ARM小主机 root + frida)
- **AWS Graviton**(按量付费, 专用不被抢占)
