# 第二十四轮代码恢复收口报告

> 分支：`test1`
>
> 角色：纯静态分析。本轮按用户要求，停止扩散到新工具和新入口，直接吸收 `backup / test2 / test3` 最新成果，并用本地 `libapp.so` 做字节级复核，收口到三项目标：
>
> 1. `故障弹窗` 继续压实稳定方案
> 2. `喵喵饿了` 给出稳定兜底和“尝试不显示”方案
> 3. `VIP / 特权页面` 继续拆细到可命名的页面子块，给出比单纯 `0x7e861c` 更贴近页面内容的观察点

## 1. 本轮结论

这轮之后，`test1` 这条线的主推荐可以继续收敛，不需要再发散：

1. **故障弹窗主推荐不变，继续以 `0x920d90` 为首选单点 choke point。**
   - 我本地再次复核，`0x920d7c` 仍只有 3 个直连 caller：
     - `0x920d1c`
     - `0xc4cc7c`
     - `0xc4d298`
   - 这与 `test3` 第八轮一致，因此 `0x920d90: 692c0054 -> 29000014` 仍是当前最干净、最窄的故障正文链截断点。

2. **`喵喵饿了` 仍分成两个层次处理：**
   - 稳定兜底：`0xbc5dd0..0xbc5dd8` 让点击链立即返回 null，块可见但不可点击、不卡加载。
   - 实验性“不显示”：吸收 `test3` 最新深挖结果后，主候选变成 `0xae32d4` / `0xae32c8`，而不是继续误以为 `0x8cf36c` 能直接控制块显示。

3. **`VIP / 特权页面` 内部结构又往前拆了一层。**
   - 以前只确认：
     - `0xa54d3c`：设备绑定提示块
     - `0xa55130`：账号绑定提示块
     - `0xa55c80`：每日喂喵子树起点
   - 这轮本地复核后可以再加两组明确块：
     - `0xa54e54 / 0xa54e80`：`已获取 / 未获得` 状态文案块
     - `0xa55584`：浏览次数 / 特权说明段的上游 gate
   - 也就是说，`0xa54178` 已经不只是“复杂页面装配器”，而是能稳定拆成若干个页面子块。

## 2. 交叉验证后的统一认识

### 2.1 与 `backup` 的一致点

`origin/backup:backup/第八轮-去广告去首启弹窗/ROUND8_FACTS.md` 仍然决定优先级：

1. 去广告后的根因是广告 channel 周期调用失败，不是普通 `showDialog`
2. 故障弹窗根因是 `MissingPluginException -> 未捕获 -> 全局错误处理`
3. `V3/V4 stub` 的统一返回值路线已被运行时证伪，不再回到主线
4. 所以真正稳定的路线仍然是：
   - 截断故障正文链
   - 让 `喵喵饿了` 退出用户路径
   - 在特权 / reward 层保留更窄的实验项，而不是追“单一 isVip 总闸”

### 2.2 与 `test2` 的一致点

`origin/test2:分析报告8.md` 和更早几轮里，和本轮继续一致的点有：

1. `0xae30e0` 的确是 `喵喵饿了` 常驻块 builder
2. `0xbc5dc8` 是 reward-prompt / 点击链，而不是普通页面构造函数
3. `0x8758bc` 是广告 dispatcher，不等于单一 VIP 逻辑
4. `0xca8bdc` 虽然是 `MissingPluginException` 构造函数，但不应替代 `0x920d90` 成为主推荐 patch 点

### 2.3 与 `test3` 的一致点

`origin/test3` 最新两份第八轮报告里，我认可并采用的结论是：

1. **故障弹窗：**
   - `0x920d7c` 是故障正文唯一构造点
   - `0x920d90: 692c0054 -> 29000014` 仍是最干净的 choke point

2. **喵喵饿了块：**
   - `0xae30e0` 是唯一明确的 build 入口
   - `0xae30e0` 所在 class 方法群无 null 早返回
   - `0x8cf36c` **不直接**控制这个块的 build 显隐
   - 因此上轮把 `0x8cf374` 升格为“隐藏块”方案是过度乐观，这一条需要修正

3. **reward / 反作弊：**
   - `0x875efc` 仍是 reward 反作弊旁路的最窄候选
   - 但它仍应留在实验项，而不是覆盖主推荐

## 3. 本地新增字节复核

### 3.1 `0xa54e54 / 0xa54e80` 是“已获取 / 未获得”状态块

本地反汇编确认：

```asm
0x00a54e54: ldr x17, [x17,#0xd38]   ; ref=9292  "已获取"
...
0x00a54e80: ldr x0,  [x0,#0xd40]    ; ref=29757 "未获得"
```

这两处位于 `0xa54d3c` 设备绑定提示块之后，同属 `0xa54178` 内部的状态文案分流段。

因此可以把 `0xa54178` 的前半段再细化成：

1. `0xa54930` 附近：账号绑定操作块
   - `点击头像绑定账号`
   - `解绑`
2. `0xa54d3c`：设备绑定提示块
   - `通过捐赠绑定设备获取`
3. `0xa54e54 / 0xa54e80`：获取状态块
   - `已获取`
   - `未获得`
4. `0xa55130`：账号绑定提示块
   - `功能，通过捐赠绑定账号获取`

### 3.2 `0xa55584` 是“浏览次数 / 特权说明段”的上游 gate

本地反汇编确认：

```asm
0x00a55584: tbnz w0, #4, #0xa55c2c
```

对应原字节：

```text
0x00a55584: 40352037
```

它所在大段内继续加载了：

```asm
0x00a555ac: ldr x0,  [x0,#0xd90]   ; ref=2878  "24小时内累计浏览"
0x00a555e4: ldr x17, [x17,#0xd98]  ; ref=13199 "次推荐信息，即可获取"
0x00a5571c: ldr x17, [x17,#0xda0]  ; ref=13928 "，未获得特权将在0点重置浏览次数！"
```

这说明 `0xa55584` 不是简单样式点，而是：

- 在 `0xa55c2c` 这段“每日喂喵 / 特权内容段”之前，
- 决定是否跳过一大段“浏览次数 / 获取特权说明”的页面装配逻辑。

因此 `0xa55584` 可以升格为一个新的页面级观察点，但仍不建议直接作为主 patch：

1. 它作用范围明显大于单个按钮或标签
2. 它前后混杂了“浏览次数说明 + 获取特权说明 + 进入每日喂喵段”
3. 粗暴改它的分支，有较高概率影响整页信息结构

### 3.3 `0xa55878` / `0xa55a28` 的页面级地位继续成立

这两点本轮继续复核：

```text
0x00a55878: 80002037
0x00a55a28: 01042036
```

对应反汇编：

```asm
0x00a55878: tbnz w0, #4, #0xa55888
0x00a55a28: tbz  w1, #4, #0xa55aa8
```

含义仍然和第 22/23 轮一致：

1. `0xa55878`
   - 上游内容级 gate
   - false 分支直接落到一个对象池对象
   - true 分支会继续走较重的子树构造

2. `0xa55a28`
   - 更贴近 `每日喂喵` 子树
   - 它之后继续串到 `0xa55c2c -> 0xa55c80`

### 3.4 `0xa55c80` 后续子树继续成立，但仍不够安全到能直接 patch

本地复核后，`0xa55c80` 之后仍然能稳定看出一串连续子树起点：

1. `0xa55c80`
2. `0xa55de0`
3. `0xa56070 / 0xa560f0`
4. `0xa563c0`
5. `0xa56544`
6. `0xa56690 / 0xa566c0`

其中已知明确字符串锚点包括：

```text
0x00a55c80 -> ref=21707 "每日喂喵"
0x00a55cbc -> off=0x2cde0 -> ref=28525 "喵喵饿了"
0x00a55dec -> off=0x2ce00 -> ref=31475 "让喵喵更智能"
0x00a5607c -> off=0x2ce08 -> ref=91698 (未定名对象)
0x00a560bc -> off=0x2ce10 -> ref=3838 "分享喵喵"
```

但这里仍有两个限制：

1. 这些块里混杂了字符串对象和未定名对象
2. 子树内部大量是共享 UI helper，单看一两个 `BL` 还不足以给出安全 patch

所以它们现在更适合作为“页面级结构图”的补充，而不是替代 `0x7e861c` 的新主推荐 patch。

## 4. `喵喵饿了` 的方案分级

### 4.1 稳定兜底：保持 `0xbc5dd0..0xbc5dd8`

这一点不变，继续作为主推荐：

```text
0xbc5dd0..0xbc5dd8
efa100d1 a00b40f9 027041b8
-> e00316aa fd79c1a8 c0035fd6
```

语义仍然是：

- 让 reward 点击链直接返回 null
- 块仍可能可见
- 但点击不再卡 `正在加载`

这是当前唯一同时满足：

1. 字节明确
2. 风险可控
3. 已有跨分支共识

的稳定方案。

### 4.2 实验性“不显示”：吸收 `test3` 最新结论后的分级

`test3` 最新第八轮已经把 `喵喵饿了块不显示` 方案收敛到两个 build 内部点：

1. **方案 B：**
   - `0xae32d4`
   - `e00301aa -> e00316aa`
   - 让 build return null
   - 最彻底，但最有可能触发 Widget 契约崩溃

2. **方案 A：**
   - `0xae32c8`
   - `20f000b8 -> 1f2003d5`
   - nop 内容写入，尽量让 build 返回空容器
   - 比 return null 温和，但不能静态保证实际渲染结果

因此现在对 `喵喵饿了` 的正确表述应该是：

1. **稳定主推荐：**
   - `0xbc5dd0..0xbc5dd8`
   - 目标是“不可点击不卡”，不是“绝对消失”

2. **实验项：**
   - `0xae32d4`
   - `0xae32c8`
   - 目标是“尝试不显示”

3. **撤回误判：**
   - 不再把 `0x8cf374` 升格为“隐藏块”主方案

### 4.3 `0x8cf374` 的地位修正

本轮之后，这个点的定位应修正为：

```text
0x8cf374: ef8101d1 -> 2a000014
```

它仍可保留为实验项，但只应描述成：

- 更广覆盖的共享特权态强制为 true
- 可能影响 reward privilege / 阅读期广告 / 部分 VIP 页面状态

而**不再**描述成：

- 可以直接隐藏 `喵喵饿了` 常驻块

## 5. 当前推荐 patch 栈

### 5.1 稳定主推荐

在既有基线上，继续推荐下面三项：

1. `0x920d90: 692c0054 -> 29000014`
   - 故障正文链 choke point
2. `0xbc5dd0..0xbc5dd8: efa100d1 a00b40f9 027041b8 -> e00316aa fd79c1a8 c0035fd6`
   - `喵喵饿了` 点击链直接返回
3. `0x7e861c: c0062036 -> 32000014`
   - 更窄的上层特权 gate

### 5.2 实验项

按风险从低到高继续保留：

1. `0x875efc: 40012037 -> 0a000014`
   - reward 反作弊旁路
2. `0x8cf374: ef8101d1 -> 2a000014`
   - 更广覆盖的共享特权态强制 true
3. `0xae32c8: 20f000b8 -> 1f2003d5`
   - 尝试空容器化 `喵喵饿了`
4. `0xae32d4: e00301aa -> e00316aa`
   - 尝试让 `喵喵饿了` build return null

## 6. 当前不再主推的方向

这轮之后，下面几条在 `test1` 报告里应保持降级：

1. `0xa54178` 入口级 patch
2. `0xb9fc84` 入口级 patch
3. `V3 / V4` stub 回到主线
4. 把 `0x8cf374` 直接说成“隐藏喵喵饿了块”的主推荐

第 24 轮的价值不在于发明新 patch，而在于把现有三条主线进一步压实并纠偏：
