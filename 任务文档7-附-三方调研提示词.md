四、★★★ 核心困境:故障弹窗(主控穷尽失败,求你们突破)
═════════════════════════════════════════════

### 4.1 机制(logcat + Dart 栈帧铁证)
app 有一个广告定时器(约每 60s),在主界面/书架页挂机时自动调用以下广告方法:
`showSplashAd`, `register`(ksad), `initAd`, `interstitial`, `showFullScreenVideoAd`,
`loadFullAd`, `loadInsertAd`, `loadSplashScreenAd`, `MobileAds#initialize`。
因为去广告(插件注册被注释),这些 channel 调用全部抛 **`MissingPluginException`**,
而且是 **`Unhandled Exception`(未捕获)**,冒泡到 app 的【全局错误处理】,
弹故障窗(ref=27673 "无法正常联网,导致广告无法显示!")。主控实测 **T+260s 必触发**,点确定可关。

### 4.2 ★ 最关键发现:故障窗【不走 showDialog BL 调用链】
主控 NOP 了下面这些 showDialog 编排器,**故障窗全部仍在 T+260s 触发**:
- 0xbc0f40(3BL) + 0xbd5a24(4BL)[第七轮]
- **第三源 0xc4cb30**(第八轮新发现:它调用故障body builder 0x920d7c,含4个showDialog BL:0xc4cc2c/ccac/ce0c/ce60)
- **22个广告失败 catch 函数的 showDialog BL**(通过 logcat Dart 栈帧定位的6个函数:
  0x883bd0 / 0x87b714 / 0x87b120 / 0x8789e8 / 0x87ccd4 / 0x881960,共22个BL,全安全模式)
→ 结论:故障窗由 app【全局错误处理】(类似 FlutterError.onError / runZonedGuarded / zone onError)
**异步**弹出,不经 `bl 0x47369c` / `bl 0x472c98` 这些常规 showDialog 入口。这就是 NOP showDialog 无效的原因。

### 4.3 故障body 与调用链(主控已查清)
- 故障body = ref 27673 "无法正常联网,导致广告无法显示!…"。
- 真·故障body builder = **0x920d7c**(加载 ref27673 + "adNum:" + "后台无响应" + "tips" 等)。
- 0x920d7c 的 3 个 caller(主控用全 libapp BL 扫描定位):
  `0x920b3c`(含0x920d1c)、`0xc4cb30`(含0xc4cc7c)、`0xc4d22c`(含0xc4d298)。
- **这 3 个 caller 全部没有 showDialog BL**,且全部 `callers=0`(走对象池 pool 间接 BLR 调用,即 Dart 虚方法/闭包)。
- 即:故障body 对象被构造后,通过 pool 间接分发交给【全局错误显示组件】。这条闭包链是主控静态追不下去的——正是你们三方(Ghidra / 全 xref DB / 对象池)的强项。

### 4.4 AdNoOpPlugin V3 stub 失败原因(主控已验证,别重复)
V3(Java侧注册无操作 MethodCallHandler)能消除 MissingPlugin(主控加日志确认接管了 showSplashAd/register/initAd/interstitial/showFullScreenVideoAd/loadFullAd/loadInsertAd/loadSplashScreenAd/MobileAds#initialize)。
**但返回值必崩**(不同广告方法的 Dart 端期望不同类型):
- `success(null)` → reward 路径:`NoSuchMethodError: []("channel_name") called on null`
- `success(空HashMap)` → `type 'HashMap' is not subtype of 'bool'`(showSplashAd 期望 bool)
- `success(Boolean.TRUE)` → 下游 `type 'bool' is not subtype of 'qsa?'`(期望自定义对象)
- `success(null)` 在主界面(非reward) → `Null check operator used on a null value`(T+66s,比 MissingPlugin 更快崩)
→ 结论:**不同广告方法期望 bool/Map/自定义对象(qsa)等不同类型,且部分用 `!` 强制非 null**。统一返回值无法满足。

### 4.5 根本矛盾
去广告(注释插件)必然 MissingPlugin → 故障窗。消除异常需精确模拟每个广告方法响应,或截断全局错误处理。这是三分支 6 轮 + 主控第八轮都没根治的**核心难点**。

五、求三方调研的 4 个方向(按主控判断的优先级,你们可自行判断调整)
═════════════════════════════════════════════

### 方向 A(主控最看好):定位 app 全局错误处理并截断
故障窗由全局错误处理弹。找到"接收未捕获异常 → 构造故障body → 显示"的地方并截断。
- 线索1:故障body builder 0x920d7c 的返回值(故障信息对象)被【谁】接收并显示。追 0x920d7c 返回值的数据流(pool 间接调用链)。0x920d7c 的3个 caller(0x920b3c/0xc4cb30/0xc4d22c)调它后,把结果传给高地址 runtime 函数(0xf216e4/0xf83fec/0xf8482c),再分发。
- 线索2:Dart 的 `runZonedGuarded` / `FlutterError.onError = ...` / `Isolate.current.addErrorListener` 注册点。这些在 libapp 启动代码(很早执行)。错误监听器的闭包就是弹故障窗的地方。
- 线索3:app 可能把"广告调用"包在 try-catch 里,catch 块构造故障body。但主控 NOP 了所有 showDialog 都没用,说明 catch 处理用的是【自定义弹窗组件】(非 showDialog)。找这个自定义弹窗组件。
- ★ 用你们已解的对象池(test3)+ Ghidra(test1)+ xref DB 精确定位全局错误处理闭包,给截断 patch。
- 风险:截断全局错误处理可能让所有真错误静默(掩盖 app 真实问题),需评估。但目标是"本地纯净运行",可接受。

### 方向 B:逐方法精确 V3(返回 Dart 端期望的类型)
V3 按 method 名返回该方法 Dart 端【期望的精确类型】。需反编译每个广告方法的 Dart 端回调解析代码,确定返回类型:
- 方法清单(主控 logcat 实证):showSplashAd / register / initAd / interstitial / showFullScreenVideoAd / loadFullAd / loadInsertAd / loadSplashScreenAd / MobileAds#initialize / showRewardVideoAd / rewardVideo
- 反编译目标:Pangle handler(C0984,test1 已有基础)、GDT、Ksad、AdMob 插件的【Dart 端 invokeMethod 回调解析】(即 `result as XxxType` 的地方),确定每个方法期望 bool/Map/自定义对象 及具体字段。
- 难点:自定义对象(如 `qsa`)在 Java 侧无法直接构造,需找可接受的替代(空对象?特定字段 Map?)。test1 的 Ghidra 反编译 + test3 的对象图脚手架可能帮上。
- test1 报告6曾建议"对 initAd/showSplashAd 短路 success(Boolean.TRUE)",但主控实测 showSplashAd 返回 bool 后下游崩(qsa?),请精确复核每个方法的真实期望。

### 方向 C:重新启用插件 + 改内部空操作
不注释 registerWith(避免 MissingPlugin),而是改每个广告插件的 onMethodCall 内部:跳过真实广告加载,直接返回【原插件格式的正确响应】。因为就是为该 channel 写的,类型天然正确。
- 工作量大(4 插件 × 多方法),但类型正确性最有保障。
- 风险:插件内部可能仍触发广告 SDK 初始化活动(即使不展示)。
- 需要:反编译每个插件的 onMethodCall,找到"真实广告加载"调用点,在它之前 return 正确格式的 success。

### 方向 D:Dart 侧 NOP 广告 invoke 调用
在广告方法的 async 入口,NOP 掉对 channel.invoke 的调用,让方法直接返回(不调用 channel = 不抛异常)。
- async 入口(主控 Dart 栈帧定位):showSplashAd→func 0x888b18,另两个→0x8880c0 / 0x88862c。
- 调用链示例:0x8863e8(开屏控制器,加载 showSplashAd)→ 0x888b18(showSplashAd async)→ invoke → MissingPlugin。
- 难点:async 状态机 NOP invoke 需保证栈平衡 + 返回合理值,否则状态机损坏。需精确定位 invoke BL 点。
