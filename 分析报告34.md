# 分析报告34

> 分支：`test1`  
> 角色：纯静态分析。  
> 本轮目标：对账 `test3` 第十轮新增发现，判断哪些结论成立、哪些仍需降级；同时回答“喵喵饿了到底还有没有更好的解决办法”。

## 1. 对 `test3` 第十轮的裁决

### 1.1 这次 `test3` 有三条结论是成立的

#### 1.1.1 `0x913bf8` 确实是配置/缓存加载器，而且字段名可以精确落地

本地字节复核 `0x913bf8` 后，可以把三条关键字段和对象池 key 对上：

```asm
0x00913ddc  ldr x16, [x16, #0xc70]
0x00913df8  ldr x0, [x0, #0x1028]
0x00913e00  str x0, [x1, #0x1028]
```

```asm
0x009141c0  ldr x16, [x16, #0xcf8]
0x009141dc  ldr x0, [x0, #0xed0]
0x009141e4  str x0, [x1, #0xed0]
```

```asm
0x00914274  ldr x16, [x16, #0xd10]
0x00914290  ldr x0, [x0, #0xef8]
0x00914298  str x0, [x1, #0xef8]
```

对象池 ref 对应：

- `ref=18152 "isFavChangePos"`
- `ref=8549 "autoCleanCacheType"`
- `ref=7606 "rewardTime"`

因此这一轮可以把字段名落得更实：

- `[G+0x1028] = isFavChangePos`
- `[G+0xed0] = autoCleanCacheType`
- `[G+0xef8] = rewardTime`

这直接说明：**`0xa52920` 读的 `0x1028/0xed0` 不是“今日已喂/是否隐藏喵块”字段。**

#### 1.1.2 `0xa52920 -> 0xa52bb0` 内部没有“当天隐藏”分支，这一点仍成立

本地复核 `0xa52920`：

```asm
0x00a52954  ldr x2, [x2, #0x1028]
...
0x00a5296c  tbnz w0, #4, #0xa529b4
...
0x00a52a98  ldr x2, [x2, #0xed0]
...
0x00a52ab0  tbnz w0, #4, #0xa52abc
0x00a52ab4  ldr x3, [x27, #0xbb8]
0x00a52abc  ldr x3, [x27, #0xbc0]
```

它只是在两套内容/外观之间切换，然后继续一路构造块，并没有出现“直接 return null / return 空 widget / 跳过构造”的隐藏门。

#### 1.1.3 reward 侧的新写入点 `0xa7fdb4` 是真的

本地复核：

```asm
0x00a7fdac  ldur x0, [x29, #-0x70]
0x00a7fdb0  ldr  x1, [x26, #0x80]
0x00a7fdb4  str  x0, [x1, #0xef8]
```

也就是说 `0xa7fcd4` 这条函数确实在写：

- `[G+0xef8] = rewardTime`

而且它自身字符串就是：

- `rewardTime`

这比此前“只知道 `0xa56c34` 会写 `0xee0/0xea0`”又多了一层 reward 时间戳链。

### 1.2 `test3` 这次也有两条需要降级

#### 1.2.1 “状态法完全走不通”这个结论还不能下死

当前能确认的是：

1. `0xa52920` 本身不读 `rewardTime`
2. `0xa52920` 读的 `0x1028/0xed0` 不是喂喵隐藏字段
3. 真正的隐藏门不在 builder 本体

但这只能推出：

- **builder 层状态法走不通**

还不能推出：

- **整个 app 的原生隐藏链静态上绝对无解**

因为用户的运行时事实已经证明“奖励完成后当天块消失”是真实存在的，只是读取判断发生在更上层调用者，也就是 reader build 墙后。

#### 1.2.2 `obj44988` 方案仍然是“值得试”，但不能宣传成已收敛

`test3` 推荐：

- `0xa52bb8` 覆写返回 `obj44988`
- 或 `0xa52b94` 单指令替换为 `ldr x0,[x27,#0x3a98]`

这仍然只是**最佳视觉根除候选**，不是已经证明的最终方案。  
它的价值在于：

1. `obj44988` 确实是 app 自己会返回的“空渲染 widget”
2. 比乱换别的 widget 更像同体系对象

但它仍需主控装机验证“是完全消失，还是留空白占位”。

## 2. 当前对 `喵喵饿了` 的最准判断

### 2.1 写入侧已经越来越清楚

现在能稳定确认三条写入：

1. `0xa56c34`
   - `0xa56c50: [G+0xee0] = true`
   - `0xa56ca4: [G+0xea0] = time-like`
2. `0xa7fcd4`
   - `0xa7fdb4: [G+0xef8] = rewardTime`
3. `0xb68234`
   - `0xb68288: object.field@0x33 = false`

其中 `0xb68234` 很像“猫从饿变饱”的对象内状态翻转：

```asm
0x00b68268  ldur w1, [x0, #0x33]
0x00b68270  tbnz w1, #4, #0xb68284
...
0x00b68284  add  x17, x22, #0x20
0x00b68288  stur w17, [x0, #0x33]
```

而 `0xa7fdb4` 则像“奖励完成时间戳”。

### 2.2 读取侧仍然卡在 reader build 墙后

现在反而更清楚为什么会卡：

1. `0xa52920` 不是总门
2. `0x846b1c / 0x846f30` 更像设置页 daily 入口门，不是阅读页块 caller
3. 喵块真正的“要不要挂载”判断，应该发生在比 `0xa52920` 更上层的 reader build
4. 这个 reader build 目前仍是 pool 间接调用墙

所以现在的真实局面是：

- **写入侧基本清楚**
- **读取/隐藏判断侧没有精确地址**

### 2.3 `0x846` 链不能再当成阅读页块总门

本地复核 `0x846ad0 -> 0x846b1c -> 0x846f30` 后，我认为它更像“设置页 `每日喂喵` 那个入口项的显示门”，而不是阅读页悬浮块总门。

原因：

1. 它强依赖 param 链上的对象字段 `#0x47`
2. 函数内部有大量双精度/比例计算与 item 构造
3. 没有看到任何与 `0xa52920 / 0xa52bb0` 的直接联系

因此它对“设置页和阅读页共享一套状态”是有帮助的，但不能再被抬成“直接隐藏悬浮块的 patch 点”。

## 3. 现在还有没有更好的办法解决喵块

### 3.1 有，但要区分“业务复原”和“用户可接受结果”

#### 路线 A：业务复原

目标：

- 让 app 自己走“今日已投喂，所以不再显示喵块”的原生逻辑

现状：

- 写入侧已逐渐清楚
- 读取判断侧没有精确分支地址

结论：

- **这条路线还没彻底死，但短期内很难靠纯静态拿到精确 patch**

#### 路线 B：用户可接受结果

目标：

- 让用户在阅读页看不到这个块

现状：

- 这条路线其实是可做的

最值得继续试的顺序，我现在建议是：

1. `obj44988` 返回方案
   - 优先级最高
   - 因为它是 app 自己的“空渲染 widget”
2. `Patch D`
   - `0xa52c68: 01 -> 16`
   - 必要时连 `0xa52c60` 一起 null
3. 最后才是继续盲猜状态门

### 3.2 为什么我现在更偏向 `obj44988`

因为 `test3` 第十轮最大的新价值不是“找到了总门”，而是：

1. 排除了 `0xa52920` 里那两项字段是隐藏状态的可能
2. 进一步证明状态门不在 builder 本体

这反而让“直接借 app 自己的空 widget 来替换块”变得更务实。

换句话说：

- **自然隐藏链没有完全找死，但短期最靠谱的解决方案，已经不是继续猜状态字段，而是让块直接渲染为空。**

## 5. 本轮结论

`test3` 第十轮这次不是白忙，它至少帮我们确认了一件非常关键的事：**`0xa52920` 读取的 `G+0x1028 / G+0xed0` 不是喂喵完成态，而是普通 app 设置项。** 这意味着继续从 builder 内部找“当日隐藏开关”基本没意义。与此同时，它新增的 `0xa7fdb4 -> [G+0xef8] = rewardTime` 也是真命中，说明 reward 完成写入侧又清楚了一层。

但更深的阅读页隐藏判断，仍然在 reader build 墙后。也就是说：**自然隐藏链没有被完全证伪，只是还没有被静态定位出来。** 所以如果你问“难道真的没有更好的办法解决这个喵喵饿了吗”，我的答案是：**有，当前最现实的更好办法不是继续硬猜状态门，而是优先把 `obj44988` 这个 app 自己的空 widget 方案拿去装机验证；如果它成功，就能在不碰 reader build 墙的前提下，把块真正做成不可见。**
