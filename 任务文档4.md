### 2.1 已知:广告插件注册入口

```
:try_start_N

  move-result-object v0
  new-instance v1, L<插件类>;
  invoke-direct {v1}, L<插件类>;-><init>()V

:try_end_N
.catch Ljava/lang/Exception; {:try_start_N .. :try_end_N} :catch_N
```

### 2.3 ★要产出什么

**对每个广告插件类**,解密其内部加密字符串,产出:
1. **SDK 归属确认**:它绑定的是哪个广告 SDK(AdMob/Pangle/Kwad/GDT/Umeng 中的哪个)
2. **MethodChannel channel 名**:它在 Flutter 侧注册的 channel 名(明文,如 "ppcat/admob" 之类)。这些插件用 hash-based 加密字符串(sparse-switch + xor 解密),需逆向解密算法
3. **初始化方法**:哪个方法调用 SDK 的 initialize/init/create(给出方法全签名 + 那行 invoke)
4. **回调接口**:初始化是否接收 OnInitializationCompleteListener / 其他回调,回调接口全名
5. **广告展示方法**:load/show/display/FullScreen/onAdShow 等方法的方法名 + 触发条件

### 2.4 加密字符串解密(关键)

这些 ส 混淆类的字符串用 **Arabic Unicode 范围(U+06D6-U+06EC)加密**,运行时通过解密函数还原。已知解密模式(来自 test2 第三轮):
- 字符串常量 → 经 `xor + hash dispatch` 解密
- 一个示例解密片段(AdMob 类 `สۥ۠ۦ/สۥ۟ۡ۟.smali` 的方法):
  ```
  const-string v0, "<arabic加密串>"
  invoke-virtual {v0}, String->hashCode()I
  xor-int/2addr v1, 0xe7
  xor-int/lit16 v1, v1, 0x1e7
  ... (一串 xor)
  sparse-switch v1, :sswitch_data_0
  ```
- 请逆向这个 sparse-switch + xor 解密算法,**离线解密出每个广告类的所有字符串常量**(channel 名、SDK 包名、方法名),输出明文表

参考:test2 第三轮分析报告4.4 节提到解密函数 `Lสۦ۠ۦۨ/สۥ۟۟ۡ;.ۥ้้۟۟ۡ()`,hash 值 198/235/541/993。请在 `main_dex/classes2.dex` 里定位这个解密类,反编译其算法。

### 2.5 ★★最终方案要求

基于解密结果,给出**最干净的去广告 patch 方案**,满足:
1. **去除所有广告展示**(开屏广告 + 应用内 banner/插页/激励)
2. **不触发故障窗**(广告失败检测)—— 这要求理解"广告失败→故障窗"的判定
3. **不影响其他功能**(只动广告插件,不碰 file_picker、connectivity 等正常插件)

候选方案(请评估哪个最干净):
- **方案 A**:在 registerWith 里,把这 5 个广告 try 块的 `new-instance + invoke-direct + invoke-interface` 三行改成 no-op(直接删,保留 try-catch 空块)。广告插件不注册 → Dart 层 channel 调用无人响应。**风险:app 可能检测广告初始化失败 → 弹故障窗**
- **方案 B**:保留广告插件注册,但在每个广告插件类内部,把 SDK initialize 调用短路(改成立即返回成功回调但不实际初始化)
- **方案 C**:mock 广告 SDK 的回调接口,让广告"加载/展示"永远报告成功但不实际展示

请基于分析,推荐方案 + 给出精确 patch(smali 哪个文件哪个方法改成什么)。

## 三、★任务 B:广告失败检测 → 故障窗触发逻辑(Dart 层)

### 3.1 现象
启动 60-90s 后弹"故障:无法正常联网,导致广告无法显示!..."(正文 ref=27672/27673),标题"故障"(ref=29112)。触发原因 = 广告加载失败。这是反广告屏蔽机制。

### 3.2 已知
- 故障正文 ref=27673 → pool slot 0x13c68 → .text 加载点 0xbbc31c(函数 0xbbc22c,588 字节,正文 Text 构建器)
- 0xbbc22c 有 **0 个直接 BL caller**,通过 Flutter vtable 间接调度
- test1 候选:故障窗上层编排器 = 0xbc0f40,条件分支 = 0xbc1020(机器码 `a0 12 00 54`,前 CMP W0,W22 @ 0xbc101c)。**未证实**
- 0x913bf8 是 22 杂字符串列表构建器,**勿动**(patch 会卡启动,前轮踩过坑)

### 3.3 ★要找的
基于核心任务产物或直接反汇编:
1. **广告失败检测函数**:Dart 侧哪个函数检测"广告加载是否失败"(可能是计时器 + 状态检查)。给出 .text 入口
2. **故障窗触发条件**:0xbc0f40(若确认是编排器)里的 `CMP W0,W22 @ 0xbc101c; B EQ 0xbc1020` —— 这个 W0 是什么状态?广告成功时 W0 取什么值才能跳过故障窗?
3. **showDialog 调用点**:故障窗实际弹出的 showDialog 调用 .text 地址
4. **patch 方案**:让故障窗永不弹的精确地址 + 字节(参照篡改窗手法)。两个子方案:
   - 把故障窗编排器入口改 return null(若所有出口返回 null)
   - 或把触发条件分支改为无条件跳过(W0 bit 设置 / 分支反转)

### 3.4 与任务 A 的协同
- 如果任务 A 用方案 A(不注册广告插件),广告必失败 → 故障窗可能必弹。必须同时 patch 故障窗(任务 B)
- 最优组合:任务 A 去广告展示 + 任务 B patch 故障窗不弹 = 完全无广告无故障窗
