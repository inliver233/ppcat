# 分析报告33

> 分支：`test1`  
> 角色：纯静态分析。  
> 本轮目标：围绕用户确认存在的“看广告/跳转成功后，当日隐藏 `每日喂喵/喵喵饿了`”真实业务路径，继续深追共享状态链；同时把 `VIP / 特权 / 机器码绑定` 从“页面混合状态机”里拆成更清晰的功能链。

## 1. 本轮结论

### 1.1 对 `分析报告32.md` 的两处关键纠偏

1. `0x913bf8 / 0x914190` 目前**只能**定性为一组 `SharedPreferences/缓存刷新/惰性初始化` 层。  
   这里确实唯一写回了：
   - `[G+0x1028]` at `0x913e00`
   - `[G+0xed0]` at `0x9141e4`
   - `[G+0xed8]` at `0x914220`
   - `[G+0xea0]` at `0x914310`
   但从函数内部模式看，它是“按 key 读取，再把结果刷新回同名缓存槽位”的通用层，不足以证明它在做 `0xee0 / 0xea0 -> 0x1028 / 0xed0 / 0xed8` 的直接翻译。

2. `[G+0xee0]` 不是单向“喂饱成功位”，而是**可变共享日状态位**。  
   本轮新增两处明确清零：
   - `0xc4d020: str x0, [x1, #0xee0]`，其中 `x0 = X22 + 0x20 = false`
   - `0xc4d284: str x0, [x1, #0xee0]`，其中 `x0 = X22 + 0x20 = false`
   所以它不能再被描述成“一旦成功就永久置 true”的位。

### 1.2 用户新增运行时事实，与当前静态证据是吻合的

用户确认：

- 设置页 `每日喂喵` 和阅读页 `喵喵饿了` 是同一套业务。
- 完成一次推荐/跳转奖励后，设置页显示 `喵喵喵~`，阅读页喵块也会一起消失。

当前最稳的静态解释是：

1. 两个 UI 入口确实共用同一条 reward/daily 后处理链。
2. 共用的是**业务状态传播**，不是同一个 widget builder。
3. 阅读页 `0xa52920 -> 0xa52bb0` 是消费层，不是根状态源。

### 1.3 `test3` 哪些结论可吸收，哪些需要降级

可直接吸收：

1. `0xa52920 -> 0xa52bb0` 的确是阅读页喵块消费层。
2. `0xa52a98` 读取 `[G+0xed0]`，只在 `0xbb8 / 0xbc0` 两套外观之间切换。
3. `0xa52bb0` 不是“当天是否显示”的根判定层。

需要降级：

1. 围绕 `0xa52bb0` 的 safe-hide 方案，不再是本轮主线。
2. `0x913e00` 那类“读自己再写自己”的片段，不能再被宣传成 `0xee0 -> 0x1028` 的同步翻译。
3. `VIP=true` 不能直接等价为“喂喵块自然消失”。

## 2. `每日喂喵 / 喵喵饿了` 共享业务状态链

### 2.1 共享入口桥：`0xbc5dc8 / 0xb68234 -> 0xa56e00`

对象池锚点：

- `ref=21707 "每日喂喵" -> slot 0x59bb`
- `ref=22466 "无法加载数据...每天喂一次喵..." -> slot 0x9912`
- `ref=28525 "喵喵饿了" -> slot 0x59bc`

本轮继续确认，两个最关键入口都汇入 `0xa56e00`：

1. 阅读页 reward 提示编排器 `0xbc5dc8`
   - 文案包含：`饿了喵~`、`是否看推荐信息获取喵粮喂喵？`
   - 调用点：`0xbc5e98: bl #0xa56e00`
2. 设置页 / 失败提示正文链 `0xb68234`
   - 唯一加载 `ref=22466`
   - 调用点：`0xb682e8: bl #0xa56e00`

另有两个共享近邻：

- `0xa56d94 -> 0xa56e00`
- `0xbc6148 -> 0xa56e00`

这说明“设置页每日喂喵”和“阅读页喵喵饿了”确实不是两套独立系统，而是两条 UI 前端接入同一条 reward/daily 后处理桥。

### 2.2 `0xa56e00` 的职责：共享桥，不是最终隐藏门

关键片段：

```asm
0x00a56e30  ldr x0, [x26, #0x80]
0x00a56e34  ldr x0, [x0, #0xb38]
...
0x00a56e64  ldr x1, [x26, #0x80]
0x00a56e68  ldr x1, [x1, #0x2538]
...
0x00a56e8c  ldr x1, [x1, #0xec8]
0x00a56ea0  bl  #0x87c0dc
...
0x00a56eb4  ldr x1, [x1, #0xed0]
0x00a56ec4  ldr x1, [x1, #0xed8]
0x00a56ee0  bl  #0x89281c
```

更稳的定性：

1. `0xa56e00` 先看 `[G+0xb38]`。
2. 再看 `[G+0x2538]`。
3. 之后分成：
   - `0xec8 -> 0x87c0dc`
   - `0xed0 + 0xed8 -> 0x89281c`

所以它是**共享桥接器**，负责把多个入口整理后送往统一后处理；它本身并不是“阅读页喵块是否挂载”的最终门。

### 2.3 成功写状态 sink：`0xa56c34`

最明确的成功入口仍然是：

- `0xa56be4`：唯一加载 `ref=26746 "喵喵喂饱了~"`
- `0xa56c14: bl #0xa56c34`

本轮补全了它的两组兄弟入口：

- `0xa56f60 -> 0xa56c34`
- `0xa57010 -> 0xa56c34`

这三条入口共用同一个 sink：

```asm
0x00a56c3c  add x0, x22, #0x30
0x00a56c4c  ldr x1, [x26, #0x80]
0x00a56c50  str x0, [x1, #0xee0]
...
0x00a56ca0  ldr x1, [x26, #0x80]
0x00a56ca4  str x0, [x1, #0xea0]
...
0x00a56cbc  ldr x16, [x16, #0xd20]
0x00a56cc8  bl  #0x9051cc
```

结论：

1. `0xa56c34` 会把 `[G+0xee0]` 写成 canonical `true`。
2. 它会把一个时间样值写入 `[G+0xea0]`。
3. 它随后通过 `[G+0xf90]` 和池常量 `0x13d20` 调 `0x9051cc` 做持久化/桥接。
4. 该池常量落到 `ref=23817 "favGridSortInt"`，因此这里只能定性为“共用持久化桥”，不能把 key 名直接当成喂喵语义名。

### 2.4 `0xa56c34` 周围还有一组回调/完成事件扇出

`0xa56c34` 之后，附近还有三组“从对象 field#0xf 取值，再经 `0xee0 / 0xef8 / 0xf10` 回调槽分发”的函数：

- `0xa56f20`
- `0xa56fd0`
- `0xa57060`

例如：

```asm
0x00a56f34  add x16, x27, #0x2c, lsl #12
0x00a56f38  add x16, x16, #0xee0
0x00a56f3c  ldp x5, x30, [x16]
0x00a56f40  blr x30
```

这说明 `0xa56c34` 不只是“写两个字段然后返回”的点，它后面还挂着一小簇完成事件分发。

### 2.5 `[G+0xee0]` 的清零路径已经明确存在

两处关键清零：

```asm
0x00c4d018  add x0, x22, #0x20
0x00c4d01c  ldr x1, [x26, #0x80]
0x00c4d020  str x0, [x1, #0xee0]
```

```asm
0x00c4d27c  add x0, x22, #0x20
0x00c4d280  ldr x1, [x26, #0x80]
0x00c4d284  str x0, [x1, #0xee0]
0x00c4d298  bl  #0x920d7c
```

其中第二处紧接着进入 `0x920d7c` 故障正文路径。  
因此 `[G+0xee0]` 更像“当前 daily/reward 状态位”，而不是不可逆成功锁。

### 2.6 `0x913bf8` 的正确位置：通用缓存刷新层

本轮对 `0x913bf8` 的定位更谨慎：

1. 它确实包含：
   - `rewardTime`
   - `remoteConfigSign`
   - `gitRuleMap`
   - 一大串 `isAuto... / isDark... / httpProxy ...`
2. 它确实唯一写了：
   - `[G+0x1028]`
   - `[G+0xed0]`
   - `[G+0xed8]`
   - `[G+0xea0]`
3. 但其内部模式是：
   - 先看 `[G+0xf90]`
   - 按 key 调 `0x7f89d8` 一类 accessor
   - 如果结果为 null，再回退到“该槽位自己的旧值”
   - 然后写回同一个缓存槽

所以它更像：

- `SharedPreferences / 缓存刷新 / 惰性初始化` 总管

而不是：

- `0xee0 / 0xea0 -> 0x1028 / 0xed0 / 0xed8` 的专用翻译器

### 2.7 阅读页当前能确认的只是消费层：`0xa52920`

关键片段：

```asm
0x00a52954  ldr x2, [x2, #0x1028]
...
0x00a52a98  ldr x2, [x2, #0xed0]
0x00a52ab4  ldr x3, [x27, #0xbb8]
0x00a52abc  ldr x3, [x27, #0xbc0]
```

它说明：

1. 阅读页喵块确实消费 `[G+0x1028]` 与 `[G+0xed0]`。
2. `[G+0xed0]` 目前只看到外观二选一：`0xbb8 / 0xbc0`。
3. 这与 test3 的观察一致：`0xa52920 -> 0xa52bb0` 不是“今天是否完全不显示”的根门。

### 2.8 `0x846b1c -> 0x846f30` 仍然是最可信的“设置页上层显示门”

关键片段：

```asm
0x00846b3c  bl   #0x846f30
0x00846b44  tbnz w0, #4, #0x846b58
0x00846b48  mov  x0, x22
0x00846b4c  ... ret
```

即：

1. 若 `0x846f30` 返回 false，`0x846b1c` 直接 `return null`。
2. 这很像“当天不再显示设置页 `每日喂喵` 入口”的上层门。

但 `0x846f30` 本身不是简单字符串比较，而是在看某个嵌套对象的字段：

```asm
0x00846f5c  ldur w2, [x1, #0x47]
...
0x00846f70  add  x16, x27, #0xb, lsl #12
0x00846f74  ldr  x16, [x16, #0x5c0]
0x00846f84  csel x0, x16, x17, eq
```

因此更安全的表述是：

- `0x846b1c -> 0x846f30` 是当前最强的“设置页 daily 显示门”候选
- 但它不等于“阅读页喵块的最终总门”

### 2.9 本轮新增的高层聚合器：`0x59d8d8 / 0x59d964 -> 0x8cf36c`

这是本轮比前几轮更值得继续追的点。

关键片段：

```asm
0x0059d964  ldr x16, [x16, #0x2b8]   ; getRewardTime
0x0059d978  bl  #0xc833ac
0x0059d980  tbnz w0, #4, #0x59d9b4
...
0x0059d994  ldr x4, [x27, #0x2bb0]
0x0059d998  bl  #0x8cf36c
0x0059d9ac  stur w1, [x0, #0x13]
```

结合对象池：

- `ref=7606 "rewardTime" -> slot 0x27a2`
- `ref=8067 "getRewardTime" -> slot 0x2a57`

这说明：

1. 这里存在一条更上层的 `getRewardTime` 聚合读取链。
2. 若条件满足，它会调 `0x8cf36c`，再把结果写回某个对象字段 `#0x13`。
3. 若条件不满足，则走另一条 fallback 路线（后续靠 `[G+0x10b0]` 一类对象继续）。

这比继续盯 `0xa52bb0` 更接近用户描述的“看完一次后整套 UI 都消失”的真实业务逻辑。

## 3. `VIP / 特权 / 机器码绑定` 功能链

### 3.1 先给总判断：它不是一个单点 `isVip()` 布尔

当前最稳的拆法是四层：

1. **绑定/身份层**：`[G+0x1108]`、`0x871320`
2. **共享门控层**：`0x911788`
3. **奖励/特权状态层**：`0x8cf36c`
4. **到期/日期/跳过计数层**：`0x8ba3d0`、`0x67994c`

页面级大函数 `0xb9fc84` 只是把这四层揉到了一起，它不是单一 `isVip()`。

### 3.2 `0xb9fc84`：绑定 + daily + reward + 页面显示 的混合状态机

其直接 caller 为：

- `0xb9fafc -> 0xb9fc84`

它的字符串同时包含：

- `isDaily:`
- `未获取`
- `未登录`
- `喵喵喵~`
- `饿了喵~`
- `通过捐赠绑定设备获取`
- `功能，通过捐赠绑定账号获取`
- `nextTime`
- `skipCount`

前半段关键流程：

```asm
0x00b9fcb4  bl  #0x911788
...
0x00b9fce4  ldr x0, [x26, #0x80]
0x00b9fce8  ldr x0, [x0, #0x1108]
...
0x00b9fe74  bl  #0x8cf36c
```

可确认：

1. 它先过 `0x911788` 共享门控。
2. 然后大量读取 `[G+0x1108]` 对象并做虚调用。
3. 再调用 `0x8cf36c` 奖励/特权状态对象。
4. 在 `0xba0060..0xba0080` 处二选一绑定文案：
   - `0xba0070`：`功能，通过捐赠绑定账号获取`
   - `0xba007c`：`通过捐赠绑定设备获取`

所以 `0xb9fc84` 最多只能叫：

- `VIP / 绑定 / daily / reward 页面混合状态机`

不能直接叫：

- `isVip()`

### 3.3 `0x911788`：共享门控，而不是纯 VIP 判断

关键字符串：

- `remoteConfigSign`
- `Main`

关键事实：

1. 它的 direct callers 非常多：
   - `0xb9fc84`
   - `0x920d7c`
   - `0x7e8534`
   - `0xa58068`
   - `0xbca240`
   - 等等
2. 它也会读 `[G+0x1108]`。
3. 它还会读 `[G+0x2550]` 对应 `remoteConfigSign` 路径。

因此更合理的定位是：

- `共享业务门控`

而不是：

- `VIP 专用门`

### 3.4 `0x8cf36c`：奖励/特权剩余时间对象

本轮继续压实它的语义：

字符串：

- `adNum`
- `lastTime`
- `today`
- `totalNum`
- `totalTime`
- `已过期！`
- `已生效！\\n剩余`
- `秒`

并且 callers 覆盖：

- `0x59d8d8`
- `0x7e8054`
- `0x7e8534`
- `0xa54178`
- `0xb9fc84`
- 等

所以它更像：

- `奖励特权 / 剩余时效 / 今日累计` 对象

而不是：

- “付费 VIP 总开关”

这也解释了为什么历史上单独去改 `0x8cf36c` 并不能自然消掉阅读页喵块：它只是共享状态的一层，不是整条隐藏链的唯一根。

### 3.5 `0x871320`：设备/账号绑定 URL 构造器

这是当前最强的“机器码绑定”锚点。

它包含的常量：

- `https://ppcat.gentle.com/?u=`
- `https://ppcat.gentle.com/?i=`
- `deviceInfo`
- `machine`
- `https://ia.51.la/go1?id=21260551&rt=`
- `&rl=`

函数内部可见：

1. 在 `?u=` 与 `?i=` 两条 URL 路线间分支。
2. 明确把 `machine`、`deviceInfo` 一类字段装入对象。
3. 同时消费：
   - `[G+0xb38]`
   - `[G+0x2540]`
   - `[G+0x2548]`
   - `[G+0x2558]`
   - `[G+0x2570]`

因此它不是普通页面文案函数，而是一条真实的：

- `设备/账号标识 -> gentle.com URL -> 远端绑定/统计`

路径。

### 3.6 `0x8ba3d0` 与 `0x67994c`：到期/日期/跳过计数层

对象池锚点：

- `ref=7406 "expiresDate" -> slot 0x2605 -> func 0x8ba3d0`
- `ref=12056 "expires" -> slot 0x3b12 -> func 0x67994c`

`0x67994c` 同时出现：

- `date`
- `expires`
- `skipCount`

因此这两条更像：

- 到期时间读取/格式化
- 跳过次数 / 日期窗口

而不是页面本身。

## 4. 交叉验证与分歧说明

### 4.1 与 `test3` 一致的部分

1. `0xa52920 -> 0xa52bb0` 是阅读页消费层，不是共享业务根层。
2. 用户所说“设置页与阅读页是同一套 daily 子系统”是对的。
3. 单纯继续做 `0xa52bb0` safe-hide，已经不是最高杠杆方向。

### 4.2 对 `test3` / 历史报告的修正

1. `0x913e00` 这类“`ldr [G+0x1028]; str [G+0x1028]`”片段，更像缓存刷新，不应宣传成字段翻译。
2. `0xee0` 有明确 false 写回，不应再写成永久成功位。
3. `VIP/特权` 不能再浓缩成一个 `isVip` 分支点，至少要拆成：
   - 绑定身份
   - 共享门控
   - 奖励特权
   - 到期/skipCount

## 5. 面向代码恢复的方案建议

### 5.1 `每日喂喵 / 喵喵饿了` 主线建议

如果目标是“恢复自然业务链”，而不是单纯视觉隐藏，优先级应是：

1. `0xa56e00`
   - 共享入口桥
2. `0xa56c34`
   - 成功态写入 sink
3. `0x59d8d8 / 0x59d964 -> 0x8cf36c`
   - 更上层 reward/daily 聚合链
4. `0x846b1c -> 0x846f30`
   - 设置页 daily 上层显示门
5. `0xa52920`
   - 阅读页消费层

具体建议：

1. 不再把主精力放在 `0xa52bb0` 本体。
2. 继续追 `0x59d964 -> 0xc833ac -> 0x8cf36c` 这条聚合链，看它如何把“今日已完成/剩余时间/奖励次数”写回上层对象。
3. 继续追 `0xc4d020 / 0xc4d284` 的上游 caller，搞清 `[G+0xee0]` 何时、为何被清零。
4. 若只想恢复“当天投喂后两处入口一起消失”的真实业务，应优先模拟 `0xa56c34` 成功态，而不是继续硬抠喵块 builder。

### 5.2 `VIP / 机器码绑定` 主线建议

若目标是恢复 `特权 / 绑定 / 到期` 相关功能，不建议继续从 `0xb9fc84` 整页入口硬改。

更稳的拆法：

1. **绑定层**
   - `0x871320`
   - `[G+0x1108]`
2. **共享门控层**
   - `0x911788`
3. **奖励/时效层**
   - `0x8cf36c`
   - `0x59d964`
4. **到期/skipCount 层**
   - `0x8ba3d0`
   - `0x67994c`

恢复路线建议：

1. 若目标是“本地特权可用”，优先看 `0x8cf36c / 0x59d964 / 0x8ba3d0 / 0x67994c`。
2. 若目标是“设备/账号绑定链恢复”，优先看 `0x871320 + [G+0x1108]`。
3. 不建议把 `0xb9fc84` 直接当 `isVip()` 去做粗暴恒真处理，因为它同时还掺着 `isDaily`、登录、绑定、更新、故障提示等多路状态。
