### 1.2 Dart AOT 调用约定(已确认)
- X15 = shadow stack 指针;X26 = Thread 寄存器;X27 = Pool 指针(PP);X22 = null sentinel;X28 = heap base(compressed pointer 高位)
- 函数 prologue:`STP X29,X30,[X15,#-16]!`,随后 stack overflow check
- 对象池访问模式:`ADD Xn,X27,#page,LSL#12; LDR Xm,[Xn,#off]` 或 `LDR Xm,[X27,#off]`

### 1.3 已验证的 ref→对象池 slot→.text 锚点(★用于校验你的反序列化是否正确)

通过"在 snapshot data 区搜 VLE 编码的 ref"已确认(test2 方法):

| ref | 内容 | VLE 编码 | snapshot 内 byte 偏移 | pool idx | slot offset | .text 加载点 |
|---|---|---|---|---|---|---|
| 26842 | 短版正文2 | `5a 51 81` | 0x2eb53d | 5538 | 0xad88 | (多处) |
| 30947 | "非法篡改"标题 | `63 71 81` | 0x2eb541 | 5539 | 0xad90 | (多处) |
| 30922 | 长版正文 | `4a 71 81` | 0x2f0416 | 11819 | 0x17168 | 0x8e52d0, 0x8ef414 |
| 27673 | 故障窗正文 | `19 58 81` | 0x2eeecb | ~10059 | 0x13c68 | 0x913d3c, **0xbbc31c** |

**VLE 编码规则(UNSIGNED,已验证)**:首字节 b0,若 bit7=1 则单字节值=b0-128;否则多字节,r=b0(shift0),循环 r|=b<<shift;shift+=7,遇 bit7=1 字节终止 r|=(b-128)<<shift。

**slot offset = (pool_idx + 2) × 8**。但注意:type 编码未完全破解(0x80=对象引用,0x81=8字节double,其他待定),byte→idx 非严格线性,插值有误差。**从 .text 指令直接读 slot offset 是精确的。**

## 二、★核心任务:Dart 对象池完整反序列化(master key)

**这是本轮最高优先级。完成它,下面所有应用任务都能基于它的产物解决。**

### 2.1 问题
blutter 因定制 VM(cid 84 / 基对象计数偏差)无法反序列化 snapshot。但 **unflutter 已能解析 snapshot**(产出 ref→string 表),说明 unflutter 内部已建立对象图。前两轮没把"对象池 slot → Code 对象 → .text 入口 → 函数名"这条链完整提取出来。

### 2.2 要产出什么
一张完整的映射表(文本或 json),至少包含:

```
Code 对象(函数):
  pool_slot / ref  →  .text 入口地址  →  Code 注释里的函数名(若有,如 "checkAppMetaData::")
  →  函数大小 / BL 调用数 / 被谁调用(caller 的 .text 地址)
```

也就是:把对象池里的每一个 **Code 对象**反序列化,拿到它对应的 .text 指令入口地址(Instructions 指针),以及它的名字(对象池里 Code 注释 `xxx::\n`)。

### 2.3 校验(必须通过)
你的产物里,以下必须自洽:
- `checkAppMetaData::` 的 Code 对象 → .text 入口应能解释为何紧邻 ref=30922(长版正文)。前两轮推断其入口在 0x8e1dd0 附近,请用反序列化**确认或纠正**。
- ref=30922(长版正文)在对象池的 slot = 0x17168,Code 对象引用它的 .text = 0x8e52d0 / 0x8ef414,应落在某个 Code 对象的范围内。
- 7 个 check* 的 .text 入口(checkAppMetaData / checkAppComponentFactory / checkApplication / checkJiagu / checkPMProxy / checkApk / checkIO)应能给出确切地址。

### 2.4 若 unflutter 也无法完整反序列化
请尝试:
- 基于 `snapshot.json` 的 CID 表 + 对象池格式(OldPoolFormat),写 Python 解析器,**重点处理 type 字节 0x80/0x81/0x00 等的不同长度**,使 byte→idx 精确(消除插值误差)。
- 或用 dex2jar 之外的方式:直接对 isolate snapshot data 做正向反序列化(cluster by cluster),输出每个 Code 对象的 {name, pool_slot, instructions_offset}。

## 三、应用任务 A:故障窗 showDialog 编排器定位

### 现象
启动约 60-90s 后弹"故障:无法正常联网,导致广告无法显示!..."(正文 ref=27673),标题"故障"(ref=29112),按钮"确定"。触发原因是广告加载失败(反广告屏蔽机制)。点"确定"可关,但希望让它不弹。

### 已知
- 故障正文 ref=27673 → slot 0x13c68 → .text 加载点 **0xbbc31c**(函数 0xbbc22c,588 字节,故障正文 Text 构建器)
- 故障标题"故障" ref=29112 → slot 未精确定位(VLE byte 0x2ece06,插值 slot~0xeb10)
- 与篡改弹窗不同:故障窗的正文通过**独立的 Text 构建器**(0xbbc22c)加载,showDialog 编排器在别处调用它

### 要找的
基于核心任务的产物(Code→.text→caller 映射):
1. 函数 **0xbbc22c**(故障正文构建器)的 **caller**(谁 BL/BLR 调用它)——这就是故障窗 builder
2. 故障窗 builder 里的 **showDialog 调用点**,以及**触发它的条件分支**(广告是否失败的判定)
3. 给出"让故障窗不弹"的精确 patch 地址 + 字节(参照篡改窗的成功手法:若 builder 所有出口都返回 null,可把入口改 `e0 03 16 aa c0 03 5f d6`;若是条件分支,改条件跳转)

### 注意
- 不要 patch 0x913bf8(它是列表构建器,patch 会卡启动)
- 故障窗根因是广告失败检测。如果能从广告侧让它"报告成功"(见任务 B),故障窗自然不弹,可能比 patch builder 更干净。两种方案都请给出。

## 五、应用任务 C:导入书源功能报错(★用户最关注)

### 现象
在 app 里"导入书源/源"时,报错:
```
程序遇到错误:No field 了 of type Ljava/lang/String; in class L了/了; or its superclasses
(declaration of '了了了' appears in base.apk!classes2.dex)
```
然后整个 app 未响应(ANR)卡死。

### 已知
- 报错来自 **classes2.dex**,类 `L了/了;`,字段 `了`(String 类型),声明 `了了了`
- `了`(U+4E86)在 classes2.dex 出现 9 次,classes3.dex 1 次,classes.dex 0 次
- apktool **未能**把 `了/了` 类反编译为 smali(所以 smali 树里没有),需用 jadx/dexdump 直接读 dex
- Dart 侧有导入相关通道方法名:`bookSource`、`import`、`importall`、`importonline`、`subscribe`(对象池已确认)
- 推断流程:Dart(import bookSource)→ Flutter 平台通道 → Java ส 通道处理器 → 反射访问 `了/了` 类字段 `了` → NoSuchFieldError → ANR

### 要找的
1. 用 jadx/dexdump 分析 `classes2.dex`,找到类 `L了/了;`:它有哪些字段?是否有字段 `了`(String)?为什么反射找不到(是字段被重命名了?类是多 dex 跨引用?还是类被混淆/抽取了?)
2. 找到执行反射 `getField("了")` / `getDeclaredField("了")` 的代码(哪个类、哪个方法,在导入流程里)
3. 判断这是**原版就有的 bug**,还是**重打包/脱壳/混淆引入的**(对照:原版能否正常导入?字段名是否在不同 dex 间不一致?)
4. 给出**修复方案**:让导入书源功能恢复正常。可能的方向:
   - 补回缺失的字段(在 `了/了` 类里)
   - 或修正反射访问的字段名
   - 或定位是哪个 dex 的类链接出错并修复

### 特别要求
这是用户当前最影响使用的功能,请**重点投入**,给出可执行的修复步骤(精确到改哪个 dex/类的哪个方法/字段)。
