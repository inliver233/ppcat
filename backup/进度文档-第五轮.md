## 2. libapp.so 基本信息（定制 Dart 引擎）

- 文件：`lib/arm64-v8a/libapp.so`，16352152 字节，**vaddr == file offset**
- ELF：`.text@0x460000`；`_kDartIsolateSnapshotData@0x3330`(size 0x45c930)；`_kDartIsolateSnapshotInstructions@0x464ce0`
- 定制 Dart 2.19 引擎：compressed-pointers, no-null-safety, use_bare_instructions, snapshot_hash 抹零
- **blutter 无法反序列化**（cid 84 / 基对象计数偏差），unflutter 能解析 snapshot 产 ref→string 表
- Dart AOT 调用约定：X15=shadow stack, X26=Thread, X27=Pool(PP), X22=null sentinel, X28=heap base
- 函数 prologue：`STP X29,X30,[X15,#-16]!`；入口后 stack overflow check

## 3. ★篡改弹窗 patch（已验证成功，核心成果）

### 弹窗特征
- 启动约 7s 后弹"应用已被非法篡改或被第三方APP禁用某些模块..."（长版正文）
- Flutter showDialog（uiautomator 抓 content-desc），标题"非法篡改"，按钮"下载"
- **阻断主界面**，必须破

### 文案 ref（对象池）
| 内容 | ref | 类型 | char data 偏移 |
|---|---|---|---|
| 长版正文 | 30922 | TwoByteString | 0x1c326e |
| 短版正文1 | 6917 | TwoByteString | 0x4d17a |
| 短版正文2 | 26842 | TwoByteString | (0xad88 slot) |
| 标题"非法篡改" | 30947 | TwoByteString | (0xad90 slot) |

### pool slot（ref→slot，用 VLE 法定位，见 §6）
- ref=30922(长版) → **slot 0x17168**（pool idx 11819）
- ref=26842(短版2) → slot 0xad88（idx 5538）
- ref=30947(标题) → slot 0xad90（idx 5539，与 26842 相邻）

### .text 引用点（ref=30922 长版正文的加载点）
| PC | 所在函数 | 说明 |
|---|---|---|
| **0x8e52d0** | **0x8e1dd0**(15KB) | 主 dialog 编排器（加载标题0xad90+短版0xad88+长版0x17168+showDialog，返回null） |
| 0x8ef414 | 0x8ef2b8(1.8KB) | 另一 dialog 构建器（同型，加载长版+0x970+0x15258） |

### ★patch 方法（关键！前几轮失败的原因）
**直接把函数入口改成 `MOV X0, X22; RET X30`（立即返回 null）。**

为什么有效（而前几轮的 patch 失败）：
- 前几轮 patch 的是函数**内部的"长/短版选择"分支**（TBNZ W0,#4），但那些分支跳转后**又继续构建弹窗**（0x8e53a8 不是真正跳过，是另一个构建分支）。所以无效。
- 0x8e1dd0 / 0x8ef2b8 是"弹窗编排器"——它们的**所有出口都返回 null（MOV X0,X22; RET）**，弹窗是通过内部副作用（showDialog）显示的。
- 把入口改成 `MOV X0,X22; RET`：保留正常返回值（null），跳过整个构建+显示副作用。调用者完全不感知，不会崩。
- X15(shadow stack) 不变（入口和返回的 X15 相同），调用约定平衡，安全。

### patch 字节
| 地址(file offset=vaddr) | 原始字节 | 改为 | 含义 |
|---|---|---|---|
| **0x8e1dd0** | `fd 79 bf a9 fd 03 0f aa` | `e0 03 16 aa c0 03 5f d6` | MOV X0,X22; RET |
| **0x8ef2b8** | `fd 79 bf a9 fd 03 0f aa` | `e0 03 16 aa c0 03 5f d6` | MOV X0,X22; RET |

`e0 03 16 aa` = MOV X0, X22；`c0 03 5f d6` = RET X30

### 验证
- 装机后启动，篡改弹窗**不再出现**
- app 正常进入主界面，进程存活，稳定运行无崩溃
- 备份：`libapp/libapp_arm64_dialogpatched.so`（已含上述 patch）

### slot 与加载点
- ref=27673(故障正文) → **slot 0x13c68**（pool idx ~10059）
- slot 0x13c68 的 .text 加载点：
  - 0x913d3c（在函数 0x913bf8，加载 22 个连续杂字符串 slot 的列表构建器，**非故障窗函数**）
  - **0xbbc31c（在函数 0xbbc22c，588 字节，故障正文 Text widget 构建器）**
- "故障"标题 ref=29112 → slot ~0xeb10 区（插值，精确 slot 未定）

### ★重要教训（避免重蹈覆辙）
1. **0x913bf8 是误判**：它加载 22 个连续 slot（0x13c58-0x13d00），ref=27673 只是其中一项。检查相邻 ref（27670-27685）全是无关杂字符串（language/本地化/ICU），证明 0x913bf8 是资源/列表构建器，不是故障窗函数。patch 它入口 return null → 破坏启动流程 → 卡开屏。**已回退。**
2. **"加载文案的函数 ≠ 显示文案的函数"**：篡改窗 builder 直接加载 slot（0x8e1dd0），但故障窗架构不同——正文通过独立的 Text 构建器（0xbbc22c）加载，showDialog 编排器在别处调用它。不能只凭"加载 ref=27673"定位 showDialog。
3. **pool idx 插值有误差**：因 type 编码不一（见 §6），byte→idx 非严格线性，插值 slot 误差可达几百。0x13c68 是精确值（从 .text 指令直接读）。

- 找故障窗 builder（showDialog 编排器）：需要 0xbbc22c 的 Code 对象的 pool slot → 调用者。需正确解析 pool（见 §6）。
- 或：让广告 SDK"报告成功"（Java 层 สۥ 广告管理类），骗过广告失败检测，故障窗不弹（见 §5）。

## 6. ★Dart 对象池 VLE 解析方法（可复用于任何 ref→slot 定位）

### VLE 编码（test2 方法，已验证）
`read_unsigned`（Dart snapshot 的 UNSIGNED VLE）：
```
读首字节 b0：
  若 b0 > 127 (bit7=1)：单字节，值 = b0 - 128
  若 b0 <= 127：多字节。r = b0 (shift 0)，循环读 b：
     r |= b << shift; shift += 7
     若 b > 127：r |= (b-128) << shift; 返回 r
```
**注意**：多字节的最后一字节 bit7=1（终止），中间字节 bit7=0。和标准 LEB128 的 continuation 位"相反"。

### ref → VLE bytes → pool slot 步骤
1. 算 ref 的 VLE 编码（见编码规则）
2. 在 isolate snapshot data 区（0x3330-0x45fc60）搜该 VLE bytes，找唯一匹配的 byte 偏移
3. 该 byte 偏移对应一个 pool slot；slot offset = (idx+2)*8，idx = 该 slot 在 pool 序列里的序号

### 锚点（ref → byte → slot，已 .text 验证为真）
| ref | VLE | byte 偏移 | pool idx | slot offset |
|---|---|---|---|---|
| 26842 | 5a 51 81 | 0x2eb53d | 5538 | 0xad88 |
| 30947 | 63 71 81 | 0x2eb541 | 5539 | 0xad90 |
| 30922 | 4a 71 81 | 0x2f0416 | 11819 | 0x17168 |
| 27673 | 19 58 81 | 0x2eeecb | ~10059 | 0x13c68(从.text) |

### pool entry 结构（每个 slot 序列化）
- 绝大多数：`[type byte 0x80][VLE value]`（0x80 = 对象引用）
- type=0x81：8 字节 double
- type<0x80：可能是单字节 Smi（待完全确认）
- **type 编码未完全破解**，导致 byte→idx 非严格线性，插值有误差。从 .text 指令读 slot offset 是精确的。

### .text → slot 映射（现成产物）
`analysis/t2_pool_accesses.txt`（122,780 条）：格式 `pc  pool_offset  full_line`，所有 `LDR Xn,[X27,#off]` 和 `ADD Xn,X27,#page,LSL#12; LDR` 模式的池访问。

### 完整反汇编
`asm.txt`（127MB，解压自 test1 的 unflutter_dump/asm.txt.gz）：格式 `0xADDR  bytes  MNEMONIC operands`，覆盖全部 .text。
