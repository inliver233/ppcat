## 2. 新主收获：首页 overlay/banner 的最佳挂点已经收敛到 `0xbd2e1c -> 0xbd3540`

这一轮最有价值的新进展，不是再回头抠 `0xae30e0`，而是吸收 `test3` 的首页链结果后，再用我本地 capstone 复核，确认：

- `0xbd2e1c` 是首页分发器
- `0xbd3540` 是首页里带 `立即获取 + admobBannerFurtureView` 的那条子树
- `0xbd3084` 是它的同级替代子树，不带这两个广告/推广锚点

### 2.1 本地复核到的关键分支

`0xbd2e1c` 内部关键指令如下：

```asm
0x00bd2e50: a2 00 20 37    tbnz w2, #4, #0xbd2e64
0x00bd2e58: ba 01 00 94    bl   #0xbd3540
0x00bd2e68: 87 00 00 94    bl   #0xbd3084
```

这说明：

1. `0xbd2e1c` 的这个位置，本来就在两条首页子树之间二选一。
2. `0xbd3540` 与 `0xbd3084` 在这里是 ABI 对等的兄弟分支，调用后在 `0xbd2e70` 汇合。
3. 因此，“绕开 `0xbd3540`，改走 `0xbd3084`”是当前最像样的首页层 patch 思路。

### 2.2 交叉验证 `0xbd3540` 的字符串和角色

`test3 xref_db` 与我本地复核一致：

- `0xbd3540` 字符串：
  - `立即获取`
  - `admobBannerFurtureView`
  - `商店版`
  - `站点类型`
  - `站点名称`
  - `站点分组`
  - `全能的自定义检索阅读工具`
- caller 只有：
  - `0xbd2e1c`

相对地，`0xbd3084` 的字符串只有：

- `商店版`
- `极速版`
- `站点名称`
- `站点分组`
- `全能的自定义检索阅读工具`

不含：

- `立即获取`
- `admobBannerFurtureView`

这使得 `0xbd3540` 成为当前最精确的“首页 overlay/banner 子树”静态挂点。

### 2.3 `0xbd3540` 与喵块数据簇的关系

我本地复核到 `0xbd3540` 内部还有这条访问：

```asm
0x00bd4188: 00 90 46 f9    ldr x0, [x0, #0xd20]
```

对应 `pool offset = 0x2cd20 = slot 0x59a4`。  
`test3` 的对象池结果显示：

- `slot 0x59a4 -> ref 121017`

它位于和以下内容同一 `0x59xx` 数据簇：

- `0x59b2` -> `24小时内累计浏览`
- `0x59b7` -> `已生效`
- `0x59bb` -> `每日喂喵`
- `0x59bc` -> `喵喵饿了`
- `0x59bf` -> `训练喵喵`
- `0x59c2` -> `分享喵喵`

这还不足以证明“首页 overlay 和阅读页喵块一定同源”，但足以说明：

- `0xbd3540` 不是一个普通的首页文本块
- 它确实碰到了和 `VIP / 喵 / 特权` 同一簇的对象池区域

## 3. 首页 overlay/banner 的可执行方案

### 3.1 当前最值得主控试的实验项

我更推荐以下 patch，而不是直接 NOP `0xbd2e58`：

#### 方案 H1：把 `0xbd2e58` 的调用目标从 `0xbd3540` 改成 `0xbd3084`

- 地址：`0xbd2e58`
- 原字节：`ba 01 00 94`
- 改字节：`8b 00 00 94`
- 含义：把 `bl #0xbd3540` 改成 `bl #0xbd3084`

理由：

1. `0xbd3084` 本来就是这个分发器的另一条兄弟子树。
2. 这样比直接 NOP `BL` 更稳，因为返回对象的形状仍来自一个真实 builder。
3. 它保留了 `0xbd2e1c` 的整体框架，只是把“带 overlay/banner 的子树”替换成“普通首页子树”。

#### 方案 H2：强制首页分发器永远走 `0xbd3084` 那条现成分支

- 地址：`0xbd2e50`
- 原字节：`a2 00 20 37`
- 改字节：`05 00 00 14`
- 含义：把 `tbnz w2,#4,#0xbd2e64` 改成 `b #0xbd2e64`

这和 H1 的目标一致，只是改在分支层而不是改 `BL` 目标。

### 3.2 我对 H1 / H2 的分级

这两条都属于：

- `静态高置信`
- `需要主控 patch-and-test`

而不是“已运行时验证”。

我当前更偏向：

1. 先试 `H1`
2. 如果 `H1` 异常，再试 `H2`

原因：

- `H1` 更局部
- `H2` 改的是上层分支条件，虽然也合理，但影响面稍宽

### 3.3 不推荐的首页 patch

以下不建议直接升格：

1. NOP `0xbd2e58`
2. `0xbd2e1c` 入口 return null
3. `0xbd3540` 入口 return null

原因是：

- `0xbd2e1c` / `0xbd3540` 都是首页对象构造链的一部分
- 直接清空返回，风险比“改走同级兄弟子树”大得多

## 4. VIP / 特权链继续收敛：`0x8cf36c` 是共享状态，`0xbca170 -> 0xbca240` 是共享前置链

### 4.1 `0x8cf36c` 仍是最核心的共享特权状态对象

`test3` 和我本地复核一致，`0x8cf36c` 的字符串包括：

- `adNum`
- `lastTime`
- `today`
- `totalNum`
- `totalTime`
- `leaveTime:`
- `已过期！`
- `已生效！\n剩余`
- `秒`

caller 包括：

- `0x7e8534`
- `0xa54178`
- `0xb3ff04`
- `0xb9fc84`
- `0x89afa0`

这说明它不是“某个页面自己的小开关”，而是：

- reward 特权
- 首页剩余状态
- VIP 页
- 某些阅读/每日喂喵链路

共享消费的 canonical 状态对象。

### 4.2 `0x7e8534` 仍是最窄、最像消费侧总闸的点

我本地复核：

```asm
0x007e8618: bl   #0x8cf36c
0x007e861c: c0 06 20 36    tbz w0, #4, #0x7e86f4
```

所以：

- `0x7e861c: c0062036 -> 32000014`

仍然是当前最稳的共享特权门 patch。

### 4.3 新确认的共享前置链：`0xbca170 -> 0x8a0fd4 -> 0xbca240`

这一轮我把 `test3` 的补链结论又本地反汇编了一遍，确认：

```asm
0x00bca1a0: bl   #0x8a0fd4
0x00bca1a4: a0 00 20 37    tbnz w0, #4, #0xbca1b8

0x00bca1b8: bl   #0xbca240
0x00bca1bc: a0 00 20 37    tbnz w0, #4, #0xbca1d0
```

并且 `0xbca240` 内部不是简单 bool，而是：

1. 先走 `0x911788`
2. 再走 `0xbca42c`
3. 再读共享状态
4. 中途还会在一条路径上把 `x0` 设成 `5`
5. 后面继续比较计数/阈值并调用 `0x7e41c0`

也就是说：

- `0xbca240` 不是干净的 `isVip()`
- 它更像“共享资格/次数/特权前置链”

### 4.4 对 VIP patch 的建议分层

#### 继续保留为主推荐

- `0x7e861c: c0062036 -> 32000014`

#### 继续保留为实验项

- `0x875efc: 40 01 20 37 -> 0a 00 00 14`
  - reward 反作弊旁路
- `0x8cf374: ef 81 01 d1 -> 2a 00 00 14`
  - 更广的共享状态强推

#### 暂不升格

以下虽然很值得继续观察，但我不建议现在就写成主 patch：

1. `0xbca1a4: a0002037 -> 05000014`
2. `0xbca1bc: a0002037 -> 05000014`
3. `0xbca270: 200d2036 -> 1f2003d5`
4. `0xbca2d0: 60002036 -> 1f2003d5`

原因：

- 这组 patch 会把 `bca170/bca240` 这条共享前置链整体强行推开
- 但它的副作用边界还没有被运行时验证
- 相比之下，`0x7e861c` 的消费侧意义更清晰

## 5. `喵喵饿了` 与首页 overlay 的当前边界

### 5.1 我当前仍不把“视觉根除喵块”升格为已解

虽然 `test3` 证明了：

- `0xbc5dc8` 是点击后 reward prompt/onTap handler
- `0xbd3540` 是首页 overlay/banner 最佳挂点

但它并没有把“阅读页右下角 Button 的真实挂载父级”完全解穿。

所以目前我仍然维持：

1. **稳定 fallback**：
   - `0xbc5dd0..0xbc5dd8`
   - 让块不可点击
2. **不再回头**：
   - `0xae30e0`
   - `0xa54178`
   - 它们都已经被运行时证伪过

### 5.2 我对“同源”的判断

当前最稳妥的说法是：

- 首页 overlay/banner 已经收敛到 `0xbd2e1c -> 0xbd3540`
- 阅读页 `喵喵饿了` 的真正 mount point 仍未彻底解出
- 两者共享 `0x59xx` 一簇对象池区域，但**还不能直接写成同一个 widget**

所以：

- 首页 overlay/banner 的 patch 方案已经可以独立推进
- 阅读页喵块仍以 `0xbc5dd0..0xbc5dd8` 作为稳定兜底
