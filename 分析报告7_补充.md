## 〇、一句话结论

喵喵饿了"消失"需父级 Visibility 显示条件，但**经 pool 间接调用的父级定位依赖 Code 簇解析——而 blutter 在本定制 VM 失败（cid84），三方都拿不到，故精确"消失"静态受限**。
转而给出**高价值的"不可点击"方案**（点击喵喵饿了无反应、不卡"正在加载"，直击用户痛点），并据此重评 reward：**放弃 V3 stub（类型模拟是无解硬骨头），改用 0x920d7c + 0xbc5dc8 双 patch 组合**——一劳永逸解掉故障弹窗 + 喵喵饿了卡顿，无需 V3。

## 一、喵喵饿了块的三层结构（已查清）

| 层 | 位置 | 性质 |
|---|---|---|
| **widget build**（叶子） | `0xae30e0` | 喵喵饿了块 widget 构造，唯一加载"喵喵饿了"(ref28525) 之一。`cmp w0,#0x14; b.ne` 是 widget 类型 switch，非显隐开关。无干净早返空路径（return null 必崩 widget 树）。callers=0（pool 间接/框架虚调用）。 |
| **点击→reward** | `0xbc5dc8` | reward-prompt："是否看推荐信息获取喵粮喂喵？"。async 闭包，callers=0。点击喵喵饿了→onTap→此函数弹询问→点"看"→reward广告→喵粮。 |
| **饥饿/特权状态** | `0x8cf36c` | 奖励特权状态机（bool，16-17 callers，TRUE 在 0x8cf3b4/0x8cf41c）。喵喵饿了"是否显示/猫是否饿"可能依赖它（VIP ctor 0xa54178 调它）。 |

### 1.1 为什么"消失"静态受限（机制铁证）
- 0xae30e0 是叶子 build，return null 崩 widget 树；return 空 widget 需 pool 现成 const 空实例（无可识别项）。
- 父级（阅读页 build 里把喵喵饿了块加进 children、用 Visibility 包裹的地方）经 **pool 间接调用**（callers=0）。
- 反查父级需 Code 簇（ref→.text 闭包解析）。**test2 已证 blutter 在本 VM 失败（"cid 84 not in class table"），三分支均无 Code 簇**。
- 本轮额外验证：pool 的 167 个 kImmediate **无一落在 .text 范围**（Code 入口不以 kImmediate 存于 ObjectPool）→ 无法用 pool 反查 callers=0 的父级。
- → 精确"消失"（找父级 Visibility bool）静态受阻。务实转向"不可点击"。

## 二、★ 主交付：喵喵饿了块"不可点击"（0xbc5dc8 入口 null-return）

### 2.1 patch
```
0xbc5dd0:  ef a1 00 d1  →  e0 03 16 aa   (mov x0, x22 = 返 null)
0xbc5dd4:  a0 0b 40 f9  →  fd 79 c1 a8   (ldp x29,x30,[x15],#0x10)
0xbc5dd8:  02 70 41 b8  →  c0 03 5f d6   (ret)
```
- 0xbc5dc8 = reward-prompt 函数入口。前两条（0xbc5dc8 STP / 0xbc5dcc mov x29,x15）建帧保留；从 func+8（0xbc5dd0）起改为"mov x0,null; ldp; ret"。
- 原第 3-5 条：`sub x15,x15,#0x28; ldr x0,[x29,#0x10]; ldur w2,[x0,#0x17]`（分配栈+取参），被 null-return 取代。

### 2.2 效果
- 点击喵喵饿了 → onTap 调 0xbc5dc8 → **立即 return null（async 闭包正常完成）→ 不弹"是否看推荐信息"询问窗 → 不进 reward 广告流程 → 不卡"正在加载"**。
- 块仍可见，但点击无任何反应、无卡顿、无故障。满足用户诉求"不卡/不可点击"。

### 2.3 安全验证（强）
1. **栈平衡**：0xbc5dc8 的 `STP x29,x30,[x15,#-0x10]!`（x15-=0x10）+ patch 的 `ldp x29,x30,[x15],#0x10`（x15+=0x10）完全对称回卷。mov x29,x15（0xbc5dcc）的临时设置被 ldp 覆盖，无害。
2. **null 安全**：0xbc5dc8 是 async `Future<...>` 闭包（reward-prompt 弹窗），正常路径本就可能"用户点取消→不弹窗→正常完成"。return null = async 正常完成，调用方（onTap）await 得到正常完成，**不读返回值、不会崩**。
3. **隔离**：0xbc5dc8 仅被 reward-prompt 触发链调用，与阅读/开屏/广告门控无关。
4. **与 0x920d7c 正交**：两 patch 互不影响。

### 2.4 局限
- 块仍可见（不"消失"）。若主控要求严格"消失"，见第三节实验性方案；若接受"不可点击/不卡"，此 patch 足够。

## 三、实验性：或能"消失"——强制特权 TRUE（0x8cf36c）

喵喵饿了块按猫饥饿状态显示；猫"已喂/特权生效"应隐藏。0x8cf36c 是"特权是否生效"判定（多 TRUE 返回）。

```
地址 0x8cf374:  ef 81 01 d1  →  2a 00 00 14   (b #0x8cf41c = 跳自带 TRUE 尾声)
```
- 原 `sub x15,x15,#0x60`（0x8cf36c 第 3 条）→ `b 0x8cf41c`（TRUE 尾声 `add x0,x22,#0x30; mov x15,x29; ldp; ret`）。
- **潜在双重收益**（与 backup"方案2"一致）：强制特权生效可能 ①隐藏喵喵饿了块 ②抑制阅读期广告 ③解锁部分 VIP 功能。
- **风险（中）**：0x8cf36c 有 16-17 callers（奖励计时/特权/广告门控/过期判定），强制 TRUE 影响面广，可能副作用（如奖励计时异常、广告门控反常）。
- **【需主控运行时验证】**：是否真能隐藏块 + 抑广告 + 无异常。作实验项，**与 0xbc5dc8 二选一**（不要同时上，否则叠加不可测）。

## 四、★ 任务三：reward/喵粮链路评估（定论）

### 4.1 现状链路
点击喵喵饿了 → ksad `register` channel → ~~MissingPluginException~~（去广告注释插件）→ reward 流程中断 → 卡"正在加载"。

### 4.2 ★ 为什么放弃 V3 stub（与 backup 三次实验一致）
backup 已穷尽验证 V3 返回值**必崩**（类型无法统一满足）：
- success(null) → reward: `NoSuchMethod []("channel_name") on null`
- success(空HashMap) → `type 'HashMap' is not subtype of 'bool'`（showSplashAd 期望 bool）
- success(Boolean.TRUE) → `type 'bool' is not subtype of 'qsa?'`（下游期望自定义对象）
- 不同广告方法期望 bool / Map / 自定义对象(qsa)，且部分用 `!` 强制非 null → **无统一返回可满足**。

### 4.3 ★ 关键：V3 引入的崩溃走不同故障路径，0x920d7c 堵不住
- 0x920d7c 只构造 ref27673（"广告无法显示"）故障 body。
- V3 返回类型错 → Dart 端 `TypeError`/`Null check` 崩溃 → 冒泡全局错误处理 → 可能构造**不同文案**的故障 body → **0x920d7c 堵不了** → 引入新故障窗。
- backup fact："V3-null 主界面 → Null check used on null (T+66s, 比 MissingPlugin 更快崩)" 印证。

### 4.4 ★ 推荐方案：放弃 V3，0x920d7c + 0xbc5dc8 组合
0x920d7c 的突破让"**不装 V3**"成为干净选项：
- 插件注释（去广告）→ 广告方法抛 MissingPluginException。
- MissingPlugin 未捕获 → 全局错误处理 → 本要构造故障 body(0x920d7c) → **0x920d7c 被堵 → 不构造 → 不弹故障窗**。MissingPlugin 异常被全局错误处理**静默吞掉**（不弹窗=无害），app 继续运行。
- 喵喵饿了：**不靠 reward 得喵粮**，而用 0xbc5dc8 让块**不可点击**（点击无反应，不卡）。

→ **无需 V3 的复杂类型模拟，无新崩溃，无新故障窗。**

### 4.5 reward"得喵粮"是否还有可能？
- 若坚持点喵喵饿了得喵粮：需 V3 按**每个方法精确返回类型**（方向 B/C），需反编译 Dart 端每个广告方法的 `invokeMethod` 回调转型（`result as bool/Map/qsa`），其中自定义对象 qsa Java 侧无法构造（需对象图脚手架，受 blutter 失败限制）。**硬骨头，ROI 低**。
- 建议：用户诉求是"块消失/不遮挡/不可点击"（非"必得喵粮"）→ **0xbc5dc8 不可点击已满足**，reward 得喵粮可放弃。

## 七、方法论
- scan_slot.py（上轮新增）字节级扫描对象池槽位加载者，**反向定位"谁构造某文案/widget"**，绕开闭包解析。
- 本轮新增验证：pool kImmediate 不含 .text 地址 → 确认 callers=0 父级静态不可解 → 据此选不可点击方案而非冒险 null 化叶子 build。

> 一句话：喵喵饿了"消失"受 blutter 失败限制（父级间接、无 Code 簇），故给"不可点击"（0xbc5dd0 null-return，点击无反应不卡，直击痛点）+ 实验性"或消失"（0x8cf374 force 特权 TRUE）。
> reward 定论：放弃 V3（类型崩溃引新故障窗，0x920d7c 堵不住），改 **0x920d7c（堵故障窗）+ 0xbc5dd0（喵喵饿了不可点击）** 组合，无需 V3 即解全部三主任务。
