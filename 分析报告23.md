# 第二十三轮代码恢复补充报告

> 分支：`test1`
>
> 角色：纯静态分析。本轮继续沿着第 22 轮的页面级思路推进，不再停在“`a54178` 是页面装配器”这一层，而是继续把它内部拆成更明确的子块分区。

## 1. 本轮结论

这轮新增的有效结论有三条：

1. **`0xa54178` 内至少可以稳定分成三类子块：**
   - donation 设备绑定提示块
   - donation 账号绑定提示块
   - 以 `0xa55c80` 为起点的“每日喂喵”子树
2. **`0xa55c80` 不是简单的字符串加载点，而是“每日喂喵”后续大子树的起点。**
3. 这意味着后续如果要继续往“VIP 页面内容进一步精细 patch”推进，`0xa55c80` 比 `a54178` 入口、也比纯看 `a55a28` 更像真正可操作的子树起点记录点。

## 2. `a54178` 内部的页面分区图

本轮把三类证据叠在一起：

1. 本地 `libapp.so` 反汇编
2. `origin/test2:analysis_workdir/pool_accesses.txt`
3. `origin/test3:analysis_workdir/KEY_REFS_MAP.txt` 与 `xref_db.json`

在此基础上，可以把 `0xa54178..0xa56828` 内部粗分成下面几段。

### 2.1 donation 设备绑定提示块

关键装配起点：

```asm
0x00a54d30: bl   #0x535040
0x00a54d3c: ldr  x17, [x27 + 0x2c000 + 0xd28]
```

对象池锚点：

- `0xa54d3c -> ref=8910 "通过捐赠绑定设备获取"`

随后紧接着是一整段标准 UI 装配：

```asm
0x00a54d44: mov  x1, x22
0x00a54d4c: bl   #0xf83da8
0x00a54d70: bl   #0x46799c
0x00a54d84: bl   #0x526e08
```

所以这里不是“只塞一个字符串”，而是一整个 donation-device 提示子块的装配起点。

### 2.2 donation 账号绑定提示块

关键装配起点：

```asm
0x00a55124: bl   #0x535040
0x00a55130: ldr  x17, [x27 + 0x2c000 + 0xd58]
```

对象池锚点：

- `0xa55130 -> ref=13169 "功能，通过捐赠绑定账号获取"`

后面同样跟着一段和前者几乎同型的装配流程：

```asm
0x00a55140: bl   #0xf83da8
0x00a55164: bl   #0x46799c
0x00a55178: bl   #0x526e08
```

因此，这一段可以明确视为 **donation-account 提示子块**。

### 2.3 “每日喂喵”子树起点

第 22 轮里，我只确认了 `a55a28` 是内容级 gate。本轮进一步确认：

**真正把“每日喂喵”这一段大子树拉起来的起点，是 `0xa55c80`。**

对象池锚点：

- `0xa55c80 -> ref=21707 "每日喂喵"`

对应反汇编：

```asm
0x00a55c80: ldr  x16, [x27 + 0x2c000 + 0xdd8]
0x00a55c88: fcmp d0, d1
...
0x00a55cb0: bl   #0x7e0788

...
0x00a55d5c: bl   #0x654980
0x00a55d7c: bl   #0x517b48
```

这不是一个“标题打点”，而是一段连续的大子树构造流程开头。后面还继续串到：

1. `0xa55de0` 一段新子树
2. `0xa560f0` 一段新子树
3. `0xa563c0` 一段新子树

因此，`0xa55c80` 更像：

- **每日喂喵页面段 / 喂喵相关大子树的起始装配点**

而不是单纯字符串引用点。

## 3. `a55a28` 与 `a55c80` 的关系

这轮可以把两者关系说得更清楚：

### 3.1 `a55a28` 是上游 gate

关键片段：

```asm
0x00a55a24: ldur x1, [x29, #-0x18]
0x00a55a28: tbz  w1, #4, #0xa55aa8
```

也就是说：

1. `a55a28` 根据 `0x8cf36c` 的结果决定是否跳过前半段装配
2. 它是一个**上游页面内容门**

### 3.2 `a55c80` 是下游子树起点

`a55a28` 之后继续走了不少装配逻辑，但真正把 `"每日喂喵"` 拉出来的起点是 `a55c80`。

所以可以把两者关系写成：

1. `0xa55a28`
   - 上游特权态内容门
2. `0xa55c80`
   - 下游“每日喂喵”子树起点

这比之前只说“`a55a28` 在喂喵区域附近”要精确一层。

## 4. `a54178` 内 donation / 喂喵 / 喵块 的结构关系

本轮可以把第 22 轮的混合页面判断再细化：

### 4.1 donation 提示和喂喵子树确实在同一个页面装配器里

目前已经能明确钉住：

1. `0xa54d3c`
   - donation-device 提示块起点
2. `0xa55130`
   - donation-account 提示块起点
3. `0xa55c80`
   - 每日喂喵子树起点

它们都落在 `0xa54178..0xa56828` 内。

因此，`a54178` 不是“页面顶部一个公用壳子”，而是真正在同一个函数里把 donation / 喂喵几块大内容拼起来。

### 4.2 `喵喵饿了` 仍只确认在更高层文本集合里出现

本轮没有把 `喵喵饿了` 在 `a54178` 内压到像 `a54d3c / a55130 / a55c80` 这样具体的起点。

当前最稳的事实仍是：

1. `喵喵饿了` 只落在：
   - `0xa54178`
   - `0xae30e0`
2. 其中：
   - `0xae30e0` 是阅读页常驻块 builder
   - `0xa54178` 是页面装配器

所以：

- 对“阅读页常驻块”的主处理仍应放在 `0xbc5dd0..0xbc5dd8`
- 不应误把 `a54178` 内 donation/每日喂喵子树等同于阅读页常驻块显隐

## 5. 为什么 `a55c80` 值得记录

第 21 轮和第 22 轮已经有：

1. `0xa55878`
2. `0xa55a28`

这轮新增记录点：

3. `0xa55c80`

它的价值在于：

1. `a55878`
   - 说明页面在切内容
2. `a55a28`
   - 说明页面在基于特权态切子树
3. `a55c80`
   - 终于把其中一个子树的主题钉成了 `"每日喂喵"`

也就是说，如果以后继续做页面级 patch，顺序上应该更像：

1. 先看 `a55a28` 是否决定“要不要进喂喵段”
2. 再看 `a55c80` 这段喂喵子树是否可以单独被削掉或替换

而不是直接碰 `a54178` 入口。
