# 第二十一轮代码恢复收口报告

> 分支：`test1`
>
> 角色：纯静态分析。本轮按主控要求修正方向，不再扩散到新工具，直接吸收 `backup / test2 / test3` 的现有成果，并用本地 `libapp.so` 做字节级复核，收口到三件事：
>
> 1. `去广告 + 稳定运行`
> 2. `喵喵饿了` 的实用处置
> 3. `VIP / reward 特权` 的可执行方案

## 1. 本轮结论

本轮之后，`test1` 这条线应收敛成两套方案，而不是继续追“单一 isVip 总闸”：

1. **稳定纯净版**
   - 在既有基线之上，补 `0x920d90`、`0xbc5dd0..0xbc5dd8`、`0x7e861c`
   - 目标是：故障弹窗不弹，`喵喵饿了` 不再点击卡死，特权门走更窄上层 gate
2. **reward / 特权实验版**
   - 保留点击链，不打 `0xbc5dd0..0xbc5dd8`
   - 改试 `0x875efc`（reward 反作弊旁路）+ `0x7e861c`
   - `0x8cf374` 只保留为更激进的实验项，不升格为主推荐

同时，跨分支和本地复核已经足够说明：

1. **没有单一 `isVip=true` 开关**
2. `0xa54178 / 0xb9fc84` 是厚页面 / 厚状态机，不适合入口 return-null
3. `V3/V4 stub` 仍不该回到主线，`backup` 的运行时事实已经把“统一返回值模拟广告 SDK”证伪

## 2. 交叉验证后的统一认识

### 2.1 与 `backup` 运行时事实一致

`origin/backup:backup/第八轮-去广告去首启弹窗/ROUND8_FACTS.md` 继续决定优先级：

1. 去广告后的根问题是广告 channel 周期调用失败，不是普通 `showDialog`
2. 故障弹窗根因是 `MissingPluginException -> 未捕获 -> 全局错误处理`
3. `V3/V4 stub` 的统一返回值路线已被运行时证伪，不再作为主线
4. 因此真正有价值的仍是：
   - 截断故障正文链
   - 让 `喵喵饿了` 退出用户路径
   - 在 reward / 特权层找更窄、更安全的 gate

### 2.2 与 `test2` 的一致点

`origin/test2:分析报告8.md` 对本轮最有用的几点：

1. `0xca8bdc` 确是 `MissingPluginException` 构造函数，但它**不是**当前最好的 patch 点
2. `0xae30e0` 确是 `喵喵饿了` 常驻块 builder
3. `0x8758bc` 确是广告 dispatcher，上游并不等于单一 VIP 逻辑
4. `0xae30e0` / `0xbc5dc8` 都是 `callers=0` 类函数，走 pool 间接或闭包路径，不应再用“找直连 caller”思路强推

### 2.3 与 `test3` 的一致点

`origin/test3:分析报告6_补VIP链路.md`、`分析报告7.md`、`分析报告7_补充.md`、`分析报告7_喵喵饿了块根治深挖.md` 给出的高价值结论，我在本地字节上已复核并接受：

1. **VIP / 广告抑制是三套并行机制**
   - 奖励特权计时：`0x8cf36c`
   - 逐源 noAd：`0x839e80 / 0x890c6c / 0x920a60`
   - 捐赠 VIP 页/状态机：`0xa54178 / 0xb9fc84`
2. `0x59d964` 是 `getRewardTime` getter，并直接调用 `0x8cf36c`
3. `0x88aacc` 是 `onReward` 回调分发器，但**不是**当前可直接给主控装机的最窄 patch 点
4. `0x8758bc` 尾段比较 `ref=11545 "onReward Cheat Triggered!"`，`0x875efc` 是当前最干净的 reward-only 反作弊旁路点

### 2.4 一个必须说明的分歧

`test3` 最新 `KEY_REFS_MAP.txt` 与更早几轮对象池锚点，对 `ref=27673` 的 slot 号仍存在漂移：

1. 旧锚点：`slot 0x278d / off 0x13c68`
2. `test3` 最新映射：`slot 0x276e / off 0x13b70`

这说明对象池反序列化的 entry 计数仍有版本差异。**本轮不再依赖这个 slot 号来给 patch**，而是直接依赖本地 `libapp.so` 的字节和 caller 扫描。尽管 slot 有漂移，各分支仍收敛到同一 code site：`0x920d7c / 0x920d90`。

## 3. 本地独立字节复核

### 3.1 `0x920d7c` 与故障正文链

本地反汇编确认：

```asm
0x00920d7c: stp  x29, x30, [x15, #-0x10]!
0x00920d80: mov  x29, x15
0x00920d84: sub  x15, x15, #0x48
0x00920d90: b.ls #0x92131c
...
0x00920dac: tbz  w0, #4, #0x920e34
...
0x00920e34: mov  x0, x22
0x00920e38: mov  x15, x29
```

本地逐词扫描 `.text` 中所有 `BL` 目标，结果是：

1. `0x920d7c` 有且仅有 **3 个直连 caller**
   - `0x920d1c`
   - `0xc4cc7c`
   - `0xc4d298`
2. 这与 `test3` 对故障正文链的判断一致

因此：

- `0x920d90: 692c0054 -> 29000014`
- 语义是：**直接跳到函数自带的 null 返回尾声 `0x920e34`**

这仍然是故障弹窗正文链的首选 choke point。

### 3.2 `喵喵饿了` 点击链

本地反汇编确认：

```asm
0x00bc5dc8: stp  x29, x30, [x15, #-0x10]!
0x00bc5dcc: mov  x29, x15
0x00bc5dd0: sub  x15, x15, #0x28
...
0x00bc5ea0: mov  x0, x22
0x00bc5ea4: mov  x15, x29
0x00bc5ea8: ldp  x29, x30, [x15], #0x10
0x00bc5eac: ret
```

本地 `BL` 扫描结果：

1. `0xbc5dc8` **没有直连 caller**
2. 说明它确实更像 reward-prompt / 点击链闭包，而不是普通直接调用函数

当前最实用的 patch 仍然是：

- `0xbc5dd0..0xbc5dd8`
- `efa100d1 a00b40f9 027041b8 -> e00316aa fd79c1a8 c0035fd6`

这不是“隐藏块”，而是让点击链**立即正常返回 null**。结果是：块仍可能可见，但点击不再卡 `正在加载`。

### 3.3 reward 反作弊链

本地反汇编确认：

```asm
0x00875ee8: ldr   x16, [x27, #0x19078]   ; ref=11545
0x00875ef4: bl    #0xc833ac
0x00875efc: tbnz  w0, #4, #0x875f24
0x00875f18: bl    #0x875f78
```

本地 `BL` 扫描结果：

1. `0x875f78` 只有 **1 个直连 caller**
   - `0x875f18`
2. `0x875efc` 正是这个 cheat-only helper 的门口

因此：

- `0x875efc: 40012037 -> 0a000014`

应继续表述为：

**reward 反作弊旁路**

而不是泛泛的“reward 候选点”。

### 3.4 特权 gate

本地反汇编确认：

```asm
0x007e8618: bl   #0x8cf36c
0x007e861c: tbz  w0, #4, #0x7e86f4
...
0x007e86e4: add  x0, x22, #0x30
0x007e86f4: add  x0, x22, #0x20
```

对应 patch：

- `0x7e861c: c0062036 -> 32000014`

语义是：

- **直接跳到 canonical true 尾声 `0x7e86e4`**

这仍是目前最窄、最干净的上层特权 gate。

### 3.5 `0x8cf36c` 是共享特权态，不是单一 `isVip`

本地反汇编确认：

```asm
0x008cf374: sub x15, x15, #0x60
...
0x008cf3b4: add x0, x22, #0x30
...
0x008cf41c: add x0, x22, #0x30
```

以及：

```asm
0x0059d998: bl #0x8cf36c    ; getRewardTime 内部
0x00a541c8: bl #0x8cf36c    ; VIP ctor 内部
```

这再次说明：

1. `0x8cf36c` 是**共享特权态 / reward privilege** 计算
2. 它被 `getRewardTime`、`VIP ctor`、阅读期广告门等多处共同消费
3. 它不是“单一 donation VIP 开关”

因此：

- `0x8cf374: ef8101d1 -> 2a000014`

仍只应保留为**更激进的实验项**，不升格为主推荐。

## 4. 推荐 patch 栈

### 4.1 稳定纯净版

这套是当前最适合主控直接装机验证的组合。默认叠加在已验证基线上：

1. 篡改窗：
   - `0x8e1dd0`
   - `0x8ef2b8`
2. 已有故障门 / dialog NOP 基线：
   - `0xbc1020`
   - `0xbc1058 / 0xbc1134 / 0xbc1234`
   - `0xbd60ac / 0xbd6264 / 0xbd6300 / 0xbd6130`

本轮新增主推荐：

1. `0x920d90: 692c0054 -> 29000014`
   - 故障正文链 choke point
   - 直接跳 `0x920e34` 自带 null 尾声
2. `0xbc5dd0..0xbc5dd8: efa100d1 a00b40f9 027041b8 -> e00316aa fd79c1a8 c0035fd6`
   - `喵喵饿了` 点击链立即返回
   - 目标是“不可点击不卡”，不是“完全消失”
3. `0x7e861c: c0062036 -> 32000014`
   - 更窄的上层特权 gate
   - 直接跳 canonical true 尾声

### 4.2 reward / 特权实验版

如果主控目标从“稳定纯净运行”切到“尽量保留喂喵 / reward / 特权效果”，则不要同时使用 `0xbc5dd0..0xbc5dd8`。

改试：

1. `0x920d90: 692c0054 -> 29000014`
2. `0x875efc: 40012037 -> 0a000014`
3. `0x7e861c: c0062036 -> 32000014`

含义：

1. `0x920d90` 继续压住故障正文链
2. `0x875efc` 绕开 reward 反作弊 helper
3. `0x7e861c` 让上层 privilege 消费门更容易走 true

这套的目标不是“绝对稳”，而是给主控一个**值得试的 reward / 特权版**。

### 4.3 更激进但不推荐默认使用的实验项

1. `0x8cf374: ef8101d1 -> 2a000014`
   - 强制跳 `0x8cf41c` true 尾声
   - 影响面明显大于 `0x7e861c`
2. `0x8a044c: c0082036`
3. `0x8a05f0: 60102036`

`0x8a044c / 0x8a05f0` 这两个是 `0x89fe80` 内部的局部消费门，我本地已复核存在，但上下文仍明显比 `0x7e861c` 更重，因此继续只做记录，不升格为主推荐。

## 5. 不再推荐的方向

### 5.1 不再把这些点当“VIP 单开关”

1. `0xa54178`
2. `0xb9fc84`

原因：

1. 本地反汇编和 `test3` 对象池链都说明它们是混合 UI / 混合状态机构造器
2. 不是单一 `isVip` bool 门
3. 入口 return-null 或粗暴强改风险过高

### 5.2 不再把这些思路放回主线

1. `V3/V4 stub`
2. 继续扩大 `showDialog` NOP 覆盖面
3. 回到 `0x913bf8`、`0x7e85f4`、`0xae312c` 这类已降级点

原因：

1. `backup` 已给出运行时反证
2. 当前更高杠杆的路已经清楚：
   - `0x920d90`
   - `0xbc5dd0..0xbc5dd8`
   - `0x7e861c`
   - `0x875efc`

## 7. 一句话收口

当前最实际的方向已经不是“再找一个单一 VIP 总开关”，而是：

1. 用 `0x920d90` 压住故障正文链
2. 用 `0xbc5dd0..0xbc5dd8` 让 `喵喵饿了` 退出用户路径
3. 用 `0x7e861c` 作为最窄的上层特权 gate
4. 若要保留 reward，再单独试 `0x875efc`

这四个点已经比继续追 `a54178 / b9fc84 / V3 stub` 更符合当前代码恢复目标：**去广告、稳定运行、尽可能恢复完整功能**。
