### 1.3 关键结论

1. **0xa52920 经虚分派调用**：lr=0xd93204 在 func 0xd931c0，该函数是分派桩：
   ```
   0xd931e8: ldurh w2,[x1,#1]      ; 读对象cid
   0xd931f8: add x30,x0,x17(#0x8dbd) ; cid+偏移
   0xd931fc: ldr x30,[x21,x30,lsl#3]; 从分派表(x21)取函数指针
   0xd93200: blr x30                ; ★虚分派→0xa52920
   ```
   这解释了11轮静态分析callers=0的原因——**0xa52920是某widget类的build方法，经cid虚分派调用，无BL直调**。

2. **喵喵块是普通Flutter widget**：调用链经标准build/rebuild（BuildOwner.buildScope→Element.attachRenderObject→...→build）。**没有"显示guard"在调用0xa52920之前**——它是widget树常驻节点，和别的widget一样被rebuild。

3. **"投喂后消失"机制**：不是运行时guard，而是**父widget根据投喂状态改变子树结构**（猫饿时children含喵喵块，喂饱后不含）。guard在**喵喵块的父widget build**（决定是否构造喵喵块widget实例加入children）。

4. **0x9d4b74内的tbnz w0,#4@0x9d4b94**（读field@0x43）是build内的**缓存标志**（首次算→存TRUE→跳过重算），非显隐guard。

## 二、给主控的精确线索（本地反汇编定位guard）

### 2.1 找guard的方向
guard = **喵喵块widget被构造并加入children的源头**。0xa52920是喵喵块build，它的**widget实例**在父build里被`new`并加入children列表。需找：
- 哪个函数**构造0xa52920对应的widget实例**（alloc cid对应widget类）
- 那个函数根据状态决定是否构造（`if(猫饿) children.add(喵喵块)`）

### 2.2 关键地址（本地反汇编用）
- **lr=0xd93204 / func 0xd931c0**：直接调0xa52920的分派桩（96B）
- **0x9d4c04**（1648B, skipCount, bool）：喵喵块build的framework辅助
- **0xd93a6c**（296B, 多blr虚分派）：children遍历器候选
- **0x9dbe1c**（2876B, attaching to render tree）：Flutter Element层
- **0x4eed8c**（2088B, rebuilding dirty elements）：BuildOwner.buildScope

建议主控：反汇编0xd93a6c（children遍历）和它的调用者，找"遍历到喵喵块widget前根据状态skip"的逻辑。或反汇编喵喵块widget类的**构造函数**（alloc点），找它的调用者=父build。
