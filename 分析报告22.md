## 一、候选A修正(原版崩→修正不崩)

### 原版崩因
- 主控给的 `52800000`(movz w0,#0) 让 w0=0
- **Dart bool false ≠ 0**! Dart FALSE = x22+0x20 = 0x8061(tagged)
- w0=0 不是合法 Dart bool → signal 4(Illegal instruction, Dart VM检查)

### ★ 修正patch
```
偏移0x466e18: 原 e00301aa (mov x0,x1, x0=isHungry值)
            → 新 c0820091 (add x0,x22,#0x20, x0=FALSE=0xffeb00008061)
```
- capstone验证: `add x0, x22, #0x20` ✓
- x22=null_base=0xffeb00008041, +0x20 = 0xffeb00008061 = FALSE ✓
- 效果: tbz w0,#4 @0x466e2c, bit4=0(FALSE) → 恒跳过(不调0x466e84塞猫)
- 等效于 runtime flip isHungry→FALSE(报告20验证猫消失)

### 装机结果
- **不崩**(PID稳定12080)
- **不白屏**(117KB截图有内容, 满屏深色渲染)
- **反篡改弹窗挡住**(A+B没完全抑制redroid检测, spawn也不稳)
- → patch安全验证通过(不崩不白屏), 但猫验证被反篡改阻碍

## 二、guard patch逻辑确认

```
0x466e14: ldr x1,[sing,#0x20]        ; 读isHungry
0x466e18: ★ add x0,x22,#0x20 (PATCH) ; x0=FALSE(原mov x0,x1=isHungry)
0x466e1c: stur x1,[x29,#-8]          ; 存原值到栈(不影响, x1仍是原isHungry)
0x466e20: tbnz w0,#5,...             ; null检查(x0=FALSE=0x8061, bit5=1→跳过lazy-init)
0x466e28: ldur x0,[x29,#-8]          ; ★ x0=栈里的原isHungry(不是我们改的FALSE!)
```

**等等! 0x466e28 从栈读回原值!** stur x1,[x29,#-8] 存的是x1(原isHungry), ldur x0,[x29,#-8] 读回的是**原值不是FALSE**!

→ **我改0x466e18只影响0x466e20的null检查, 不影响0x466e2c的tbz判断(它用0x466e28的栈值=原isHungry)!**

**我的patch位置错了!** 0x466e18改了x0, 但0x466e28从栈恢复原值, 0x466e2c用栈值(原isHungry)做判断.

正确patch位置应该在 **0x466e28**(ldur x0,[x29,#-8])之后, 或 **0x466e2c之前**注入FALSE.

→ **0x466e2c tbz用0x466e28的x0. 改0x466e28为add x0,x22,#0x20才对!**
