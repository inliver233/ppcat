# 第二十二轮代码恢复补充报告

> 分支：`test1`
>
> 角色：纯静态分析。本轮不再重复“没有单一 isVip 开关”这个总论，而是继续向用户最关心的方向收口：
>
> 1. `VIP / 特权页面`内部，到底哪些点是真正在切换页面内容
> 2. `0x7e861c / 0x8cf374 / 0xa54178` 三者应该如何分级
> 3. `喵喵饿了 / 每日喂喵 / 捐赠绑定文案` 在页面装配层的关系

## 1. 本轮结论

这轮最重要的新结论有三条：

1. **`0xa54178` 不是单一 VIP 页标题函数，而是一个把“捐赠绑定提示 + 每日喂喵 + 喵喵饿了 + 已生效/未获得状态”一起装配出来的页面级 builder。**
2. **`0xa55878` 和 `0xa55a28` 不是单纯 spacing/样式点，而是基于 `0x8cf36c` 的结果切换页面子树/文案的局部门。**
3. 因此：
   - `0x7e861c` 仍是默认主推荐
   - `0x8cf374` 只适合实验性“更广覆盖”尝试
   - `0xa54178` 本身依然不适合入口 return-null，但它内部的局部特权门可以单独记录

## 2. `0xa54178` 的新定性

### 2.1 交叉验证来源

本轮把三类证据合在了一起：

1. 本地 `libapp.so` 反汇编
2. `origin/test2:analysis_workdir/pool_accesses.txt`
3. `origin/test3:analysis_workdir/xref_db.json` 与 `KEY_REFS_MAP.txt`

### 2.2 `0xa54178` 不是“单一 VIP 开关页”，而是混合页面装配器

`xref_db` 给出的字符串集合，已经把它的页面性质钉得很死：

`0xa54178` 内同时出现：

1. `通过捐赠绑定设备获取`
2. `功能，通过捐赠绑定账号获取`
3. `每日喂喵`
4. `喵喵饿了`
5. `已生效`
6. `已获取`
7. `未获得`
8. `次推荐信息，即可获取`
9. `点击头像绑定账号`
10. `解绑`
11. `训练喵喵`
12. `分享喵喵`
13. `，未获得特权将在0点重置浏览次数！`

这说明 `0xa54178` 不是“单一 donation VIP 页”或“单一喂喵页”，而是一个**把 donation、reward privilege、喵块入口、账号绑定说明混在一起的装配函数**。

### 2.3 本地反汇编确认它直接消费 `0x8cf36c`

关键片段：

```asm
0x00a541c4: ldr x4, [x27, #0x2bb0]
0x00a541c8: bl  #0x8cf36c
0x00a541cc: stur x0, [x29, #-0x18]
```

因此，后面凡是反复读取 `[x29, #-0x18]` 并对 bit4 做 `tbz/tbnz` 的点，本质上都是在消费“当前特权状态是否生效”。

## 3. `0xa54178` 内部两个真正有价值的页面级门

第 21 轮里我已经把 `0xa55688 / 0xa55878 / 0xa55a28` 作为候选记录过。这轮进一步细化后，结论如下。

### 3.1 `0xa55688` 只是局部尺寸/间距门，不升格

窗口：

```asm
0x00a55684: ldur x1, [x29, #-0x18]
0x00a55688: tbnz w1, #4, #0xa55694
0x00a5568c: fmov d0, #16.0
0x00a55694: fmov d0, #8.0
```

这里切的是 `16.0 / 8.0`，更像 spacing 或 padding，**不是内容级 gate**。

因此：

- `0xa55688` 继续降级
- 不再当作 VIP/特权页面解锁主候选

### 3.2 `0xa55878` 是页面内容级门

关键片段：

```asm
0x00a55870: ldur x0, [x29, #-0x18]
0x00a55878: tbnz w0, #4, #0xa55888
0x00a5587c: ldr  x1, [x27 + 0x2c000 + 0xdc8]
0x00a55884: b    #0xa559c8
...
0x00a55940: bl   #0x89b300
...
0x00a559c8: bl   #0x46fedc
```

这个分支和 `0xa55688` 最大不同在于：

1. false 分支直接装载一个对象池对象并跳入后续子树装配
2. true 分支会继续走一大段运行时/文案构造路径
3. 里面还会调用 `0x89b300`

而 `0x89b300` 本身的 callers 包含：

1. `0x89afa0`
2. `0x89b5d0`
3. `0x8cf36c`
4. `0xa54178`
5. `0xb9fc84`

这说明 `0xa55878` 已经不是“调个数字/样式”那么轻，而是在切一段真正的内容装配逻辑。

### 3.3 `0xa55a28` 也是页面内容级门，而且比 `0xa55878` 更重

关键片段：

```asm
0x00a55a24: ldur x1, [x29, #-0x18]
0x00a55a28: tbz  w1, #4, #0xa55aa8
0x00a55a2c: bl   #0x535040
...
0x00a55aa8: mov  x2, x0
0x00a55abc: bl   #0x526e08
```

如果 bit4=false，它直接跳到 `0xa55aa8`，跳过前面一整段对象装配；如果 bit4=true，则会多走一段构造和写字段逻辑。

更重要的是，本函数范围内还能看到：

1. `0xa55c80 -> ref=21707 "每日喂喵"`
2. 前面同函数体里还有：
   - `0xa54d3c -> ref=8910 "通过捐赠绑定设备获取"`
   - `0xa55130 -> ref=13169 "功能，通过捐赠绑定账号获取"`

所以 `0xa55a28` 所在的后半段，已经非常接近“喂喵/特权页面子块总装配”的区域，而不只是一个局部样式门。

## 4. `0xb9fc84` 的作用进一步压实

### 4.1 它的核心不是 `8cf36c`，而是 donation / login / bind 文案分流

`xref_db` 显示：

`0xb9fc84` 内部出现：

1. `通过捐赠绑定设备获取`
2. `功能，通过捐赠绑定账号获取`
3. `未登录`
4. `未获取`
5. `检测更新`
6. `喵喵喵~`
7. `次推荐信息，即可获取`
8. `skipCount`
9. `remoteConfigSign`

本地反汇编里最关键的一段是：

```asm
0x00ba0064: cmp  w1, w16
0x00ba0068: b.ne #0xba0078
0x00ba0070: ldr  x3, [x27 + 0x2c000 + 0xd58]   ; ref=13169
0x00ba0074: b    #0xba0080
0x00ba0078: ldr  x3, [x27 + 0x2c000 + 0xd28]   ; ref=8910
```

也就是说：

1. `0xba0070` 和 `0xba0078/0xba007c` 这组不是“特权 true/false”
2. 它们是在两个 donation 提示文案之间分流：
   - `功能，通过捐赠绑定账号获取`
   - `通过捐赠绑定设备获取`

这进一步说明 `0xb9fc84` 更像：

- donation / bind / login / update 混合状态机

而不是应该被直接 patch 成恒真或入口 null-return 的东西。

## 5. `0x7e861c`、`0x8cf374`、`0xa54178` 的正确分级

### 5.1 默认主推荐仍是 `0x7e861c`

原因没有变化，但这轮多了一条新支撑：

1. `0xa54178` 内部确实有更具体的页面级消费门
2. 但这些门都依赖更大的页面装配上下文
3. `0x7e861c` 仍然是**离共享特权态最近、同时上下文最轻**的默认 patch 点

因此默认仍推荐：

- `0x7e861c: c0062036 -> 32000014`

### 5.2 `0x8cf374` 只作为更广覆盖实验项

本轮没有推翻第 21 轮的判断，反而更支持它：

1. `0x8cf36c` 被 `a54178 / 59d964 / 89fe80 / 7e8618` 等多处共享消费
2. 强行把 `0x8cf374` 跳到 true 尾声，影响面天然更大
3. 它可能同时改变：
   - 阅读期广告门
   - VIP 页面子树
   - 喂喵状态表现

因此：

- `0x8cf374: ef8101d1 -> 2a000014`

仍只保留为实验项。

### 5.3 `0xa54178` 只建议记录局部门，不建议做入口 patch

这轮的新发现不是“现在可以 patch `a54178` 入口了”，而恰恰相反：

1. 它内部承载的文案和子树太多
2. 至少 `a55878 / a55a28` 这两个内容门都表明它在做真正的页面装配
3. 继续对它做入口级别 return-null 或粗暴跳转，风险依然过高

所以正确姿势是：

1. 把它作为**页面装配器**来理解
2. 记录它内部的几个页面级局部门
3. 默认 patch 仍优先落在：
   - `0x7e861c`
   - `0x875efc`
   - `0x920d90`
   - `0xbc5dd0..0xbc5dd8`

## 6. 对 `喵喵饿了` 的新影响

这轮还有一个额外收获：

1. `每日喂喵` 只落在：
   - `0x846b1c`
   - `0xa54178`
   - `0xae71bc`
2. `喵喵饿了` 只落在：
   - `0xa54178`
   - `0xae30e0`

这说明：

1. `a54178` 的确是把“喂喵入口”和“VIP/特权说明”绑在一起的页面总装配器
2. `ae30e0` 才是阅读页常驻块 builder
3. `a54178` 里的内容门更像“页面展示层”，不是阅读页常驻块的直接显隐开关

这也进一步支持当前主线：

- `常驻块` 仍优先用 `0xbc5dd0..0xbc5dd8` 让点击退出用户路径
- 不把 `a54178` 当成“隐藏阅读页喵块”的主 patch 点

## 7. 本轮新增建议

### 7.1 推荐等级不变

主推荐仍是：

1. `0x920d90: 692c0054 -> 29000014`
2. `0xbc5dd0..0xbc5dd8: efa100d1 a00b40f9 027041b8 -> e00316aa fd79c1a8 c0035fd6`
3. `0x7e861c: c0062036 -> 32000014`

### 7.2 新记录的页面级 gate

新增高价值记录点：

1. `0xa55878: 80002037`
   - `a54178` 内的页面内容级 gate
   - 不再只当 spacing/样式门
2. `0xa55a28: 01042036`
   - `a54178` 内更重的页面子树 gate
   - 与 donation/每日喂喵子树处在同一后半段装配区
3. `0xba0070 / 0xba0078`
   - `b9fc84` 内 donation 文案分流
   - 不是 reward privilege gate

### 7.3 暂不升级为主 patch

这三个点虽然现在语义更清楚了，但我仍不建议直接升格成主控默认 patch：

1. 它们都在厚页面函数内部
2. 静态上已经能证明“是内容门”，但还缺主控的运行时逐项实测
3. 当前默认方案已经足够支撑“纯净稳定运行”

## 8. 一句话收口

第 22 轮最大的价值不是再找一个“更大的总闸”，而是把 `a54178` 从“复杂函数”进一步压实成了**页面装配器**，并确认它内部至少有两个真正基于 `0x8cf36c` 的内容级分支：

1. `0xa55878`
2. `0xa55a28`

这让后续如果主控要继续冲“VIP 页面内容全解锁”，就有了比入口 patch 更合理的页面级切入点；但在此之前，默认主推荐仍应保持 `0x7e861c`，而不是回到 `a54178 / b9fc84` 的粗 patch 路线。
