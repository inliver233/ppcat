## 二、test3 本轮新增深化（在 test1/test2 之上）

### 2.1 ★ 共享门控 0x911788 全解剖（最接近"总开关"的东西，但非干净单点）
test1 发现 0x911788，test3 独立反汇编全函数（928字节）+ 全 caller 群，结论：

- **18 个 caller**，覆盖：0x8a0fd4(可以捐赠)、0x920d7c(故障body builder)、0xb9fc84(VIP状态机)、0x7e2adc/0x7e8534/0x7eb884/0x7f7858/0x80a2fc(广告/UI)、0x838e2c、0xa58068、0xa92b0c、0xaf1710、0xb04188、0xbca240、0xbe4418、0xc46a68 等。
- 其中 **14 个 caller-func 自己也返回 canonical bool**（gate 下游）。
- 它加载 `remoteConfigSign`(26968)、`Main`(18250)，做 **时间阈值比较**（[x26,#0x80]+0xea8 时间字段，sdiv /1000 秒级）。
- **返回约定**（canonical bool，可复用不变量）：
  - `add Xd, x22, #0x30` = **True**（bit4 置位，调用方 `tbnz w0,#4` 命中=通过）
  - `add Xd, x22, #0x20` = **False**（bit4 不置位）
  - 尾部 `csel x0, x16(false), x17(true), lt` @ 0x911aec：**elapsed_sec >= 阈值 → True(通过)**
- **但它有多个非-bool 出口**（0x911b00+ 返回 x1 对象、0x911b24 null 路径），所以**不是干净的单一 return-true patch 目标**。强改 csel 只影响时间比较那一路，其它路不受控 → 风险高，不推荐盲改。
- **语义**：它是"remote-config-sign + 时间阈值"复核门，被篡改/超时返回 false → 14+ 路径被 gate（捐赠禁用、故障显示等）。这解释了篡改后为何多路受限。要真正绕过得逐路处理或动态确认，**不能单字节强改这个共享函数**。

### 2.2 可复用不变量：Dart canonical bool 返回约定（app 全局）
扫描证实：本 app 所有 Dart bool 返回走 `add Xd, x22, #{0x20|0x30}` + 调用方 `tbnz/tbz w0,#4` 判定。bit4=1=True, bit4=0=False。**这是识别/构造任意 isXxx 判定 patch 的通用工具**（改 `add #0x20`↔`add #0x30` 即可翻转单点 bool，或改调用方 `tbnz`↔`tbz`）。

### 2.3 喵喵/Cat 体系完整字符串簇（VIP 页 0xa54178 内，test1 补 + test3 复核）
0xa54178 是 VIP/喵喵 内容页主构造区，包含：
- 喵喵饿了(28525 @0xa55cbc)、每日喂喵(21707 @0xa55c80)、训练喵喵(21925 @0xa55f30)、让喵喵更智能(31475)、每日一次(17788 @0xa56200)
- VIP 态：已获取(9292)、未获得(29757)、已生效(3524)、通过捐赠绑定设备(8910)/账号(13169)、已绑定账号(10073 @0xa5796c)、点击头像绑定账号(4505)、解绑(24111)
- 限制：未获得特权将在0点重置浏览次数(13928)、24小时内累计浏览(2878)、次推荐信息即可获取(13199)

→ **喵喵饿了块是 VIP 页的一部分**，与每日喂喵/reward 共用 0x8cf36c 特权状态机。

## 三、奖励反作弊 + reward 事件格式（test1 解码，test3 复核用于 stub 升级）

test1 解码了 Pangle reward 回调完整格式（`analysis_round6/reward_events_decoded.txt`）：
- 事件：onAdShow / onAdClicked / onAdExposure / onVideoComplete / onAdComplete / onAdClose(d) / onAdSkip / onSkippedVideo
- reward 成功：**onRewardArrived**(verify/amount/name)、**onRewardVerify**、onAdReward
- error：onError(code) / onAdError(errMsg,errCode) / onVideoError
- 字段：rewardType / rewardName / rewardVerify / rewardAmount / userId / customData

→ **任务D stub 升级方向**：当 Dart 调 `showRewardVideoAd`，stub 除回 success 外，需回传 `onRewardArrived`(verify=true,amount=1) + `onAdClose` 事件（经 flutter_pangle_ads_event 或 invokeMethod 回灌），Dart 侧才认"奖励到账"。但反作弊(11545)可能仍判即时reward为作弊 → **建议任务D stub 对 reward 调用加一个最小延时**（模拟观看时长）以绕过反作弊时间阈值检查。

## 五、确定性分级总结

- **可直接装机信任**（三方一致+运行时已证）：对象池4映射、故障body=0x920d7c、0xbbc22c纠正、任务D no-op stub、0x7e1464/0xbc0f40 入口禁改。

- **明确证伪/否决**：单一 isVip 字节开关（不存在）、0x911788 单点强改（多出口不干净）、0xbbc22c=故障builder、旧27673→0x278d。
