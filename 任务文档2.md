# 第二轮分析任务:精确定位"长版提示框"的 showDialog 触发点

> 本任务为纯静态分析。第一轮两位分析者已产出高质量结果(test1/test2 分支),本任务是**针对性收口**一个关键缺口。

## 一、第一轮成果回顾(已知可靠信息)

### 1.1 已验证可靠的地址(test1 用 unflutter 产出,我已逐字节核对全部匹配)

- `.text` 起 `0x460000`,本 so 文件 **vaddr == file offset**
- 聚合判定函数候选 `0x00592fd8`(内部有 22 个 `b.eq` 失败分支,汇聚到 thunk 表 `0x005a9200–0x5a9258`,每个 thunk `bl 0xf8482c`)
- `0xf8482c` 是 Dart 通用 throw 函数(全库被调用 13224 处)
- tamper 短版2 文案构造 helper `0x0051dfa8`,外层 `0x0051d764`,混合层 `0x006418a4`

### 1.2 ★关键反馈:patch `0x00592fd8` 无效

我把 `0x00592fd8` 内的 **22 个 `b.eq` 全部改成 `nop`**(已验证字节进了装机包),但**启动提示框仍然弹出**。

结论:`0x00592fd8` 的 throw 路径**不是**提示框的来源。提示框是 Flutter 的 `showDialog`(正常 Dart 调用),走的是**另一个函数**。

### 1.3 提示框实际显示的文案 = 长版正文(ref=30922)

通过 unflutter 的对象池 dump(test1),已确认启动时弹出的提示框正文是:

```
[ref=30922] (2b/UTF-16LE) "应用已被非法篡改或被第三方APP禁用某些模块或安装在第三方虚拟环境！请重新下载正版并直接安装在真实环境，而且不要使用第三方APP禁用某些模块！如果是环境异常导致的，你可以尝试重启设备和重装APP！"
```

该字符串在 `libapp.so` 的 char data 偏移 `0x1c326e`(file offset = vaddr)。

test2 进一步指出:该长版正文在对象池中紧邻 `checkAppMetaData::` 的 Code 注释(注释偏移 `0x1c3346`),强烈提示 **`checkAppMetaData` 就是构造/弹出该长版提示框的函数**。对象池里还有 7 个 check\*:`checkAppMetaData / checkAppComponentFactory / checkApplication / checkJiagu / checkPMProxy / checkApk / checkIO`。

## 二、本任务要解决的问题(就一个)

**精确找到装载 ref=30922(长版正文)的那条/那几条 .text 指令,从而定位弹窗函数,再定位它的调用者(判定函数),最终给出"决定是否弹框"的条件分支的精确 vaddr + file offset + 机器码字节。**

之所以是缺口:第一轮因为定制引擎 blutter 反序列化失败,没能把对象池 ref 映射到 .text 引用。但 **unflutter 已经成功解析了 snapshot**(它产出了带 ref 编号的字符串表 `unflutter_strings.txt`),说明 unflutter 内部具备 ref→对象→代码引用的映射能力,只是第一轮没把它提取出来。

## 三、建议的方法(任选其一或组合)

### 方法 A(推荐):扩展 unflutter 输出 ref 的 xref

unflutter 已能 dump 出 `[ref=N]` 字符串表。请利用 unflutter 内部已建立的对象图,提取:
- ref=30922(长版正文)被哪些 **对象池 slot** 引用
- 每个 pool slot 又被哪些 **.text 指令**(`ldr xN, [x27, #offset]`,x27=pool pointer)引用

也就是给出:**ref=30922 的所有 .text 引用点地址清单**。

同样的方法也用于校验:ref=6917(短版1"应用已被非法篡改！请重新下载正版！")、ref=26842(短版2"无法更改设置…")的 xref,用来交叉验证方法正确性(test1 已知短版2 的构造在 `0x0051dfa8`/`0x006418a4`,若你算出的 ref=26842 的 xref 落在这两个函数内,说明方法对了)。

### 方法 B:基于 `snapshot.json` 的完整 VM 结构写解析器

`unflutter_dump/snapshot.json` 已给出本定制 VM 的完整结构(compressed-pointers、OldPoolFormat、PreV32Format、CID 表如 TwoByteString=86/Code=17/ObjectPool=21 等)。可据此写一个 Python 解析器,反序列化 isolate snapshot,建立 ref→pool slot→.text xref 映射。

### 方法 C(备选,若有 Android 设备/模拟器):frida 动态

在运行中的 app 里,读 `_kDartIsolateSnapshotData` 的实际 mmap 地址作为 heap_base,反推 ref=30922 字符串对象的运行时地址,再在 .text 内存里搜索引用它的指令。(注:本任务环境无 Android 设备,此方法仅作记录。)

## 四、具体要找的东西(输出)

请在 `分析报告2.md` 里给出:

1. **ref=30922(长版正文)的所有 .text 引用点**(地址清单)。这些指令所在函数 = 长版提示框的构造/builder 函数。
2. 校验:ref=6917 / ref=26842 的 .text 引用点(确认方法正确)。
3. **弹窗 builder 函数的入口地址**(vaddr)。
4. **谁调用了弹窗 builder**(调用者 = 判定函数):给出调用点地址 + 判定函数入口。
5. ★**判定函数里"决定是否弹框"那条条件分支的精确 vaddr + file offset + 机器码(hex)+ 前一条比较指令**。(参照 test1 报告 3.3 节的格式。)
6. 若能顺带定位 7 个 check\* 函数各自的 .text 入口,一并列出(便于后续逐个处理)。

## 五、为什么这个点很关键

拿到"决定弹框"的分支地址和机器码后,我就可以像处理 `0x00592fd8` 那样,把它(们)精确 patch 成 nop,让提示框永不弹出 —— 从而完成应用的完整可用。所以**分支点的精确地址和机器码是最重要的交付物**。
