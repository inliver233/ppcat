# 分析报告29

> 分支：`test1`  
> 角色：纯静态分析。  
> 本轮目标不是回到已证伪的 `0xae30e0 / 0xa54178 / obj44988 直接替换`，而是沿着主控已确认的 `0xa52920 -> 0xa52bb0` 真链路，给出一个比 `entry-return-null` 更稳的 `喵喵饿了` safe-hide 候选。

## 1. 交叉验证结论

与主控反馈、`test3` 第 8 轮/第 9 轮结果对账后，本轮确认并补强了三件事：

1. `0xa52bb0` 这条线是当前唯一值得继续深挖的阅读页喵块链路。
   - 主控运行时硬事实：`0xa52bb0 entry-null -> 阅读页空白`。
   - `0xa52920` 是它唯一的直调包装器：`0xa52b94 -> bl 0xa52bb0`。
2. `obj44988` 绝不能再当作“整块替代 widget”使用。
   - 正确映射：`slot 0x753 -> pool offset 0x3a98 -> ref 44988`。
   - 实二进制中它在多个函数里都只是被写入复合对象的单个字段。
   - 关键例子：`0xb79c60 ldr x17,[x27,#0x3a98]`，随后 `0xb79c64 stur w17,[x0,#0x33]`。
   - 这和主控“把 `obj44988` 直接作为 `0xa52bb0` 返回值会不稳定/经常空白”的运行时现象完全一致。
3. 以后这条线必须以真实 `libapp.so` 反汇编为准，不再盲信 `asm.txt.gz` 的相对跳转文本。

## 2. `0xa52bb0` 的真实结构

实二进制注释反汇编见 [analysis_round17/meow_builder_probe.py](/root/ppcat/repo/analysis_round17/meow_builder_probe.py)。

### 2.1 核心片段

```asm
0x00a52c18: 420080d2  mov     x2, #2
0x00a52c1c: 63c41494  bl      #0xf83da8
0x00a52c20: e10300aa  mov     x1, x0
0x00a52c24: a0035ff8  ldur    x0, [x29, #-0x10]
0x00a52c28: 20f000b8  stur    w0, [x1, #0xf]
0x00a52c2c: 706f4091  add     x16, x27, #0x1b, lsl #12
0x00a52c30: 103642f9  ldr     x16, [x16, #0x468] ; slot 0x368d ref 91683
0x00a52c34: e141bfa9  stp     x1, x16, [x15, #-0x10]!
0x00a52c38: 5953e897  bl      #0x46799c
0x00a52c3c: ef410091  add     x15, x15, #0x10
0x00a52c40: 510080d2  mov     x17, #2
0x00a52c44: 11b000b8  stur    w17, [x0, #0xb]
0x00a52c48: e08d1ff8  str     x0, [x15, #-8]!
0x00a52c4c: 11000094  bl      #0xa52c90
```

这段说明：

1. `0xa52bb0` 不是简单 leaf widget。
2. 它先造一个中间对象，把某个子结果写到 `[#0xf]`。
3. 然后再把 **`[#0xb]` 设成 `2`**，再喂给 `0xa52c90`。

### 2.2 `0xa52c90` 如何解释 `[#0xb]`

```asm
0x00a52cbc: 02b040b8  ldur    w2, [x0, #0xb]
0x00a52cc0: 42801c8b  add     x2, x2, x28, lsl #32
0x00a52cc4: 437c4193  sbfx    x3, x2, #1, #0x1f
0x00a52cd8: 050080d2  mov     x5, #0
0x00a52cf4: bf0003eb  cmp     x5, x3
0x00a52cf8: 8a050054  b.ge    #0xa52da8
```

这里 `0xa52c90` 明确把 `[#0xb]` 当作 **Smi 编码的元素个数** 来读：

- 当前写入值是 `2`，解出来就是 `1`
- `x5` 从 `0` 开始
- 只有当 `count > 0` 时才进入循环
- 若 `count == 0`，会直接走 `0xa52da8` 的空路径，然后仍然从 `0xa52eec` 正常返回

这意味着：**把 `0xa52c40` 的 `mov x17,#2` 改成 `mov x17,#0`，不会破坏返回类型链，但会让这一层 children 聚合器收到空集合。**

## 3. 为什么这比 `entry-return-null` 更安全

主控已经验证过：

- `0xa52bb0 entry-return-null` 会让阅读页空白
- 随便塞 `obj964 / obj51369 / obj91683 / obj44988` 都不稳

本轮静态结论是：

1. `0xa52bb0` 的真正返回对象来自 `0xa52c80` 这一层固定类型包装，不应整体跳过。
2. 更稳的切法不是“让函数不返回 widget”，而是“保留同一返回类型，只让内部子项数变成 0”。
3. `0xa52c90` 自身就包含零元素短路路径，所以这不是盲 patch，而是顺着现有控制流走。

## 4. 主推荐补丁

### H1：把 `0xa52bb0` 传给聚合器的 child 数量从 1 改成 0

- 地址：`0x00a52c40`
- 文件偏移：`0x00a52c40`
- 原字节：`51 00 80 d2`
- 补丁字节：`11 00 80 d2`
- 语义：`mov x17,#2` -> `mov x17,#0`

#### 机制

原流程：

- `0xa52c44 stur w17,[x0,#0xb]`
- `0xa52c90` 读取 `[#0xb]` 并解出 `count=1`
- 进入一次循环，把喵块子项装进最终包装对象

补丁后：

- `[#0xb] = 0`
- `0xa52c90` 在 `0xa52cf8 b.ge 0xa52da8` 直接走空路径
- `0xa52c80` 的最终返回包装类型保持不变，只是内部 children 为空

#### 静态判断

- `静态确定`：这是 `0xa52bb0` 内部最干净的 safe-hide 候选
- `需主控验证`：空 children 最终是彻底不渲染，还是留下空占位，需要装机确认

#### 风险评估

- 低于 `entry-return-null`
- 低于“整个返回值替换成别的 pool 对象”
- 主要风险不是类型崩，而是“空 children 仍保留一个空容器/占位”

#### 主控验证步骤

1. 进入阅读页，确认页面不空白
2. 用 `uiautomator dump` 看 `content-desc="喵喵饿了"` 是否消失
3. 翻页、回退、重进阅读页，确认没有二次崩溃

## 5. 明确证伪：`obj44988` 只能当成员对象

本轮补充的关键反证如下：

```asm
0x00f3a204: 604f5df9  ldr     x0, [x27, #0x3a98] ; slot 0x753 ref 44988
0x00b79c60: 714f5df9  ldr     x17, [x27, #0x3a98] ; slot 0x753 ref 44988
0x00b79c64: 113003b8  stur    w17, [x0, #0x33]
0x004f2420: 714f5df9  ldr     x17, [x27, #0x3a98]
0x004f2424: 11b003b8  stur    w17, [x0, #0x3b]
0x0053f4f4: 714f5df9  ldr     x17, [x27, #0x3a98]
0x0053f4f8: 11f003b8  stur    w17, [x0, #0x3f]
```

结论：

- `obj44988` 在多个地方都是“被塞进别人字段里的成员”
- 它不是“可稳定直接返回的完整 widget”
- 主控之前对它的装机结论是对的，本轮静态分析予以确认

## 6. 不推荐的方向

以下方向本轮不再推荐给主控：

1. `0xa52bb0 entry-return-null`
   - 主控已证：阅读页空白
2. `0xa52bb0 -> 直接返回 obj44988`
   - 本轮已静态证伪：`obj44988` 是成员对象，不是安全整替代
3. 回到 `0xae30e0 / 0xa54178`
   - 主控多轮已证伪
4. 再把 `0xb799b8` 当作阅读页主 build 去 patch
   - 主控已证伪
   - 本轮只保留它作为 `obj44988`“成员而非整对象”的证据来源
