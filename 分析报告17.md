# 第十七轮代码恢复收口报告

> 分支：`test1`
>
> 角色：纯静态分析。本轮不再扩工具链，直接吸收 `backup / test2 / test3` 的现成成果，收敛到“去广告 + reward 特权 + VIP 页面”这条更实用的主线。

## 1. 本轮结论

这轮最重要的修正有四点：

1. 当前目标不该再继续泛搜新的弹窗编排器，而应围绕“**去广告后的残余副作用**”收口：
   - 残留故障窗
   - `喵喵饿了` 点击链
   - reward 特权 / VIP 相关门控
2. `test3` 的对象池数据库已经足够成熟，后续 `test1` 可以直接把它作为一等输入，再用本地 `libapp.so` 逐字节复核关键地址。
3. `VIP` 仍然**不是单一 `isVip` 布尔开关**。更准确的结构是三套并行机制：
   - reward 特权状态链：`0x8cf36c`
   - 逐源 noAd / 广告门控链
   - 捐赠 / 绑定 / 页面分流链：`0xa54178 / 0xb9fc84 / 0x911788`
4. 当前最务实的方案，不是硬改 `0xa54178 / 0xb9fc84`，而是：
   - 先把 app 稳定跑净
   - 再沿着更窄的 reward / 特权门控逐层实验

## 2. 交叉验证后的稳定认识

### 2.1 与 `backup` 运行时硬事实一致

我重新对齐了 `origin/backup:backup/第八轮-去广告去首启弹窗/ROUND8_FACTS.md`，以下判断继续成立：

1. 去广告后的根问题是广告 channel 调用失败，而不是普通 `showDialog`。
2. 残留故障窗最稳的 Dart 侧 choke point 仍然是：
   - `0x920d90: 69 2c 00 54 -> 29 00 00 14`
3. `喵喵饿了` 点击链当前最稳的缩窄点仍然是：
   - `0xbc5dd0..0xbc5dd8: ef a1 00 d1 a0 0b 40 f9 02 70 41 b8 -> e0 03 16 aa fd 79 c1 a8 c0 03 5f d6`

本地 `libapp.so` 已逐字节复核：

- `0x920d90 = 69 2c 00 54`
- `0xbc5dd0 = ef a1 00 d1 a0 0b 40 f9 02 70 41 b8`

### 2.2 与 `test3` 的一致点

`test3` 现在最有价值的不是某个单一 patch，而是它已经把下面这条链打穿：

- `ref -> slot -> .text -> 函数`

这对 `test1` 现在最有用的三条映射是：

1. `ref=27673` 故障正文 -> `0x920d7c`
2. `ref=28525` `喵喵饿了` -> `0xa54178 / 0xae30e0`
3. `ref=11545` `onReward Cheat Triggered!` -> `0x875ee8 / 0x89219c / 0xaa62f4 / 0xaacee8`

这直接把 reward / VIP / 喂喵块三条线串起来了。

### 2.3 与 `test2` 的分歧点

`test2` 早期一些结构判断仍有参考价值，但这轮需要明确两类降级项：

1. `0x913bf8` 不再应被当作故障窗主候选。
2. `0x7e85f4` 过粗，只能当历史候选，不应再放在高优先级。

## 3. reward / 特权 / VIP 的当前正确表述

结合 `test3` 的 `分析报告6_补VIP链路.md` 和我本地的逐字节复核，现在更准确的说法是：

### 3.1 `0x8cf36c` 是底层共享“特权态”计算

本地复核可见：

- `0x8cf36c` 有完整函数序言
- `0x8cf374` 是该函数早期关键分支点
- 它不是一个局部页面专用小函数，而是被多个上层调用者共享消费

这解释了为什么它不能被简单描述成“单一 VIP 开关”。

### 3.2 `0x7e8534 -> 0x7e861c` 是更窄的上层 bool gate

本地反汇编窗口：

```asm
0x007e8618: bl   #0x8cf36c
0x007e861c: tbz  w0, #4, #0x7e86f4
0x007e86e4: add  x0, x22, #0x30
0x007e86f4: add  x0, x22, #0x20
```

这说明：

1. `0x7e861c` 比 `0x8cf374` 更接近上层门控返回
2. 它自带 canonical true / false 尾声
3. 它比直接改 `0x8cf36c` 更窄

所以，reward / 特权方向目前最值得先试的仍然是：

- `0x7e861c: c0 06 20 36 -> 32 00 00 14`

### 3.3 `0xa54178 / 0xb9fc84` 仍不适合粗暴硬改

本地原始字节：

- `0xa54178: fd79bfa9 fd030faa ...`
- `0xb9fc84: fd79bfa9 fd030faa ...`

这两个点我现在仍然不建议直接做入口 return-null 或“猜一个分支就强改”。

原因没变：

1. `0xa54178` 更像 VIP 内容页 / 每日喂喵 / 绑定相关 UI 装配区
2. `0xb9fc84` 更像页面侧厚状态机
3. 它们都不适合被压缩成“单一 VIP 分支”

## 4. 当前推荐方案

### 4.1 第一层：纯净运行基线

这一层的目标不是“拿满特权”，而是先让 app 在去广告后尽量稳定、少弹、少卡。

推荐顺序：

1. `0x920d90: 69 2c 00 54 -> 29 00 00 14`
2. `0xbc5dd0..0xbc5dd8: ef a1 00 d1 a0 0b 40 f9 02 70 41 b8 -> e0 03 16 aa fd 79 c1 a8 c0 03 5f d6`
3. `0xae3560: 40 0c 00 54 -> 1f 20 03 d5`

这三点的角色现在应这样理解：

1. `0x920d90`：压残留故障正文构造链
2. `0xbc5dd0..0xbc5dd8`：直接掐掉 `喵喵饿了` 点击后那条容易卡住的链
3. `0xae3560`：只作为更细的显示链增强项，不再当主解

### 4.2 第二层：reward / 特权实验顺序

如果第一层已经把 app 跑稳，下一层才值得逐步加。

我现在推荐的实验顺序是：

1. `0x875efc: 40 01 20 37 -> 0a 00 00 14`
2. `0x7e861c: c0 06 20 36 -> 32 00 00 14`
3. `0x8cf374: ef 81 01 d1 -> 2a 00 00 14`

排序理由：

1. `0x875efc` 最贴近 `onReward Cheat Triggered!` 这条反作弊链
2. `0x7e861c` 是更窄的上层特权 gate
3. `0x8cf374` 虽然重要，但它在更底层、更共享，应该放到最后

### 4.3 第三层：VIP 页面方向的正确切入

如果后续继续攻 VIP，我建议方向改成：

1. 把 `VIP 页面功能` 和 `reward 特权` 分开验证
2. 继续围绕：
   - `0x911788`
   - `0x8cf36c`
   - `0x7e8534 / 0x7e861c`
   - `0x875efc`
3. 暂时不要再把 `0xb9fc84` 强行写成“恒 VIP true”的单点 patch

更现实的目标是：

- 先让 reward / 特权链成功
- 再观察 VIP 页面里哪些能力随特权态自然放开

## 5. 当前明确不推荐的点

这轮需要把一些旧候选正式降级，避免继续偏航：

1. **不推荐** `0x7e85f4`
   - 过粗
   - 早于 `0x8cf36c` 结果门控
2. **不推荐** `0x885ee0 / 0x886290` 入口 return-null
   - 返回对象会被上层立即消费
   - 不是简单 bool
3. **不推荐** `0xa54178 / 0xb9fc84` 入口 return-null
   - 副作用面太大
4. **不推荐**继续追 `0x913bf8` 作为故障窗主解
   - 这一条已经被 `test3` 的对象池映射纠正

## 6. 本轮收口后的主张

如果把当前所有跨分支成果压成一句话，我现在的结论是：

1. `test3` 提供对象池和函数关系数据库
2. `backup` 提供运行时铁事实
3. `test1` 负责把这些信息压成更保守、可直接 patch-and-test 的顺序

所以，当前最务实的代码恢复路线已经很清楚：

1. 先用 `0x920d90 + 0xbc5dd0..0xbc5dd8 + 0xae3560` 稳定“纯净运行”
2. 再按 `0x875efc -> 0x7e861c -> 0x8cf374` 的顺序逐步试 reward / 特权
3. `VIP` 先别继续追“大一统 isVip”，而应把它当成页面装配层 + 页面分流层 + 共享特权层的组合问题

## 7. 一句话结论

这轮之后，`test1` 的方向已经明确修正：不再继续发散找新工具或新弹窗链，而是以 `test3` 的对象池数据库为底，围绕 `去广告后纯净运行 + reward 特权 + VIP 页面` 三条线做最小、最稳、可回滚的 patch 组合；其中新增最值得优先试的窄点仍然是 `0x7e861c`，而不是继续粗切 `0x7e85f4` 或硬动 `0xa54178 / 0xb9fc84`。
