## 三、★ 签名校验(篡改弹窗)根因+修复现状

### 根因(已完全查清)
- app 自带 `com.gentle.ppcat.hook.PmsHook` 签名伪造框架(App.smali: attachBaseContext调install, onCreate调hookMPM+hookLoadedApk)
- PmsHook.install: 反射 ActivityThread.sPackageManager + Proxy 注入, cert硬编码(原版Gentle Kwan, 正确)
- PmsHook.invoke: 拦截getPackageInfo, 替换signatures=[origSig] + 调makeSigningInfo伪造signingInfo
- **Android14失效点**: makeSigningInfo用旧内部类 `PackageParser$SigningDetails` 反射构造SigningInfo → Android14该构造签名变了 → 抛异常 → catch返回null → signingInfo=null → 反篡改读signingInfo.getApkContentsSigners()拿null → 判定篡改
- **这就是LDPlayer(老系统, 反射有效)能用、平板(Android14, 反射失效)不行的最终原因**

### 已做修复(PmsHook.smali makeSigningInfo)
- 弃反射, 改用Android14 public构造: `new SigningDetails(sig[],2); new SigningInfo(sd)` (类名 android.content.pm.SigningDetails)
- 验证: signature spoof installed + mPM hooked 成功, 篡改弹窗消失
- **但hookLoadedApk仍失败(No field mPackageInfo in LoadedApk), 修复版黑屏卡住**

### 签名校验完整链(frida抓到)
```
Java: getPackageInfo("com.gentle.ppcat", flags=0x8000040) 读签名
  → MessageDigest MD5 算签名hash
  → native(libapp) 经platform channel比对
  → 不匹配 → 0x913bf8(isCheckCode) → 0x91ca6c → 0x9126b0(弹窗触发) → 0x91280c(正文构造)
  → 弹"非法篡改/下载"
```
- 原版签名 hashCode=-1414421055 (Gentle Kwan)
- frida runtime伪造签名(Signature.$new(hex DER), 替换getPackageInfo返回)成功绕过(用frida-java-bridge, frida17需frida-compile打包)

### 待新对话解决
- **PmsHook修复版黑屏**: 诊断为何黑屏(hookLoadedApk失败? 签名伪造引入?). 可能需要:
  a. 也修复hookLoadedApk(Android14 LoadedApk.mPackageInfo改名/换路径)
  b. 或检查makeSigningInfo是否返回了坏SigningInfo导致后续崩
  c. 黑屏可能不是PmsHook导致, 是别的原因(对比: 原版v0.9.0能正常启动, patch版才黑屏——但patch版libapp的overlay patch(0xbd2e1c)在LDPlayer正常, 平板应该也正常)

## 四、★ 喵喵饿了块(猫块)完整认知

### 架构(已完全摸清)
- 猫wrapper = `0xa52920`(经cid虚分派桩0xd931c0调用, blr x30, 这是静态callers=0的根因)
- 猫builder = `0xa52bb0`(wrapper内tail-call), 分配cid-0xaf7 widget, 写field@0xb/0xf
- cat widget cid = **0x85e**, ctor = `0xc62438`, alloc thunk = `0xc62464`
- 显示开关 isHungry = `singleton(thread.field@0x80).field@0x20`
  - Dart bool格式: TRUE=0x...8071(null_base+0x30), FALSE=0x...8061(null_base+0x20)
  - guard = `0x466e2c`(tbz w0,#4, 在共享函数0x466dc8内)
- 猫是framework延迟inflate的overlay, 调用链全是framework(buildScope/attachRenderObject), app build不在backtrace
- "喵喵饿了"文案 ref28525, "饿了喵~"ref19128, "喵喵喂饱了~"ref26746

### 为什么所有"直接patch"都失败(三环境验证)
| 方法 | 结果 | 墙 |
|---|---|---|
| 改0xa52bb0返回值(null/异widget) | 崩/阅读页空白 | cid敏感(父级cid3315严格期望cid0xaf7) |
| patch guard 0x466e2c/0x466e28(改tbz/喂FALSE) | **白屏(LDPlayer+redroid双确认)** | guard在共享函数0x466dc8(所有overlay走) |
| 翻转sing+0x20→FALSE | 藏猫但**触发篡改** | sing+0x20是猫显隐+反篡改共享标志 |
- runtime flip sing+0x20→FALSE藏猫(test3 redroid证实, 平板也证实), 但**转静态就白屏/触发篡改** — sing+0x20/guard都是共享的, 不可碰
