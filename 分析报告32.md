# 分析报告32

> 分支：`test1`  
> 角色：纯静态分析。  
> 本轮目标：根据用户新增的运行时事实，纠正上一轮把“设置页每日喂喵状态”直接等同于“阅读页喵喵饿了挂块门控”的过度推断；吸收 `test3` 最新成果，但把主线转回共享业务状态链。

## 1. 本轮结论

### 1.1 对用户新事实的吸收结果

用户新增事实可以静态上解释成两层：

1. **设置页 `每日喂喵` 与阅读页 `喵喵饿了` 确实是同一子系统的两个入口。**
2. **但它们不是同一个直接 UI gate。**

也就是：

- 设置页入口会走 `每日喂喵 -> 获取 -> reward/跳转 -> 喂饱成功`
- 阅读页入口会消费同一套“喂饱/日状态”派生字段，决定块的外观或是否继续挂载
- 两边共享的是**业务状态链**，不是同一个 widget builder

这和 `test3` 对 `0xa52920 -> 0xa52bb0` 的结论并不冲突：  
`0xa52bb0` 内部没有“今天是否完全不显示”的 hide 分支，说明最终“是否挂这个块”不在它本体里。

### 1.2 `test3` 哪些结论可用，哪些应降级

可直接吸收的部分：

1. `0xa52920 -> 0xa52bb0` 是阅读页喵块消费层。
2. `0xa52a98` 读取 `[thread+0xed0]`，再在 `0xbb8 / 0xbc0` 两套外观之间切换。
3. `0xa52bb0` 本体确实更偏向“构建块”，不是“判定今天是否还应出现”。

需要降级的部分：

1. 继续围绕 `0xa52bb0` 做 safe-hide，不再是本轮主线。
2. 把设置页 `每日喂喵` 的状态写入直接等同于阅读页最终挂块门控，证据还不够。
3. “单纯走 VIP 就能覆盖喂喵块”仍然过宽。当前更扎实的是 reward/daily 子系统，不是整体 `VIP=true`。

一句话：**test3 在 UI 消费层上是对的，但主线应前移到 reward 成功后的共享状态写入与同步层。**

## 2. 共享子系统的确定性链路

### 2.1 两个入口共用同一后续桥：`0xa56e00`

本轮新增的直接 `BL` 调用者扫描结果：

- `0xbc5e98 -> 0xa56e00`
- `0xb682e8 -> 0xa56e00`
- 另外还有 `0xa56de0`、`0xbc61cc`

其中：

1. `0xbc5dc8` 已知是阅读页点击 `喵喵饿了` 后的 reward prompt 入口。
2. `0xb68234` 唯一加载 `ref=22466`，是“无法加载数据...每天喂一次喵...”那条提示正文链。

这说明：

- **阅读页 prompt 链** 和 **每日喂喵提示链** 最终都会汇入 `0xa56e00`
- 所以用户说“设置页每日喂喵”和“阅读页喵喵饿了”是同一套系统，这一点已经有静态铁证支撑

### 2.2 `0xa56e00` 是共享桥，不是最终挂块门

`0xa56e00` 的关键片段：

```asm
0x00a56e34: ldr x0, [x0, #0xb38]
...
0x00a56e64: ldr x1, [x1, #0x2538]
...
0x00a56e8c: ldr x1, [x1, #0xec8]
0x00a56ea0: bl  #0x87c0dc
...
0x00a56eb4: ldr x1, [x1, #0xed0]
0x00a56ec4: ldr x1, [x1, #0xed8]
0x00a56ee0: bl  #0x89281c
```

静态可确认：

1. `0xa56e00` 先看 `[thread+0xb38]`
2. 再看 `[thread+0x2538]`
3. 之后二分：
   - 一支走 `0xec8 -> 0x87c0dc`
   - 一支走 `0xed0 + 0xed8 -> 0x89281c`

因此更稳妥的表述是：

- `0xa56e00` = **daily/reward 子系统的共享桥接器**
- 它负责把不同入口整理后送到统一的后处理
- 它本身还不是“阅读页块是否挂载”的最终判定点

## 3. 成功写状态层与同步层

### 3.1 `0xa56be4 -> 0xa56c34` 仍然是成功 sink

这部分上一轮成立，本轮保留：

- `0xa56be4` 唯一加载 `ref=26746 "喵喵喂饱了~"`
- `0xa56c34` 写：
  - `[thread+0xee0] = true` at `0xa56c50`
  - `[thread+0xea0] = <time-like object>` at `0xa56ca4`

而且新增了一个更重要的事实：

- `0xa56c34` 的直接 `BL` caller 只有 3 个：
  - `0xa56c14`
  - `0xa56f90`
  - `0xa57040`

这说明 `0xa56c34` 是一组“完成/成功包装入口”的共用 sink，而不是单一 toast helper。

### 3.2 真正把阅读页消费字段落盘/同步的是 `0x913bf8` 这一组

本轮最关键的新证据，是对 `0x1028 / 0xed0 / 0xed8` 写回点的全 `.text` 扫描。

结果：

- `str [x1,#0x1028]` 只有一处：`0x913e00`
- `str [x1,#0xed0]` 只有一处：`0x9141e4`
- `str [x1,#0xed8]` 只有一处：`0x914220`

对应片段：

```asm
0x00913df8: ldr x0, [x0, #0x1028]
0x00913e00: str x0, [x1, #0x1028]
```

```asm
0x009141dc: ldr x0, [x0, #0xed0]
0x009141e4: str x0, [x1, #0xed0]
```

```asm
0x00914218: ldr x0, [x0, #0xed8]
0x00914220: str x0, [x1, #0xed8]
```

这组代码有两个含义：

1. `0x1028 / 0xed0 / 0xed8` 不是在 `a56c34` 成功时直接写出的。
2. 它们是**后续同步层**从别处取值，再写回到全局 state 的派生字段。

所以必须修正上一轮的说法：

- `0xee0 / 0xea0` 更像“成功/完成态”的共享字段
- `0x1028 / 0xed0 / 0xed8` 更像“供 UI 消费的同步后字段”
- 阅读页 `0xa52920` 读到的是后者，不是前者

### 3.3 `0xa52920` 只是消费层，不是成功写入层

阅读页链核心片段：

```asm
0x00a52954: ldr x2, [x2, #0x1028]
...
0x00a52a98: ldr x2, [x2, #0xed0]
0x00a52ab4: ldr x3, [x27, #0xbb8]
0x00a52abc: ldr x3, [x27, #0xbc0]
```

可确认：

1. `0x1028` 是更上游传下来的门或状态对象
2. `0xed0` 控制 `0xbb8 / 0xbc0` 两套外观切换
3. 所以 `0xa52920 -> 0xa52bb0` 解决的是“块长什么样/按什么状态渲染”
4. 它不能单独解释“为什么看完奖励后当天完全不见”

因此：

- **test3 把它定位为 builder 核心是对的**
- **但阅读页真正的“是否挂块”门，应在消费它之前的更上层**

## 4. 设置页 daily 状态与阅读页喵块的关系

### 4.1 设置页显示链仍然成立，但应降级为“同子系统状态展示”

`0xb9fc84` 内：

- `ref=17423 "isDaily:"` at `0xb9fdb8`
- `ref=26299 "未获取"` at `0xb9fe54`
- `ref=8726 "喵喵喵~"` at `0xba09e8`

`0xa54178` 内：

- `ref=21707 "每日喂喵"` at `0xa55c80`
- `ref=28525 "喵喵饿了"` at `0xa55cbc`
- `ref=2878 "24小时内累计浏览"` at `0xa555ac`
- `ref=13199 "次推荐信息，即可获取"` at `0xa555e4`

这说明：

1. 设置页确实展示这套 daily/reward 状态
2. 但它更多是**状态展示页 / 页面混合状态机**
3. 不能再把 `0xb9fc84` 直接说成“阅读页块最终隐藏门”

### 4.2 对用户新事实的最佳静态解释

结合用户“设置页喂完后，阅读页块也消失”的事实，当前最稳的静态解释是：

1. 设置页 `每日喂喵` 和阅读页 `喵喵饿了` 共用 `0xa56e00` 后处理桥
2. reward 成功后，`0xa56c34` 写入一组共享完成态字段（至少 `0xee0 / 0xea0`）
3. 后续同步层 `0x913bf8 ... 0x914220` 把某些派生状态刷新到 `0x1028 / 0xed0 / 0xed8`
4. 阅读页 `0xa52920` 再消费这些派生字段，决定块的状态外观或上游是否继续挂载

也就是：

- **同一子系统：是**
- **同一个直接 gate：不是**
- **共享的是“reward 成功后的状态传播”链：是**

## 5. 对 `VIP / 特权` 方向的更新

### 5.1 用户思路“VIP/特权可能帮助去掉喵块”并不算错

因为当前静态上能确认：

1. `rewardTime -> getRewardTime -> 0x8cf36c` 是一条真实的奖励特权链
2. `0x7e8534` / `0xa54178` / `0xb9fc84` 都会消费这一链
3. reward 成功后确实会影响 daily/reward 子系统

所以“从特权/奖励成功方向入手”是合理的。

### 5.2 但仍不建议把目标泛化成“整体强开 VIP”

原因：

1. `0xb9fc84` 是混合状态机，不是单点 `isVip`
2. `绑定设备 / 捐赠 VIP / 奖励特权 / daily 喂喵` 不是一回事
3. 当前与喵块最直接相关的，仍是：
   - `0xbc5dc8 -> 0xa56e00`
   - `0xa56be4 -> 0xa56c34`
   - `0x913bf8 / 0x914190` 同步层
   - `0xa52920` 消费层

所以后续若要继续深入 `VIP / 特权`，建议只围绕：

- `rewardTime`
- `isDaily`
- `0x8cf36c`
- `0xa56e00`

而不是再回到宽泛的 `VIP=true`。

## 6. 本轮修正后的方案建议

### 6.1 若目标是“恢复自然业务链”

最值得继续追的不是 `0xa52bb0`，而是：

1. `0xa56e00`
   - 共享桥接器
2. `0xa56c34`
   - 成功 sink
3. `0x913bf8 -> 0x914220`
   - 状态同步层
4. `0xa52920`
   - 阅读页消费层

其中当前最缺的一块，是：

- `0x913bf8 / 0x914190` 这组从哪里取值，如何把成功态翻译成 `0x1028 / 0xed0 / 0xed8`

### 6.2 若目标是“给主控一个可试的业务向 patch 方向”

当前更符合代码恢复方向的建议优先级：

1. 继续围绕 reward 成功链，而不是继续只做 safe-hide。
2. `0x875efc` 仍可保留为 reward anti-cheat 关键分支候选，但它解决的是“假成功为何不生效”，不是 UI 消费层。
3. 若要进一步做字节级试验，应优先观察：
   - `0xa56c34` 成功后
   - `0x913bf8 / 0x914190` 是否同步刷新 `0x1028 / 0xed0 / 0xed8`

## 7. 本轮最终结论

本轮最重要的修正是：

1. **接受用户新事实：设置页 `每日喂喵` 与阅读页 `喵喵饿了` 是同一子系统。**
2. **否定上一轮的过度推断：设置页 daily 状态链不等于阅读页最终挂块门控。**
3. **明确分层：**
   - `0xbc5dc8 / 0xb68234 -> 0xa56e00`：共享入口桥
   - `0xa56be4 -> 0xa56c34`：成功写状态
   - `0x913bf8 / 0x914190`：同步出 `0x1028 / 0xed0 / 0xed8`
   - `0xa52920 -> 0xa52bb0`：阅读页消费这些派生字段

一句话收口：**阅读页喵块与设置页每日喂喵确实同源，但真正共享的是 reward 成功后的状态传播链，而不是同一个直接 UI gate；因此后续最有价值的突破口是 `a56e00 -> a56c34 -> 913bf8/914190 -> a52920` 这条业务状态线，而不是继续只在 `a52bb0` 上做视觉 safe-hide。**
