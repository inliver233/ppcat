4. 把 `0xbc0f40 / 0xbd5a24 / 0x7e1464` 入口改 `return null`

## 3. 新增主线 A：故障 body `0x920d7c`

### 3.1 为什么这一条比继续 NOP `showDialog` 更值得试

`test3` 共享 xref 库与本地 `capstone` 一致：

- `0x920d7c` callers 只有 3 个：
  - `0x920b3c`
  - `0xc4cb30`
  - `0xc4d22c`
- 中文正文字符串 `ref=27673` 的唯一 `.text` 落点就是它
- 它函数内部**本身就有现成的 `return null` epilogue**：

```asm
0x00920e34: e00316aa    mov     x0, x22
0x00920e38: ef031daa    mov     x15, x29
0x00920e3c: fd79c1a8    ldp     x29, x30, [x15], #0x10
0x00920e40: c0035fd6    ret
```

这意味着：相比粗暴改入口，更稳的做法是让它**走自己原生的 null-return 路**。

### 3.2 三个 caller 的静态差异

#### caller 1: `0x920b3c`

关键点：

```asm
0x00920d18: f001bfa9    stp     x16, x0, [x15, #-0x10]!
0x00920d1c: 18000094    bl      #0x920d7c
0x00920d20: ef410091    add     x15, x15, #0x10
0x00920d24: a0835bf8    ldur    x0, [x29, #-0x48]
```

特点：

- 调用前已经完成 shadow-stack push
- 调用后无条件 `add x15,#0x10`
- **返回值没有被直接消费**

所以：

- `0x920d1c: 18 00 00 94 -> 1f 20 03 d5`

是一个高价值候选：

- 栈仍平衡
- 不依赖 callee 返回值
- 只切掉这一个 caller 对故障 body 的构造调用

#### caller 2: `0xc4d22c`

关键点：

```asm
0x00c4d294: f001bfa9    stp     x16, x0, [x15, #-0x10]!
0x00c4d298: b94ef397    bl      #0x920d7c
0x00c4d29c: ef410091    add     x15, x15, #0x10
0x00c4d2a0: e00316aa    mov     x0, x22
```

特点：

- 同样是“先 push、后 add x15”
- **返回值立刻被丢弃**

所以：

- `0xc4d298: b9 4e f3 97 -> 1f 20 03 d5`

也属于高价值候选。

#### caller 3: `0xc4cb30`

关键点：

```asm
0x00c4cc7c: 4050f397    bl      #0x920d7c
0x00c4cc84: e10300aa    mov     x1, x0
0x00c4ccac: 7c9ae097    bl      #0x47369c
```

特点：

- 这条 caller **直接消费返回值**
- 不能像前两个 caller 一样把 `BL` 直接改 `NOP`

若只想把它局部压成 null body，最多可试：

- `0xc4cc84: e1 03 00 aa -> e1 03 16 aa`

即：

- `mov x1, x0 -> mov x1, x22`

但这条不是本轮主推荐，因为 `ROUND8_FACTS` 已证明此类 `showDialog` 路线不是故障窗根因。

### 3.3 `0x920d7c` 本体的两档候选

如果只 patch caller 仍不够，本轮建议优先试“让 callee 走自己原生 null-return 路”，而不是改入口。

#### 候选 A1：较保守

- 地址：`0x920e10`
- 原字节：`a0 01 20 37`
- 原指令：`tbnz w0,#4,#0x920e44`
- patch：`09 00 00 14`
- 含义：改成 `b 0x920e34`，强制走本函数自带 null-return epilogue

理由：

- 这条位于多个前置检查之后
- 保留了前面的 `0x7e41c0 / 0x911788 / runtime type` 路径
- 只在“准备真正拼故障 body 内容”前强制返回 null

#### 候选 A2：更强硬

- 地址：`0x920dac`
- 原字节：`40 04 20 36`
- 原指令：`tbz w0,#4,#0x920e34`
- patch：`22 00 00 14`
- 含义：改成 `b 0x920e34`

理由：

- 更早地把这个 builder 固定为 null-return
- 仍然复用本函数原生 epilogue，栈逻辑自洽

#### 不推荐：入口直接 `return null`

和前几轮已经被证伪的入口 patch 教训一致，本轮不再推荐把 `0x920d7c` 入口直接改成 `MOV X0,X22;RET`。这里更稳的静态选择是：

- 让函数建帧后，跳到它自己原生的 `0x920e34`

### 3.4 主控建议的测试顺序

对故障窗，本轮给主控的顺序是：

1. 先试 caller 层
   - `0x920d1c -> NOP`
   - `0xc4d298 -> NOP`
2. 如果故障窗仍在，再升到 callee 层
   - 先试 `0x920e10 -> 09000014`
   - 再试更强的 `0x920dac -> 22000014`

这比回头再碰 `0xbc0f40 / 0xbd5a24 / 0xc4cb30` 的 `showDialog BL` 更符合第八轮的运行时事实。

## 4. 新增主线 B：“喵喵饿了”常驻块 `0xae30e0`

### 4.1 为什么这一条比 reward 弹窗更适合先做

用户第八轮的明确诉求之一是：

- 常驻“喵喵饿了”块如果短期还做不到奖励链完整恢复，就先让它**消失 / 不遮挡 / 不可点击**

这条线上，`test3` 的共享 xref 库给出了一个比前几轮更明确的落点：

- `0xae30e0`
  - `callers = 0`
  - `bool_ret = True`
  - 命中字符串包含 `喵喵饿了`

本地 `capstone` 再补了一步：

- 真正装载 `ref=28525 "喵喵饿了"` 的点在 `0xae36fc`
- 它只出现在 `cmp w0,#0x14` 之后进入的专用支路内

### 4.2 关键分支

```asm
0x00ae3128: 1f500071    cmp     w0, #0x14
0x00ae312c: c10d0054    b.ne    #0xae32e4
```

对应的特殊支路中，最终会走到：

```asm
0x00ae36f8: 71b34091    add     x17, x27, #0x2c, lsl #12
0x00ae36fc: 31f246f9    ldr     x17, [x17, #0xde0] ; ref=28525 "喵喵饿了"
0x00ae3700: 11b000b8    stur    w17, [x0, #0xb]
```

这说明：

- `w0 == 0x14` 时，函数会进入一个“喵喵饿了”专用 widget 构造支路
- 否则走 `0xae32e4` 的通用 fallback 分支

### 4.3 候选 patch

- 地址：`0xae312c`
- 原字节：`c1 0d 00 54`
- 原指令：`b.ne #0xae32e4`
- patch：`6e 00 00 14`
- 含义：改成 `b #0xae32e4`

效果预期：

- 无论 `w0` 是否等于 `0x14`
- 都强制跳过“喵喵饿了”专用支路
- 继续走函数原本的 fallback 构造路径

### 4.4 风险评级

这条 patch 的静态信号很强，但我仍把它定为“**高信号候选，需主控运行时确认**”，理由是：

1. `0xae30e0` 仍是零-caller 的间接分发 widget builder
2. `w0 == 0x14` 对应的 case 语义虽然从字符串上高度像“喵喵饿了”，但还没有运行时 case-id 佐证

所以本轮建议是：

- 可以把它排在故障 body patch 之后，作为“隐藏常驻块”的第一优先尝试
- 但不要把它包装成已证实结论

## 5. VIP：本轮没有新的单点突破

这轮 `test1` 不再假装有新的单点 `isVip` patch。跨分支仍然一致：

- `0xa54178`：VIP / 喂喵内容页装配
- `0xb9fc84`：VIP / 绑定 / daily / reward 混合状态机
- `0x911788`：共享门控器
- `0x8cf36c`：奖励特权 / 免广告状态对象
- `0xaa5848`：隐藏测试面板 / reward anti-cheat 同簇函数

所以本轮对 VIP 的结论仍然是：

- **没有干净单点**
- 真要继续推进，最值得投的是：
  - `0xaa5848` 隐藏面板如何进入
  - `0x8758bc` / `0xaa5848` / `0xaace80` 这一组 anti-cheat / reward 链

## 6. 本轮给主控的建议顺序

如果按“最少试错、最贴近当前运行时事实”的顺序，本轮推荐：

1. **先测喵块隐藏**
   - `0xae312c: c10d0054 -> 6e000014`
2. **再测故障 body caller 层**
   - `0x920d1c: 18000094 -> 1f2003d5`
   - `0xc4d298: b94ef397 -> 1f2003d5`
3. **若故障窗仍在，再升到 callee 层**
   - `0x920e10: a0012037 -> 09000014`
   - 若还不够，再试：
   - `0x920dac: 40042036 -> 22000014`
4. **Java 侧长期仍走精确 stub**
   - 逐方法返回精确类型
   - reward 走 `EventChannel`
   - 继续绕 anti-cheat，而不是再回头碰 `0x7e8540`
