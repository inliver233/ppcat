### 1.4 喵喵饿了常驻块 Widget Builder：0xae30e0

**结构分析：**

```
0xae30e0: STP X29, X30, [X15,#-16]!   ← 函数入口
0xae3120: LDR X0, [X29,#16]            ← 加载第3参数 (配置枚举)
0xae3124: STUR W0, [X1,#15]            ← 存入字段 #15
0xae3128: CMP W0, #0x14                ← ★ 关键判断：参数 == 20?
0xae312c: B NE, 0xae32e4               ← 不等于 0x14 → 跳转 PATH_B
0xae3130-0xae32e0: PATH_A              ← 等于 0x14 → 构建完整 widget 树
    (含大量 pool 加载 + BL 子 widget 构造)
0xae32e4-0xae33fc: PATH_B              ← 不等于 0x14 → 构建简化 widget
```

**调用方式：** callers=0（全部通过 pool 间接 BLR，即 Dart 闭包/virtual dispatch）

**PATH_A (0xae3130-0xae32e0) 行为：**
- 调用 0xae3130: BL closure → 构建主容器
- 加载 pool page 0xe000, 0xd000, 0x2a000, 0xf000 的多个子 widget
- 组装成完整的喵喵训练/饥饿状态 widget（含图片、状态文本、按钮等）
- 返回组装好的 widget 树

**PATH_B (0xae32e4+) 行为：**
- 加载不同的 pool 配置
- 构建简化/备选 widget
- 调用 BLR X30 间接分发

**关键发现：** 0xae30e0 是一个 **switch builder**，根据配置值（0x14=20 或其他）构建不同 widget。如果 0x14 对应"喵喵饿了"常驻块，则：
- 将 0xae312c 的 `B NE` 改为 `B`（无条件跳转）→ 永远不走 PATH_A → 喵喵饿了块不构建
- 或：修改 CMP 条件让比较永远不成立

**风险：** PATH_A 可能也构建了其他有用的 UI（如喵喵训练页）。需要确认 0x14 是否**仅**对应该常驻块。

## 二、推荐 Patch 方案

### 2.1 ★ 喵喵饿了常驻块消失（推荐先验证）

**方案 A-1（高风险高杠杆）：篡改 builder 条件**

```python
# 0xae30e0: 强制永远走 PATH_B（不构建 PATH_A widget）
# 原指令: B NE, .+0x1b8 (0xae312c: c1 0d 00 54)
# 改为:   B .+0x1b8 (无条件跳转)
d[0xae312c:0xae312c+4] = bytes.fromhex('c1 0d 00 14')  # B.NE → B
# 或直接改为 CMP W0, #0x14 → CMP W0, #0x0 (永不等于被传入的非零值)
# 原: 1f 50 00 71
# 改为: 1f 00 00 71
d[0xae3128:0xae3128+4] = bytes.fromhex('1f 00 00 71')  # CMP #0x14 → CMP #0x0
```

⚠ **风险：** PATH_A 可能不仅构建喵喵饿了块。如果它也构建 VIP/训练页面，那些也会消失。

**方案 A-2（低风险保守）：在 caller 层判断**

由于 0xae30e0 通过 pool BLR 调用且 callers=0（无直接 BL caller），无法直接找到并 patch caller。此方案不可行。

**方案 A-3（中风险精细）：找喵喵饿了文本的精确加载点**

ref=28525 "喵喵饿了" 在 pool page 0x2c000 (slot 0x59bc, low_off=0xde0)。需要精确反汇编 PATH_A 找到哪个子函数加载此 ref 并构建文本 widget，然后 NOP 那个子函数调用。**需 capstone 级精细分析，下轮深入。**

### 2.2 ★ 全局错误处理截断（方向A）

**方案 A-4（最高杠杆但高风险）：截断全局 onError 回调**

在 0xcc0010-0xcc67f8 区间定位 `FlutterError.onError` 设置点。如果回调是一个单入口函数，可以将其第一条指令改为 `RET`（return immediately），使所有未捕获异常静默。

```python
# 候选：如果回调函数在 0xccXXXX
# 原: STP X29, X30, [X15,#-16]!  (fd 79 bf a9)
# 改为: RET X30                     (c0 03 5f d6)
# d[0xccXXXX:0xccXXXX+8] = bytes.fromhex('e00316aac0035fd6')
```

⚠ **需要运行时确认：** 回调函数的确切地址需要 test1 Ghidra 反编译或主控 logcat Dart 栈帧定位。截断所有未捕获异常可能让**真错误**也被静默。

**方案 A-5（中风险）：截断 MissingPluginException 的 throw**

在 0xca8bdc（MissingPluginException 构造函数）入口改为直接返回 null：
```python
# 原: STP X29, X30, [X15,#-16]! (fd 79 bf a9)
# 改为: MOV X0, X22; RET           (e0 03 16 aa c0 03 5f d6)
d[0xca8bdc:0xca8bdc+8] = bytes.fromhex('e00316aac0035fd6')
```

⚠ **注意：** 这会让所有 MissingPluginException **都**变成 null。调用者收到 null 后可能在其他地方崩溃（类似 V3 stub 的 null 崩溃问题）。但可能比抛异常→全局错误处理→故障窗要好。

### 2.3 方向D：Dart 侧 NOP 广告 invoke

**方案 D-1：在方法名级别截断**

所有广告方法（showSplashAd/loadFullAd 等）的方法名字符串集中在 slots 0x3232-0x32e8。广告引擎 dispatcher 0x8758bc 使用这些方法名查找配置并发起 invokeMethod。

如果能精确定位 dispatcher 中的 **invokeMethod 调用点**，NOP 它就能阻止所有广告 channel 调用，且不会影响非广告 channel。

**当前进展：** test3 已定性 dispatcher 为 0x8758bc（4SDK×4类型 dispatch），但 invokeMethod 的精确调用地址尚未被任何分支静态确认。需要反汇编 0x8758bc 全文搜索 BLR X30（间接调用 invokeMethod）模式。

### 2.4 总结：安全 NOP 模式（已验证）

所有 patch 遵循主控验证过的安全 NOP 模式：
```
2×STP push → BL → ADD X15 pop → MOV X0,X22 → RET
```
等价字节：`fd79bfa9 fd030faa [8-byte BL] ef034191 e00316aa fd79c1a8 c0035fd6`

NOP 替换：将中间 BL 改为 `1f2003d5` (NOP)

## 五、方向D：Dart侧NOP广告invoke调用（关键突破）

### 5.1 广告 async 函数统一调用模式

通过反汇编 6 个广告函数发现统一调用链模式：

```
STP X29,X30 → 状态机序言 → ... → 
BL <invoke_channel>  (0x886290 或 0x885ee0) → 
BL <showDialog>      (0x47369c)             → 
BL <alloc>           (0xf83da8)             → 
BL <another_dialog>  (0x4676c4 或 0x4670b0) →
MOV X0,X22; RET
```

| 函数 | 地址 | invoke | showDialog | alloc | altDialog |
|------|------|--------|------------|-------|-----------|
| showSplashAd | 0x888b18 | 0x888c10→0x886290 | 0x888c3c→0x47369c | 0x888cc8→0xf83da8 | 0x888cec→0x4676c4 |
| ad_async_1 | 0x8880c0 | 0x8881c4→0x885ee0 | 0x8881f0→0x47369c | 0x88827c→0xf83da8 | 0x8882a0→0x4676c4 |
| ad_async_2 | 0x88862c | 0x888724→0x885ee0 | 0x888750→0x47369c | 0x8887dc→0xf83da8 | 0x888800→0x4676c4 |
| ad_controller | 0x8863e8 | — | 0x886564→0x47369c | 0x8864c4→0xf83da8 | 0x8865e4→0x4670b0 |
| ad_dispatcher | 0x8758bc | — | 0x875934→0xf82494 | 0x8758d8→0xf82f38 | 0x875a00→0xf841ec |

**关键发现：** 0x47369c（showDialog）在所有广告函数中被直接 BL 调用。但主控已验证 NOP 这些 showDialog 无法阻止故障窗（故障窗走全局错误处理，不走这里的 showDialog）。

### 5.2 方案D-1：入口 return-null NOP（推荐先试）

每个广告 async 函数都已经在特定路径返回 `MOV X0, X22` (null)。将入口直接改为 return null 是安全的：

```python
# 方案D-1a: 激进——所有广告 async 函数入口 return null
ad_funcs = {
    0x888b18: 'showSplashAd',
    0x8880c0: 'ad_register_ksad_like',
    0x88862c: 'ad_initAd_like',
    # 这些函数在正常情况下都返回 null (MOV X0,X22)
}

for addr in ad_funcs:
    # 原: STP X29, X30, [X15,#-16]! + MOV X29, X15
    # 改为: MOV X0, X22; RET
    d[addr:addr+8] = bytes.fromhex('e00316aac0035fd6')
```

⚠ **风险：** 如果这些函数也被非广告路径调用，会直接返回 null 可能导致其他功能异常。需运行时确认每个函数是否**仅**用于广告。

### 5.3 方案D-2：NOP invokeMethod 调用（更精细）

每个函数内的第一个 BL 是 channel invoke 调用。NOP 它 + 让函数继续走到 return null：

```python
# showSplashAd @ 0x888b18:
# 0x888c10: BL 0x886290 → 改为 NOP (1f2003d5)
d[0x888c10:0x888c10+4] = bytes.fromhex('1f2003d5')

# ad_async_1 @ 0x8880c0:
# 0x8881c4: BL 0x885ee0 → 改为 NOP
d[0x8881c4:0x8881c4+4] = bytes.fromhex('1f2003d5')

# ad_async_2 @ 0x88862c:
# 0x888724: BL 0x885ee0 → 改为 NOP
d[0x888724:0x888724+4] = bytes.fromhex('1f2003d5')
```

⚠ **风险：** NOP invoke 后 X0 可能是未定义值（invoke 的返回值），后续代码可能基于 X0 做分支判断导致不可预测行为。

### 5.4 方案D-3（推荐！）：保留 invoke，但 NOP showDialog

```python
# 在 showDialog BL 处 NOP（主控已验证安全模式）
for addr in [0x888c3c, 0x8881f0, 0x888750, 0x886564]:
    d[addr:addr+4] = bytes.fromhex('1f2003d5')
```

✅ 这些 NOP 在主控的 0xbc0f40/0xbd5a24 上已证明安全（2×STP push + BL + ADD X15 pop + MOV X0,X22 + RET = 安全模式）。但主控已验证 NOP showDialog **无法阻止故障窗**。此方案仅用于消除广告弹窗的 UI 展示，不能解决 MissingPluginException。

## 六、方向B：逐方法精确V3 返回类型（基于反编译+Dart语义）

```java
success(result);
```

⚠ **注意：** 此方案虽然解决了"类型匹配"问题，但主控已验证返回 Boolean.TRUE 仍可能导致下游 `type 'bool' is not subtype of 'qsa?'`。这个 'qsa' 报错可能来自 METHODS 列表中某个未知方法。需要将 showSplashAd 等方法改为返回 **正确结构的 HashMap**（模仿 qsa 对象）而非 Boolean.TRUE。

### 6.4 方案B-2：统一返回空 HashMap + 消除 qsa 依赖

如果所有方法都返回空 HashMap（而不是 bool/null/Map），Dart 端索引 HashMap key 时最多返回 null 而不会类型崩溃。这是最小可行方案：

```java
// 所有方法统一返回值
Map<String, Object> emptyMap = new HashMap<>();
success(emptyMap);  // Dart 端: result["key"] → null, 不崩溃
```

风险：Dart 端有 `result!` 强制非空，空 Map 本身非 null 所以 `!` 不会崩，但 `result["field"]!` 会崩。

## 七、补充：喵喵饿了常驻块 NOP 候选补充

### 7.1 新增备选方案

除了方案 A-1（改 0xae312c 的 CMP/B.NE），还发现 0xa54178（VIP ctor）也加载 ref=28525 "喵喵饿了"（通过 pool page 0x2c000）：

```
0xa542ec: ADD X17, X27, #0x2c, LSL #12  ← pool page 0x2c000
0xa542f0: LDR X17, [X17,#3264]           ← slot 0x5998
```

slot 0x5998 在喵喵饿了区域（0x59bb-0x59cf），0xa54178 是 VIP 页面构造函数。如果 NOP 掉对 0xae30e0 的 BLR 调用（在最上游的 widget 构建时），常驻块不会出现在 widget 树中。但 0xae30e0 callers=0（走 pool），静态无法直接定位 BLR。

### 7.2 推荐先试方案

优先顺序：
1. **方案 A-1**：改 0xae312c B.NE→B（最简单，4 字节）
2. 如果无效：说明 PATH_A 不构建喵喵饿了块 → 试 0xae3128 CMP→CMP #0x0
3. 如果仍无效：喵喵饿了块的构建在其他函数中 → 需下轮 capstone 精细追踪
