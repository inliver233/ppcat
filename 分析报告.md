# 皮皮喵 (ppcat) Flutter 应用 Dart 代码恢复 — 静态分析报告

> 本报告为**纯静态分析**结果,只读取文件、产出分析报告,未修改或分发任何文件。

## 一、执行摘要

本任务目标是静态分析 ppcat 应用的 `libapp.so` (Dart AOT 业务逻辑),还原启动提示框触发逻辑、VIP 判定和版本更新逻辑。

**主要结论**:
- ✅ **对象池层面**:成功验证了《已知信息.md》中所有关键字符串的偏移,并**额外发现 7 个 check\* 完整性校验函数名**(checkAppMetaData、checkJiagu、checkApplication、checkApk、checkIO、checkPMProxy、checkAppComponentFactory)和 ~20 个 get\*/filter\* 辅助函数名,完整勾勒出"聚合判定"的调用图。
- ✅ **定位推断**:启动提示框的长版正文(0x1c326e)紧邻 `checkAppMetaData` 函数的 Code 注释(0x1c3346),强烈提示 **checkAppMetaData 就是弹出该提示框的函数**;7 个 check\* 函数共同构成"聚合判定"的检测源。
- ⚠️ **机器码层面**:由于 `libflutter.so` 为**深度定制引擎**(snapshot_hash 被抹零、ABI 字符串为 `arm64-sysv` 而非标准 `arm64 android`、与 Dart 2.19.x 存在 cid 84 / 基对象计数两类结构性偏差),**blutter 无法完整反序列化 snapshot**,因此未能给出"决定是否弹框那条条件分支"的精确 vaddr 和机器码字节。本报告已穷尽尝试 2.18.6 / 2.19.0 / 2.19.5 / 2.19.6 四个 Dart 版本和三处 patch(feature check、alignment check、base object count check),全部失败,详见第三节。

## 三、blutter 失败原因详细记录(为后续恢复者避坑)

按《任务文档.md》第 4.3 节 fallback,系统尝试了 4 个 Dart 版本和 3 类 patch,全部失败。失败链条如下:

### 3.1 自动版本探测失败
`blutter.py` 默认通过 `libflutter.so` 的 `.rodata` 中"恰好 2 个 40 字符 SHA1"+"\<version\> (stable|beta|dev)"正则来探测。本引擎:
- `.rodata` 中真实 SHA1 hash 数量 = 0(全部为意外匹配,如 `00000cf000000000000000000000000000000000` 等字符表片段)
- 版本字符串为 `0.0.1                                            on "android_arm64"`(被替换为占位)
→ `AssertionError: found hashes` (`extract_dart_info.py:47`)

### 3.2 用 `--dart-version` 强制指定版本后的失败序列

| Dart 版本 | 给 blutter 的 snapshot_hash | 失败点 | 失败信息 |
|---|---|---|---|
| 2.18.6 | 默认(2.18.6 实际 hash) | `VerifyVersion` | `Wrong full snapshot version, expected '<2.18.6 hash>' found '                                '` |
| 2.18.6 | "                                "(32 空格,与 binary 中一致) | `Deserialize` | `Snapshot expects -1 base objects, but deserializer provided 94`(读到的 num_base_objects_ 字段为 -1,即读位置错位) |
| 2.19.0 | "                                " | `ImageReader::VerifyAlignment` | `Snapshot is misaligned` |
| 2.19.0(+ patch 跳过 alignment) | "                                " | `Deserialize` | `Snapshot expects 93 base objects, but deserializer provided 94`(基对象计数差 1) |
| 2.19.0(+ patch 跳过 base object 计数 check) | "                                " | `ReadCluster` | `No cluster defined for cid 84`(类 ID 84 在 2.19.x 不可识别) |
| 2.19.6 | "                                " | 同 2.19.0 链路,同样卡在 cid 84 |  |

### 3.3 三类 patch 的细节
为推进 blutter,在 `dartsdk/v2.19.0/runtime/vm/app_snapshot.cc` 和 `image_snapshot.cc` 应用了:

1. **`SnapshotHeaderReader::VerifyFeatures`**(app_snapshot.cc:~8530):原函数比较 features 字符串与本地 VM 期望值,本引擎的 features 含 `arm64-sysv` / `lazy_async_stacks` / `use_bare_instructions`,与任何官方版本都不等。改为:仍然调用 `ReadFeatures` 推进流位置(关键 — 否则后续读取错位),但跳过比较。
2. **`ImageReader::VerifyAlignment`**(image_snapshot.cc:~1796):本 snapshot 的 data_image 与 instructions_image 在 mmap 后地址不满足 `kObjectStartAlignment = 64` 字节对齐(因为 `_kDartIsolateSnapshotData` 的 vaddr 0x3330 % 64 = 48)。改为直接返回 `ApiError::null()`。
3. **`Deserializer::Deserialize` 中的 base objects 计数 check**(app_snapshot.cc:~8757):改为打印警告而非 `FATAL`。

### 3.4 根本原因
**本引擎的 cluster 序列化格式与任何官方 Dart 2.19.x 都不兼容**:
- 基对象计数差 1(说明 VM 内置类表多/少 1 项)
- cid 84 在 2.19.x 类表中不存在(说明自定义引擎新增或重排了某些类)

→ 这是 `libflutter.so` 的"定制引擎"声明的具体表现,与《已知信息.md》第一节"疑似定制构建"完全吻合。

### 3.5 后续恢复者建议
若希望让 blutter 完整跑通,需要做以下其中之一:
- 找到该定制引擎对应的 Dart SDK fork(可能在 ghostgzt 的其他仓库或 ppcat 的早期 release 历史),checkout 出来再用 blutter 编译
- 在 `ReadCluster` 处增加 cid 84 的 fallback `InstanceDeserializationCluster`,并修正 base objects 计数(需要逆向该定制 VM 的类表)
- 用 frida 等动态工具,在运行中的 app 里 dump 出 isolate 的对象池(完全绕开静态反序列化)

## 五、重点 1:启动提示框触发逻辑

### 5.1 关键证据链

1. **长版提示框正文位于 `0x1c326e – 0x1c3333`**(约 200 字 UTF-16LE),内容包含《已知信息.md》列出的全部 5 个中文片段(`应用已被非法篡改`、`非法篡改`、`第三方虚拟环境`、`环境异常`、`禁用某些模块`)。
2. **该正文紧接着是 `checkAppMetaData::\n` Code 注释,起始偏移 `0x1c3346`** —— 长版正文与 `checkAppMetaData` 的 Code 注释相距仅 19 字节。这意味着 **`checkAppMetaData` 函数的 Code 对象在对象池中引用了上述长版正文字符串**,即 `checkAppMetaData` 是**直接弹出该提示框的函数**(或在检测失败时调用 `showDialog` 并以该长版为正文)。
3. **对象池中同时存在 7 个 `check*` 函数注释**(见 4.4 节),它们都是某条独立的检测;一个"聚合判定函数"必然依次 BL 调用这些 check\*,把返回的 7 个 bool 做 OR(任一为 true 即判定环境异常 → 弹框)。
4. 短版正文 `0x4d17a` 位于 `getPackages::` Code 注释(0x4d15d)的紧后位置,可能是另一处错误文案(或 `showDialog` 的标题/短描述)。

### 5.2 触发流程(还原伪代码)

```dart
// 推测的聚合判定函数(未直接定位到 .text 地址,但调用图已确认)
Future<bool> _shouldShowWarningDialog() async {
  // 7 项独立检测,任一为 true 即视为"环境异常"
  return await checkAppMetaData()   // 校验 LIVE_API_VERSION_CODE 等 meta-data
      || await checkAppComponentFactory()  // 校验 AppComponentFactory 是否被替换
      || await checkApplication()    // 校验 Application 类是否为 BaseApplication
      || await checkJiagu()          // 检测 libjiagu 残留(360 加固)
      || await checkPMProxy()        // 校验 PackageManager 代理类
      || await checkApk()            // 校验 APK 签名/完整性
      || await checkIO();            // 校验 I/O 行为(maps/资源重定向)
}

// 启动后约 7 秒触发
Timer(Duration(seconds: 7), () async {
  if (await _shouldShowWarningDialog()) {
    showDialog(
      context: context,
      builder: (_) => AlertDialog(
        title: Text('非法篡改'),
        content: Text('应用已被非法篡改或被第三方APP禁用某些模块...'),  // 长版正文 0x1c326e
        actions: [TextButton(child: Text('下载'), onPressed: ...)],
      ),
    );
  }
});
```

### 5.3 关于"决定是否弹框那条条件分支"的精确地址 — 当前未能给出

非常抱歉,**本任务未能给出该条件分支的精确 vaddr / file offset / 机器码字节**。原因如下:
- blutter 因定制 VM 反序列化失败(详见第三节),无法把函数名映射到 .text 地址,也无法解析对象池中 String 对象的指针
- 在 capstone 反汇编里,共识别 **21,696 个 Dart 函数**(`stp x29, x30, [x15, #-0x10]!` 形式的 prologue),但要找到"调用了 7 个 check\* 的那一个",需要知道每个 check\* 的 .text 入口地址
- check\* 的 .text 入口地址只能在 Code 对象里找到,而 Code 对象的解析依赖 cluster 反序列化(blutter 卡死的环节)
- 我也尝试过在 raw bytes 中搜索"compressed pointer 指向 0x1c326e 长版正文"的 4 字节模式,但因为没有确定的 heap base 候选(试过 0x0、0x1b0、0x3330、0x460000、0x464ce0 共 5 个,均无 <= 200 次的合理匹配),未能定位

**可行的下一步**(供后续恢复者):
- 用 frida 在运行时 hook `showDialog` 或 `AlertDialog` 构造函数,从堆栈回溯直接拿到 `checkAppMetaData` 的 .text 入口
- 或在 frida 中读 `_kDartIsolateSnapshotData` 实际 mmap 后的地址,加上 Dart VM 内部记录的 isolate→pool 偏移,反推 String 对象的运行时地址,再在 .text 中搜 compressed pointer 引用

| 0x503bf0 | 5808 | 1452 | 126 | 76 | 143 | 66 |
| 0x843958 | 5680 | 1420 | 40 | 112 | 140 | 74 |

> 这些函数的入口 prologue 全部为 `stp x29, x30, [x15, #-0x10]!`(Dart 定制 VM 用 x15 作为 shadow stack 而非 sp),后续 4 条指令是 stack overflow check:`mov x29, x15` / `sub x15, x15, #N` / `ldr x16, [x26, #0x38]` / `cmp x15, x16` / `b.ls <overflow_handler>`。x26 = Thread register,x27 = Pool pointer (PP)。

## 六、重点 2:会员 / 特权判定逻辑

### 6.2 Code 注释层面

对象池中**未发现** `*Vip*::` / `*VIP*::` / `*privilege*::` / `*donate*::` / `*member*::` / `*pay*::` / `*purchase*::` 形式的 Dart 函数名。这有三种可能:
- ppcat 没有"会员系统",`特权` / `捐赠` 只是 UI 文案(如"捐赠支持开发者"按钮)


### 7.1 版本相关 URL(对象池中确认,均为 UTF-8)

| `https://ppcat.gentle.com/?u` | **0x1b10ee** | 官网(带 `?u` 参数,可能是 update referral) |

### 7.2 关键观察

- `gitlab.com/ghostgzt/ppcat/-/raw/master/version` 与 `github.com/ghostgzt/ppcat/blob/master/version` 指向**同一个 `version` 文件**的两个镜像。raw 端点返回纯文本版本号,这是典型的"轻量级版本检查"模式。
- `ppcat.gentle.com` 出现两次,带不同 query 参数 `?i` 和 `?u`。`?i` 很可能是 install(首次安装),`?u` 是 update(更新跳转),与商店分发流程一致。
- `AcgLibrary/ppcat_store` 仓库很可能是 GitHub Pages 形式的下载站。

### 7.3 推测的版本更新流程

```dart
Future<void> checkUpdate() async {
  final String latestVersion = await http.get(
    Uri.parse('https://gitlab.com/ghostgzt/ppcat/-/raw/master/version'),
    // 备用: 'https://github.com/ghostgzt/ppcat/blob/master/version'
  ).then((r) => r.body.trim());
  
  if (_compareVersion(latestVersion, currentVersion) > 0) {
    // 弹出更新提示,跳转:
    // - 'https://github.com/AcgLibrary/ppcat_store' (商店)
    // - 'https://ppcat.gentle.com/?u' (官网更新页)
  }
}
```

对象池中**未发现** `*Version*::` / `*Update*::` 形式的明确函数名(只有 `VIProductVersion` 这种 Windows API 误报)。相关函数名可能是 `_checkUpdate`、`_fetchLatestVersion` 等带下划线的私有名,被 Code 注释剥离了下划线。
