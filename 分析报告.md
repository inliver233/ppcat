## 3. 重点1: 启动约 7 秒后的“非法篡改”提示框

### 3.1 结论摘要

- 高置信度“聚合判定函数”候选: `0x00592fd8`
- 高置信度“弹框构造层”函数:
  - 外层 `0x0051d764`
  - 内层 helper `0x0051dfa8`
- 另一处高置信度“检测结果 + 弹框文案混合构造层”函数: `0x006418a4`
- 当前 fallback 产物下，尚未恢复出一个带符号的、可直接命名为 `showDialog` 的调用点
- 因此本报告把“聚合判定层”和“弹框构造层”分开给出；两者都已定位，但最终桥接调用链尚未完全闭合

### 3.2 聚合判定函数候选: `0x00592fd8`

文件: `unflutter_dump/asm.txt`

这个大函数中已经确认同时处理多项平台通道检测结果，符合“收集多个检测值，再做布尔聚合”的形态:

- `0x0059a708` 载入 `getPMProxy`
- `0x0059a850` 载入 `getApplication`
- `0x0059caf4` 载入 `getMapsList`
- `0x0059e1e0` 载入 `getAppInfo`

这些块的共同特征是:

- 先以字符串常量选择某个检测方法
- 调用一段通用 helper 读取 / 转换返回值
- 与 `W22` 或对象字段比较
- 条件满足时跳向一组公共分支 thunk

这正是“若任一检测异常则进入失败路径”的典型 AOT 形态。

### 3.3 已证实的关键判定分支点

这三处是当前最强的“决定进入失败路径”的候选分支。它们都位于 `0x00592fd8` 内，并且都汇聚到 `0x005a920c..0x005a9214` 这段公共 thunk。

### 3.7 关于 `showDialog` 调用点的当前结论

我没有在当前 fallback 反汇编里恢复出一个能被直接命名为 `showDialog` 的高层符号，也没有找到从 `0x00592fd8` 直接 `BL` 到 `0x0051d764` 或 `0x006418a4` 的明确外部 xref。

已额外确认:

- 指向 `0x0051dfa8` 的直接调用只有两处: `0x0051dae0` 与 `0x0051dd30`
- 指向 `0x0051d764` 的直接外部 `BL` 未恢复
- 指向 `0x006418a4` 的直接外部 `BL` 未恢复

因此，当前最诚实的结论是:

- “决定是否走失败路径”的聚合判定函数，已经高置信度定位到 `0x00592fd8`
- “构造非法篡改提示框内容”的函数，已经高置信度定位到 `0x0051d764` / `0x0051dfa8`，以及更大的混合构造层 `0x006418a4`
- 但“聚合判定层最终跳到 Flutter `showDialog` 封装层”的最后一跳，当前仍未在 fallback 产物中完全闭合

### 5.3 当前结论

- `0x00dac684` 是当前最强的“更新源 URL 选择器”
- `0x0085f09c` 是当前最强的“更新字段拼装器”

- 但“最终弹出更新提示”的 UI 函数以及完整版本号比较流程，本轮未像重点1那样收敛到单一函数

## 6. 最终结论

按当前证据强度排序:

- 最高置信度:
  - 启动 tamper 聚合判定候选 `0x00592fd8`
  - 关键失败分支点 `0x0059a1e4` / `0x0059a2d8` / `0x0059a3bc`
  - tamper 提示构造 helper `0x0051dfa8`
  - tamper 外层调用者 `0x0051d764`
  - 混合检测 + tamper 构造层 `0x006418a4`
  - VIP 提示构造层 `0x00a54178`
  - VIP 分支选择点 `0x00b9fc84` -> `0x00ba0070` / `0x00ba007c`
  - 更新 URL 选择器 `0x00dac684`
- 中等置信度:
  - `0x00592fd8` 与 `0x0051d764` / `0x006418a4` 之间存在运行时桥接，但在当前 fallback 反汇编中未完全恢复
  - `0x00b9fc84` 比 `0x00a54178` 更接近真正的 VIP 类型布尔分支
- 仍未解决:
  - 精确、可命名的 Flutter `showDialog` 调用点
  - 单独的 `isVip` / 本地 key / 服务器字段名
  - 精确 Dart 语义版本号
