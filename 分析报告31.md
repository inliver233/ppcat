# 分析报告31

> 分支：`test1`  
> 角色：纯静态分析。  
> 本轮按用户新要求修正主线：不再把 `喵喵饿了` 主要当成视觉 safe-hide 问题，而是优先沿“点击 reward 成功后，当天自然不再出现”这条业务状态链推进；同时把 `VIP / 特权 / 绑定设备` 相关状态拆开，判断哪些能与这条链复用。

## 1. 本轮结论

### 1.1 对 `test3` 最新进展的裁决

`github/test3` 最新第 9 轮对 `0xa52bb0` 的内部结构拆解大体成立：

1. `0xa52bb0` 仍是当前阅读页 `喵喵饿了` 真实 builder 核心。
2. `0xa52c40` 的 `mov x17,#2` 的确向 `0xa52c90` 传入了 `Smi(1)` 子项数量。
3. `0xa52c60 / 0xa52c68` 确实是在最终返回对象上写两个字段。

这些事实仍然有价值，但**不再是本轮主线**。原因是：

1. 这条线解决的是“如何不显示”，不是“为什么当天看过 reward 后自然不再显示”。
2. 用户明确指出：这块在正常业务下本来就会因为 reward 成功而当天消失，因此优先恢复这条业务状态比强做 safe-hide 更符合代码恢复目标。

### 1.2 新主线已经足够明确

本轮新增并独立复核后的核心链是：

1. `0xa56be4 -> 0xa56c34`
   - `0xa56be4` 唯一加载 `ref=26746 "喵喵喂饱了~"`。
   - `0xa56c34` 会把 `[thread+0xee0] = true`，并把一个时间样值写到 `[thread+0xea0]`。
2. `0x846b1c -> 0x846f30`
   - `0x846b1c` 在前置 `BL 0x846f30` 后，若结果为 false 就直接 `return null`。
   - 这比 `0xa52bb0` 更像“每日喂喵/喵块上层显示门”。
3. `0x7e8534` 语义必须重评
   - 之前多轮把 `0x7e861c` 当“最窄上层特权门”推荐，现在要降级。
   - 因为 `0xa56c34` 把 `[thread+0xee0]` 置 true 后，`0x7e8534` 在 `0x7e85bc` 反而会直接走 `false` 尾声。
   - 这说明 `[thread+0xee0]` 更像“本轮已经满足 / 不再继续某条广告或提示链”，而不是一个简单的“特权开启”布尔。

一句话：**`0xa56c34` 更像“成功写状态”，`0x846b1c -> 0x846f30` 更像“是否还要显示每日喂喵”门控，`0x7e861c` 则不应再被简单宣传成默认主推荐。**

## 2. `喵喵饿了` 的自然消失链

### 2.1 成功提示与状态写入 sink：`0xa56be4 -> 0xa56c34`

#### 2.1.1 `0xa56be4`：唯一加载 `"喵喵喂饱了~"`

```asm
0x00a56c04: 61b34091  add     x1, x27, #0x2c, lsl #12
0x00a56c08: 213c47f9  ldr     x1, [x1, #0xe78]
0x00a56c0c: 0eb11494  bl      #0xf83044
0x00a56c10: e08d1ff8  str     x0, [x15, #-8]!
0x00a56c14: 08000094  bl      #0xa56c34
```

这里 `0xe78` 对应的对象池项不是普通调试字符串，而是 `"喵喵喂饱了~"` 所在 toast/消息链的实际对象入口。  
静态上已经足够确认：**`0xa56c34` 是这个成功提示后的直接状态写入 sink。**

#### 2.1.2 `0xa56c34`：写 `[thread+0xee0]` 与 `[thread+0xea0]`

```asm
0x00a56c3c: c0c20091  add     x0, x22, #0x30
0x00a56c4c: 414340f9  ldr     x1, [x26, #0x80]
0x00a56c50: 207007f9  str     x0, [x1, #0xee0]
...
0x00a56c84: 007d80d2  mov     x0, #0x3e8
0x00a56c88: 220cc09a  sdiv    x2, x1, x0
...
0x00a56ca0: 414340f9  ldr     x1, [x26, #0x80]
0x00a56ca4: 205007f9  str     x0, [x1, #0xea0]
0x00a56ca8: 414340f9  ldr     x1, [x26, #0x80]
0x00a56cac: 21c847f9  ldr     x1, [x1, #0xf90]
```

新增结论：

1. `[thread+0xee0]` 被显式写为 canonical `true`。
2. `[thread+0xea0]` 被写入一个由时间/计时值计算出的对象。
3. 后面还显式消费 `[thread+0xf90]`，说明这不是单独的 toast helper，而是会回流到更大状态树。

#### 2.1.3 `0xa56c34` 的 3 个直接 caller

全 `.text` 直接 `BL` 扫描结果：

1. `0xa56c14` from `0xa56be4`
2. `0xa56f90` from `0xa56f60`
3. `0xa57040` from `0xa57010`

对应反汇编：

```asm
0x00a56f84: 217847f9  ldr x1, [x1, #0xef0]
0x00a56f88: 2fb01494  bl  #0xf83044
0x00a56f90: 29ffff97  bl  #0xa56c34
```

```asm
0x00a57034: 218447f9  ldr x1, [x1, #0xf08]
0x00a57038: 03b01494  bl  #0xf83044
0x00a57040: fdfeff97  bl  #0xa56c34
```

因此 `0xa56c34` 不是单次 toast 专用，而是**至少 3 条成功/完成包装入口共用的写状态 sink**。  
这进一步支持：**它比 `0xa52bb0` 更接近“当天自然消失”的业务核心。**

### 2.2 上层显示门：`0x846b1c -> 0x846f30`

#### 2.2.1 `0x846b1c`：失败直接 `return null`

```asm
0x00846b3c: fd000094  bl      #0x846f30
0x00846b44: a0002037  tbnz    w0, #4, #0x846b58
0x00846b48: e00316aa  mov     x0, x22
0x00846b4c: ef031daa  mov     x15, x29
0x00846b50: fd79c1a8  ldp     x29, x30, [x15], #0x10
0x00846b54: c0035fd6  ret
```

这段比 `0xa52bb0` 更像一个正经的“是否显示这段 widget”门：

1. 若 `0x846f30` 返回 false，直接返回 `null`。
2. 若返回 true，才继续构造后面一整段 UI。

这里加载的标题正是：

```text
ref=21707 "每日喂喵" -> slot 0x59bb -> func 0x846b1c
```

所以 `0x846b1c` 现在应升格为：

- **每日喂喵上层显示门 / builder**
- 而不是仅仅“一个加载标题字符串的 helper”

#### 2.2.2 `0x846f30`：不是简单字符串比较，而是状态对象比较

```asm
0x00846f5c: 227044b8  ldur    w2, [x1, #0x47]
0x00846f60: 42801c8b  add     x2, x2, x28, lsl #32
0x00846f64: 701740f9  ldr     x16, [x27, #0x28]
0x00846f68: 5f00106b  cmp     w2, w16
0x00846f6c: 80010054  b.eq    #0x846f9c
0x00846f70: 702f4091  add     x16, x27, #0xb, lsl #12
0x00846f74: 10e242f9  ldr     x16, [x16, #0x5c0]
0x00846f78: 5f00106b  cmp     w2, w16
0x00846f7c: d0820091  add     x16, x22, #0x20
0x00846f80: d1c20091  add     x17, x22, #0x30
0x00846f84: 0002919a  csel    x0, x16, x17, eq
```

补充定性：

1. `0x846f30` 比较的不是普通字符串 ref。
2. 它前面还有一层 `cmp w2,[x27,#0x28] -> b.eq 0x846f9c` 的特殊值分流。
3. 真正决定 true/false 的，是 `field@0x47` 与 `slot 0x16b8 / pool off 0xb5c0 / val=105024` 的比较。

因此最稳妥的表述应是：

- **`0x846f30` 是一个读取状态对象内部字段、与特定状态常量比较的 bool gate**
- 它比“纯 UI builder”更接近业务态
- 也比 `0x7e861c` 更贴近“每日喂喵当天是否应继续显示”

### 2.3 与 `0x7e8534` 的新关系：旧推荐需要降级

之前多轮把 `0x7e861c` 当默认主推荐，是因为它位于 `0x7e8534` 这个共享 gate 内部：

```asm
0x007e8618: 559b0394  bl   #0x8cf36c
0x007e861c: c0062036  tbz  w0, #4, #0x7e86f4
```

但本轮新增的硬事实是：

```asm
0x00a56c50: str x0, [x1, #0xee0]   ; 写 true
```

而 `0x7e8534` 紧接着有：

```asm
0x007e85a4: 217047f9  ldr x1, [x1, #0xee0]
0x007e85bc: c0092037  tbnz w0, #4, #0x7e86f4
```

也就是：

1. `0xa56c34` 把 `[thread+0xee0]` 写 true
2. `0x7e8534` 看到 `[thread+0xee0] == true` 时，会提前走 `0x7e86f4`
3. `0x7e86f4` 是 canonical false-return 尾声

所以必须纠偏：

1. `[thread+0xee0]` 很可能代表“这条广告/喂喵流程今天已满足，无需继续走展示/提示链”
2. 这会让 `0x7e861c` 的旧说法“强制特权消费门放开”变得过于粗糙
3. **`0x7e861c` 不再适合作为默认第一推荐**

新的优先级应改成：

1. 先理解并利用 `0xa56c34` 的成功写状态
2. 再看 `0x846b1c -> 0x846f30` 是否正是“当天是否显示每日喂喵”
3. `0x7e861c` 只保留为更宽、更共享的广告/特权消费门候选，不再作为首选

## 3. `VIP / 特权 / 绑定设备` 的拆分

### 3.1 `0xb9fc84` 仍然不是单一 `isVip`

本轮再次复核 `0xb9fc84` 前半段，结论不变，但更具体：

```asm
0x00b9fcb4: b5c6f597  bl      #0x911788
...
0x00b9fce8: 008448f9  ldr     x0, [x0, #0x1108]
...
0x00b9fdb8: 004442f9  ldr     x0, [x0, #0x488]   ; ref=17423 "isDaily:"
...
0x00b9fe74: 3ebdf497  bl      #0x8cf36c
```

这说明它同时混合了三层：

1. `0x911788` 上游共享状态门
2. `[thread+0x1108]` 这类绑定/账户/页面态对象
3. `isDaily:` 与 `0x8cf36c` 奖励特权状态对象

因此对 `VIP / 特权` 的更稳说法是：

- `0xb9fc84` = **绑定设备/账号 + daily 状态 + reward privilege 的页面混合状态机**
- 不是单点 `isVip()`

### 3.2 `VIP / 特权` 应继续拆成 3 类

#### A. 奖励特权类

这一类和 `喵喵饿了` 最相关：

1. `rewardTime`：`0x913bf8 / 0xa7fcd4`
2. `getRewardTime`：`0x59d964`
3. 共享状态对象：`0x8cf36c`
4. 反作弊专属旁路：`0x875efc`

其中 `0x875efc` 仍然成立：

```asm
0x00875efc: 40012037  tbnz  w0, #4, #0x875f24
```

它只是在本轮被重新定位为：

- **reward-only 反作弊旁路**
- 不是“总特权门”

#### B. 绑定设备/账号类

这一类更多和“是否已绑定”“是否显示捐赠绑定文案”相关：

1. `0xb9fc84` 中多次读取 `[thread+0x1108]`
2. `0xa54178 / 0xb9fc84` 中都有“通过捐赠绑定设备获取 / 通过捐赠绑定账号获取”
3. `0x871320` 一带还明确出现 `?i=`、`?u=`、machine/device 相关 URL 片段

所以这条线更像：

- **远端绑定状态 / 本地绑定对象 / 文案分流**
- 不应直接拿来解释 `喵喵饿了` 当天消失

#### C. 过期/本地期限类

这条线独立于上面两类：

1. `expiresDate`：`0x8ba3d0`
2. `expires/date/skipCount`：`0x67994c`

它更像：

- 本地期限/过期缓存
- 不等于 reward privilege 当天状态

### 3.3 对“VIP 能否帮助喵块自然消失”的当前判断

当前最稳妥的说法是：

1. **有帮助，但不是一把总钥匙。**
2. 与 `喵喵饿了` 最直接相关的是“奖励特权 / daily 成功状态”这条线：
   - `0xa56c34`
   - `0x846b1c -> 0x846f30`
   - `0x59d964 / 0x8cf36c`
3. `绑定设备/账号 VIP` 更像并行的另一套状态，不应替代主线。

换句话说，**对去掉喵块最有帮助的不是“把所有 VIP 功能强开”，而是先把 reward 成功后的本地状态写入与 daily 显示门打通。**

## 4. 对主控的建议排序

### 4.1 主线优先级修正

本轮后，`test1` 建议顺序改成：

1. **主线 A：`0xa56c34` 成功写状态链**
   - 继续查它的上游调用条件与写入后的消费者
2. **主线 B：`0x846b1c -> 0x846f30` 每日喂喵显示门**
   - 重点理解 `field@0x47` 与 `slot 0x16b8` 常量的语义
3. **辅助线：`0x875efc` reward anti-cheat**
   - 只负责“成功不被判作弊”
4. **降级线：`0x7e861c`**
   - 不再默认作为首选
5. **页面/视觉线：`0xa52bb0` safe-hide**
   - 保留作 fallback，不再当主线

### 4.2 现阶段最值得继续压实的地址

1. `0xa56c34`
   - 成功写 `[thread+0xee0] / [thread+0xea0]`
2. `0xa56be4 / 0xa56f60 / 0xa57010`
   - 3 个直接上游包装入口
3. `0x846b1c`
   - 上层显示门：false 直接 `return null`
4. `0x846f30`
   - 业务状态比较门
5. `0x875efc`
   - reward anti-cheat 尾部分支
6. `0xb9fc84`
   - 继续只做拆层，不再粗暴压成单点 `isVip`

## 5. 一句话交付

本轮最重要的修正不是又找到了一个新的视觉 patch，而是把主线从 `0xa52bb0 safe-hide` 转到了更符合代码恢复目标的业务状态链：**`0xa56c34` 是“喂饱/成功”后的共用状态写入 sink，`0x846b1c -> 0x846f30` 是最像“每日喂喵当天是否还要显示”的上层门。** 同时，本轮新增硬证据表明：`0xa56c34` 把 `[thread+0xee0]` 置 true 后，`0x7e8534` 会提前走 false 尾声，因此此前把 `0x7e861c` 长期当作默认主推荐的说法需要降级。`VIP / 特权 / 绑定设备` 这条线也应拆成“奖励特权 / 绑定状态 / 过期缓存”三类继续看，其中真正对 `喵喵饿了` 自然消失有直接帮助的，是前者而不是整体强开 VIP。
