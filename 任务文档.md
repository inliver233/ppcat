# Flutter 应用 Dart 代码恢复 — 静态分析任务

> 本任务为**纯静态分析 / 代码理解**,只读取文件、输出分析报告,不修改、不分发任何文件。

## 一、项目背景

这是一个 **Flutter 移动应用**(包名 `com.gentle.ppcat`,中文名"皮皮喵",一个本地阅读工具)。**原作者本人**多年前将其开源,后因故丢失了 Dart 源代码。现在网上流传的安装包是第三方重新打包的版本,其中包含一些原作者不掌握的运行时逻辑。

本任务的目标:**通过静态分析,把应用的 Dart 业务逻辑还原成可读形式**,帮助作者理解和恢复自己应用的代码。应用使用 Flutter,业务逻辑被编译为 Dart AOT 机器码,存放在 `libapp.so` 里。要还原它,需要使用开源工具 **blutter**(https://github.com/worawit/blutter ),它能把 Dart AOT 反汇编成带符号标注的可读汇编,并 dump 出完整的对象池(字符串、常量、类名、方法名)。

这是合法的代码恢复研究(恢复作者本人应用),感谢协助。

## 二、环境要求(Linux)

推荐 **Ubuntu 24.04** 或 **Debian 13(trixie)**(这两个发行版自带够新的 C++ 编译器)。需要机器能访问 GitHub(blutter 会 clone Dart SDK 源码编译运行时库)。

### 2.1 系统依赖(以 apt 为例)

```bash
sudo apt update
sudo apt install -y \
    build-essential git cmake ninja-build pkg-config \
    python3 python3-pip python3-venv \
    libicu-dev libcapstone-dev \
    gcc-13 g++-13 clang-16 clang++-16
```

> **关键**:blutter 是 C++20 工程,需要 **gcc ≥ 13** 或 **clang ≥ 16**。Ubuntu 24.04 默认 gcc 就是 13+,可直接用默认 `g++`。如果你的发行版默认 gcc 较旧,务必用上面的 `g++-13`/`clang++-16`,并通过 `CC/CXX` 环境变量指定(见 2.3)。

### 2.2 Python 依赖

```bash
python3 -m pip install --break-system-packages pyelftools requests
```

### 2.3 验证

```bash

clang++ --version 2>/dev/null
git --version

ninja --version

```

## 三、输入文件说明

本仓库已包含待分析的文件:

```
lib/
├── arm64-v8a/
│   ├── libapp.so        ← 主分析目标(Dart AOT 业务逻辑),约 16MB
│   └── libflutter.so    ← Flutter 引擎(blutter 用它自动探测 Dart 版本),约 9MB
└── armeabi-v7a/
    ├── libapp.so        ← 同一份逻辑的 32 位版本(备用,通常分析 arm64 即可)
    └── libflutter.so
```

- `libapp.so` = 应用的 Dart 业务代码(AOT 机器码 + 对象池)
- `libflutter.so` = Flutter 运行时引擎。blutter 需要它来探测 Dart 版本号,然后会 checkout 对应版本的 Dart SDK 源码并编译运行时库,再据此反汇编 libapp.so

## 四、分析步骤

### 4.1 clone 本仓库

```bash
git clone https://github.com/inliver233/ppcat.git
cd ppcat
```

### 4.2 clone 并运行 blutter

```bash
git clone https://github.com/worawit/blutter.git
cd blutter
git submodule update --init --recursive   # 若有子模块

# (可选)若默认 g++ 较旧,强制用 clang-16
# export CC=clang-16 CXX=clang++-16

# 运行(blutter.py 接受一个含 libapp.so+libflutter.so 的目录)
python3 blutter.py ../lib/arm64-v8a ../out_arm64

cd ..
```

首次运行时 blutter 会:
1. 从 `libflutter.so` 探测 Dart 版本
2. clone 对应版本的 Dart SDK 源码(github.com/dart-lang/sdk,约几百 MB)
3. 编译 Dart 运行时库(这一步较慢,可能 10~40 分钟,取决于机器)
4. 用编译好的运行时库反汇编 `libapp.so`

完成后在 `out_arm64/` 产出:
- `pp.txt` —— 对象池里所有 Dart 对象(字符串/常量/类名/方法名)**★ 最有用**
- `objs.txt` —— 嵌套的对象 dump
- `asm/*.S` —— 带符号标注的反汇编(按函数分文件)
- `blutter_frida.js` —— frida 脚本模板

### 4.3 若 blutter 自动探测 Dart 版本失败(已知风险)

本应用的 `libflutter.so` 可能是定制引擎。如果 `blutter.py` 在探测阶段报类似 `found hashes`(engine_ids 数量异常)的错误,fallback 步骤:

```bash
# A. 手动从 libflutter.so 找 Dart 版本号
strings lib/arm64-v8a/libflutter.so | grep -iE 'dart.*[0-9]+\.[0-9]+\.[0-9]+' | head
# 或找 snapshot hash:
strings lib/arm64-v8a/libflutter.so | grep -oE '[0-9a-f]{40}' | head

# B. 若能确定版本(例如 2.x.x / 3.x.x),尝试用 dartvm_fetch_build.py 直接构建运行时库
cd blutter
python3 dartvm_fetch_build.py <版本号> android arm64
# 然后 blutter.py 会复用已构建的库

# C. 备选开源工具(若 blutter 完全无法构建):
#    - reFlutter  (https://github.com/Aspect-Dev/reFlutter)
#    - darter     (https://github.com/mildsunrise/darter)
#    也能 dump Dart 对象池,功能弱于 blutter 但依赖少、易装。
```

无论用哪种工具,最终都要拿到 **Dart 对象池(字符串表) + 反汇编** 这两类产物。

## 五、需要定位的具体逻辑(分析重点)

请在 `pp.txt` 和 `asm/` 产物里,定位以下三块逻辑。关键词见同目录的 **已知信息.md**(含已确认的字符串偏移、期望值等,可直接加速定位)。

### 重点 1:启动提示框的触发逻辑(最高优先级)

应用启动约 7 秒后,会显示一个**提示框**(Flutter 渲染),标题为「非法篡改」,正文开头为「应用已被非法篡改或被第三方 APP 禁用某些模块或安装在第三方虚拟环境…」,底部一个「下载」按钮。

需要查清:
- 这个提示框是哪个 Dart 函数调用 `showDialog` 弹出的(在 `asm/` 里找引用了上述字符串的函数)
- **触发它的判定条件是什么** —— 应该有一个聚合判定函数,读取一组检测结果(通过 Flutter 平台通道从 Java 侧获取,通道方法名有 `getAll`/`getMaps`/`getAppInfo`/`getApplication`/`getPMProxy` 等),做布尔运算后决定是否弹框
- 找到这个**聚合判定函数的入口地址** 和 **判定分支的机器码偏移**

### 重点 2:会员 / 特权判定逻辑

应用内有"喵喵特权/捐赠 VIP"相关功能。定位:
- 判断当前用户是否为 VIP 的函数(对象池里搜 `vip`/`VIP`/`isVip`/`会员`/`特权`/`捐赠`)
- 其判定字段(本地存储 key?服务器返回字段?)

### 重点 3:版本更新逻辑

应用会检查更新,涉及外部 URL。定位:
- 版本检查请求的 URL 构造逻辑(对象池里已知有 `https://gitlab.com/ghostgzt/ppcat/-/raw/master/version`、`https://github.com/ghostgzt/ppcat/blob/master/version`、`https://github.com/AcgLibrary/ppcat_store`、`https://ppcat.gentle.com/`)
- 比对版本号、提示更新的函数

## 六、输出要求

请在仓库根目录(或 `/output/` 目录)产出一份 **`分析报告.md`**,包含:

1. **环境与产物**:实际用的工具(blutter 还是备选)、Dart 版本探测结果、产物路径
2. **对象池关键发现**:`pp.txt` 里与上述三块逻辑相关的类名/方法名/字符串清单(带在 pp.txt 中的偏移)
3. **重点 1 的结论**(最重要):
   - 触发提示框的 showDialog 调用所在函数的**反汇编片段**(从 `asm/` 里摘出来,带文件名和行号/偏移)
   - 聚合判定函数的**完整反汇编片段**
   - **判定分支点的精确虚拟地址(vaddr)+ 文件偏移(file offset)+ 该处的几条机器码(hex)**
   - 判定依据了哪些检测结果(列出)
4. **重点 2、3 的结论**:函数定位 + 关键反汇编片段 + 偏移
5. **附:反汇编片段文件**:把重点 1/2/3 涉及的 asm 函数原文另存为 `asm_关键片段.txt` 一起提交

这些偏移和反汇编信息是后续工作的关键输入,请尽量精确(给到具体地址和字节)。
