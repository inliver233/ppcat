# 第十六轮代码恢复补充报告

> 分支：`test1`
>
> 角色：纯静态分析。本轮目的有两个：一是把 `test2` 早期提出的 `0x7e85f4` 总闸方案与主控后续运行时事实重新对齐；二是继续解释为什么 `0x885ee0 / 0x886290` 这类 async invoke helper 仍然不适合直接入口 return-null。

## 1. 本轮结论

本轮没有推翻前两轮的主结论，反而把它们再压实了一层：

1. `0x7e8534` 仍然是 reward privilege / 广告决策上层门控之一，但 `test2` 早期把它当“全 app 总闸”来粗切的 `0x7e85f4` 方案，粒度太大。
2. 在 `0x7e8534` 内部，**更窄的点仍然是 `0x7e861c`**：
   - `0x7e8618: bl 0x8cf36c`
   - `0x7e861c: tbz w0,#4,#0x7e86f4`
   - `0x7e86e4: canonical true`
   - `0x7e86f4: canonical false`
3. `0x885ee0 / 0x886290` 这两个 invoke helper 虽然是 reward / splash async 链的核心，但它们**返回的是后续字符串/UI链立即消费的对象，不是简单 bool**。因此仍然不能粗暴入口 return-null。

所以，本轮对当前推荐栈的影响是：

- `0x7e861c` 继续保留为 reward privilege 方向更窄的实验项
- `0x7e85f4` 只作为历史粗粒度候选记录，不上调优先级
- `0x885ee0 / 0x886290` 暂不进入推荐 patch 栈

## 2. `0x7e85f4` 与 `0x7e861c` 的关系

### 2.1 `test2` 早期发现的价值

`test2` 在 `4828053` / `ccff110` 那批工作里，确实抓到了两个对当前分析仍有价值的事实：

1. `0x7e8534` 是一个被大量 caller 复用的上层广告/特权门控函数
2. `0x7e8054` 与 `0x8863e8` 是更贴近广告控制的上层封装器

这两点我本轮再次独立复核后仍然成立。

### 2.2 为什么 `0x7e85f4` 现在不应作为首选

`test2` 当时的主张是：

- `0x7e85f4` 附近的 `TBZ` 决策点是全局广告总闸
- 可以强制跳“免广告”路径

但结合主控后续运行时硬事实，这个思路的问题不是“完全错”，而是**粒度过粗**：

1. `0x7e8534` 确实有 41 个 caller
2. 它内部不仅涉及 `isNoAdLock`
3. 还串着：
   - `0x7e870c`
   - `0x911788`
   - `0x7e41c0`
   - `0x8cf36c`
   等多个前置检查

也就是说：

- 直接在较早层的 `0x7e85f4` 粗切，副作用面会更大
- 主控后来的运行时反馈也已经说明：把 `isNoAdLock` 当简单全局总闸，并不足以解释所有可见广告行为

### 2.3 `0x7e861c` 为什么更好

本地复核的关键窗口：

```asm
0x007e85f8: f2eeff97  bl   #0x7e41c0
...
0x007e8618: 559b0394  bl   #0x8cf36c
0x007e861c: c0062036  tbz  w0, #4, #0x7e86f4
...
0x007e86e4: c0c20091  add  x0, x22, #0x30
...
0x007e86f4: c0820091  add  x0, x22, #0x20
```

这比 `0x7e85f4` 更好的原因是：

1. 它已经经过前面的 `0x7e41c0` 等检查
2. 直接只对 `0x8cf36c` 的 bit4 结果做门控
3. 目标尾声就是函数自带的 canonical true / false

因此：

- `0x7e861c: c0 06 20 36 -> 32 00 00 14`

仍然是 reward privilege 方向比 `0x7e85f4` 更精细、更可解释的实验点。

## 3. `0x7e8054` 的意义：上层广告控制器，不等于 reward privilege 总闸

`0x7e8054` 的关键结构：

```asm
0x007e80f8: 0f010094  bl   #0x7e8534
0x007e80fc: e0002036  tbz  w0, #4, #0x7e8118
...
0x007e813c: 8c9c0394  bl   #0x8cf36c
0x007e8140: c0072036  tbz  w0, #4, #0x7e8238
...
0x007e8220: 1f2df297  bl   #0x47369c
```

这说明：

1. `0x7e8054` 是更上层的广告控制 / 提示封装器
2. 它同时消费：
   - `0x7e8534` 的 bool 门控结果
   - `0x8cf36c` 的 reward privilege 结果
3. 它后面还会直接进入 `0x47369c` 这类 UI / 文本链

所以：

- `0x7e8054` 本身是理解整条广告门控链的重要上层入口
- 但若目标是“更窄地强制特权态成功”，还是应优先改它下面的 `0x7e861c`
- 而不是继续回到更早的 `0x7e80fc` 或更粗的 `0x7e85f4`

## 4. `0x885ee0 / 0x886290` 仍然不适合入口 return-null

### 4.1 结构复核

本轮重新完整反汇编后，`0x885ee0` 与 `0x886290` 的结构几乎镜像：

```asm
0x00885ef8: 410180d2  mov x1, #0xa
0x00885efc: 0ff41b94  bl  #0xf82f38
...
0x00885f14: 61c2ef97  bl  #0x476898
...
0x00885f88: 2ff41b94  bl  #0xf83044
...
0x00885f9c: ccb1ef97  bl  #0x4726cc
...
0x00886004: 643740f9  ldr x4, [x27, #0x68]
0x00886008: 02f041f8  ldur x2, [x0, #0x1f]
0x0088600c: 40003fd6  blr x2
...
0x00886018: d1820091  add x17, x22, #0x20
0x0088601c: 317001b8  stur w17, [x1, #0x17]
0x00886020: a0035ff8  ldur x0, [x29, #-0x10]
0x00886024: ret
```

`0x886290` 同理。

### 4.2 为什么这和 `0x7e861c` 不一样

`0x7e861c` 的好处是：

- 直接命中一个 bool gate
- 下游是函数自带的 canonical true / false 尾声

而 `0x885ee0 / 0x886290` 不同：

1. 它们先构造对象
2. 再把对象字段喂给：
   - `0x4726cc`
   - `0x472520`
   - 最后 `BLR X2`
3. 返回给 caller 的是一个**后续马上被 `0x47369c` 消费的对象**

对应上层 async caller：

- `0x8880c0` 在 `0x8881c4` 调 `0x885ee0`
- `0x88862c` 在 `0x888724` 调 `0x885ee0`
- `0x888b18` 在 `0x888c10` 调 `0x886290`

之后都会立刻：

```asm
stp x2, x0, ...
bl 0x47369c
```

也就是说：

- `x0` 不是“可随便 null 化的无关返回值”
- 而是后续字符串/界面链立即消费的对象

这正是我之前没有把这两个 helper 上调为推荐 patch 点的原因，本轮复核后这个结论继续成立。

## 5. `0x8863e8` 与 `0x7e8054`：仍然只适合做结构说明，不适合重新升格为主 patch

`test2` 当时提出过：

- `0x886564 / 0x886638 / 0x8868ec / 0x8869d4`
- `0x7e8220 / 0x7e83cc`

这些 `BL 0x47369c / 0x472c98` 的 SAFE NOP 点。

静态上它们确实栈平衡，但主控后续的运行时事实已经说明：

1. 围绕普通 `showDialog` / 提示 BL 的 NOP 不是根因方案
2. 开屏与广告控制器也不能轻易混成一类 patch

所以这批点的定位仍应是：

- 帮助理解广告控制器结构
- 不是当前主推荐栈的中心

## 6. 对当前推荐栈的影响

本轮不改主栈，只做层级上的再确认：

### 6.1 仍然保留的主栈

1. `0x920d90`
2. `0xbc5dc8`
3. `0xae3560`
4. `0x875efc`
5. `0x7e861c`
6. `0x8cf374`

### 6.2 本轮明确不升格的候选

1. `0x7e85f4`
2. `0x7e80fc`
3. `0x885ee0` 入口 return-null
4. `0x886290` 入口 return-null

原因分别是：

- `0x7e85f4 / 0x7e80fc` 过早、过粗
- `0x885ee0 / 0x886290` 返回对象被后续立即消费，不是简单 bool

## 7. 一句话结论

`test2` 早期对 `0x7e8534 / 0x7e8054 / 0x8863e8` 的结构挖掘是有价值的，但主控后续运行时事实已经要求我们继续把粒度收窄；本轮再次确认，reward privilege 方向现在最值得优先试的仍然是 `0x7e861c`，而不是更粗的 `0x7e85f4`，也不是更危险的 `0x885ee0 / 0x886290` 入口空返回。
