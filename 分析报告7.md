> 皮皮喵 com.gentle.ppcat Flutter/Dart AOT 代码恢复。test3 分支（ARM64 原生 + 全量对象池 + xref DB）。
> 中性措辞"代码恢复"。只读静态分析，产出精确 patch 建议，主控负责 patch-and-test。
> 本轮攻克：**故障弹窗第三源 0x920d7c 的精确截断点**（解释了主控 NOP showDialog 全失败的根因）。

## 〇、一句话结论

故障弹窗的"无法正常联网…广告无法显示"文案**仅由 0x920d7c 唯一构建**，3 个故障分发 caller 都调用它，
而**显示走的是 0x920d7c 内部的 0xf923d4 调用（运行时 stub），不是 showDialog BL**——这正是主控 NOP 所有 showDialog
编排器全部失败的根因。**截断 0x920d7c 入口（→ 直接返回 null）= 单点根治**，且比截断全局错误处理更精准、更安全。

## 一、★★★ 核心突破：为什么主控的 showDialog NOP 全失败

### 1.1 铁证：故障文案的唯一构建者

用 test3 新工具 `scan_slot.py`（字节级扫描所有 `.text` 中加载某对象池槽位的指令）扫描故障文案：

- 故障字符串 "无法正常联网…" = 对象池 ref **27673** → 槽位 **0x276e**（偏移 0x13b70）。
- **全 libapp（16352152 字节、21792 函数）扫描：ref27673 仅被 1 处加载** —— `0x921080`，位于函数 **0x920d7c** 内。

> 即：0x920d7c 是故障 body 文案的**唯一**来源。T+260s 弹窗若显示该文案，**必经 0x920d7c**。

### 1.2 0x920d7c 的 3 个 caller（全部 callers=0，pool 间接）

| caller | bl 站点 | callees | 说明 |
|---|---|---|---|
| 0x920b3c | 0x920d1c | 8 | 故障分发：调 0x920d7c 后经运行时 **0xf216e4**(109 callers) 再分发 |
| **0xc4cb30** | **0xc4cc7c** | 12 | 主控命名的"故障弹窗编排器"，有 4 个 showDialog BL |
| 0xc4d22c | 0xc4d298 | 2 | 写全局标志 [Thread+0x80]+0xee0=FALSE 后调 0x920d7c |

### 1.3 ★ 决定性证据：0xc4cb30 调 0x920d7c 的 BL 站点 0xc4cc7c，主控**没有 NOP**

主控第八轮 NOP 了 0xc4cb30 的 4 个 showDialog BL：
`0xc4cc2c / 0xc4ccac (bl 0x47369c)`、`0xc4ce0c (bl 0x4670b0)`、`0xc4ce60 (bl 0x472c98)`。

**但 0xc4cb30 调用 0x920d7c（故障 body 构造）的 BL 在 0xc4cc7c —— 主控没 NOP 它！**
故 0xc4cb30 仍构造故障 body（0x920d7c → 内部 `bl 0xf923d4` 传故障文案），弹窗照弹。

> 这与主控"NOP showDialog 全无效"的运行时结论**完全自洽**：故障显示根本不走 showDialog BL，
> 而走 0x920d7c 内部的运行时分发（0xf923d4 等 0xf… stub）。NOP showDialog 当然无效。

### 1.4 caller 忽略 0x920d7c 返回值（副作用调用）

- 0xc4d22c：`bl 0x920d7c` 后 `add x15,x15,#0x10`（弹影子栈）→ **直接 `mov x0,x22`（null）返回**，不读 0x920d7c 的 x0。
- 0x920b3c：`bl 0x920d7c` 后从 `[x29,#-0x48]`（自身栈帧）重载 x0，**不读返回值**。
- → 0x920d7c 是**为副作用而调**（构造故障 body 并经 0xf923d4 分发）。截断其入口 = 副作用不发生 = 无故障 body。

## 二、★ 主交付：截断 0x920d7c（单点根治）

### 2.1 patch

```
地址 0x920d90:  69 2c 00 54  →  29 00 00 14
```

- 原字节 `0x54002c69` = `b.ls #0x92131c`（栈溢出检查分支，0x920d7c 第 6 条指令，prologue+栈检查之后）。
- patch `0x14000029` = `b #0x920e34`（**无条件跳到函数自带的 null 返回尾声**）。
- 0x920e34 尾声（已存在）：`mov x0,x22(null); mov x15,x29; ldp x29,x30,[x15],#0x10; ret` —— 完整、栈平衡的 null 返回。

### 2.2 效果

- 0x920d7c 入口即跳尾声 → **永远返回 null，跳过全部内部逻辑**（含 0x921084 处 `bl 0xf923d4` 传故障文案、各字段写）。
- 3 个 caller 都拿不到故障 body → 无"无法正常联网"文案可显示 → **T+260s 故障弹窗应消失（或无内容不弹）**。

### 2.3 安全验证（强）

1. **null 返回本就是合法路径**：0x920d7c 自身已有 null 返回路径（0x920e34，经 `tbz w0,#4` 正常可达）。
   → caller 在正常运行中本就处理过 0x920d7c 返回 null 的情况，**不会因 null 崩溃**（否则 app 早已崩）。
2. **栈平衡**：prologue（0x920d7c STP / 0x920d80 mov x29,x15）已建帧；尾声 `mov x15,x29; ldp…ret` 完整回卷。Dart AOT 不用 x19-x28 传统被调用者保存寄存器（x26=Thread/x27=PP/x28=heap 约定寄存器），无需额外恢复。
3. **影子栈平衡**：caller 在 `bl 0x920d7c` 前压参数（如 0xc4d22c 的 `stp x16,x0,[x15,#-0x10]!`），调用后自行 `add x15,x15,#0x10` 弹出——与 0x920d7c 是否消费参数无关，**始终平衡**。
4. **副作用最小**：0xc4d22c 的全局写 `[Thread+0x80]+0xee0=FALSE`（0xc4d284）发生在 `bl 0x920d7c` **之前**，仍执行；0x920d7c 只是读该全局，跳过读取无影响。
5. **功能域隔离**：0x920d7c 仅被 3 个故障分发函数调用，文案故障专属；null 化只影响故障 body，不涉开屏/阅读/VIP。

### 2.4 与既有 patch 的关系（叠加，不冲突）

本 patch 与 TestC 基线（篡改窗 0x8e1dd0/0x8ef2b8、故障门控 0xbc1020、7 点 dialog-show NOP、去广告 smali）**正交**。
建议：在 TestC 基础上**追加** 0x920d90 这一点。原 7 点 dialog-show NOP 可保留（无害）也可移除（因 0x920d7c 已堵源）。

## 三、备选：若 0x920d7c 单点不足，null 化 3 个 caller

万一某 caller（如 0x920b3c）经 0xf216e4 独立弹不同文案的窗，可加 null 化 3 个 caller 入口。
**通用"入口返 null"配方**（覆盖任意 Dart AOT 函数前 3 条之后的 3 条指令，保留 STP+mov x29 建帧）：

```
func+8:  e0 03 16 aa   mov x0, x22                 (返 null)
func+12: fd 79 c1 a8   ldp x29, x30, [x15], #0x10  (回卷 STP)
func+16: c0 03 5f d6   ret
```

应用于 3 个 caller（`func+8` 即各自偏移 +8）：
- 0x920b3c+8 = **0x920b44**：原 `ef4101d1` → `e00316aa`；+12 `a00f40f9`→`fd79c1a8`；+16 `017041b8`→`c0035fd6`
- 0xc4cb30+8 = **0xc4cb38**：同配方
- 0xc4d22c+8 = **0xc4d234**：同配方

> 注：先用 0x920d90 单点测；不够再加 caller。三个一起 null 最稳但改动面大。

## 四、方向 A2（全局错误处理）与方向 D 评估

### 4.1 方向 A2：定位 runZonedGuarded / FlutterError.onError 注册点并截断
- **评估：被本报告的 0x920d7c 截断取代，无需单独做。**
- 理由：全局错误处理闭包是 pool 间接存储的全局字段闭包，解析它需 Code 簇（ref→.text）。
  **test2 已证实 blutter 在本定制 VM 失败（"cid 84 not in class table"）**，三分支都没拿到 Code 簇/闭包解析。
  而 0x920d7c 是故障文案唯一源、是所有错误处理路径的共同下游——**截断 0x920d7c 等价于截断错误处理的"显示"输出，却更精准**（不掩盖其它真错误的全局副作用，只去掉故障 body 构造）。
- 风险对比：截断全局 onError 会吞掉**所有**未捕获异常（掩盖真错误）；截断 0x920d7c 只去故障 body。后者更优。

### 4.2 方向 D：Dart 侧 NOP 广告 invoke（0x888b18/0x8880c0/0x88862c）
- **评估：可行但复杂，作为 0x920d7c 的后备，本轮不优先。**
- 0x888b18 实测：是 **async 状态机**（入口即按 state index 从帧恢复局部变量：`add xN,x29,w0,sxtw#2; ldr…`）。
  NOP invoke 需定位状态机深处具体 invoke BL + 保证状态机带合法返回值走完—— delicate。
- 0x920d7c 截断更干净（单点、非状态机），故优先 A。

## 五、喵喵饿了常驻块（次要）

### 5.1 结构（已查清）
- "喵喵饿了"文案 = ref **28525** → 槽位 **0x59bc**。
- **全 app 仅 2 函数加载**：0xae30e0（widget builder）与 0xa54178（VIP/donation ctor）。
- 0xae30e0 = 阅读页"喵喵饿了"常驻块的 **widget 构造函数**（callers=0，Flutter 框架经 build 管线间接调）。
  - 入口 `cmp w0,#0x14; b.ne 0xae32e4` 是 **widget 类型 switch**（type==0x14 才建本块），**非**简单显隐开关。
  - 函数体庞大：建 title/body/按钮等子 widget（bl 0x550bb4/0x4c9ba4/0x647300/0x654980/0x535040/0x654cd8/0x527768/0x654940/0x6527b8…），**无干净的"早返空 widget"路径**——直接 null 化会破坏 widget 树（build 返 null 必崩）。
- 0xa54178（VIP ctor）调奖励特权状态机 **0x8cf36c**（bool，16-17 callers）。

### 5.2 实验性 patch：强制奖励特权"已生效"（0x8cf36c → 恒 TRUE）
"喵喵饿了"块按猫饥饿状态显示；猫"已喂/特权生效"应隐藏。0x8cf36c 是"特权是否生效"判定（多 TRUE 返回 0x8cf3b4/0x8cf41c）。

```
地址 0x8cf374:  ef 81 01 d1  →  2a 00 00 14
```
- 原 `0xd10181ef` = `sub x15,x15,#0x60`（0x8cf36c 第 3 条，mov x29,x15 之后）。
- patch `0x1400002a` = `b #0x8cf41c`（跳到自带 TRUE 返回尾声 `add x0,x22,#0x30; mov x15,x29; ldp; ret`）。

**潜在双重收益**（与 backup"方案2"一致）：强制特权生效可能**同时**①隐藏喵喵饿了块 ②抑制阅读期广告。
**风险（中）**：0x8cf36c 有 16-17 callers，强制 TRUE 影响面广（奖励计时/特权判定/广告门控等），可能副作用。
**【需主控运行时验证】**：是否真能隐藏块 + 不引发其它异常。置信中，作实验项。

### 5.3 兜底
若上述均不理想：喵喵饿了块是装饰性的（reward 本就坏，点了也卡"正在加载"），可接受其可见但不可用。
用户诉求"消失/不遮挡/不可点击"的最稳静态解需找到包含该 widget 的父级 Visibility 条件——但 0xae30e0 经框架间接调，父级条件需 Code 簇解析（blutter 失败，暂不可得）。

## 九、本轮方法论小结

主控追不下去的"pool 间接闭包链"，test3 用**对象池槽位反向扫描**绕开了"闭包解析"这个硬问题：
不求"谁经闭包调了 0x920d7c"（需 Code 簇，blutter 都失败），而求"**故障文案这个 pool 槽被谁加载**"（字节级可扫），
直接锁定 0x920d7c 是唯一文案源 → 单点截断。这是 test3 池+xref 优势对第八轮核心难点的直接突破。

> 一句话：故障文案仅由 0x920d7c 唯一构建，显示走其内部运行时 stub（非 showDialog，故主控 NOP 全失败）；
> 截断 0x920d7c 入口（0x920d90: 692c0054→29000014，跳自带 null 尾声）= 单点根治故障弹窗，安全（null 本是合法返回）。
> 喵喵饿了块：0xae30e0 是唯一 widget builder（无干净隐藏路径）；实验性 force 0x8cf36c→TRUE 或可双效（隐藏块+抑广告），需主控验证。
