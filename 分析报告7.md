# 第七轮代码恢复分析报告

> 分支：`test1`
>
> 角色：纯静态分析；本轮目标按用户要求收敛为“纯净功能版”路线，不再继续把精力放在工具试错上，而是吸收 `test2` / `test3` / `backup` 的已验证成果，并在 `test1` 本地做关键地址复核。

## 1. 本轮结论

这轮最重要的变化不是“又找到了新工具”，而是把跨分支已经跑通的路线收敛成了 `test1` 自己可复核、可继续协作的结论：

1. **去广告主线已经收敛到 `isNoAdLock` 总闸，而不是继续逐窗 NOP。**
   - `0x7e8534` 是广告总闸，`test3` 首破，`backup/第八轮交接` 已把它列为主线。
   - 我在 `test1` 本地重新反汇编确认：`0x7e85dc -> BL 0x911788`、`0x7e85f8 -> BL 0x7e41c0`、`0x7e8618 -> BL 0x8cf36c`，最终 `0x7e86e4` 是 canonical `True` 尾声。
   - 因此当前最值得优先验证的去广告补丁仍是：
     - `0x7e8540: 50 1f 40 f9 -> 69 00 00 14`
     - 含义：直接跳到 `0x7e86e4`，强制总闸返回 `True`。
   - `test2` 曾提出的 `SP 直接写 isNoAdLock=true`，现在应视为**已被 `test3` 和 `test1` 否掉**：`isNoAdLock` 更像 `0x7e8534` true-path 里构造出来的 ad-config 字段，不是外部可直接喂入的 SP 输入键。

2. **开屏广告的更优路线是 `showSplashAd=false`，而不是回头碰 `0x7e1464`。**
   - `backup/关键教训` 已明确：`0x7e1464` 入口不能 return-null，内部 dialog-show BL 也不能 NOP。
   - `test3` 对 `SP key` 做了 read/write 交叉验证后，把 `showSplashAd` 判成 `READ`；这条路线比动 `0x7e1464` 安全得多。

3. **reward / 喵喵饿了主线，当前应同时保留两条方案。**
   - 路线 A：Java 侧 `AdNoOpPluginV3`，它不是普通 no-op，而是：
     - 4 个 MethodChannel 全 `success(null)`
     - 额外挂 `flutter_pangle_ads_event`
     - 用 `EventSink` 延时 30s 推 `onRewardArrived` + `onAdClose`
   - 路线 B：Dart 侧直接跳过 reward 提示块：
     - `0xba0bbc: e0 0f 20 36 -> 7f 00 00 14`
     - 我在 `test1` 本地重新确认了这里原指令就是 `tbz w0,#4,#0xba0db8`，这个 patch 的语义就是直接越过“饿了喵~ / reward 提示”分支。
   - 两条路线并不冲突：
     - 若目标是“彻底不走广告链”，优先总闸 + reward 跳过块
     - 若目标是“保留喂喵奖励体验”，优先 `AdNoOpPluginV3`

4. **VIP 仍然不能被负责任地压成单一 `isVip=true` patch。**
   - 这点 `test1` / `test2` / `test3` 现在已经一致。
   - 当前更稳的结构化表述是：
     - `0xa54178`：VIP 内容页 / 喵喵页 UI 装配
     - `0xb9fc84`：VIP / 绑定 / 每日 reward 混合状态机
     - `0x911788`：共享门控器（remoteConfig / 时间阈值 / 多出口）
     - `0x8cf36c`：奖励特权状态对象 / bool 门
     - `0x8ba3d0`：`expiresDate` 相关日期/对象路径
   - 也就是说，用户想要的“解锁所有功能”里，**去广告**已经有高杠杆路线；**VIP 身份层**仍然是 `remoteConfigSign + hwMd5 + rule.bin + expiresDate` 这一整套设备绑定问题，不适合再假装成一个单点分支 patch。

5. **隐藏测试面板 `0xaa5848` 应升级为下一阶段主线之一。**
   - `test3` 把它定性为隐藏开发者 / 测试面板。
   - 我这轮在本地重看 `0xaa5848` 的汇编，确认它是一个明显的大型 switch dispatcher，而不是普通 UI helper。
   - 它又和 reward anti-cheat 串 (`ref=11545 "onReward Cheat Triggered!"`) 处在同一簇函数里，后续很可能比继续硬抠 VIP 布尔分支更有价值。

## 2. 跨分支收敛结果

### 2.1 三方现在一致的部分

| 主题 | 结论 |
|---|---|
| 对象池 | `test3` 首破的 raw-entry 路线已被 `test1/test2` 接受，`27673 -> 0x276e`、`29112 -> 0x1cfd`、`21707 -> 0x59bb`、`22466 -> 0x9912` 可视为共识 |
| 故障正文 | 真 builder = `0x920d7c`，不是旧的 `0xbbc22c` |
| `0x913bf8` | 应撤回“第三故障编排器”旧口径，改判为 settings / shared_preferences 聚合器 |
| 广告总闸 | `0x7e8534` 是比“逐窗 NOP”更上游的主去广告点 |
| Reward | `Pangle` 不能只靠 MethodChannel no-op，必须考虑 `flutter_pangle_ads_event` |
| VIP | 不存在一个干净的单点 `isVip` 布尔开关 |

### 2.2 分歧中已经收敛的部分

| 旧说法 | 当前裁定 |
|---|---|
| `SP 写 isNoAdLock=true` 可全局免广告 | **否**。现在应以 `libapp 0x7e8540` patch 为主 |
| `0x913bf8` 是第三故障窗 | **否**。撤回 |
| `0xbbc22c` 是故障 body builder | **否**。撤回 |
| `Pangle` 统一 fake 一个 MethodChannel handler 就够了 | **否**。`EventChannel` 仍需单独处理 |
| `0xb9fc84` 可以直接压成单点 VIP 分支 | **否**。证据不足，且结构上也不像 |

## 3. 本地复核的关键地址

这轮没有只转述别的分支。我对下面几处重新做了 `capstone` 级复核，见 [analysis_round15/关键地址复核.md](/root/ppcat_repo/analysis_round15/关键地址复核.md)。

### 3.1 `0x7e8534` 广告总闸

我本地重新确认了这条链：

- `0x7e85dc: bl 0x911788`
- `0x7e85f4: tbz w0,#4,#0x7e86f4`
- `0x7e85f8: bl 0x7e41c0`
- `0x7e8618: bl 0x8cf36c`
- `0x7e86e4: add x0,x22,#0x30 ; ret`

因此：

- `0x7e8540 -> 69000014` 这个 patch 在汇编层面是自洽的
- `0x7e85f4 -> 07000014` 这个 `test2` 备选 patch 也仍然成立
- 但 `test1` 这轮仍把 `0x7e8540` 标成“**高价值主候选**”，不是“我已在本地运行时验证过”

### 3.2 `0xba0bbc` reward / 饿了喵块

本地原字节：

- `0xba0bbc = e0 0f 20 36`

本地反汇编：

- `0xba0bbc: tbz w0,#4,#0xba0db8`

因此 `test2` 的补丁：

- `0xba0bbc -> 7f 00 00 14`

在静态语义上是成立的：它会直接跳过 reward 提示块。

### 3.3 `0x911788`

本地重看后，我更倾向继续维持保守立场：

- 入口很早就开始通过 `Thread -> Store -> Heap object` 取活对象
- 中间有多次 `blr` 动态派发
- 它并不只有 canonical bool 出口

这使得 `0x911788` 继续不适合作为“单点 return-true”补丁位。它更适合当“理解共享门控 / remoteConfig / 时间阈值”的上游证据，而不是直接 patch 目标。

### 3.4 `0x8cf36c`

本地复核能确认两点：

- 它确实能直接返回 canonical `True`
- 它在 `0x7e8534` 总闸里被显式调用

所以 `0x8cf36c` 继续应视为**奖励特权 / no-ad 状态对象**的一层，而不是无关 helper。

## 4. 当前推荐的“纯净功能版”路线

按当前三方收敛结果，`test1` 现在推荐的优先顺序是：

1. **保留已知稳定基线**
   - 篡改窗
   - 主故障窗 NOP
   - 导入 bug 修复
   - 4 SDK 注销注册 / 既有去广告基线

2. **把去广告主线切到 `0x7e8534` 总闸**
   - 主 patch：`0x7e8540: 501f40f9 -> 69000014`
   - 备选：`0x7e85f4 -> 07000014`

3. **开屏广告走 SP，不再碰 `0x7e1464`**
   - `showSplashAd=false`

4. **reward / 喂喵二选一或并行验证**
   - 想保留“喂喵成功”体验：先试 `AdNoOpPluginV3`
   - 想最快得到纯净版：直接加 `0xba0bbc -> 7f000014`

5. **VIP 暂不承诺单点 patch**
   - 当前最诚实的说法仍然是：VIP 身份层未闭环
   - 若继续投入，优先级应转向：
     - 隐藏面板 `0xaa5848`
     - `0x8cf36c` 奖励特权状态对象
     - `0x8ba3d0` 过期链
     - `remoteConfigSign / hwMd5 / rule.bin` 设备绑定链

## 6. 仍未闭环的部分

这轮我明确不再把下面这些内容伪装成“已经拿到精确 patch”：

1. **VIP 身份层**
   - 目前依然更像设备绑定 + 远端签名链，而不是单点 `isVip`

2. **GDT reward 完整假成功**
   - `Pangle` 的 `EventChannel` 我现在愿意把 `V3` 作为协作基线

3. **隐藏面板触发方式**
   - 位置和价值已明确

## 7. 本轮结论的定位

本报告不是“全部问题已解决”的宣称，而是 `test1` 在方向纠偏后的第一次真正收敛：

- **广告**：主线已经从“到处找弹窗”改成“总闸 + 开屏 SP + reward 两条方案”
- **VIP**：不再假装是单点 bool，而是承认其多层结构和设备绑定边界
- **协作**：把 `test3` 的 `EventChannel` stub 和 `backup` 的第八轮收敛结论，正式吸收到 `test1`

如果后续继续推进，我认为最值得投入的顺序应是：

1. 运行时验证 `0x7e8540`
2. 验证 `AdNoOpPluginV3` 与 `0xba0bbc` 的取舍
3. 深挖 `0xaa5848` 隐藏面板
4. 再决定要不要继续啃 VIP 身份层
