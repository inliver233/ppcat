# 已确认的技术细节(请先读,避免重复劳动)

本文件汇总前期已查清的事实,供分析时直接使用。所有偏移基于本仓库 `lib/arm64-v8a/libapp.so`(16352152 字节)和 `lib/armeabi-v7a/libapp.so`(17629776 字节)。

## 一、技术架构

- **框架**:Flutter(Dart AOT)
- **业务逻辑**:全部在 `libapp.so`(Dart AOT 机器码)
- **引擎**:`libflutter.so`,疑似**定制构建**(标准 blutter 的版本探测在此引擎上可能失败,见任务文档 4.3 fallback)
- **本仓库的 libapp.so / libflutter.so 与网上原版字节完全一致**(SHA256 已核对),所以分析结果可直接对应线上版本

ELF 布局(arm64):
```
.kDartVmSnapshotData           vaddr=0x3330   size=0x45c930   (对象池/字符串在此)
.kDartIsolateSnapshotData      vaddr=0x3330   size=0x45c930
.kDartVmSnapshotInstructions   vaddr=0x460000 size=0x4ce0
.kDartIsolateSnapshotInstructions vaddr=0x464ce0 size=0xb329d0  (Dart 机器码在此)
.text(整段)                    vaddr=0x460000 size=0xb376b0
```

> 注意:Dart 字符串有两种编码 —— **OneByte(Latin-1,按 UTF-8 字节存)** 和 **TwoByte(UTF-16LE)**。中文串是 UTF-16LE,英文串多数是 UTF-8。下面的偏移已标注编码。

## 四、Flutter 平台通道(Java↔Dart 桥)的检测方法清单

Dart 启动约 7 秒内,通过一个平台通道(plugin,通道前缀 `plugins.flutter.io/`)向 Java 侧依次调用以下方法收集环境信息。**这些是聚合判定的输入来源**,在 asm 里应能看到对这些方法名字符串的引用:

```
getAll  getAppInfo  getApplication  getAppMetaData  getActivity  resolveActivity
getPackages  getMaps  getAssets  getAssetsFileSize  getFileSize
getPMProxy  getBinder  getCreator  getClassLoader  getSuperclass
checkFileRedirect  checkMapsRedirect  getStackTrace  getZipFileList
getManifestXml  runCommand
```

各方法的典型返回(供理解 asm 逻辑):
- `getAll` → 一个 Map,含 `appName/version/buildNumber/packageName/sdk/androidId/hardwardInfo/signature/path`
- `getAppInfo` → 多行字符串,每行 `字段名：值`(注意冒号是全角 `：` U+FF1A),含 `className`/`processName`/`sourceDir` 等约 40 个字段
- `getApplication` → 字符串 `类名|父类名`
- `getPMProxy` → 字符串 `代理类名|类型`
- `getMaps` → `/proc/self/maps` 内容(加载的 .so 路径列表)
- `getBinder/getCreator/getClassLoader` → 类似的 `类名|类型` 串
- `getAssetsFileSize` → 8 个数字,对应 8 个 asset 文件大小
- 多数 `check*/getStackTrace/getZipFileList/getManifestXml/runCommand` 返回 null

## 五、关键事实(已验证,避免重复劳动)

1. **提示框是 Flutter 渲染的**(UI dump 里是 `content-desc` 而非 `text=`),不是原生 Android Dialog → 判定逻辑**确实在 Dart(libapp.so)里**,不在 Java/dex 里。
2. **提示框文案不在 Java 侧**(Java 侧的字符串解密日志里没有这些文案),也不以明文 UTF-8 存在于 .so(是 UTF-16LE)。
3. **没有网络验证** —— 判定是纯本地的,聚合上述检测结果后做。
4. **触发时机**:应用启动后约 7 秒,且只在这一刻收集一次(后续不再重复)。
5. **libapp.so 与原版字节一致** → 线上原版应用跑同一套判定;原版不弹框是因为它的检测结果恰好全部符合期望(没有被改动)。本仓库这份 libapp.so 的逻辑完全等价。

前期已逐一核对,以下**不是**弹框触发原因,无需重复分析:

## 七、给分析者的提示

- 先 `grep` 上面第二节、第三节的字符串,在 `pp.txt` 里确认它们的对象池偏移,再到 `asm/` 里找 xref
- 重点找:**一个函数,它读取多个检测结果(getAll/getAppInfo/getApplication/getPMProxy/getMaps 等的返回),做一连串比较/布尔运算,最后条件跳转去调 showDialog**。这就是聚合判定函数
- 找到后,记录该函数在 `asm/` 中的文件名、起始地址、以及"决定弹框"那条条件分支的**精确地址和机器码**
- 若 blutter 对这个定制引擎输出质量不佳(符号缺失),至少把对象池 dump(`pp.txt`)拿到,字符串层面就能定位大部分逻辑
