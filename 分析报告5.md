## 一、对象池局部反序列化

### 1.1 Snapshot 格式确认

Isolate snapshot header (fixed-format, from unflutter Go source):
```
+0x00: magic   [4]byte {0xf5, 0xf5, 0xdc, 0xdc}
+0x04: length  int64 LE
+0x0c: kind    int64 LE (3=FullAOT)
+0x14: hash    [32]byte
+0x34: features null-terminated string
+0xed: cluster list start (588 clusters)
```

VM Isolate snapshot alloc header:
```
num_base_objects = 1006
num_objects = 163332
num_clusters = 588
instr_table_len = <value>
instr_data_off = <value>
```

### 1.2 关键 CID

| CID | 名称 | alloc count | 说明 |
|---|---|---|---|
| 84 | String | 32135 | OneByteString + TwoByteString |
| 17 | Code | 31387 | AOT 编译后的代码对象 |
| 7 | Function | 13207 | Dart 函数元数据 |
| 5 | Class | ~10000+ | Dart 类定义 |
| 21 | ObjectPool | 1 (内含大量 sub-array) | **核心: ref→slot 映射** |
| 43+ | Instance | 多个 | 运行时对象实例 |

### 1.3 锚点验证 ✓

| ref | 内容 | VLE | snapshot file offset | pool slot | .text 加载点 | 验证 |
|---|---|---|---|---|---|---|
| 26842 | 短版正文2 | `5a 51 81` | 0x2eb53d | **0x0ad88** | 多处 | ✓ |
| 30947 | "非法篡改"标题 | `63 71 81` | 0x2eb541 | **0x0ad90** | 0x5xxxxx 等多处 | ✓ |
| 30922 | 长版正文 | `4a 71 81` | 0x2f0416 | **0x17168** | 0x8e52d0, 0x8ef414 | ✓ |
| 27673 | 故障正文 | `19 58 81` | 0x2eeecb | **0x13c68** | 0xbbc31c, 0x913d3c | ✓ |
| 29112 | "故障"标题 | `38 63 81` | 0x2ece06 | **待精确** | 待确认 | ⚠️ |
| 21707 | "每日喂喵"标题 | `4b 29 81` | 0x2f8c82 | **待精确** | 待确认 | ⚠️ |
| 22466 | 每日喂喵正文 | `42 2f 81` | 0x305c5b | **待精确** | 待确认 | ⚠️ |

所有 ref 的 VLE 编码均在 snapshot 中**唯一出现**,且前导 type byte = 0x80 (kTaggedObject),确认了 snapshot 中的位置。

### 1.4 为什么本轮仍未完成 ref→pool_slot 精确映射

1. **ObjectPool fill 数据编码差异**: pool 条目内的 ref 使用 **ReadRefId**(signed byte accumulation,b`result = b_signed + (result << 7); if bit7: return result + 128`),而非 fill 数据中其他位置的 unsigned VLE
2. **Fill 数据组织方式**: 588 个 cluster 的 fill 数据按 cluster 顺序交错存储。ObjectPool (cluster[572]) 位于 fill 数据的末尾,需要正确解析前 571 个 cluster 的所有 fill 数据才能到达
3. **定制 VM cluster 格式差异**: 部分 cluster 的 fill 格式与标准 Dart 2.19 不同 (如 Instance cluster 的 bitmap 格式),导致解析器在某处越界
4. **现有工具局限性**: 
   - `extract_entry_points.py` 能解析 String+Function+Code cluster,但跳过 ObjectPool (仅 `pass`)
   - `dump_object_pool.py` 只做正则提取字符串,不解析 pool 结构
   - `unflutter` Go 工具只导出 ref→string,不导出 ObjectPool 索引

### 1.5 改进行动

要完成 ref→pool_slot 的精确映射,需要:
1. 从 `extract_entry_points.py` 出发,正确解析 Instance (CID≥43) 的 fill 数据 (bitmap 格式是主要障碍)
2. 或在 fill 数据中定位 ObjectPool 的特征模式 (每个 sub-array 以 unsigned VLE length 开头)
3. 已知锚点 ref=27673→slot=0x13c68 可作为验证标准

**本轮产出**: 改进的 snapshot 解析脚本 `parse_pool_fill.py` (见附件),能解析到 CID=21 之前的 cluster,但在 Instance cluster 的 bitmap 处 blocked。

## 二、弹窗 1: 每日喂喵窗 (★ 重点)

### 2.1 弹窗特征

- **触发条件**: 点漫画页"喵喵饿了"→ reward 激励广告尝试加载 → 广告失败 → 弹窗
- **标题**: ref=21707 "每日喂喵"
- **正文**: ref=22466 "无法加载数据,请稍后再试!\n如果开启了屏蔽广告的相关软件或插件,请关闭然后重试!\n每天喂一次喵,喵喵就不来找你了~"
- **按钮文本**: ref=28525 "喵喵饿了"
- **奖励提示**: ref=18266 "是否看推荐信息获取喵粮喂喵？"
- **重试**: 每 30-45 秒重试 reward 加载

### 2.2 Java 层 Reward 机制 (来自 logcat + backup 分支)

去广告前 logcat:
```
rewardVideo on channel plugins.hetian.me/gdt_plugins
showRewardVideoAd on channel flutter_pangle_ads
```

reward 涉及两个 SDK:
- **GDT**: `plugins.hetian.me/gdt_plugins` → reward video
- **Pangle**: `flutter_pangle_ads` → showRewardVideoAd

去广告后这两个 channel 的 Java handler 不存在 → reward 调用永远无响应 → Dart 层超时/失败 → 弹每日喂喵窗。

### 2.4 ★★ Patch 方案

#### 方案 A (推荐★): Mock reward 成功回调 — "点喵喵饿了直接喂喵"

**原理**: 在 Dart 侧找到 reward 的 onReward 成功回调路径,让其永远触发成功。效果是: 用户点"喵喵饿了"→ 直接获得喵粮奖励 → 无需看广告 → 无弹窗。

**实施**:
1. 搜索 ref=2839 (`onAdReward`) 和 ref=6808 (`onReward`) 的 .text 加载点
2. 定位 reward 状态机的成功/失败判定分支
3. 将"失败"路径重定向到"成功"路径

**优点**: 
- 用户体验最好 (直接获得奖励,无等待)
- 不需要 patch 弹窗本身

**缺点**: 
- 需要精确找到 reward 判定分支 (当前因 pool 未完全反序列化受阻)

#### 方案 B: Patch 每日喂喵窗 orchestrator entry return null

**原理**: 找到每日喂喵弹窗的 showDialog 编排器,入口改为 return null。

**实施步骤**:
1. 扩大搜索: 在 asm.txt 中搜索除 0x8b1390 外的其他 showDialog 相关调用
2. 搜索所有 `MOV X1, #0x6` 附近的函数 (Flutter showDialog 常见参数)
3. 在 pool_accesses.txt 中搜索: 是否有函数同时访问两个相邻的中等 slot (title + body 对)
4. 在函数 0xbbc22c (故障正文 builder) 附近搜索每日喂喵正文 builder
5. 候选区域: 0xba0000-0xbf0000 中具有 CMP W0,W22 + B EQ/NE 模式的函数

**一旦定位 orchestrator**:
```
地址: <orchestrator_entry>
原始: fd 79 bf a9 ...  (STP X29,X30,[X15,#-16]!; ...)
改为: e0 03 16 aa c0 03 5f d6  (MOV X0,X22; RET X30)
```
前提: 验证该函数所有出口返回 null (X22)。

#### 方案 C: Java 层 patch Reward 回调 (备选)

在 Java 层让 GDT/Pangle 的 reward 回调直接返回成功。但需解密相关 smali 方法名 (Arabic 加密),且两个 SDK 的回调接口不同。

### 2.5 ★ 每日喂喵 orchestrator 的 .text 候选

由于 pool 反序列化未完成,给出以下**搜索策略**以在 asm.txt 中定位:

**策略 1 — 搜索正文构建器**: 搜索同时加载两个相邻 pool slot (title+body) 的函数。已知篡改窗加载 0x0ad88+0x0ad90 (距离 8 字节),每日喂喵的 title+body slot 也应邻近。

**策略 2 — 搜索 showDialog 调用模式**: 已知故障窗 orchestrator (0xbc0f40) 在调用 showDialog 前设置 `MOV X1, #0x6`。在 asm.txt 中搜索所有类似的 `MOV X1, #0x6` 前导指令,交叉 pool_accesses。

**策略 3 — 搜索 reward 字符串引用**: 计算 ref=2839/6808/9164 的近似 pool slot:
- ref=2839 在文件 0x47bc5 (接近 snapshot 开头)
- ref 越小,pool slot 越小
- 线性估计: slot ≈ 0x125a (pool index ~587)
- 在 pool_accesses.txt 中搜索 0x01250-0x01300 范围的访问

## 三、弹窗 2: 残留故障窗

### 3.1 现象与证据

- 0xbc0f40 entry return null 已 patch → 启动路径故障窗不弹 ✅
- 但**搜索页/漫画页**偶尔仍弹"故障:无法正常联网..."弹窗
- **全 .text 仅 1 处加载故障正文 ref=27673**: 0xbbc31c (在 function 0xbbc22c 内)
- **全 .text 仅 1 处调用 showDialog helper 0x8b1390**: 0xbc133c (在 function 0xbc0f40 内)
- **全 .text 仅 1 处 BL 到 0x8b6100** (call bridge→fault body builder): 在 function 0xbc0f40 内

### 3.2 推理

既然 0xbc0f40 已 patch (entry return null),但故障窗仍偶尔出现,说明:
1. 残留故障窗使用了**不同的 showDialog 路径**(不是 0x8b1390)
2. 可能通过 vtable 直接 dispatch 到 Flutter 的 `showDialog` 函数
3. 或使用了另一个独立的弹窗编排函数

### 3.3 定位策略

1. **搜索所有 call 0xbbc22c 的路径**: 虽然 0xbbc22c 是间接调用的 (0 direct BL),但 call bridge 0x8b6100 可能有其他调用者。搜索 asm.txt 中所有 `BLR` 到 0xbbc22c 附近 PC 的指令。
2. **搜索第二个故障标题加载点**: ref=29112 ("故障") 的 pool slot 若能被定位,其 .text 加载点就是第二个故障窗的标题 builder。
3. **扩大搜索**: 搜索所有使用 `MOV X1, #0x6` 的函数 (showDialog 常见参数),排除已知的故障窗和篡改窗路径。

### 3.4 Patch 方案

一旦定位第二个故障窗 orchestrator,使用已验证的安全手法: 若所有出口返回 null,入口改为 `MOV X0,X22; RET X30`。

如果用不同 showDialog 路径,也可尝试在 showDialog 调用前改条件分支,强制跳过弹窗。
