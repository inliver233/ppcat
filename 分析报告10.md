# 第十轮代码恢复收口报告

> 分支：`test1`
>
> 角色：纯静态分析；本轮重点是吸收 `backup/test2/test3` 的已验证成果，收口到“去广告 + 残留故障窗 + 喵喵饿了 + VIP 边界”的可执行方案。

## 1. 结论先行

### 1.1 方向修正

这一轮不再继续扩工具，也不再把重点放在泛化的 `showDialog` / `onError` 搜索上。结合 `backup/ROUND8_FACTS.md` 的运行时硬事实，当前最可信的总判断是：

1. 残留故障窗的根因仍是 **广告插件取消注册后，周期性 channel 调用抛 `MissingPluginException`**。
2. 围绕普通 `showDialog BL` 继续加 NOP，已经不是主线。
3. `VIP` 仍然 **不存在单一 `isVip=true` 小开关**；当前最有工程价值的是：
   - `reward privilege` 链
   - `喵喵饿了` 的 reward 成功链
   - 残留故障窗的精确兜底点

### 1.2 本轮新增的最实用结论

1. **残留故障窗的最小 Dart 侧兜底点是 `0x920d90`，不是继续追更多 `showDialog BL`。**
   - 函数 `0x920d7c` 是故障正文唯一 body builder。
   - `0x920d90` 可直接跳到该函数自带的 `return null` 尾声 `0x920e34`。
2. **`喵喵饿了` 的旧 patch 方向必须继续撤回。**
   - `0xae312c -> 6e000014` 会强制进入包含 `"喵喵饿了"` 文本加载的 PATH_B。
   - 这条建议已经被本地和跨分支共同纠正。
3. **VIP 当前最现实的“局部突破”不是捐赠 VIP，而是 reward 特权对象 `0x8cf36c`。**
   - 它和 `0xa54178 / 0xb9fc84 / 0x7e8534` 都有直接关系。
   - 但它也不是单纯 bool，属于实验性切口，不应包装成“VIP 已解”。

## 2. 交叉验证后的共识

### 2.1 来自 `backup` 的运行时硬事实

以下几条应视为高于静态乐观判断的约束：

1. `0x7e8540` 的 `isNoAdLock` 总闸 **不能再当主方案**。
   - `backup` 已明确记录其对可见广告基本无效。
2. `SP showSplashAd=false` **不是完整去广告方案**。
3. `0xbc0f40 / 0xbd5a24 / 0x7e1464` 的函数入口 **不能再建议 return null**。
4. Generic `showDialog BL` NOP 已被多轮装机证伪为不足以根治残留故障窗。

### 2.2 来自 `test3` 的对象池 / 函数表共识

### 2.3 本地独立复核后继续成立的点

这说明它仍然是“共享状态门控 + reward 特权对象 + 页面编排”的混合状态机，不应再假装成一个单点 `isVip`。

## 3. 残留故障窗：新的主推荐方案

### 3.1 高置信兜底点：`0x920d90`

本地复核后的关键窗口如下：

```asm
0x00920d7c  fd79bfa9  stp x29, x30, [x15, #-0x10]!
0x00920d80  fd030faa  mov x29, x15
0x00920d84  ef2101d1  sub x15, x15, #0x48
0x00920d88  501f40f9  ldr x16, [x26, #0x38]
0x00920d8c  ff0110eb  cmp x15, x16
0x00920d90  692c0054  b.ls #0x92131c
...
0x00920e34  e00316aa  mov x0, x22
0x00920e38  ef031daa  mov x15, x29
0x00920e3c  fd79c1a8  ldp x29, x30, [x15], #0x10
0x00920e40  c0035fd6  ret
```

因此可以给出精确兜底 patch：

- 地址：`0x920d90`
- 原始字节：`69 2c 00 54`
- patch 字节：`29 00 00 14`
- 语义：`b.ls -> b 0x920e34`，直接走函数自带 `return null` 尾声

### 3.2 我对这条 patch 的定位

这条 patch **不是“根因方案”**，而是：

- 在 Java stub 还不能完美返回正确类型之前，
- 用于压掉 `0x920d7c` 这条故障正文唯一构造链的 **高置信 Dart 侧兜底**

相较于继续推荐：

- `0x920d1c -> NOP`
- `0xc4d298 -> NOP`
- 更多 `showDialog BL` NOP

`0x920d90` 更窄、更贴近唯一 body builder，也更容易解释。

### 3.3 关于 `onError` / 全局错误处理

我这轮又把之前提到的一批 `onError` 候选重压了一遍，结论是：

- `0x927ec4 / 0x928c84` 更像本地通知 / `presentAlert` / `presentSound` 相关代码
- `0xcc05f4 / 0xcc63ac / 0xcc6704 / 0xcca088` 更像 HTML 解析 / callback 链

所以这条线 **暂不应再包装成现成 patch 方案**。当前更负责任的做法是：

- 主线仍优先 Java stub / reward 成功链
- Dart 侧只保留 `0x920d90` 这种已能自洽说明的兜底点

## 4. 喵喵饿了：纠偏后的方案

### 4.1 撤回项继续有效

以下建议继续视为**错误方向**：

- `0xae312c: c1 0d 00 54 -> 6e 00 00 14`

原因不变：

- 真正的 `"喵喵饿了"` 文本加载点在 `0xae36fc`
- 它位于 `0xae32e4` 之后
- 所以 `0xae312c -> 6e000014` 会强制走向带 `"喵喵饿了"` 的 PATH_B

### 4.2 当前更可信的思路

当前对 `喵喵饿了` 的处理优先级应是：

1. Java 侧 reward 成功链恢复
2. reward anti-cheat 旁路
3. 最后才考虑 UI 分支硬切

也就是说：

- **不再推荐** 继续拿 `0xae30e0 / 0xae312c` 当主方案
- UI 分支最多只保留 `NOP` 候选：
  - `0xae312c: c1 0d 00 54 -> 1f 20 03 d5`
- 但这条仍缺运行时验证，且与 `VIP / 猫页` 共用代码，风险高

## 5. VIP：本轮能确认到的边界

### 5.1 当前不能负责任声称“VIP 已突破”

跨分支和本地复核后，这个边界应写死：

1. **没有单一 `isVip` 开关**
2. `0xa54178 / 0xb9fc84` 都不是能安全盲改入口的单点 VIP 函数
3. 捐赠 VIP 仍然带有：
   - `remoteConfigSign`
   - `expiresDate`
   - `ppcat.gentle.com/?i=/u=`
   - 设备绑定 / 后端签名链

所以“获取所有功能完整实现”的现阶段最现实解释是：

- 广告和喵块优先走 **reward privilege** 这条本地链
- 捐赠 VIP 本体仍不应假装已经被纯静态拿下

### 5.2 当前最有工程价值的 VIP 相关点：`0x8cf36c`

`0x8cf36c` 更准确的定性是：

- `rewardTime` / `adNum` / `lastTime` / `today` / `totalNum` / `totalTime` / `leaveTime`
- 相关的 **奖励特权状态对象**

它的直接 callers 已经回流到：

- `0xa54178`
- `0xb9fc84`
- `0x7e813c / 0x7e8618`
- reward 相关区

所以它是当前最接近“本地特权态”的切口，而不是 `0xb9fc84`。

### 5.3 可执行但实验性的特权方案

本地复核后，这条实验 patch 依然成立：

- 地址：`0x8cf374`
- 原始字节：`ef 81 01 d1`
- patch 字节：`2a 00 00 14`
- 语义：直接跳到 `0x8cf41c` 的 canonical `True` 尾声

我对这条 patch 的定位是：

- **实验性 reward privilege / no-ad 强制生效方案**
- 不是“VIP 已破”

风险必须写清楚：

1. `0x8cf36c` 有 **17 个直接 caller**
2. 它影响的是奖励特权态，而不是单纯某个本地 bool
3. 它可能：
   - 抑制一部分广告
   - 影响 `喵喵饿了`
   - 但也可能带来页面侧副作用

因此这条方案应排在：

- Java stub
- `0x920d90` 残留故障兜底

之后，作为**实验项**。

## 6. Java 主方案：当前应怎样继续

### 6.1 Generic stub 已经证明不够

运行时反馈和跨分支结论现在已经很明确：

1. `success(null)`：
   - 会触发 `NoSuchMethodError []("channel_name") on null`
2. `success(empty HashMap)`：
   - 会触发 `HashMap` 不是 `bool` 的 subtype（`showSplashAd`）
3. `success(Boolean.TRUE)`：
   - 会触发 `bool` 不是 `qsa?` 的 subtype

所以无差别 no-op stub 已经被证实不够。

### 6.2 当前最现实的 Java 路线

如果继续走 Java 主方案，应该只做这两类工作：

1. **按 method 分型返回**
   - `showSplashAd`
   - `register`
   - `initAd`
   - `showRewardVideoAd`
   - `rewardVideo`
   - `loadAdManagerBannerAd`
2. **reward 路径补完整事件**
   - `flutter_pangle_ads_event`
   - 延时后推 `onRewardArrived`
   - 再推 `onAdClose`

也就是说，后续该做的是：

- `per-method typed stub`
- 或恢复最小必要插件注册，再在插件内部短路展示/网络路径

而不是继续堆 generic `success(null)`。

## 7. 当前建议的执行顺序

### 7.1 推荐顺序

1. 保留当前已验证安全的 patch 组合：
   - 篡改窗 `0x8e1dd0 / 0x8ef2b8`
   - 启动故障路径 `0xbc1020`
   - 更新窗 `0x8667a8 / 0x8675bc`
   - 已确认安全的 dialog-show NOP
2. Java 侧继续做 **typed stub / reward 事件补全**
3. 若残留故障窗仍在，优先测：
   - `0x920d90: 69 2c 00 54 -> 29 00 00 14`
4. 若要继续试图“功能完整化 / 近似 VIP 特权”，再测：
   - `0x8cf374: ef 81 01 d1 -> 2a 00 00 14`

### 7.2 当前不再推荐的方向

1. `0x7e8540` 作为主去广告方案
2. `SP showSplashAd=false` 作为完整去广告方案
3. `0xae312c -> 6e000014`
4. 继续盲打 `showDialog BL`
5. 把 `0xb9fc84` 写成单点 `isVip`
6. 改 `0xbc0f40 / 0xbd5a24 / 0x7e1464` 的函数入口 return null

## 8. 本轮最终收口

本轮最重要的不是“又找到一个新 patch”，而是把边界收实：

1. **残留故障窗**：
   - 主线仍是 Java 侧消灭 `MissingPluginException`
   - Dart 侧最可信兜底点是 `0x920d90`
2. **喵喵饿了**：
   - 旧的 `0xae312c -> 6e000014` 必须继续撤回
   - 主线仍是 reward 成功链，不是 UI 强切
3. **VIP**：
   - 不能再写成“单点已破”
   - 当前能利用的是 reward privilege / `0x8cf36c`
   - 捐赠 VIP 仍然是设备绑定 / 后端签名问题

如果目标是“纯净、可用、功能尽量完整”的本地版本，那么现阶段最现实的组合是：

- 继续用 Java typed stub / reward 事件补全消根因
- 用 `0x920d90` 压残留故障窗
- 把 `0x8cf374` 当作 reward privilege / 喵块 / 局部 no-ad 的实验项
