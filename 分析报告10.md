## 〇、一句话定论（三方交叉，高置信）

**用户"看广告→奖励→块消失"的隐藏路径运行时真实存在，但静态上无法精确定位**——喵喵块 builder `0xa52920→0xa52bb0` 内部无任何隐藏分支（字节铁证，C2/wf_state多路确认），隐藏判断在 builder 的调用者层（reader build），而 reader build 经 pool 间接 + blutter cid84 崩溃，**静态不可定位**。这是真实的天花板，不是方法缺陷。

**所以"让 app 自己走原生隐藏路径"的静态 patch 走不通**。但 ground truth 证明该逻辑运行时存在——只是我们够不到它。能做的：用 app 自己的"渲染空"widget（obj44988）替换块（绕开 cid 墙），让块恒空。这是当前静态能给的最优解，主推装机验证。

### 1.1 关键字段的权威命名（C1 突破，test3 独有）

C1 用 config-loader `0x913bf8` 的"载key slot→解析→写G偏移"模式，反查 G 字段名（权威，非猜）：

| G 字段 | 写入点 | key slot | ref | 字段名 | 是隐藏开关? |
|---|---|---|---|---|---|
| G+0x1028 | 0x913e00 | 0x278e | 18152 | **isFavChangePos** | ❌ 收藏位置设置 |
| G+0xed0 | 0x9141e4 | 0x279f | 8549 | **autoCleanCacheType** | ❌ 清缓存设置 |
| G+0xef8 | 0x914298 | 0x27a2 | 7606 | **rewardTime** | ❌ reward时间戳(独立) |

→ 喵喵块 builder 0xa52920 读的 0x1028/0xed0 **都是普通 app 设置项**，与猫/广告/隐藏**无关**。这推翻了报告9 的"猫饿/猫饱切换"猜测（其实只是内容变体 slot0xbb8/0xbc0 的选择）。

### 1.2 reward→喂猫→隐藏的完整业务链（已定位写入侧）

```
reward-prompt 0xbc5dc8 ("是否看推荐获取喵粮喂喵？获取/取消")
    │ (两入口: 阅读页块onTap + 设置页"每日喂喵" 都汇入)
    ↓
奖励完成回调 → 写 rewardTime G+0xef8 (setter 0xa7fdb4)
    ↓
喂猫 handler 0xa56e00 ← 被 0xbc5dc8 调用
    ↓
写猫对象 field@0x33 = FALSE (喂饱) @0xb68288
    ↓
reader build (cid84墙后) 读猫饿/日期 → 当天不构造/不挂载块
```

**写入侧（奖励→喂饱）静态完全清楚**。**读取侧（reader build 决定隐藏）静态不可定位**——这就是天花板。

### 1.3 对 test1 报告32 的纠偏（交叉验证价值）

test1 报告32 称"同步层 `0x913bf8→0x914220` 把成功态翻译成 0x1028/0xed0 消费态"——**本轮证伪**：0x913bf8 的串是 `isAutoLayout/isTranslucentNavBar/isFavChangePos/isScaleFont...`，它是**app 设置项的 config-loader**（把设置字典刷到 G），与 reward 无关，函数体**不含 0xee0**。test1 的"翻译"过度推断。三方交叉避免了 test1 这个误判扩散。

## 三、★ 给主控的 patch 清单（按推荐顺序）

### 【第1优先，最强未测】obj44988 返回（app 原生渲染空 widget，绕开 cid 墙）

obj44988（slot0x753）是 app 在 0x546760"尺寸==0"分支自用的"渲染空"const Widget（231 处复用，确定非 String）。让 builder 返回它 → 块被 app 原生空 widget 替换。

**方案①（覆写 builder 入口，保留序言，推荐）**：
```
地址 0xa52bb8 (func+8, 16 字节覆写):
  原: ef 41 00 d1 | 50 1f 40 f9 | ff 01 10 eb | a9 05 00 54
      (sub x15,#0x10; ldr x16,[x26,#0x38]; cmp x15,x16; b.ls 0xa52c78)
  改: 60 4f 5d f9 | ef 03 1d aa | fd 79 c1 a8 | c0 03 5f d6
      (ldr x0,[x27,#0x3a98]=obj44988; mov x15,x29; ldp x29,x30,[x15],#0x10; ret)
```
- 保留序言 `0xa52bb0 stp + 0xa52bb4 mov x29,x15`，从 0xa52bb8 覆写。
- 栈对称：func+8 时 x15==x29，尾声 mov x15,x29 复位 → ldp 弹出正确帧。
- 返回 idiom 与 app 自用的 0x5467ac 逐字节一致。

**方案②（单字替换 tail-call，备选）**：
```
地址 0xa52b94 (1 字替换):
  原: 07 00 00 94   bl 0xa52bb0
  改: 60 4f 5d f9   ldr x0,[x27,#0x3a98]=obj44988
```
- 0xa52b90 已 push 8 字节（str x1,[x15,#-8]!），0xa52b98 `add x15,#8` 配平栈，0xa52b9c mov x15,x29 规范化，ldp/ret 返回 obj44988。
- 注：方案②使 0xa52bb0（真 builder）成为死代码，更彻底。

**为何比 Patch D 强**：主控已测 obj51369/obj91683 返回→**空白不崩**，证明该 Stack 槽是 Widget 多态（接受任意 Widget）。obj44988 是 app 对**这类槽的原生空 widget**，比那两个更有可能"真正消失"而非"留空白占位"。需装机确认。

### 【第2优先】Patch D（field-null，cid-safe，已测类别安全）
```
地址 0xa52c68 (1 字节): 原 01 f0 00 b8 → 16 f0 00 b8
  (stur w1,[x0,#0xf] → stur w22,[x0,#0xf], 内容 child field@0xf 置 null)
可选 + 0xa52c60: 原 01 b0 00 b8 → 16 b0 00 b8 (field@0xb 也 null)
```
- 返回对象仍是合法 cid0xaf7（reader 不报类型错），但内容 null。
- 属"动块本身"，可能留空白占位（取决于 reader 对 null content 的处理）。

### 【第3，状态法】已证伪，不推荐
见第二节。rewardTime/猫饿bool 的消费点在 reader build 墙后，静态够不到。

若主控顺带想要 VIP 功能（浏览次数/免广告等，**不**隐块）：

## 四、诚实边界（务必告知主控）

1. **所有"动块/返回空"方案都使块【恒空】，不等价于 ground truth 的"看广告后当日隐藏"**——静态无法精确复现该业务逻辑（reader build 墙后）。
2. **obj44988 方案需装机确认**：是"真正消失"还是"留空白占位"。主控已测同类槽（obj51369/obj91683）空白不崩 → obj44988 崩点概率低，但视觉未验证。
3. **reader build 不可静态定位**（pool 间接 + blutter cid84 崩溃）——三方 + test1 共识，非本次方法缺陷。要真正复现"当日隐藏"，需动态 hook 0xa52920 入口看调用栈（但主控 frida 不可用）。
4. **reward 成功的精确触发**：reward-prompt 0xbc5dc8 → 喂猫 0xa56e00 → 写 field@0x33=FALSE，但用户描述的"跳转外部→10秒→手动返回→恭喜奖励"这条**外部跳转回调链**（淘宝/微信小程序）静态无法追穿（跨应用 URI）。
