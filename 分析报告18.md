# 第十八轮代码恢复补充报告

> 分支：`test1`
>
> 角色：纯静态分析。本轮只收口 `喵喵饿了` 这条线，直接吸收 `test2 / test3` 最新提交，并用本地 `libapp.so` 做字节级复核。

## 1. 本轮结论

这轮对 `喵喵饿了` 的判断已经可以进一步收紧：

1. **当前最稳的主方案仍然是“不可点击不卡”**，而不是继续把“完全消失”当主交付。
2. `test3` 最新提交 `573581f` 把三层结构已经压得很清楚：
   - `0xae30e0`：常驻块 widget build
   - `0xbc5dc8`：点击后的 reward prompt / 喂喵编排器
   - `0x8cf36c`：饥饿 / 特权 / reward privilege 状态层
3. `test2` 最新提交 `8f2c622` 继续把重点放在 `V4 stub / Dart 侧 async invoke`，但结合 `backup` 的运行时铁事实，这条线仍然只能排在次级。
4. 如果用户目标是“根除”；但真正根除难度高，那么**“不显示和不被点击，相当于消失”**这个标准下，当前最可交付的是：
   - **先用 `0xbc5dd0..0xbc5dd8` 让它不可点击**
   - **再把 `0xae3560` / `0x8cf374` 作为显示侧实验项**

## 2. 跨分支最新进展

### 2.1 `test3` 最新结论：从“硬消失”退到“不可点击”

`origin/test3` 最新相关提交：

- `573581f` `test3 第八轮·下半: 喵喵饿了不可点击(0xbc5dc8)+reward链路评估(放弃V3组合方案)`

它给出的核心判断，我这边认为是成立的：

1. `0xbc5dc8` 是最贴近用户痛点的点击链 patch 点。
2. `0x8cf374` 只能算实验性“可能消失 / 可能抑广告”的特权态方案。
3. 真正依赖父级 `Visibility` 的“完全消失”仍然受限于：
   - `0xae30e0 callers=0`
   - 无 Code 簇
   - `blutter` 在当前定制 VM 上失败

所以，`test3` 这次实际是在把路线从“试图静态完全隐藏”收回到“先让它不可点击不卡”。

### 2.2 `test2` 最新结论：V4 stub / Dart 侧 D 路线继续推进

`origin/test2` 最新相关提交：

- `8f2c622` `第八轮补全:方向B+D+NOP+精确V4 stub设计`

它的新增价值主要有两点：

1. 把广告 async 调用链又梳理了一遍：
   - `0x888b18 / 0x8880c0 / 0x88862c`
2. 把 `qsa` 自定义返回对象和方法级返回类型问题写得更清楚了。

但结合 `backup` 的运行时铁事实，这条线仍然有两个硬问题：

1. `V3/V4` 方向本质上还在跟“每个方法要什么类型”硬碰硬。
2. 就算把 `MissingPluginException` 压掉，也可能引入新的 `TypeError / Null check / qsa` 路径。

所以对 `喵喵饿了` 这条线来说，`test2` 的 D/B 路线现在更像备用，而不是主推荐。

## 3. 本地字节级复核

### 3.1 `0xae30e0`：常驻块 builder 确认

本地复核：

```asm
0x00ae3128: cmp  w0, #0x14
0x00ae312c: b.ne #0xae32e4
```

以及：

```asm
0x00ae3560: b.eq #0xae36e8
...
0x00ae36e8: fmov d0, #1.0
...
0x00ae36fc: ldr x17, [x17, #0xde0]   ; ref=28525 "喵喵饿了"
```

这说明三件事：

1. `0xae312c` 仍然只是共享 builder 里的早期粗分流，不适合当主方案。
2. `0xae3560 -> 0xae36e8` 这条门，确实直接贴着 `"喵喵饿了"` fallback 文本块。
3. 所以 `0xae3560` 仍可作为显示侧实验项，但不该压过 `0xbc5dc8`。

### 3.2 `0xbc5dc8`：点击链主 patch 继续成立

本地复核：

```asm
0x00bc5dc8: stp x29, x30, [x15, #-0x10]!
0x00bc5dcc: mov x29, x15
0x00bc5dd0: sub x15, x15, #0x28
...
0x00bc5e98: bl  #0xa56e00
0x00bc5ea0: mov x0, x22
0x00bc5ea4: mov x15, x29
0x00bc5ea8: ldp x29, x30, [x15], #0x10
0x00bc5eac: ret
```

交叉数据库也继续支持这个定位：

- `test3 xref_db` 对 `0xbc5dc8` 的字符串命中：
  - `是否看推荐信息获取喵粮喂喵？`
  - `饿了喵~`
  - `AdPgl_Reward|AdMob_Reward|AdNet_Reward|AdKs_Reward`

这使得 `0xbc5dc8` 的角色已经非常清楚：

1. 它不是共享页面大状态机。
2. 它就是 `喵喵饿了` 点击之后进入 reward prompt 的关键 async 闭包。
3. 它天然存在 `return null` 结尾。

因此：

- `0xbc5dd0..0xbc5dd8: ef a1 00 d1 a0 0b 40 f9 02 70 41 b8 -> e0 03 16 aa fd 79 c1 a8 c0 03 5f d6`

仍然是当前**最高性价比**的 patch。

### 3.3 `0x8cf374`：实验性“可能消失”继续保留

本地复核：

```asm
0x008cf374: sub x15, x15, #0x60
...
0x008cf41c: add x0, x22, #0x30
```

所以：

- `0x8cf374: ef 81 01 d1 -> 2a 00 00 14`

本质上仍是“直接跳到 TRUE 尾声”。

但这个点的问题没有变：

1. 它处在底层共享 `reward privilege` 状态函数 `0x8cf36c` 里。
2. `test3 xref_db` 仍然显示它有 16-17 个调用者。
3. 它可能同时影响：
   - `喵喵饿了`
   - reward 特权
   - 局部广告门控
   - VIP 页面状态消费

所以它只能是：

- **实验性的“可能消失 / 可能抑广告 / 可能副作用更大”的方案**

而不是当前主交付。

## 4. 这三条线现在该怎么排

### 4.1 主推荐

如果目标是“等效消失”，也就是：

- 不再卡
- 不再进 reward
- 不影响阅读
- 用户不必再点它

那么当前第一推荐仍然是：

1. `0x920d90: 69 2c 00 54 -> 29 00 00 14`
2. `0xbc5dd0..0xbc5dd8 -> e0 03 16 aa fd 79 c1 a8 c0 03 5f d6`

其中：

1. `0x920d90` 继续压故障正文链
2. `0xbc5dc8` 直接让 `喵喵饿了` 点击失效，不再卡在 reward

### 4.2 次级显示实验项

如果用户还想继续逼近“看起来也少一点 / 最好不显示”，再试：

1. `0xae3560: 40 0c 00 54 -> 1f 20 03 d5`
2. `0x8cf374: ef 81 01 d1 -> 2a 00 00 14`

我现在对这两个点的定位是：

1. `0xae3560`：
   - 更像显示侧的 fallback 门
   - 比 `0xae312c` 更细
   - 但仍不如 `0xbc5dc8` 那样直接解决“点了就卡”
2. `0x8cf374`：
   - 可能带来真正的“块消失 / 广告更少”
   - 但副作用面更大

### 4.3 明确撤回项

这轮继续明确撤回：

1. `0xae312c`
   - 太粗
   - 早期静态推断已经不够站得住
2. 把 `0xae30e0` 入口直接改空返回
   - 高风险
   - 容易破坏 widget 树
3. 把 `V4 stub / qsa` 方向当作 `喵喵饿了` 主解
   - 复杂度高
   - 运行时不确定性明显高于 `0xbc5dc8`

## 5. 本轮建议

如果按你现在的目标来排优先级，我建议这样理解：

1. **“根除”优先级最高，但当前静态证据还不足以把“完全不显示”当成高置信主解。**
2. **“不显示和不被点击，相当于消失”这个标准下，当前最实用的路线已经足够明确：**
   - 主线：`0xbc5dc8`
   - 次线：`0xae3560`
   - 实验线：`0x8cf374`
3. `test3` 这次最新进展本质上也支持这个排序。

## 6. 一句话结论

这轮之后，`喵喵饿了` 的路线可以正式收敛成一句话：**主交付不再追“静态彻底隐藏”的幻觉，而是先用 `0xbc5dc8` 让它不可点击不卡，等效退出用户路径；若还要逼近“消失”，再把 `0xae3560` 作为显示侧细门、`0x8cf374` 作为高风险实验项。**
