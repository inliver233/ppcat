═══════════════════════════════════
二、★★★ 核心困境:故障弹窗 (主控穷尽失败, 求三方调研)
═══════════════════════════════════

### 机制 (主控logcat+Dart栈帧铁证)
广告定时器(每~60s)调用这些方法 → 因去广告(注释插件)全部抛 MissingPluginException →
**未捕获(Unhandled Exception)** → app全局错误处理 → 弹故障窗(ref27673"无法正常联网…广告无法显示").
触发方法(logcat实证): showSplashAd, register(ksad), initAd, interstitial,
showFullScreenVideoAd, loadFullAd, loadInsertAd, loadSplashScreenAd, MobileAds#initialize.
T+260s必触发, 点确定可关.

### ★ 关键发现:故障窗【不走showDialog BL】
主控NOP了这些showDialog编排器, 故障窗【全部仍在T+260s触发】:
- 0xbc0f40(3BL) + 0xbd5a24(4BL) [第七轮]
- 0xc4cb30 第三源(4BL: 0xc4cc2c/ccac/ce0c/ce60) [第八轮新发现, 调故障body 0x920d7c]
- 22个广告失败catch函数的showDialog(0x883bd0/0x87b714/0x87b120/0x8789e8/0x87ccd4/0x881960, 栈帧定位)
→ 证实: 故障窗由 app全局错误处理(类似FlutterError.onError/runZonedGuarded)异步弹出, 不经0x47369c/0x472c98.

### ★ V3 stub(AdNoOpPlugin)失败原因
V3接管channel消除MissingPlugin(已验证ADNOP日志: 接管了showSplashAd/register/initAd等).
但返回值崩溃(不同方法期望不同类型):
- null → reward: NoSuchMethod []("channel_name") on null
- 空HashMap → showSplashAd: type 'HashMap' not subtype of 'bool'
- Boolean.TRUE → 下游: type 'bool' not subtype of 'qsa?' (期望自定义对象)
- V3-null主界面 → Null check operator used on null value (T+66s崩, 比MissingPlugin更快)
→ 不同方法期望 bool/Map/自定义对象qsa, 且部分用!强制非null. 统一返回无法满足.

### 根本矛盾
去广告(注释插件)必然MissingPlugin → 故障窗. 消除异常需精确模拟每个广告方法响应.
这是三分支6轮+第八轮都没根治的核心难点.

═══════════════════════════════════
三、求三方调研的4个方向 (按优先级)
═══════════════════════════════════

### 方向A (最高优先): 定位app全局错误处理并截断
故障窗由全局错误处理弹. 找到它(把未捕获异常转成故障body 0x920d7c并显示的地方).
- 线索: 故障body builder=0x920d7c (3 callers: 0x920b3c/0xc4cb30/0xc4d22c, 均无showDialog BL).
  body对象被传给全局组件显示. 找"谁接收0x920d7c的返回值并显示".
- 线索: Dart的runZonedGuarded/FlutterError.onError注册点(在libapp启动代码).
- 目标: 截断全局错误处理→故障窗(或让它静默). 风险:可能掩盖真错误, 需谨慎.
- ★ 用三方已解的对象池+xref DB(test3) + Ghidra(test1) 精确定位.

### 方向B: 逐方法精确V3 (返回正确类型)
V3按method名返回该方法Dart端期望的类型. 需解码每个广告方法的Dart端invoke回调期望返回:
- showSplashAd / register / initAd / interstitial / showFullScreenVideoAd /
  loadFullAd / loadInsertAd / loadSplashScreenAd / MobileAds#initialize / showRewardVideoAd
- 用Ghidra反编译Pangle(C0984 handler)/GDT/Ksad/AdMob插件的Dart端回调解析代码,
  确定每个方法期望 bool/Map/自定义对象 及具体字段.
- test1已有C0984反编译基础, test3有对象池+全xref DB.
- 难点: 自定义对象(如qsa)在Java侧无法直接构造, 需找替代.

### 方向C: 重新启用插件+改内部空操作
不注释registerWith(避免MissingPlugin), 改每个广告插件onMethodCall内部: 跳过真实广告加载,
直接返回【原插件格式的正确响应】(类型正确, 因为就是为该channel写的).
- 工作量大(4个插件), 但类型正确性有保障.
- 风险:插件内部可能仍触发广告SDK活动.

### 方向D: Dart侧NOP广告invoke调用
在广告方法async入口 NOP掉对channel.invoke的调用, 让方法直接返回(不调用channel=不抛异常).
- async入口(栈帧定位): showSplashAd→0x888b18, 另两个→0x8880c0/0x88862c.
- 调用链: 0x8863e8(开屏控制器)→0x888b18(showSplashAd async)→invoke→MissingPlugin.
- 难点: async状态机NOP invoke需保证栈平衡+返回合理值, 否则状态机损坏.
