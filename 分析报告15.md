# 第十五轮代码恢复补充报告

> 分支：`test1`
>
> 角色：纯静态分析。本轮只做一件事：把上一轮仍标成“实验项”的 reward privilege / VIP 方向继续收窄，避免把 `0x8cf36c` 这种底层共享状态函数直接当成首选 patch 点。

## 1. 本轮结论

上一轮 [分析报告14.md](/root/ppcat_repo/分析报告14.md) 里保留了两类实验项：

1. `0x875efc`：reward 反作弊分支
2. `0x8cf374`：reward privilege / 特权态强制 true

本轮继续吸收 `origin/test3` 的对象池 / xref 数据后，我对第二类实验项做了进一步分层：

1. `0x8cf36c` 仍然是 reward privilege 的**底层共享状态计算函数**
2. 但如果目标是“让广告/喵块相关上层逻辑更容易走特权成功路径”，**更窄的首选点应该是它的上层 bool gate caller**
3. 当前最值得单独尝试的窄点是：
   - `0x7e861c: c0 06 20 36 -> 32 00 00 14`

也就是说，本轮新增的核心结论不是“再加一个大 patch”，而是：

- **把 `0x8cf374` 从 reward privilege 方向的首选实验项，降级为更下游、更粗的兜底项**
- **把 `0x7e861c` 提升为更窄、更像真正门控返回的候选**

## 2. `0x8cf36c` 的真实角色：底层共享状态计算，不是上层总闸

结合 `origin/test3:analysis_workdir/xref_db.json`，`0x8cf36c` 现在可以更准确地定性为：

- 函数：`0x8cf36c`
- `bool_ret = True`
- 17 个直接调用点
- 字符串：
  - `adNum`
  - `lastTime`
  - `today`
  - `totalNum`
  - `totalTime`
  - `leaveTime:`
  - `已过期！`
  - `已生效！\n剩余`
  - `秒`

直接调用它的上层函数包括：

- `0x7e8054`
- `0x7e8534`
- `0x89fe80`
- `0xa54178`
- `0xb9fc84`
- 以及多个更底层 / 混合页面函数

这说明：

1. `0x8cf36c` 确实是 reward privilege / 特权态计算链的核心
2. 但它并不是一个“只服务于某个页面”的窄门控
3. 直接改 `0x8cf374 -> 0x8cf41c` 会影响一批不同上层逻辑，副作用面更大

所以 `0x8cf374` 仍然可以保留，但不应作为 reward privilege 方向的第一实验项。

## 3. 新增更窄候选：`0x7e8534` / `0x7e861c`

### 3.1 结构

我本轮重新完整反汇编了 `0x7e8534`，它是一个非常典型的上层 bool gate：

```asm
0x007e8534: fd79bfa9  stp x29, x30, [x15, #-0x10]!
0x007e8538: fd030faa  mov x29, x15
...
0x007e85f8: f2eeff97  bl   #0x7e41c0
...
0x007e8618: 559b0394  bl   #0x8cf36c
0x007e861c: c0062036  tbz  w0, #4, #0x7e86f4
...
0x007e86e4: c0c20091  add  x0, x22, #0x30
0x007e86e8: ef031daa  mov  x15, x29
0x007e86ec: fd79c1a8  ldp  x29, x30, [x15], #0x10
0x007e86f0: c0035fd6  ret
0x007e86f4: c0820091  add  x0, x22, #0x20
0x007e86f8: ef031daa  mov  x15, x29
0x007e86fc: fd79c1a8  ldp  x29, x30, [x15], #0x10
0x007e8700: c0035fd6  ret
```

这里的控制流很干净：

1. 调 `0x8cf36c`
2. 如果 bit4 不成立，直接走 `0x7e86f4` 返回 canonical false
3. 如果 bit4 成立，则继续构造对象 / 调 `0x906424`
4. 最后走 `0x7e86e4` 返回 canonical true

### 3.2 新候选 patch

推荐实验 patch：

- 地址：`0x7e861c`
- 原始字节：`c0 06 20 36`
- patch 字节：`32 00 00 14`

语义：

- 原逻辑：`tbz w0,#4,#0x7e86f4`
- patch 后：无条件跳到 `0x7e86e4`

也就是：

- 不再依赖 `0x8cf36c` 的返回 bit4
- 直接强制走这个上层 gate 自带的 canonical true 尾声

### 3.3 为什么这比 `0x8cf374` 更值得优先试

因为它具备三点优势：

1. **更窄**
   - 只影响 `0x7e8534` 这条门控链
   - 不会像 `0x8cf374` 那样直接改动 17 个调用者共享的底层状态函数
2. **返回约定更清晰**
   - `0x7e86e4` / `0x7e86f4` 是函数自带的 canonical true / false 尾声
   - 比“强行把底层函数跳到某个 true 尾声”更容易解释
3. **位置更接近上层广告/特权封装**
   - `origin/test3` 的字符串表已经表明 `0x7e8534` 自己就带：
     - `isNoAdLock`
     - `adNum:`

所以如果主控要在 reward privilege 方向只多试一个点，我现在更推荐：

- `0x7e861c`

而不是直接先上：

- `0x8cf374`

## 4. `0x7e8054`：上层广告/提示封装，说明 `0x7e8534` 的地位更像门控

`0x7e8054` 调 `0x7e8534` 的关键片段：

```asm
0x007e80f8: 0f010094  bl   #0x7e8534
0x007e80fc: e0002036  tbz  w0, #4, #0x7e8118
0x007e8100: ...
0x007e8104: d1c20091  add x17, x22, #0x30
0x007e8108: 11b001b8  stur w17, [x0, #0x1b]
...
0x007e8220: 1f2df297  bl   #0x47369c
...
0x007e8228: e00316aa  mov x0, x22
```

这说明：

1. `0x7e8054` 是更上层的广告/提示封装器
2. 它直接把 `0x7e8534` 当 bool gate 使用
3. `0x7e8534` 返回 true 后，`0x7e8054` 才会继续做后续对象和提示链

因此把 `0x7e8534` 自身看作“更窄的 reward privilege 门控封装”，是自洽的。

## 5. `0x89fe80`：同样调用 `0x8cf36c`，但不适合优先 patch

我本轮也重压了 `0x89fe80`，它内部有两处 `BL 0x8cf36c`：

- `0x8a0448`
- `0x8a05ec`

对应判断：

- `0x8a044c: c0 08 20 36  tbz w0,#4,#0x8a0564`
- `0x8a05f0: 60 10 20 36  tbz w0,#4,#0x8a07fc`

但这条函数链和 `0x7e8534` 相比明显更重：

1. 内部伴随大量对象分配、字符串构造、`BL 0x47369c`
2. 同时夹杂：
   - `无法正常调用！`
   - `已收到登录结果，请稍等！`
   - `而且广告无法正常显示，此功能`
   等 UI / 登录 / 提示相关语义
3. 修改它的局部门控，比修改 `0x7e8534` 更容易带来非广告副作用

所以：

- `0x89fe80` 应继续保留为“说明 `0x8cf36c` 被多个上层 UI/广告链复用”的证据
- 但不应在优先级上压过 `0x7e861c`

## 6. 对现有推荐栈的修正

上一轮我给出的实验项顺序是：

1. `0x875efc`
2. `0x8cf374`

现在建议修正为：

1. `0x875efc: 40 01 20 37 -> 0a 00 00 14`
2. `0x7e861c: c0 06 20 36 -> 32 00 00 14`
3. `0x8cf374: ef 81 01 d1 -> 2a 00 00 14`

也就是：

- `0x8cf374` 不撤回
- 但把它下调为 **更粗粒度的后备特权 patch**
