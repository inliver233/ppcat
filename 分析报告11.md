## 3. 残留故障窗

### 3.1 继续保持的主推荐

`0x920d7c` 仍是故障正文唯一 body builder，且自带合法 `return null` 尾声：

```asm
0x00920d90  692c0054  b.ls #0x92131c
...
0x00920e34  e00316aa  mov x0, x22
0x00920e38  ef031daa  mov x15, x29
0x00920e3c  fd79c1a8  ldp x29, x30, [x15], #0x10
0x00920e40  c0035fd6  ret
```

因此继续保留这条高置信补丁：

- 地址：`0x920d90`
- 原始字节：`69 2c 00 54`
- patch 字节：`29 00 00 14`
- 含义：直接跳到 `0x920e34` 的 `null` 尾声

### 3.2 定位

这条补丁的定位仍然是：

- 不是根因修复
- 而是 `去广告后 MissingPluginException -> 全局错误链` 还没完全源头解决前，最窄、最容易解释的 Dart 侧兜底

## 4. 喵喵饿了：新的精细门控点

### 4.1 关键反汇编

本地再次压 `0xae30e0` 后，真正值得关注的不是入口粗分流，而是这段：

```asm
0x00ae3554  c0033fd6  blr x30
0x00ae3558  ef410091  add x15, x15, #0x10
0x00ae355c  1f00166b  cmp w0, w22
0x00ae3560  400c0054  b.eq #0xae36e8
...
0x00ae36e8  00106e1e  fmov d0, #1.0
0x00ae36ec  614b4091  add x1, x27, #0x12, lsl #12
0x00ae36f0  21b445f9  ldr x1, [x1, #0xb68]
0x00ae36f4  25f4f397  bl #0x7e0788
0x00ae36f8  71b34091  add x17, x27, #0x2c, lsl #12
0x00ae36fc  31f246f9  ldr x17, [x17, #0xde0]   ; ref=28525 "喵喵饿了"
0x00ae3700  11b000b8  stur w17, [x0, #0xb]
```

### 4.2 语义

这说明：

1. `0xae3554` 前面的那个间接 helper 一旦返回 `null`
2. 就会在 `0xae3560` 直接跳进 `0xae36e8`
3. 然后硬编码装载 `ref=28525 "喵喵饿了"`

也就是说：

- `0xae3560` 更像“无数据/失败时，退化成喵块文案 fallback”的真门
- 比 `0xae312c` 那种入口 case 粗分流精细得多

### 4.3 新的实验性 UI 补丁候选

推荐把旧的 `0xae312c` 撤回后，新的实验项改成：

- 地址：`0xae3560`
- 原始字节：`40 0c 00 54`
- patch 字节：`1f 20 03 d5`
- 含义：`b.eq -> nop`，不再因为第一次 helper 返回 `null` 就直接进入 `"喵喵饿了"` fallback

### 4.4 为什么它比 `0xae312c` 更值得试

这条建议的依据比旧方案强：

1. 它直接贴着 `ref=28525` 的装载块
2. 它不是入口大分支，而是更靠近“文案 fallback”本身
3. 其后仍有额外的 `null` 检查与默认对象回退：
   - `0xae35b8`
   - `0xae3658`
4. 因此它更像“只压默认喵块文案”，而不是粗暴改变整个 widget case

### 4.5 风险

这条补丁仍然只能列为实验项，不应包装成已验证结论：

1. `0xae30e0` 是共享 widget builder，`callers=0`，由框架间接调度
2. 第一次 helper 返回 `null` 可能不仅代表“该显示喵块”，也可能代表“缺数据但要走另一条简化 UI”
3. 因此它的定位应是：
   - **比 `0xae312c` 明显更合理**
   - 但仍需主控装机验证

## 5. reward anti-cheat：更窄的真实分支

### 5.1 关键窗口

本地复核 `0x8758bc` 尾段，和 `test3` / `analysis_round12` 一致：

```asm
0x00875ee8  103e40f9  ldr x16, [x16, #0x78]   ; ref=11545 "onReward Cheat Triggered!"
0x00875ef4  2e351094  bl #0xc833ac
0x00875ef8  ef410091  add x15, x15, #0x10
0x00875efc  40012037  tbnz w0, #4, #0x875f24
0x00875f00  ...
0x00875f18  18000094  bl #0x875f78
0x00875f1c  ef810091  add x15, x15, #0x20
0x00875f20  0a000014  b #0x875f48
```

而 `0x875f78` 自身是独立的 bool-returning helper，只有 `0x8758bc` 一个 caller。

### 5.2 语义

当前静态控制流能说明：

1. `0x875ee8` 这段确实围着 `ref=11545 "onReward Cheat Triggered!"` 转
2. `0x875efc` 决定是否跳去 `0x875f24` 的 generic callback 路径
3. 不跳转则会显式调用 `0x875f78`
4. `analysis_round12/reward_anticheat_probe.txt` 与本地复核一致地把 `0x875f78` 定性为 dedicated cheat handler

### 5.3 实验性 anti-cheat 旁路候选

因此可以继续保留这条 **中置信实验项**：

- 地址：`0x875efc`
- 原始字节：`40 01 20 37`
- patch 字节：`0a 00 00 14`
- 含义：强制跳到 `0x875f24`，绕过 `0x875f18 -> bl 0x875f78` 这条 dedicated cheat-handler 路径

### 5.4 定位

这条补丁的定位必须写清：

- 它不是已验证的“可直接上线”方案
- 但它比泛化地找 `onReward` / `rewardTime` / `showRewardVideoAd` 更窄、更接近真正的 anti-cheat 决策位

## 7. 本轮建议的组合方案

### 7.1 方案 A：稳妥兜底

目标：先把 `残留故障窗` 压住，不扩大风险面。

- 保持现有基线：
  - 去广告 smali
  - 已验证的篡改窗 patch
  - 已验证的旧故障窗 patch
- 追加：
  - `0x920d90: 69 2c 00 54 -> 29 00 00 14`

定位：

- 这是当前最稳的补丁组合

### 7.2 方案 B：压掉喵块 fallback 文案

目标：不先碰完整 VIP / reward 成功链，先试着让 `喵喵饿了` 这块别再落默认 fallback。

- 方案 A
- 再加：
  - `0xae3560: 40 0c 00 54 -> 1f 20 03 d5`

定位：

- 实验性，但比旧的 `0xae312c` 明显更精细

### 7.3 方案 C：reward 优先路线

目标：争取让 `喂喵 -> 奖励成功 -> 特权生效 -> 广告自然压制`

优先顺序建议：

1. Java 侧继续走 typed stub，而不是 generic `success(null)`
2. 在 Dart 侧试：
   - `0x875efc: 40 01 20 37 -> 0a 00 00 14`
3. 若仍需要再试本地特权态：
   - `0x8cf374: ef 81 01 d1 -> 2a 00 00 14`

定位：

- 这是更高杠杆但更实验性的路线
- 适合在 `方案 A` 已保证基本可用后继续压

2. 把 `0xb9fc84` 当成单点 VIP 开关

## 9. 最终判断

这一轮最有价值的新增，不是再多找几个“可能点”，而是把两个旧的大范围问题各自收紧成一个更窄的门：

1. `喵块`：
   - 从 `0xae312c` 的入口粗分流
   - 收紧到 `0xae3560` 这个直接通向 `"喵喵饿了"` fallback 的门
2. `reward anti-cheat`：
   - 从泛化的 reward / VIP / noAd 猜测
   - 收紧到 `0x875efc` 这条直接决定是否走 `0x875f78` cheat-handler 的门

如果只问“本轮最值得主控直接试哪两个新点”，我的排序是：

1. `0x920d90: 692c0054 -> 29000014`
2. `0xae3560: 400c0054 -> 1f2003d5`

如果要继续冲“喂喵成功 + 特权生效 + 广告自然压制”，则下一优先级是：

3. `0x875efc: 40012037 -> 0a000014`
4. `0x8cf374: ef8101d1 -> 2a000014`
