## ★ test3 交叉验证与关键纠正 (2026-06-20)

test3 分支使用 capstone + Dart 2.19.6 源码对照完成了对象池彻底突破。以下是我对照后的关键纠正：

### 纠正1：Pool entry 格式

| | test2 (本分支，已纠正) | test3 (正确) |
|---|---|---|
| entry type | `eb & 3` (2 bits) | **`eb & 0x7f`** (7 bits, TypeBits) |
| bit7 | 忽略 | **PatchableBit** (0x80=可 patch) |
| kImmediate payload | 固定 4/8 字节 | **ReadUnsigned (VLE)** |
| Pool fill start | 未精确定位 | **0x2e7043** (length=51140, 4-anchor 全通过) |

**根因**：我把 `entry_bits = d.Read<uint8_t>()`（raw 字节）误解为 ReadUnsigned 编码。正确格式：先读 1 个 raw 字节，`eb & 0x7f` = type (0-4)，再按 type 读 payload。kImmediate 也用 ReadUnsigned（变长），不是固定 4/8 字节。这导致了 slot 编号的系统性漂移。

### 纠正2：故障 body builder 身份

| slot 0x13b70 实际内容 | — | **ref=27673** (故障 body：无法正常联网…) |

**结论**：**0xbbc22c 是设置读写函数**（加载 isAutoCheckFav、isFavChangePos 等），不是故障 body builder。真正的故障 body 由 **0x920d7c** 构建（还加载 adNum、后台无响应、tips 等）。

## 一、对象池 entry 格式突破 (任务A)

### 1.1 核心发现

经过 7 种失败方法后，第六轮发现：**entry_byte 是 RAW uint8_t（非 VLE 编码）**。test3 交叉验证确认：entry 类型由 `eb & 0x7f`（低 7 位）决定，bit7 = PatchableBit：

| eb & 0x7f | 类型 | Payload | 说明 |
|---|---|---|---|
| 0 | kTaggedObject | VLE ref | 对象引用 |
| 1 | kImmediate | **VLE 编码的立即数** | 不是固定字节！ |
| 2 | kNativeFunction | 零字节 | Native 入口 |
| 3 | kSwitchableCallMissEntryPoint | 零字节 | Switchable call miss |
| 4 | kMegamorphicCallEntryPoint | 零字节 | Megamorphic call |

**关键纠错**：kImmediate 使用 **ReadUnsigned (VLE)** 而非固定 4/8 字节！这解释了前 5 轮的所有 slot 漂移。

### 1.2 验证结果（7 anchor 全中）

```
ref=27673 @ 0x2eeeca: eb=0x80 type=0 ref=27673 ✓

ref=21707 @ 0x2f8c81: eb=0x80 type=0 ref=21707 ✓
ref=22466 @ 0x305c5a: eb=0x80 type=0 ref=22466 ✓
```

### 1.3 "未知单元" 真相

之前的 `00 81 00 6a 86` = **3 个独立 entry**：
- `eb=0x00, VLE=81` → TAG ref=1
- `eb=0x00, VLE=6a 86` → TAG ref=874

- `eb=0x00, VLE=77 55 88` → TAG ref=142071
- `eb=0x03` → NAT (type=3, zero-payload)

## 三、每日喂喵弹窗分析 (任务B-1)

### 3.1 Builder 函数

| 函数 | 地址 | 大小 | 类型 | ref | slot offset |
|---|---|---|---|---|---|
| title builder | **0x89afa0** | 684B (171 instr) | Async SM | 21707 ("每日喂喵") | 0x2cf40 |
| body builder | **0xb5ac34** | 1088B (272 instr) | Async SM | 22466 ("无法加载数据…") | 0x4cb98 |

### 3.2 Builder 调用者

- **0x89afa0** (title): 无直接 BL caller（间接调用，通过 pool dispatch）
- **0xb5ac34** (body): 2 个 callers:
  - PC 0xb3ff84（在函数 **0xb3ff04** 内，6335 行，非 async SM）
  - PC 0xb446a8（在另一大型函数内）

### 3.3 为什么不能直接 patch builders

- 0x89afa0: 2 个 RET，**都不是 MOV X0,X22; RET**（不返回 null）
- 0xb5ac34: 1 个 RET，**不是 null return**
- 所以【入口 return null】手法 **不可用**（会破坏调用者的控制流）
- 两个 builders **都不包含 dialog BL**（0x47369c/0x472c98），它们是纯 widget builder

### 3.4 编排器定位下一步

候选方向：
1. **函数 0xb3ff04**（6335 行）：调用 daily body builder，检查是否也调用 title builder 或 showDialog
2. **函数 0xb446a8 的包围函数**：同上
3. **Java stub 从根本上消除触发**（见第四节）：让 reward 不失败，每日喂喵弹窗自然不触发

### 3.5 临时替代方案

如果必须从 libapp 侧 patch：
- **NOP pool 加载**：改 0x89b000 的 LDR 偏移使其加载 null slot
  - 指令：`LDR X4, [X4, #3904]` = `84 a0 47 f9`
  - 可改为 `MOV X4, X22` = `e4 03 16 aa`（4 字节，与 LDR 长度不同）
  - ⚠️ 风险：可能影响使用同一 X4 基址的其他 pool 访问

1. 检查 SharedPreferences 中的 `expiresDate` (ref=7406, func 0x8ba3d0)
2. 检查 `isDaily:` (ref=17423, func 0xb9fc84) — 每日限制检查

### 5.8 ★ 主广告门控函数 0x7e8534（41 callers，其他分支未分析）

**0x7e8534** 是整个 app 的**广告决策总闸**函数（472 bytes），被 41 个不同函数调用：

- 返回值语义：`0x20` = 跳过广告 / `0x30` = 继续检查广告
- 决策链：加载 `isNoAdLock` → SP getter (BL 0x4655b0) → TBZ 判断 → 广告状态
- **关键决策点 0x7e85f4**: `TBZ W0, #4, .+0x100`
  - bit4=0 → 跳过全部广告检查，直接返回 0x20（免广告）
  - bit4=1 → 继续加载 isNoAdLock/adNum 等，决定是否显示广告
- 主要调用者：0x7e8054（广告集成点，同时调 0x8cf36c 特权状态机）、0xaa4f2c（测试面板）、0x7e2adc 等

```python
# 实验性 patch: 强制跳过所有广告检查（影响41 caller，需运行时验证）
d[0x7e85f4:0x7e85f4+4] = bytes.fromhex('07000014')  # TBZ→B, 总是走免广告路径
```

**推荐策略**：与其逐个 NOP 弹窗 BL，**让 isVip=true 从源头消灭广告调用**：
- 方法A: 在 0x8ba3d0 (expiresDate 检查) 中 patch 日期比较 → 永不过期
- 方法B: 在 0xb9fc84 中强制跳过 0xba0070/0xba007c (捐赠提示路径)
- 方法C (最简单): 修改 SharedPreferences 中的 `expiresDate` 为未来日期 (需运行时定位 SP key)
