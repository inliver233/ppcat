## 1. 本轮结论

本轮 `test1` 的有效新增，不是“所有问题都已闭环”，而是把三件事收口到了可直接执行的证据层级：

1. **Java 广告桥接的关键纠偏已独立证实**：`flutter_pangle_ads` 不是单一 `MethodChannel` 插件，而是 `MethodChannel + EventChannel` 双通道；其真正 handler 类是 [C0984.java](/root/ppcat_repo/analysis_round6/out/jadx_classes2/sources/p058/C0984.java:1)，同时实现 `C0932.InterfaceC0935` 与 `C0945.InterfaceC0949`。因此，前几轮那种“统一 fake 一个 MethodChannel no-op handler 就够了”的说法，在 `Pangle` 上不成立。
2. **第三故障来源有了高置信候选**：基于 `pool_accesses.txt`，`ref=27673` 对应的故障正文 slot `0x013c68` 除 `0xbbc22c` builder 外，还被 `0x913bf8` 访问；其内部两个 dialog-show `BL` 位点 `0x914e74 / 0x914f24` 与已被运行时验证安全的 `0xbc0f40/0xbd5a24` 模式一致，适合作为第三故障来源候选 patch。
3. **对象池在 test1 侧仍未彻底闭环**：`slot -> PC` 交叉验证是成立的；但“完整 ref->slot 反序列化”在 `test1` 本轮仍不能诚实宣称已完成。官方 Dart 2.19 源码确认对象池 fill entry 不是旧报告假设的 `ReadUnsigned(type)` 形式；`unflutter` 在 ppcat 定制 VM 的 pool parse 仍卡在 `type=103 / 0x67`。

因此，本轮交付策略是：

- 对 **Java no-op / reward 假成功**，只交付“已证实结构 + 可执行最小替代方案 + 未闭环卡点”。
- 对 **第三故障来源**，交付“高置信候选 patch”，明确它比前几轮更接近运行时可测，但仍保留“尚待主控实机二次验证”的标注。
- 对 **对象池**，如实报告 `test1` 当前能独立复核到哪一步，避免把 `test2` 的结论直接冒充为本分支已独立突破。

### 4.2 Pangle 不是普通单通道插件

`Pangle` 主插件类是 [C0981.java](/root/ppcat_repo/analysis_round6/out/jadx_classes2/sources/p058/C0981.java:1)。

独立解出的两个 channel 名：

- `flutter_pangle_ads`
- `flutter_pangle_ads_event`

证据：

- `C0981.mo1050(...)` 中：
  - `this.f2450 = new C0932(..., "flutter_pangle_ads")`
  - `this.f2451 = new C0945(..., "flutter_pangle_ads_event")`
- `C0981.mo2146(...)` 中：
  - `this.f2450.m2780(c0984)` 把同一个 `C0984` 挂到 `MethodChannel`
  - `this.f2451.m2793(this.f2452)` / `this.f2451.m2793(null)` 走 `EventChannel`

`C0984` 本身同时实现：

- `C0932.InterfaceC0935`：消费 `flutter_pangle_ads` 的 method call
- `C0945.InterfaceC0949`：处理 `flutter_pangle_ads_event` 的 listen/cancel，并保存 `EventSink`

对应源码：

- [C0984.java](/root/ppcat_repo/analysis_round6/out/jadx_classes2/sources/p058/C0984.java:1)
- `mo2029(Object, EventSink)`：保存 `this.f2466 = interfaceC0950`，即保存 event sink
- `mo2030(Object)`：清理 `this.f2466 = null`
- `mo28(MethodCall, Result)`：method 分发入口

这说明：

- **Pangle 的 reward 成功，不是单纯 `result.success(null)` 就完事**
- 还必须有 **EventChannel 侧的事件发送**，否则 Dart 端 reward 流程大概率仍会停在“监听不到成功事件”

这也是本轮最重要的纠偏。

## 5. Java no-op / reward 假成功：test1 的实际建议

### 5.1 本轮不再背书的说法

`test1` 本轮**不直接背书**“通用新建一个 fake 插件类 + 统一 MethodChannel success(null) + 人工 invoke 几个事件”已经足够，原因：

1. `Pangle` 已证实需要 **EventChannel 级别** 的 listen/cancel + event sink。
2. `C0984` 同时承担 method 分发、event sink 保存、以及多个 helper 的路由逻辑；不是简单单方法 stub 能完全替代。
3. 虽然已解出 reward 事件名和 map key，但 `Dart` 端对事件的时序与必填字段要求，纯静态下还没有完整闭环。

因此，`test2` 报告里“完整 Java stub 已完成”的说法，在 `test1` 看来证据仍偏强。

### 5.2 当前最稳妥的最小替代方案

如果主控要优先做“最小改动、尽量不炸”的 Java 侧代码恢复，我建议顺序如下：

1. **先恢复插件注册，消灭 MissingPluginException**
2. 对明确实现 `MethodCallHandler` 的插件，优先短路返回 `success(Boolean.TRUE)` 或 `success(null)`
3. `Pangle` 单独处理，优先复用 `C0981/C0984` 原结构，而不是新造一个总 stub

原因：

- `GDT`、`Kwad`、`AdMob` 的 `MethodChannel` 结构相对直白
- `Pangle` 的 method/event 双通道必须分开恢复

### 5.3 对 Pangle 的更小风险路线

在 `Pangle` 上，比“另起炉灶造 fake 插件”更稳妥的路线是：

- 继续使用原 `C0981 -> C0984` 结构注册 `flutter_pangle_ads` / `flutter_pangle_ads_event`
- 在 `C0984` 的 method 分发中，**只短路最关键的广告方法**
  - `initAd`
  - `showRewardVideoAd`
  - `showFullScreenVideoAd`
  - `showSplashAd`
- 对 `showRewardVideoAd`，若要做“喂喵成功”路径，必须仿照 `C1292` 的真实 event 协议，向 `this.f2466`（EventSink）推送 reward success + close 事件

但这里我仍然要如实说明：

- **test1 本轮没有把这一套 smali 直接写成可装机版本**
- 因为仅靠纯静态，还不足以证明最小字段集合与事件时序一定正确

也就是说，本轮在任务 D 上的价值是：

- 明确指出了此前方案的结构性缺口
- 给出了更可信的改造方向
- 但没有虚构“已完成可直接过机”的 stub

## 6. 对象池：test1 本轮的真实状态

### 6.1 已独立确认

官方 Dart 2.19 源码（`/root/dartsdk/runtime/vm/app_snapshot.cc` 与 `object_pool_builder.h`）表明：

- object pool fill 是“每 entry 一个原始 `uint8 entry_bits` + type-dependent payload”
- 不是旧 `test1` 报告误写的“type 本身走 `ReadUnsigned()`”

因此，旧假设需要废弃。

### 6.2 `slot -> PC` 交叉验证成立

从 `analysis_round6/pool_accesses.txt` 独立复核到的关键命中：

| PC | slot offset | 含义 |
|---|---|---|
| `0x00913d3c` | `0x013c68` | 故障 body slot |
| `0x00bbc31c` | `0x013c68` | 故障 body builder 已知点 |
| `0x006349e8` | `0x00e8a8` | `test2` 认为是故障 title slot，test1 认可 slot->PC 但不单独宣称 ref 已闭环 |
| `0x0089b000` | `0x02cf40` | 每日喂喵 title 唯一加载点 |
| `0x00b5af3c` | `0x04cb98` | 每日喂喵 body 唯一加载点 |

其中：

- `0x02cf40` 只有 `0x89b000` 这一处加载
- `0x04cb98` 只有 `0xb5af3c` 这一处加载

这使得“每日喂喵标题/正文 builder”在 `slot -> PC` 层面已经很清楚。

### 6.3 `test1` 仍未独立完成的部分

`unflutter` 在 ppcat 定制 VM 的对象池 parse 仍卡在：

- pool stream 起点附近 `0x2e0bfb`
- unknown type `103 / 0x67`

因此本轮 `test1` 不宣称：

- “完整 ref->slot 反序列化已完成”
- “21707/22466/29112 的精确 slot 已由 test1 parser 独立求出”

更准确的说法是：

- `slot -> PC` 交叉验证成立
- `ref -> slot` 仍部分依赖 cross-branch 结果

## 7. 第三故障来源：`0x913bf8` 高置信候选

### 7.1 命中故障正文 slot

`pool_accesses.txt`：

```text
0x00913d3c  0x013c68  0x00913d3c  10 36 46 f9  LDR X16, [X16,#3176]
0x00bbc31c  0x013c68  0x00bbc31c  10 36 46 f9  LDR X16, [X16,#3176]
```

已知 `0x013c68` 是故障正文 `ref=27673` 的 slot。

也就是说，`0x913bf8` 与已知故障 body builder `0xbbc22c` 确实落在同一故障正文对象上。

### 7.2 函数形态

函数入口：

```asm
0x00913bf8  fd 79 bf a9  STP X29, X30, [X15,#-16]!
0x00913bfc  fd 03 0f aa  MOV X29, X15
0x00913c00  ef 81 0e d1  SUB X15, X15, #0x3a0
```

这是典型 async state machine，形态与已知广告/故障调度函数同类。

### 7.3 两个 dialog-show BL 位点

#### 位点 1：`0x914e74`

```asm
0x914e6c  STP X2, X1, [X15,#-16]!
0x914e70  STP X4, X3, [X15,#-16]!
0x914e74  0a 7a ed 97  BL .+0xffffffffffb5e828   ; target = 0x47369c
0x914e78  ADD X15, X15, #0x20
0x914e7c  MOV X0, X22
```

#### 位点 2：`0x914f24`

```asm
0x914f1c  STP X1, X3, [X15,#-16]!
0x914f20  STP X4, X0, [X15,#-16]!
0x914f24  5d 77 ed 97  BL .+0xffffffffffb5dd74   ; target = 0x472c98
0x914f28  ADD X15, X15, #0x20
0x914f2c  MOV X0, X22
```

这两个位点与主控已经验证安全的 `0xbc0f40/0xbd5a24` 模式完全一致：

- `BL` 前有 2 个 `STP` push
- `BL` 后有配对 `ADD X15, #0x20`
- 返回值马上被 `MOV X0, X22` 覆盖

### 7.4 patch 建议

**高置信候选 patch：**

| 地址 | 原始字节 | patch 字节 | 说明 |
|---|---|---|---|
| `0x914e74` | `0a 7a ed 97` | `1f 20 03 d5` | `BL 0x47369c -> NOP` |
| `0x914f24` | `5d 77 ed 97` | `1f 20 03 d5` | `BL 0x472c98 -> NOP` |

> 注意：`0x913bf8` **入口** 不可改 `MOV X0, X22; RET`，仍遵守第七轮运行时硬约束。

### 7.5 本轮立场

`test1` 对 `0x913bf8` 的表述是：

- **高置信候选第三故障编排器**
- 证据强于前几轮
- 但尚未在 `test1` 本轮自行补出“同链故障 title slot 也命中该 caller”的完整闭环

所以我建议主控把它作为下一优先级 patch-and-test 点，而不是在报告里写成“100% 已独立最终确认”。
