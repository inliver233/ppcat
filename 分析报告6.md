## 1. 本轮结论

本轮 `test1` 的有效新增，不是“所有问题都已闭环”，而是把三件事收口到了可直接执行的证据层级：

1. **Java 广告桥接的关键纠偏已独立证实**：`flutter_pangle_ads` 不是单一 `MethodChannel` 插件，而是 `MethodChannel + EventChannel` 双通道；其真正 handler 类是 [C0984.java](/root/ppcat_repo/analysis_round6/out/jadx_classes2/sources/p058/C0984.java:1)，同时实现 `C0932.InterfaceC0935` 与 `C0945.InterfaceC0949`。因此，前几轮那种“统一 fake 一个 MethodChannel no-op handler 就够了”的说法，在 `Pangle` 上不成立。`GDT` 本地混淆实现也不是 EventChannel 形态，而是**每次 reward 请求动态创建子 `MethodChannel`**。
2. **第三故障来源有了高置信候选**：基于 `pool_accesses.txt`，`ref=27673` 对应的故障正文 slot `0x013c68` 除 `0xbbc22c` builder 外，还被 `0x913bf8` 访问；其内部两个 dialog-show `BL` 位点 `0x914e74 / 0x914f24` 与已被运行时验证安全的 `0xbc0f40/0xbd5a24` 模式一致，适合作为第三故障来源候选 patch。
3. **对象池在 test1 侧仍未彻底闭环**：`slot -> PC` 交叉验证是成立的；但“完整 ref->slot 反序列化”在 `test1` 本轮仍不能诚实宣称已完成。官方 Dart 2.19 源码确认对象池 fill entry 不是旧报告假设的 `ReadUnsigned(type)` 形式，而是每 entry 一个原始 `uint8 entry_bits` 后跟 type-dependent payload；`unflutter` 在 ppcat 定制 VM 的 pool parse 仍卡在 `type=103 / 0x67`。`test2` 所用的 `eb & 3` 启发式，本轮只能当作候选规律，不能直接升级为 `test1` 已证实规则。

因此，本轮交付策略是：

- 对 **Java no-op / reward 假成功**，只交付“已证实结构 + 可执行最小替代方案 + 未闭环卡点”。
- 对 **第三故障来源**，交付“高置信候选 patch”，明确它比前几轮更接近运行时可测，但仍保留“尚待主控实机二次验证”的标注。
- 对 **对象池**，如实报告 `test1` 当前能独立复核到哪一步，避免把 `test2` 的结论直接冒充为本分支已独立突破。

### 4.2 Pangle 不是普通单通道插件

`Pangle` 主插件类是 [C0981.java](/root/ppcat_repo/analysis_round6/out/jadx_classes2/sources/p058/C0981.java:1)。

独立解出的两个 channel 名：

- `flutter_pangle_ads`
- `flutter_pangle_ads_event`

证据：

- `C0981.mo1050(...)` 中：
  - `this.f2450 = new C0932(..., "flutter_pangle_ads")`
  - `this.f2451 = new C0945(..., "flutter_pangle_ads_event")`
- `C0981.mo2146(...)` 中：
  - `this.f2450.m2780(c0984)` 把同一个 `C0984` 挂到 `MethodChannel`
  - `this.f2451.m2793(this.f2452)` / `this.f2451.m2793(null)` 走 `EventChannel`

`C0984` 本身同时实现：

- `C0932.InterfaceC0935`：消费 `flutter_pangle_ads` 的 method call

- [C0984.java](/root/ppcat_repo/analysis_round6/out/jadx_classes2/sources/p058/C0984.java:1)
- `mo2029(Object, EventSink)`：保存 `this.f2466 = interfaceC0950`，即保存 event sink
- `mo2030(Object)`：清理 `this.f2466 = null`
- `mo28(MethodCall, Result)`：method 分发入口

这说明：

- **Pangle 的 reward 成功，不是单纯 `result.success(null)` 就完事**
- 还必须有 **EventChannel 侧的事件发送**，否则 Dart 端 reward 流程大概率仍会停在“监听不到成功事件”

这也是本轮最重要的纠偏。

## 5. Java no-op / reward 假成功：test1 的实际建议

### 5.1 本轮不再背书的说法

`test1` 本轮**不直接背书**“通用新建一个 fake 插件类 + 统一 MethodChannel success(null) + 人工 invoke 几个事件”已经足够，原因：

1. `Pangle` 已证实需要 **EventChannel 级别** 的 listen/cancel + event sink。
2. `C0984` 同时承担 method 分发、event sink 保存、以及多个 helper 的路由逻辑；不是简单单方法 stub 能完全替代。
3. 虽然已解出 reward 事件名和 map key，但 `Dart` 端对事件的时序与必填字段要求，纯静态下还没有完整闭环。

因此，`test2` 报告里“完整 Java stub 已完成”的说法，在 `test1` 看来证据仍偏强。至少 `Pangle` 和 `GDT` 就不是同一种回调拓扑，不能共用一个过度抽象的 fake handler。

### 5.2 当前最稳妥的最小替代方案

如果主控要优先做“最小改动、尽量不炸”的 Java 侧代码恢复，我建议顺序如下：

1. **先恢复插件注册，消灭 MissingPluginException**
2. 对明确实现 `MethodCallHandler` 的插件，优先短路返回 `success(Boolean.TRUE)` 或 `success(null)`
3. `Pangle` 单独处理，优先复用 `C0981/C0984` 原结构，而不是新造一个总 stub
4. `GDT` 也不建议套 `Pangle` 模板；更稳妥的是保留原 `C1103/C1083` 结构，只在 reward 关键分支里短路成功回调

原因：

- `GDT`、`Kwad`、`AdMob` 的 `MethodChannel` 结构相对直白
- `Pangle` 的 method/event 双通道必须分开恢复

### 5.3 对 Pangle 的更小风险路线

在 `Pangle` 上，比“另起炉灶造 fake 插件”更稳妥的路线是：

- 继续使用原 `C0981 -> C0984` 结构注册 `flutter_pangle_ads` / `flutter_pangle_ads_event`
- 在 `C0984` 的 method 分发中，**只短路最关键的广告方法**
  - `initAd`
  - `showRewardVideoAd`
  - `showFullScreenVideoAd`
  - `showSplashAd`
- 对 `showRewardVideoAd`，若要做“喂喵成功”路径，必须仿照 `C1292` 的真实 event 协议，向 `this.f2466`（EventSink）推送 reward success + close 事件

但这里我仍然要如实说明：

- **test1 本轮没有把这一套 smali 直接写成可装机版本**
- 因为仅靠纯静态，还不足以证明最小字段集合与事件时序一定正确

也就是说，本轮在任务 D 上的价值是：

- 明确指出了此前方案的结构性缺口
- 给出了更可信的改造方向
- 但没有虚构“已完成可直接过机”的 stub

## 6. 对象池：test1 本轮的真实状态

### 6.1 已独立确认

官方 Dart 2.19 源码（`/root/dartsdk/runtime/vm/app_snapshot.cc` 与 `object_pool_builder.h`）表明：

- object pool fill 是“每 entry 一个原始 `uint8 entry_bits` + type-dependent payload”
- `ObjectPoolSerializationCluster::ReadFill` 在 [app_snapshot.cc](/root/dartsdk/runtime/vm/app_snapshot.cc:2585) 直接 `Read<uint8_t>()` 读取 `entry_bits`
- `TypeBits` 定义在 [object_pool_builder.h](/root/dartsdk/runtime/vm/compiler/assembler/object_pool_builder.h:49)，类型位宽是 7 bit，不是 2 bit
- 不是旧 `test1` 报告误写的“type 本身走 `ReadUnsigned()`”

因此，旧假设需要废弃。

### 6.2 七个关键 ref 的独立 anchor 校验

本轮新增脚本 [scan_snapshot_pool_refs.py](/root/ppcat_repo/analysis_round6/scan_snapshot_pool_refs.py:1) 可直接在 `libapp.so` 上扫描 `0x80 + VLE(ref)`，输出 isolate snapshot 中的相对偏移与文件偏移。其作用不是解出 slot，而是把七个关键 anchor 变成可重复运行的本地证据。

脚本默认验证的 7 个 ref 及其期望文件偏移：

| ref | 文件偏移 |
|---|---|
| `26842` | `0x2eb53c` |
| `30947` | `0x2eb540` |
| `29112` | `0x2ece05` |
| `27673` | `0x2eeeca` |
| `30922` | `0x2f0415` |
| `21707` | `0x2f8c81` |

这 7 个 anchor，本轮在 `unflutter/internal/cluster` 的 `TestScanPpcatPoolTaggedWindows` 和 `TestInspectPpcatRefNeighborhoods` 上也再次独立跑通，和上表完全一致。

### 6.3 `slot -> PC` 交叉验证成立

### 6.4 `test1` 仍未独立完成的部分

`unflutter` 在 ppcat 定制 VM 的对象池 parse 仍卡在：

- pool stream 起点附近 `0x2e0bfb`
- unknown type `103 / 0x67`

因此本轮 `test1` 不宣称：

- “完整 ref->slot 反序列化已完成”
- “21707/22466/29112 的精确 slot 已由 test1 parser 独立求出”

更准确的说法是：

- `slot -> PC` 交叉验证成立
- `ref -> slot` 仍部分依赖 cross-branch 结果
- `test2` 的 `eb & 3` 解析法目前只可作线索，不可当作 `test1` 已证实的 pool 规范

## 7. 第三故障来源：`0x913bf8` 高置信候选

### 7.1 命中故障正文 slot

`pool_accesses.txt`：

```text

0x00bbc31c  0x013c68  0x00bbc31c  10 36 46 f9  LDR X16, [X16,#3176]
```

已知 `0x013c68` 是故障正文 `ref=27673` 的 slot。

也就是说，`0x913bf8` 与已知故障 body builder `0xbbc22c` 确实落在同一故障正文对象上。

### 7.2 函数形态

函数入口：

```asm
0x00913bf8  fd 79 bf a9  STP X29, X30, [X15,#-16]!
0x00913bfc  fd 03 0f aa  MOV X29, X15
0x00913c00  ef 81 0e d1  SUB X15, X15, #0x3a0
```

这是典型 async state machine，形态与已知广告/故障调度函数同类。

### 7.3 两个 dialog-show BL 位点

#### 位点 1：`0x914e74`

```asm
0x914e6c  STP X2, X1, [X15,#-16]!
0x914e70  STP X4, X3, [X15,#-16]!
```

#### 位点 2：`0x914f24`

```asm
0x914f1c  STP X1, X3, [X15,#-16]!

0x914f24  5d 77 ed 97  BL .+0xffffffffffb5dd74   ; target = 0x472c98
```

这两个位点与主控已经验证安全的 `0xbc0f40/0xbd5a24` 模式完全一致：

- `BL` 前有 2 个 `STP` push
- `BL` 后有配对 `ADD X15, #0x20`
- 返回值马上被 `MOV X0, X22` 覆盖

### 7.4 patch 建议

| `0x914f24` | `5d 77 ed 97` | `1f 20 03 d5` | `BL 0x472c98 -> NOP` |

## 9. VIP / 更新

这一轮 `test1` 的新增，不是“已经拿到单点 isVip patch”，而是把 `VIP 内容页`、`共享状态门控`、`过期字段读取`、`更新弹窗` 四条线拆开了。

### 9.1 `0x911788`：共享状态门控器，而不是 VIP 专用函数

对 `libapp.so` 的直接反汇编显示，`0x911788` 不是单纯 `isVip()`，而是一个被多路复用的共享状态读取器。它目前至少被这些函数直接 `BL` 调用：

- `0x8a0fd4`
- `0x920d7c`
- `0xb9fc84`
- `0xbca240`
- 以及 `0x590370 / 0x7e3d80 / 0x7e85dc / 0x7eba74 / 0x7f7898 / 0x80a384 / 0x8391dc / 0xa581c0 / 0xa92e04 / 0xaf20b0 / 0xb04bd8 / 0xba2848 / 0xbe4954 / 0xc46bb0`

函数内部的关键证据：

- `0x91185c` 加载 `pool[0x35a8] -> ref=26968 "remoteConfigSign"`
- `0x91191c` 加载 `pool[0x19da0] -> ref=18250 "Main"`
- `0x91196c` 加载 `pool[0x100] -> ref=1579 "Invalid internet address "`
- `0x911aa4` 读取 `[x26,#0x80]+0xea8` 的时间字段，再和当前值做秒级差值比较
- `0x911ae4` 最终通过 `CSEL` 在 `x22+0x20` / `x22+0x30` 两个 canonical bool 之间返回

更重要的是：调用方几乎都用 `tbnz/tbz w0,#4` 检查它的返回值。由于 `x22+0x30` 的 bit4 置位、`x22+0x20` 的 bit4 不置位，可以独立推出：

- `0x911788` 返回 `x22+0x30` = “通过 / 继续”
- `0x911788` 返回 `x22+0x20` = “不通过 / 提前退出”

结合 `0x911ad0..0x911aec` 的秒级差值比较，当前最稳的结论是：

- `0x911788` 是一个 **带时间阈值的共享门控器**
- 它会综合 `remoteConfigSign / Main / Invalid internet address / 上次记录时间` 等状态
- 最终决定“某条需要网络/广告/配置的路径是否允许继续”

这能解释为什么它会同时出现在：

- `0x920d7c`（故障正文构造链）
- `0x8a0fd4`（“可以捐赠”提示）
- `0xb9fc84`（VIP 页面上层状态机）
- `0xbca240`（另一条带计数/阈值的状态链）

因此，后续如果要继续攻 VIP，不该再把 `0x911788` 简化理解成“一个布尔 isVip”。

### 9.2 `0xb9fc84`：VIP 页面上层混合状态机，不宜硬猜单分支 patch

`0xb9fc84` 的前半段先调用 `0x911788`，随后进入大量 UI 构造与状态分流。关键字符串落点已经能独立复核：

- `0xb9fe54` -> `ref=26299` -> `未获取`
- `0xb9fe94` -> `ref=3877` -> `未登录`
- `0xba0070` -> `ref=13169` -> `功能，通过捐赠绑定账号获取`
- `0xba007c` -> `ref=8910` -> `通过捐赠绑定设备获取`
- `0xba027c` -> `ref=4505` -> `点击头像绑定账号`
- `0xba02e0` -> `ref=24111` -> `解绑`
- `0xb9fdb8` -> `ref=17423` -> `isDaily:`

这段代码的实际角色更像：

1. 先经过 `0x911788` 判断当前全局状态是否允许继续。
2. 再根据“未登录 / 未获取 / 账号绑定 / 设备绑定 / 每日状态”等条件，构造不同的页面区块或提示对象。
3. 非 VIP 路径会落到 `0xba0070 / 0xba007c` 这两条“需要捐赠绑定”的文案分支。

所以当前更稳的表述是：

- `0xb9fc84` 是 **VIP 页面上层混合状态机**
- 它确实包含“非 VIP 提示路径”
- 但它不是一个可以直接负责任改成“恒 true”的单比较点

本轮仍不建议对 `0xb9fc84` 粗暴做入口 return-null，也不建议凭猜测改某个早期 `b.eq/b.ne`。

### 9.3 `0xa54178 / 0x8a0fd4 / 0x8ba3d0 / 0x67994c`：四条线已经能分清

基于对象池 + 本地反汇编，VIP 相关现在可以稳定拆成四类函数：

- `0xa54178`：**VIP 内容页主构造区**
  - `0xa54d3c` -> `8910` `通过捐赠绑定设备获取`
  - `0xa54e54` -> `9292` `已获取`
  - `0xa54e80` -> `29757` `未获得`
  - `0xa55130` -> `13169` `功能，通过捐赠绑定账号获取`
  - `0xa555ac` -> `2878` `24小时内累计浏览`
  - `0xa555e4` -> `13199` `次推荐信息，即可获取`
  - `0xa5571c` -> `13928` `，未获得特权将在0点重置浏览次数！`
  - `0xa55858` -> `3524` `已生效`
  - `0xa579c0` -> `10073` `已绑定账号`
- `0x8a0fd4`：**“可以捐赠”提示构造器**
  - 入口立刻 `BL 0x911788`
  - `0x8a10d0` -> `11366` `可以捐赠`
  - 这条链还检查 `[x26,#0x80]+0xb40 / +0x2540 / +0xee0 / +0x1108`，明显不是纯文案函数

  - `0x8ba8e0` -> `7406` `expiresDate`

  - 之后会返回 `x22+0x20 / x22+0x30`，再进入后续字符串/对象构造

因此，当前最稳的 VIP 结论不是“已找到 isVip patch”，而是：

- `0xa54178` 负责 **页面内容**
- `0xb9fc84` 负责 **页面状态编排**
- `0x8a0fd4` 负责 **可捐赠提示**
- `0x8ba3d0 / 0x67994c` 负责 **过期字段读取 / 过期判定**

### 9.4 更新窗：维持 caller 侧 `BL -> NOP` 方案

更新链的结论没有变化，但现在能更明确地说明其职责拆分：

- `0x85efd4`：构造 `发现新版本 / 新版本 ...` 相关对象
  - `0x85f020` -> `5520` `发现新版本`
  - `0x85f060` -> `19264` `新版本 `
- `0x85f09c`：后续更新内容拼装
- `0xdac684`：更新 URL 选择

- 比直接改 `0x85efd4` 入口更保守
