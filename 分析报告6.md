### 4.2 Pangle 不是普通单通道插件

`Pangle` 主插件类是 [C0981.java](/root/ppcat_repo/analysis_round6/out/jadx_classes2/sources/p058/C0981.java:1)。

独立解出的两个 channel 名：

- `flutter_pangle_ads`
- `flutter_pangle_ads_event`

证据：

- `C0981.mo1050(...)` 中：
  - `this.f2450 = new C0932(..., "flutter_pangle_ads")`
  - `this.f2451 = new C0945(..., "flutter_pangle_ads_event")`
- `C0981.mo2146(...)` 中：
  - `this.f2450.m2780(c0984)` 把同一个 `C0984` 挂到 `MethodChannel`
  - `this.f2451.m2793(this.f2452)` / `this.f2451.m2793(null)` 走 `EventChannel`

`C0984` 本身同时实现：

- `C0932.InterfaceC0935`：消费 `flutter_pangle_ads` 的 method call
- `C0945.InterfaceC0949`：处理 `flutter_pangle_ads_event` 的 listen/cancel，并保存 `EventSink`

对应源码：

- [C0984.java](/root/ppcat_repo/analysis_round6/out/jadx_classes2/sources/p058/C0984.java:1)
- `mo2029(Object, EventSink)`：保存 `this.f2466 = interfaceC0950`，即保存 event sink
- `mo2030(Object)`：清理 `this.f2466 = null`
- `mo28(MethodCall, Result)`：method 分发入口

这说明：

- **Pangle 的 reward 成功，不是单纯 `result.success(null)` 就完事**
- 还必须有 **EventChannel 侧的事件发送**，否则 Dart 端 reward 流程大概率仍会停在“监听不到成功事件”

这也是本轮最重要的纠偏。

## 5. Java no-op / reward 假成功：test1 的实际建议

### 5.1 本轮不再背书的说法

`test1` 本轮**不直接背书**“通用新建一个 fake 插件类 + 统一 MethodChannel success(null) + 人工 invoke 几个事件”已经足够，原因：

1. `Pangle` 已证实需要 **EventChannel 级别** 的 listen/cancel + event sink。
2. `C0984` 同时承担 method 分发、event sink 保存、以及多个 helper 的路由逻辑；不是简单单方法 stub 能完全替代。
3. 虽然已解出 reward 事件名和 map key，但 `Dart` 端对事件的时序与必填字段要求，纯静态下还没有完整闭环。

因此，`test2` 报告里“完整 Java stub 已完成”的说法，在 `test1` 看来证据仍偏强。至少 `Pangle` 和 `GDT` 就不是同一种回调拓扑，不能共用一个过度抽象的 fake handler。

### 5.2 当前最稳妥的最小替代方案

如果主控要优先做“最小改动、尽量不炸”的 Java 侧代码恢复，我建议顺序如下：

1. **先恢复插件注册，消灭 MissingPluginException**
2. 对明确实现 `MethodCallHandler` 的插件，优先短路返回 `success(Boolean.TRUE)` 或 `success(null)`
3. `Pangle` 单独处理，优先复用 `C0981/C0984` 原结构，而不是新造一个总 stub
4. `GDT` 也不建议套 `Pangle` 模板；更稳妥的是保留原 `C1103/C1083` 结构，只在 reward 关键分支里短路成功回调

原因：

- `GDT`、`Kwad`、`AdMob` 的 `MethodChannel` 结构相对直白
- `Pangle` 的 method/event 双通道必须分开恢复

### 5.3 对 Pangle 的更小风险路线

在 `Pangle` 上，比“另起炉灶造 fake 插件”更稳妥的路线是：

- 继续使用原 `C0981 -> C0984` 结构注册 `flutter_pangle_ads` / `flutter_pangle_ads_event`
- 在 `C0984` 的 method 分发中，**只短路最关键的广告方法**
  - `initAd`
  - `showRewardVideoAd`
  - `showFullScreenVideoAd`
  - `showSplashAd`
- 对 `showRewardVideoAd`，若要做“喂喵成功”路径，必须仿照 `C1292` 的真实 event 协议，向 `this.f2466`（EventSink）推送 reward success + close 事件

但这里我仍然要如实说明：

- **test1 本轮没有把这一套 smali 直接写成可装机版本**
- 因为仅靠纯静态，还不足以证明最小字段集合与事件时序一定正确

也就是说，本轮在任务 D 上的价值是：

- 明确指出了此前方案的结构性缺口
- 给出了更可信的改造方向
- 但没有虚构“已完成可直接过机”的 stub

## 6. 对象池：test1 本轮的真实状态

### 6.1 已独立确认

官方 Dart 2.19 源码（`/root/dartsdk/runtime/vm/app_snapshot.cc` 与 `object_pool_builder.h`）表明：

- object pool fill 是“每 entry 一个原始 `uint8 entry_bits` + type-dependent payload”
- `ObjectPoolSerializationCluster::ReadFill` 在 [app_snapshot.cc](/root/dartsdk/runtime/vm/app_snapshot.cc:2585) 直接 `Read<uint8_t>()` 读取 `entry_bits`
- `TypeBits` 定义在 [object_pool_builder.h](/root/dartsdk/runtime/vm/compiler/assembler/object_pool_builder.h:49)，类型位宽是 7 bit，不是 2 bit
- 不是旧 `test1` 报告误写的“type 本身走 `ReadUnsigned()`”

因此，旧假设需要废弃。

### 9.1 `0x911788`：共享状态门控器，而不是 VIP 专用函数

对 `libapp.so` 的直接反汇编显示，`0x911788` 不是单纯 `isVip()`，而是一个被多路复用的共享状态读取器。它目前至少被这些函数直接 `BL` 调用：

- `0x8a0fd4`
- `0x920d7c`
- `0xb9fc84`
- `0xbca240`
- 以及 `0x590370 / 0x7e3d80 / 0x7e85dc / 0x7eba74 / 0x7f7898 / 0x80a384 / 0x8391dc / 0xa581c0 / 0xa92e04 / 0xaf20b0 / 0xb04bd8 / 0xba2848 / 0xbe4954 / 0xc46bb0`

函数内部的关键证据：

- `0x91185c` 加载 `pool[0x35a8] -> ref=26968 "remoteConfigSign"`
- `0x91191c` 加载 `pool[0x19da0] -> ref=18250 "Main"`
- `0x91196c` 加载 `pool[0x100] -> ref=1579 "Invalid internet address "`
- `0x911aa4` 读取 `[x26,#0x80]+0xea8` 的时间字段，再和当前值做秒级差值比较
- `0x911ae4` 最终通过 `CSEL` 在 `x22+0x20` / `x22+0x30` 两个 canonical bool 之间返回

更重要的是：调用方几乎都用 `tbnz/tbz w0,#4` 检查它的返回值。由于 `x22+0x30` 的 bit4 置位、`x22+0x20` 的 bit4 不置位，可以独立推出：

- `0x911788` 返回 `x22+0x30` = “通过 / 继续”
- `0x911788` 返回 `x22+0x20` = “不通过 / 提前退出”

结合 `0x911ad0..0x911aec` 的秒级差值比较，当前最稳的结论是：

- `0x911788` 是一个 **带时间阈值的共享门控器**
- 它会综合 `remoteConfigSign / Main / Invalid internet address / 上次记录时间` 等状态
- 最终决定“某条需要网络/广告/配置的路径是否允许继续”

这能解释为什么它会同时出现在：

- `0x920d7c`（故障正文构造链）
- `0x8a0fd4`（“可以捐赠”提示）
- `0xb9fc84`（VIP 页面上层状态机）
- `0xbca240`（另一条带计数/阈值的状态链）

因此，后续如果要继续攻 VIP，不该再把 `0x911788` 简化理解成“一个布尔 isVip”。
