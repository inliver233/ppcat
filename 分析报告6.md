#### Kwad `0x492e3c`

- `0050`：`MethodChannel.setMethodCallHandler(v6)`

#### GDT `0x4e9de8`

关键片段：

- `0028..0044`：创建 `MethodChannel("plugins.hetian.me/gdt_plugins")`
- `0044`：`MethodChannel.setMethodCallHandler(v7)`

### 3.6 对任务 D 的实际建议

#### 3.6.1 能站得住的最小方案

本轮推荐的最小可实施方向，不是“凭空新建一个完全 fake 的广告 stub 插件类”，而是：

1. **恢复广告插件注册，先消灭 `MissingPluginException`**
2. 对 **AdMob / Kwad / GDT** 这 3 个明确实现了 `MethodCallHandler` 的插件，**在各自处理方法里直接调用 `Result.success(...)` 并尽快返回**
3. **Pangle 单独处理**，不要硬套相同手法

原因：

- `MethodChannel` / `Result` / `MethodCallHandler` 的混淆 API 已经完全钉死
- 但 `Pangle` 的 handler 绑定结构和另外 3 家不同
- 直接新建总 stub 类并同时正确伪造 `flutter_pangle_ads_event` 事件，静态证据还不够

#### 3.6.2 更具体的 patch 方向

对于 3 个可直接 handler 化的插件：

| 插件 | 建议短路点 | 原因 |
|---|---|---|
| AdMob | `Lสۥ۠ۦ/สۥ۟ۢ;` 的 `code_off=0x4b5c18` 一类 `MethodCall, Result` 处理方法 | 直接吃 `MethodCall` |
| Kwad | `Lสۥ۠ۤۧ/สۥ۟۠ۢ;` 的 `code_off=0x491700` | 直接吃 `MethodCall` |
| GDT | `Lสۥۡۨ/สۥ۟۠ۢ;` 的 `code_off=0x4e8430 / 0x4e8a9c` 以及相关 `MethodCall, Result` 处理方法 | 初始化和业务调用都在本类内部 |

统一动作模式：

```smali
.locals 1
const/4 v0, 0x0
invoke-interface {p2, v0}, Lสۥ۠ۦۡ/สۥ۟۟۠$สۥۣ۟۟;->ۥ้้้้۟۟ۡ(Ljava/lang/Object;)V
return-void
```

对应的是：

- `p2 == Result`
- `Result.success(null)`

说明：

- 这只是“让调用不失败”的 no-op
- **它不能保证 reward 业务完整成功**
- 但它比“完全不注册插件”前进了一步，至少先把 `MissingPluginException` 消掉

#### 3.6.3 本轮不敢直接交付的新 stub smali

用户任务书要求“完整 reward 假成功 stub smali，可直接 apktool b 装机”。

本轮没有把这份 smali 直接写出来，原因是：

1. `Pangle` 的 `flutter_pangle_ads` 与 `flutter_pangle_ads_event` 是双通道结构
2. 其 binding 方法里没有直接暴露 `setMethodCallHandler(this)`，说明回调不在插件主类的简单路径里
3. reward 成功所需的 event payload 格式仍未通过静态证据完全落定

因此如果现在强行产出“完整 fake reward 成功 stub smali”，风险是：

- 不仅不能让“喵喵饿了”闭环成功
- 还可能因为 event 结构不符，让 Dart 侧状态机卡得更深

#### 3.6.4 本轮对任务 D 的最终结论

- **可交付的结论**：Flutter 混淆 API 已恢复，3 家广告插件可做 `Result.success(null)` 型 no-op，`Pangle` 需单独处理
- **不可诚信交付的内容**：一个已经证明可直接装机并能完整伪造 reward 成功事件的新 stub 插件类

### 7.3 backup 分支材料内部的一个时间窗口差异

`backup/第七轮弹窗攻关/第七轮交接.md` 记录了：

- `0xbc0f40` 三个内部 `BL` NOP 后，65 秒内故障窗不再复现

但 `backup/第七轮弹窗攻关/关键教训.md` 的更长观察又记录：

- 即便继续 NOP `0xbd5a24` 的 4 个 `BL`，故障窗仍可能在约 4 分钟后出现第三来源

本轮以 **`关键教训.md` 的更长观察** 为主，因为它更符合当前任务书“残留弹窗仍未完全收口”的背景。

## 8. 本轮最终交付

### 8.1 可直接采纳的结论

1. **Java 侧最小可信路线**：
   - 恢复广告插件注册，先消灭 `MissingPluginException`
   - 对 `AdMob/Kwad/GDT` 走 `Result.success(null)` 型 no-op
   - `Pangle` 单独处理，不要强行套统一 fake handler
2. **Dart 侧只保留已被运行时证实安全的内部 `BL -> NOP` patch**
3. **对象池仍未彻底打通**，`21707/22466/29112` 不能谎报成已精确 slot

### 8.2 本轮没有硬交付的内容
