## 一、任务A：对象池 ref→slot→.text 反序列化彻底突破（钥匙）✅

### 1.1 突破点：entry 格式的正确理解

对照 Dart 2.19.6 源码（已下载到 `/root/tools/`）：

- `runtime/vm/compiler/assembler/object_pool_builder.h`：
  ```
  enum EntryType { kTaggedObject=0, kImmediate=1, kNativeFunction=2,
                   kSwitchableCallMissEntryPoint=3, kMegamorphicCallEntryPoint=4,
                   kImmediate64=5, kImmediate128=6 };
  TypeBits      = BitField<uint8_t, EntryType, 0, 7>;   // 低 7 位 = type
  PatchableBit  = BitField<uint8_t, Patchability, 7, 1>; // bit7 = patchable(0x80)
  ```
- `runtime/vm/app_snapshot.cc` `ObjectPoolDeserializationCluster::ReadFill`：每个 entry 先读 **1 个 raw 字节** `entry_bits = d.Read<uint8_t>()`（**不是 ReadUnsigned！**），再按 type 读 payload：
  - kTaggedObject(0)：`ReadRef()`
  - kImmediate(1)：`Read<intptr_t>()`
  - kNativeFunction(2) / kSwitchableCallMissEntryPoint(3) / kMegamorphicCallEntryPoint(4)：**无 payload**（仅 1 字节 entry_bits）
- `runtime/vm/datastream.h` `ReadUnsigned()` / `Read<T>`：小端、低 7 位/字节、字节>0x7f 为终止符（贡献 = byte−0x80）。**ref 用 ReadUnsigned**（即 `FillRefUnsigned=true`，已由 7 个 anchor 实证；`snapshot.json` 里该字段=false 是 unflutter 启发式误判）。

**前几轮卡死的根因**：把 entry 当作 `ReadUnsigned(type) + ...`，于是把后续的 ref 数据字节误读成"type"，遇到 `0x67`（实为某 ref 的中间字节）就报"unknown type 103"。一旦按"raw 字节 entry_bits"解析，相位立刻对齐。

## 二、任务D：reward / no-op handler 假成功 stub（Java 侧）✅【最高优先·主方案】

### 2.1 思路与可行性

去广告后 Dart 调广告 channel 全 `MissingPluginException` → 失败累积 → 弹窗。**彻底解法**：Java 侧注册无操作 `MethodCallHandler`，让这些 channel 不抛异常。

**全混淆现实**：本包 Flutter API（MethodChannel、FlutterPlugin、setMethodCallHandler、onMethodCall 等）类名/方法名/字段名**全部阿拉伯字符混淆**（สۥ笔墨ۦۧ…），且既有插件用 360 加固的字符串加密（`C10539.m22107(byte[],byte[])`）+ 控制流平坦化。**盲写新类会 crash。**

**破解**：以**已知工作的 sqflite 插件**（`Lสۥ۟ۥۥ/สۥ۟۟ۢ;`，含明文调试串 "sqlite_error"/"inDatabase"）为模板，**逐字节抽取**混淆 API 符号。关键发现：**channel 名是明文 const-string**（sqflite 用 `"com.tekartik.sqflite"` 明文），所以我的新类可用明文 channel 名。

### 2.3 交付物（可直接 apktool b 装机）

### 2.4 编译验证 ✅

`apktool b` 全部 smali 成功汇编为 dex（`Smaling smali_classes2 folder into classes2.dex...` 无错）。反查 `classes2.dex`：`AdNoOpPlugin` 类 + 4 个 channel const-string 全部就位（`flutter_pangle_ads`/`google_mobile_ads`/`gdt_plugins`/`ksad` 各命中）。

> 注：apktool 在 `aapt` 资源阶段报"Unterminated quoted string"——这是**反编译 APK 资源里混淆零宽字符**的既有问题，与我的 smali 无关；主控的 apktool 流程已能正常重编（第七轮 APK 即证）。

### 2.5 生成器（保证 byte-exact）

`analysis_workdir/smali_deliverable/gen_stub.py`（生成 AdNoOpPlugin，从接口 smali 读精确符号）+ `gen_registrant_patch.py`（插入注册块）。

## 三、任务B：每日喂喵窗 + 残留故障窗

> **首要结论**：任务D（no-op stub）从源头消灭广告调用失败，**预期可一并消灭每日喂喵窗 + 残留故障窗**。本节的 Dart 侧 patch 作为**纵深防御 / 任务D失效时的兜底**。

### 3.1 每日喂喵窗

- title ref 21707"每日喂喵" @ slot 0x59bb (off 0x2cdd8)；body ref 22466 @ slot 0x9912 (off 0x4c890)。
- title 被 0xae71bc（3 处加载）、0x846b1c、0xa54178(VIP ctor) 访问；body 被 0xb68234 访问。
- **slot 已确认无误**（slot 0x59bb = kTaggedObject ref 21707，邻居 0x59ba=22939/0x59bc=28525 均字符串，自洽）。
- **编排器无法纯静态干净定位**：实测 0xae71bc / 0x846b1c 把 pool[0x2cdd8] 加载后用于 `fmov/fcmp`（**数值比较**上下文），并非 showDialog 上下文——说明这两个函数只是"引用标题串"（可能在判等/建 map/通知），不是弹窗 show 编排器。body builder 0xb68234 仅 53 指令、且**无直接 BL 调用方**（走 pool 间接 BLR），其上游编排器也难纯静态定位。
- **结论：每日喂喵窗的 Dart 侧精确 show 点本轮未能干净锁定**。这正是任务D（源头 no-op）的价值——它不依赖定位编排器，直接让广告调用不失败 → 不触发每日喂喵。**推荐任务D**；Dart 侧 patch 留作主控动态（uiautomator dump）定位 show 点后再补。

### 3.2 残留故障窗（第三源）

- 纠错：**故障 body（27673）真 builder = 0x920d7c**（0xbbc22c 是设置项 `isAutoCheckFav`，前几轮误认）。
- 故障 title 29112"故障" @ slot 0x1cfd。周边字符串簇自洽（loadUrl / 无法正常联网…此功能 / 可以捐赠 / 更多资讯 / 故障 / tips 相邻），佐证 slot 正确。
- 已堵主路：0xbc0f40（3 BL）+ 0xbd5a24（4 BL）全 NOP（第七轮）。
- 第三源：沿 0x920d7c 的调用方 `0x920d1c`（所在编排器）追，或直接靠任务D。**推荐任务D**。
