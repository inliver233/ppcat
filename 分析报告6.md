## 一、对象池 entry 格式突破 (任务A)

### 1.1 核心发现

经过 7 种失败方法后，第六轮发现：**entry_byte 是 RAW uint8_t（非 VLE 编码）**，entry 类型由 `eb & 3` 决定：

| eb & 3 | 类型 | Payload | 说明 |
|---|---|---|---|
| 0 | kTaggedObject | VLE ref | `0x80` 和 `0x00` 都有效（bit7=flags） |
| 1 | kImmediate | 4 或 8 字节 | compressed-pointers 可能用 4 字节 |
| 2, 3 | Native | 零字节 | kNativeFunction/kSwitchableCallMiss 等 |

### 1.2 验证结果（7 anchor 全中）

```
ref=26842 @ 0x2eb53c: eb=0x80 type=0 ref=26842 ✓
ref=30947 @ 0x2eb540: eb=0x80 type=0 ref=30947 ✓
ref=29112 @ 0x2ece05: eb=0x80 type=0 ref=29112 ✓
ref=27673 @ 0x2eeeca: eb=0x80 type=0 ref=27673 ✓
ref=30922 @ 0x2f0415: eb=0x80 type=0 ref=30922 ✓
ref=21707 @ 0x2f8c81: eb=0x80 type=0 ref=21707 ✓
ref=22466 @ 0x305c5a: eb=0x80 type=0 ref=22466 ✓
```

### 1.3 "未知单元" 真相

之前的 `00 81 00 6a 86` = **3 个独立 entry**：
- `eb=0x00, VLE=81` → TAG ref=1
- `eb=0x00, VLE=6a 86` → TAG ref=874

之前的 `77 55 88 03` = **2 个独立 entry**：
- `eb=0x00, VLE=77 55 88` → TAG ref=142071
- `eb=0x03` → NAT (type=3, zero-payload)

### 1.4 关键 ref → slot 映射（与 pool_accesses.txt 交叉验证）

| ref | 用途 | slot offset | slot index | .text 加载 PC | 包围函数 |
|---|---|---|---|---|---|
| 27673 | 故障 body | **0x13c68** | 0x278d (10125) | 0xbbc31c | 0xbbc22c (已知 builder) |
| 27673 | 故障 body | 0x13c68 | 0x278d | 0x913d3c | **0x913bf8** (第三编排器) |

| 21707 | 每日喂喵 title | **0x2cf40** | 0x59e8 (23016) | **0x89b000** (唯一) | **0x89afa0** (title builder) |
| 22466 | 每日喂喵 body | **0x4cb98** | 0x9973 (39283) | **0xb5af3c** (唯一) | **0xb5ac34** (body builder) |
| 26842 | (验证锚点) | 0xad88 | 0x15b1 (5553) | — | — |

**关键发现**：21707 和 22466 各自只有 **1 个 .text 加载点**，说明这些文本专用于每日喂喵弹窗，无其他用途。

## 二、第三故障编排器精确 Patch (任务B-2)

### 2.1 发现过程

已知故障 body builder = 0xbbc22c（加载 ref=27673 @ slot 0x13c68）。
通过 pool_accesses.txt 发现 **另一个函数 0x913bf8 也在同一 slot 加载 ref=27673**。

### 2.2 0x913bf8 特征

| 属性 | 值 |
|---|---|
| 地址 | 0x913bf8 — 0x914ffc (5124 bytes, 1281 instrs) |
| 类型 | **async state machine** (MOV X0,X4; LDUR W1,[X0,#19]; ADD X1,X1,X28,LSL#32; SUB X0,X1,#0x2...) |
| dialog BLs | **2** (0x47369c ×1, 0x472c98 ×1) |
| BL 目标模式 | 与 0xbc0f40/0xbd5a24 完全相同 |

### 2.3 Dialog BL 安全验证

#### BL #1: PC 0x914e74 → 0x47369c

```asm
0x914e60: ADD X3, X3, X28, LSL #32
0x914e64: LDUR W4, [X0, #51]
0x914e68: ADD X4, X4, X28, LSL #32
0x914e6c: STP X2, X1, [X15, #-16]!      ; push #1
0x914e70: STP X4, X3, [X15, #-16]!      ; push #2 ✓ (2 pushes total)
0x914e74: BL 0x47369c                     ; ★ dialog-show
0x914e78: ADD X15, X15, #0x20           ; pop ✓ (平衡 2 个 push)
0x914e7c: MOV X0, X22                    ; null return ✓ (返回值被覆盖)
0x914e80: MOV X15, X29
0x914e84: LDP X29, X30, [X15], #16
```

#### BL #2: PC 0x914f24 → 0x472c98

```asm
0x914f10: ADD X3, X3, X28, LSL #32
0x914f14: LDUR W4, [X2, #23]
0x914f18: ADD X4, X4, X28, LSL #32
0x914f1c: STP X1, X3, [X15, #-16]!      ; push #1
0x914f20: STP X4, X0, [X15, #-16]!      ; push #2 ✓ (2 pushes total)
0x914f24: BL 0x472c98                     ; ★ dialog-show
0x914f28: ADD X15, X15, #0x20           ; pop ✓ (平衡 2 个 push)
0x914f2c: MOV X0, X22                    ; null return ✓ (返回值被覆盖)
0x914f30: MOV X15, X29
0x914f34: LDP X29, X30, [X15], #16
```

**结论**：两个 BL 都符合已验证的安全模式：
- ✅ 2×STP push 前
- ✅ ADD X15 pop 后
- ✅ MOV X0,X22 null return 紧随
- ✅ 与 0xbc0f40 的 3 个已安全 NOP 的 BL 模式完全一致

**与 0x7e1464 的区别**：0x7e1464 的 BL NOP 破坏开屏广告，原因是 0x7e1464 参与启动流程。0x913bf8 不在启动路径上（其调用图不同），且位于 .text 不同区域（0x913bf8 vs 0x7e1464 差距很大）。

### 2.4 Patch 指令

```python
# ★ 第三故障编排器（0x913bf8）：NOP 内部 dialog-show BL
d[0x914e74:0x914e74+4] = bytes.fromhex('1f2003d5')  # BL 0x47369c → NOP
d[0x914f24:0x914f24+4] = bytes.fromhex('1f2003d5')  # BL 0x472c98 → NOP
# 函数入口 0x913bf8 保持原值 fd79bfa9fd030faa，绝不动！
```

### 2.5 预期效果

结合已有的 0xbc0f40(NOP 3个BL) + 0xbd5a24(NOP 4个BL) + **0x913bf8(NOP 2个BL)**，三个故障弹窗来源全部堵死。预期故障弹窗彻底消失。

## 三、每日喂喵弹窗分析 (任务B-1)

### 3.1 Builder 函数

| 函数 | 地址 | 大小 | 类型 | ref | slot offset |
|---|---|---|---|---|---|
| title builder | **0x89afa0** | 684B (171 instr) | Async SM | 21707 ("每日喂喵") | 0x2cf40 |
| body builder | **0xb5ac34** | 1088B (272 instr) | Async SM | 22466 ("无法加载数据…") | 0x4cb98 |

### 3.2 Builder 调用者

- **0x89afa0** (title): 无直接 BL caller（间接调用，通过 pool dispatch）
- **0xb5ac34** (body): 2 个 callers:
  - PC 0xb3ff84（在函数 **0xb3ff04** 内，6335 行，非 async SM）
  - PC 0xb446a8（在另一大型函数内）

### 3.3 为什么不能直接 patch builders

- 0x89afa0: 2 个 RET，**都不是 MOV X0,X22; RET**（不返回 null）
- 0xb5ac34: 1 个 RET，**不是 null return**
- 所以【入口 return null】手法 **不可用**（会破坏调用者的控制流）
- 两个 builders **都不包含 dialog BL**（0x47369c/0x472c98），它们是纯 widget builder

### 3.4 编排器定位下一步

候选方向：
1. **函数 0xb3ff04**（6335 行）：调用 daily body builder，检查是否也调用 title builder 或 showDialog
2. **函数 0xb446a8 的包围函数**：同上
3. **Java stub 从根本上消除触发**（见第四节）：让 reward 不失败，每日喂喵弹窗自然不触发

### 3.5 临时替代方案

如果必须从 libapp 侧 patch：
- **NOP pool 加载**：改 0x89b000 的 LDR 偏移使其加载 null slot
  - 指令：`LDR X4, [X4, #3904]` = `84 a0 47 f9`
  - 可改为 `MOV X4, X22` = `e4 03 16 aa`（4 字节，与 LDR 长度不同）
  - ⚠️ 风险：可能影响使用同一 X4 基址的其他 pool 访问

### 4.5 风险说明

- **Event channel**：`flutter_pangle_ads_event` 作为 MethodChannel 处理（而非 StreamHandler）。如果 Dart 侧通过 EventChannel 监听，需要额外创建 StreamHandler 实现。但从去广告后的 logcat 看，主要问题是 `MissingPluginException`，因此 MethodCallHandler 方式足够
