# 第十四轮代码恢复收敛报告

> 分支：`test1`
>
> 角色：纯静态分析。本轮目标不是继续扩工具，而是吸收 `backup/test2/test3` 的数据库、对象表与运行时硬事实，收敛到“纯净运行 + VIP/特权实验”的可执行方案。

## 1. 结论先行

### 1.1 当前主线应这样理解

结合 `origin/backup:backup/第八轮-去广告去首启弹窗/ROUND8_FACTS.md`、`origin/test3` 的全量对象池结果，以及我在原始 `libapp.so` 上的再次反汇编复核，当前最稳的方向是：

1. **残留故障窗** 的直接兜底点仍是 `0x920d90`。
2. **喵喵饿了点击卡住** 的最实用 patch 仍是 `0xbc5dc8` 入口空返回。
3. **VIP/免广告** 现在仍不能包装成“单一 isVip 已突破”，但可以给出一条更严谨的实验性特权方案：
   - `0x8cf36c` 代表 reward privilege / 特权态对象，不是单纯 bool
   - `0x8cf374 -> 0x8cf41c` 的跳转仍可作为实验项
4. **reward 反作弊** 仍是最有价值、但尚需主控验证的一条深水线：
   - `0x875efc` 是目前最贴近 `"onReward Cheat Triggered!"` 的精确分支

### 1.2 本轮给主控的推荐栈

按置信度与工程收益排序：

1. `0x920d90: 69 2c 00 54 -> 29 00 00 14`
2. `0xbc5dd0..0xbc5dd8: efa100d1 a00b40f9 027041b8 -> e00316aa fd79c1a8 c0035fd6`
3. `0xae3560: 40 0c 00 54 -> 1f 20 03 d5`
4. `0x875efc: 40 01 20 37 -> 0a 00 00 14`
5. `0x8cf374: ef 81 01 d1 -> 2a 00 00 14`

其中：

- `1`、`2` 仍是当前最高置信的“纯净运行”主线。
- `3`、`4`、`5` 属于逐步逼近“喵块更少、特权更强、广告更少”的增强项。

## 2. 跨分支交叉验证结论

### 2.1 与 `backup` 的运行时事实一致

以下事实现在应视为硬约束：

1. Generic `showDialog BL` NOP 已被运行时证伪，不是主线。
2. `isNoAdLock` 总闸与 `SP showSplashAd=false` 对可见广告无决定性效果。
3. 去广告后根因仍是周期性广告 channel 调用抛 `MissingPluginException`，再被全局错误处理链转成故障窗。
4. Java no-op stub 若不按方法精确返回类型，会落入 `null / HashMap / bool / qsa?` 类型错配。

### 2.2 与 `test3` 的对象池 / xref 数据一致

`origin/test3` 的对象池与共享表现在可直接作为稳定基线使用，我本地抽查后继续成立：

- `ref=27673` `"无法正常联网..."` -> `slot 0x276e` -> `func 0x920d7c`
- `ref=21707` `"每日喂喵"` -> `slot 0x59bb`
- `ref=22466` `"无法加载数据..."` -> `slot 0x9912`
- `ref=28525` `"喵喵饿了"` -> `slot 0x59bc` -> `0xa54178 / 0xae30e0`
- `ref=11545` `"onReward Cheat Triggered!"` -> `0x875ee8 / 0x89219c / 0xaa62f4 / 0xaacee8`

### 2.3 与 `test2` 的广告链结论一致

`test2` 第八轮的这些点我本轮继续采纳：

1. `0xca8bdc` = `MissingPluginException` 构造函数，定位是对的。
2. 广告调度中心 `0x8758bc` 是 4 SDK × 4 类型的总分发器。
3. `VIP` 不存在一个简单 `isVip=true` 小开关，而是：
   - reward 特权计时
   - 逐源 noAd
   - 捐赠 VIP
   三套并行机制。

## 3. 纯净运行基线：两个高置信 patch

### 3.1 残留故障窗兜底：`0x920d90`

本地复核：

```asm
0x00920d7c: fd79bfa9  stp x29, x30, [x15, #-0x10]!
0x00920d80: fd030faa  mov x29, x15
0x00920d84: ef2101d1  sub x15, x15, #0x48
0x00920d88: 501f40f9  ldr x16, [x26, #0x38]
0x00920d8c: ff0110eb  cmp x15, x16
0x00920d90: 692c0054  b.ls #0x92131c
...
0x00920e34: e00316aa  mov x0, x22
0x00920e38: ef031daa  mov x15, x29
0x00920e3c: fd79c1a8  ldp x29, x30, [x15], #0x10
0x00920e40: c0035fd6  ret
```

推荐 patch：

- 地址：`0x920d90`
- 原始字节：`69 2c 00 54`
- patch 字节：`29 00 00 14`
- 语义：无条件跳到 `0x920e34`，直接走函数自带 `return null` 尾声

理由：

1. `0x920d7c` 是故障正文唯一 body builder。
2. 该函数自带合法 `return null` 尾声。
3. 这条 patch 比继续追更多 `showDialog` 或 `onError` 候选更窄、更贴近唯一正文构造链。

### 3.2 喵喵饿了点击链：`0xbc5dc8`

本地复核：

```asm
0x00bc5dc8: fd79bfa9  stp x29, x30, [x15, #-0x10]!
0x00bc5dcc: fd030faa  mov x29, x15
0x00bc5dd0: efa100d1  sub x15, x15, #0x28
0x00bc5dd4: a00b40f9  ldr x0, [x29, #0x10]
0x00bc5dd8: 027041b8  ldur w2, [x0, #0x17]
```

推荐 patch：

- 地址：`0xbc5dd0..0xbc5dd8`
- 原始字节：`ef a1 00 d1 a0 0b 40 f9 02 70 41 b8`
- patch 字节：`e0 03 16 aa fd 79 c1 a8 c0 03 5f d6`

语义：

- 保留前两条建帧
- 从 `func+8` 起直接执行 `mov x0, x22; ldp x29,x30,[x15],#0x10; ret`

定位：

1. 这是 `喵喵饿了` 点击后的 reward prompt / 编排器，不是薄 gate。
2. 当前它仍是最直接解决“点击后卡正在加载”的点。
3. 与 `test3` 的 `0xbc5d10 -> 0xbc5dc8 -> 0xbc60dc` 三层结构结论一致：主 patch 仍应落在 `0xbc5dc8`。

## 4. 增强项一：喵块显示链候选 `0xae3560`

这一点来自我上一轮与 `test3` 的对象池/函数表交叉后形成的收敛候选，本轮再次复核原始字节：

```asm
0x00ae3554: c0033fd6  blr x30
0x00ae3558: ef410091  add x15, x15, #0x10
0x00ae355c: 1f00166b  cmp w0, w22
0x00ae3560: 400c0054  b.eq #0xae36e8
...
0x00ae36fc: 31f246f9  ldr x17, [x17, #0xde0] ; ref=28525 "喵喵饿了"
```

推荐 patch：

- 地址：`0xae3560`
- 原始字节：`40 0c 00 54`
- patch 字节：`1f 20 03 d5`

语义：

- 抹掉 `b.eq #0xae36e8`
- 阻止这条“回落到构造 `喵喵饿了` 文本支路”的条件跳转

定位：

1. 这不是像早前 `0xae312c -> 6e000014` 那样方向错误的硬跳。
2. 它比强改 `0xae30e0` 的主分支更局部。
3. 但它仍然缺主控运行时验证，所以应排在 `0xbc5dc8` 之后。

## 5. 增强项二：reward 反作弊分支 `0x875efc`

### 5.1 本地独立复核

`0x875ee8` 处确实加载了 `ref=11545 "onReward Cheat Triggered!"` 所在对象池项，关键窗口如下：

```asm
0x00875ee4: 70674091  add x16, x27, #0x19, lsl #12
0x00875ee8: 103e40f9  ldr x16, [x16, #0x78]
...
0x00875ef4: 2e351094  bl #0xc833ac
0x00875ef8: ef410091  add x15, x15, #0x10
0x00875efc: 40012037  tbnz w0, #4, #0x875f24
0x00875f00: ...
0x00875f18: 18000094  bl #0x875f78
0x00875f20: 0a000014  b #0x875f48
```

推荐 patch：

- 地址：`0x875efc`
- 原始字节：`40 01 20 37`
- patch 字节：`0a 00 00 14`

语义：

- 原逻辑：`tbnz w0,#4,#0x875f24`
- patch 后：无条件跳到 `0x875f24`

为什么这条仍值得保留：

1. 它位于加载 `"onReward Cheat Triggered!"` 后的紧邻分支。
2. patch 后会绕开 `0x875f18 -> bl 0x875f78` 这一支。
3. `0x875f78` 本身是一个单独的小对象构造函数：

```asm
0x00875f78: fd79bfa9  stp x29, x30, [x15, #-0x10]!
0x00875f7c: fd030faa  mov x29, x15
...
0x00875fc4: 3502f097  bl #0x476898
```

也就是说，这里更像“进入作弊对象/作弊路径的构造支路”，不是广告总调度主骨架本身。

### 5.2 风险说明

这条 patch 仍然是 **实验项**，原因不是地址不准，而是：

1. 主控尚未给出运行时证据证明它确实让 reward 立即成功且不引入新状态错误。
2. 反作弊链还有 `0x89219c / 0xaa62f4 / 0xaacee8` 三个相关加载点。
3. 因此这条 patch 的定位应该是：
   - “最贴近反作弊 gate 的优先试点”
   - 不是“已确认反作弊全解”

## 6. 增强项三：reward privilege / 特权态 `0x8cf374`

### 6.1 本地复核

```asm
0x008cf36c: fd79bfa9  stp x29, x30, [x15, #-0x10]!
0x008cf370: fd030faa  mov x29, x15
0x008cf374: ef8101d1  sub x15, x15, #0x60
...
0x008cf3b4: c0c20091  add x0, x22, #0x30
...
0x008cf41c: c0c20091  add x0, x22, #0x30
0x008cf420: ef031daa  mov x15, x29
0x008cf424: fd79c1a8  ldp x29, x30, [x15], #0x10
0x008cf428: c0035fd6  ret
```

推荐 patch：

- 地址：`0x8cf374`
- 原始字节：`ef 81 01 d1`
- patch 字节：`2a 00 00 14`

语义：

- 直接跳到 `0x8cf41c`
- 使用函数自带 canonical `True` 尾声返回

### 6.2 正确定性

这一点不能再包装成“VIP 开关”，更准确的说法是：

1. `0x8cf36c` 是 reward privilege / 特权态对象或判定链。
2. 它与：
   - `0xa54178`
   - `0xb9fc84`
   - `0x7e813c / 0x7e8618`
   等多处调用方有关。
3. 强制走 `True` 尾声，理论上可能带来：
   - 一部分广告抑制
   - 一部分喵块/特权显示变化
   - 局部功能解锁

但必须把风险写死：

1. 它有很多 caller，不是单一页面私有逻辑。
2. 它影响的是“特权态”而非“捐赠 VIP 已绑定”的完整后端链。
3. 所以它只能叫 **实验性特权 patch**，不能叫“VIP 已获取”。

## 7. Java / stub 方向，本轮不再重复乐观结论

本轮在方向选择上遵循用户的纠偏意见，不再继续把精力花在“新工具 + 泛搜索”上，也不再把 generic stub 写成现成答案。当前更负责任的结论是：

1. 无差别 `success(null)` / `success(HashMap)` / `success(Boolean.TRUE)` 已被运行时证伪。
2. 真正能从源头消灭残留故障窗与喵块点击异常的 Java 路线，仍然只有两条：
   - 按 method 精确返回正确类型
   - 或恢复插件注册后在插件内部做 no-op
3. 这条线依旧高价值，但当前我这边没有新到足以替代 `0x920d90 / 0xbc5dc8` 的确定性产物，所以本轮不再把它写成主交付。

## 9. 本轮明确撤回 / 不再主推的路线

1. 不再主推泛化 `showDialog BL` NOP。
2. 不再主推 `onError` 区域的模糊候选。
3. 不再主推 `0xae312c -> 6e000014` 这种方向已证错的喵块分支。
4. 不再主推“单一 isVip 开关”叙事。

## 10. 一句话总结

当前最务实的代码恢复路线已经很明确：先用 `0x920d90 + 0xbc5dc8` 把“纯净运行”收稳，再把 `0xae3560 / 0x875efc / 0x8cf374` 作为逐步逼近“喵块更少、reward 更通、特权更强”的增强实验，而不是继续把精力分散到新的工具和泛搜索上。
